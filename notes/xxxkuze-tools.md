Build Anything

**加密钱包**

-   **钱包管理的是账户而不是资产**：区块链上的资产始终存储在链上，钱包只是帮助用户管理账户、发起交易和与 dApp 交互，类似浏览器是访问互联网的入口。
    
-   **区块链账户由地址（Address）和私钥（Private Key）组成**：地址可以公开分享用于收款，而私钥则相当于账户密码，拥有私钥就拥有账户的全部控制权，因此绝不能泄露。
    
-   **助记词（Seed Phrase）是钱包最重要的凭证**：12 或 24 个单词可以恢复钱包中的所有账户，一旦泄露，相当于把所有资产的控制权交给别人，因此必须离线妥善保管。
    
-   **开发者应使用专门的开发钱包**：部署智能合约或测试 dApp 时，应创建一个没有真实资产的开发账户，避免因私钥泄露或误操作造成资金损失；Monad 兼容 EVM，因此可直接使用 MetaMask、Rabby、Coinbase Wallet 等主流钱包。

**数据库与文件存储**

-   **数据库（Database）负责保存结构化数据**，例如用户信息、文章、订单、任务等，让用户刷新页面后数据仍然存在；数据库通常由表（Table）、行（Row）、列（Column）组成。
    
-   **文件存储（File Storage）用于保存非结构化内容**，例如图片、视频、音频、PDF 等，因为这些数据不适合直接放进数据库。
    
-   **数据库和存储必须重视安全**：用户数据所在的位置往往也是攻击者重点目标，需要做好权限控制、访问规则和数据保护，避免数据泄露。
    
-   **现代应用通常会借助云服务快速搭建这些能力**，例如使用集成数据库和文件存储的平台，让开发者专注于产品逻辑，而不是从零维护基础设施。文章以 Supabase 为例说明这一思路。
    
-   **核心思想**：没有数据库的应用像“写在白板上的内容”，关闭或刷新后就消失；加入数据库和存储后，应用才具备长期保存信息、服务真实用户的能力。

Build Anything

**重要的区块链概念**

-   **钱包（Wallet）是用户身份和资产入口**，通过公私钥管理资产和签名交易，而不是传统应用中的账号密码。
    
-   **交易（Transaction）是用户与区块链交互的基本方式**，无论是转账还是调用智能合约，本质上都是向网络发送一笔交易。
    
-   **Gas 是执行交易所需的手续费**，用于支付网络计算和存储资源，因此每次链上操作都需要消耗 Gas。
    
-   **智能合约（Smart Contract）是部署在区块链上的程序**，负责自动执行业务逻辑，具有公开透明、不可随意篡改的特点；而代币（Token）则是在智能合约基础上发行的数字资产，可代表货币、积分、会员权益或 NFT 等。
    
-   **不同区块链网络（如 Ethereum、Monad 等）在性能、费用和生态上各有差异**，但基本开发模型一致。掌握这些基础概念后，就能够理解绝大多数 Web3 应用的工作方式。

-   **保障稳定与安全**：使用环境变量管理密钥，配置 CI/CD 和测试环境，并做好错误处理、日志追踪、备份、输入校验、HTTPS 和 API 限流，保证系统可靠且不易被攻击。
    
-   **优化性能并满足法律要求**：通过图片优化、缓存和 Core Web Vitals 提升性能，同时准备隐私政策、服务条款和 Cookie 合规等法律文件。
    
-   **核心观点**：生产级软件不仅仅是写代码，而是需要整合大量基础设施和第三方服务，共同保证产品能够稳定、安全地服务真实用户。
    

**为什么要构建去中心化应用？**

-   **传统应用依赖大量中心化服务**（如数据库、支付、云服务等），这些服务都可能宕机、封号、修改规则甚至停止运营，本质上都需要信任第三方。
    
-   **去中心化应用将信任交给网络和代码**，应用运行在由众多节点组成的区块链网络上，而不是某一家公司的服务器，因此没有单点故障，也不容易被关闭或审查。
    
-   **区块链真正有价值的应用场景**并非投机，而是解决现实问题，例如抗审查通信、不可篡改身份与投票、用户自主拥有数据、AI 内容真实性验证以及无法被平台收回的数字资产所有权等。
    
-   **目前真正面向大众的杀手级 dApp 还没有出现**。文章认为，技术已经具备，真正缺少的是既懂用户体验又懂去中心化技术的开发者，这也是未来最大的机会。
    

**什么是 Monad？**

-   **Monad 继承了以太坊生态**，支持 Solidity 和 EVM，因此以太坊上的智能合约、开发工具和教程几乎都可以直接使用，大大降低了开发门槛。
    
-   **性能是 Monad 的最大特点**：支持约 **10,000 TPS**、约 **800ms 最终确认（Finality）**，交易费用低，让 dApp 的使用体验更接近普通 Web 应用。
    
-   **速度并非靠牺牲去中心化换来**。Monad 通过较低的节点硬件要求和新的共识机制（MonadBFT），支持更多验证者参与网络，兼顾性能与去中心化。
    
-   **智能合约是 dApp 的核心**。应用逻辑部署到区块链后，由整个网络执行，而不是运行在某家公司的服务器上，因此代码公开透明、不可随意修改或下线。网络的原生代币 **MON** 用于支付 Gas 和参与质押（Staking）。

### 主要完成了两个链上的实践

### 你遇到了哪些问题

-   私钥格式不匹配**：** 从 MetaMask 导出私钥时，默认生成的是不带 `0x` 前缀的 64 位十六进制字符串。
    
-   工具链报错**：** 我直接将该字符串填入 `.env` 文件，而底层的 Web3 部署脚本（如基于 Ethers.js 或 Viem 的工具）在初始化 Wallet/Signer 时，严格要求私钥字符串必须以 `0x` 开头，导致脚本解析失败、签名初始化报错，无法顺利完成合约部署。
    

### AI 帮你解决了什么

-   基础合约代码生成**：** 快速编写了 `contracts/` 目录下的核心 `.sol` 业务代码，包括打卡逻辑、积分数据结构（Mapping 与 Array 的结合应用）、排行榜排序逻辑以及邀请人的关联存储。
    
-   部署脚本的自动兼容性修改**：** 针对 MetaMask 私钥报错问题，AI 对部署脚本进行了逻辑优化，加入私钥格式自动检测与兼容算法——在读取 `process.env.PRIVATE_KEY` 时，自动判断字符串首部是否存在 `0x`，若无则自动拼接，从代码层面彻底解决了不同钱包导出私钥格式不统一的部署痛点。
    

### 哪些地方必须由你人工判断

-   业务逻辑与需求匹配度校验**：** AI 写出的 Solidity 代码往往能通过编译，但这并不代表它符合业务逻辑。我必须人工检查 `.sol` 文件，判断其是否存在漏洞或偏离需求。例如：每日签到的时间锁是否严格限制在 24 小时？积分排行榜的更新逻辑是否会消耗过多的 Gas？
    
-   安全与环境配置约束**：** 涉及到真实的资产和账户权限（私钥管理、`.env` 的 `.gitignore` 配置）时，必须由人工主导。在部署脚本出现异常时，需要由我准确定位问题边界（是合约逻辑问题还是脚本传参问题），再向 AI 下达明确的修改指令。
    

### 你对 Monad 或 Web3 的理解发生了什么变化

-   开发范式的转移**：** 过去我认为 Web3 链上实践的最大门槛是熟练掌握 Solidity 语法；通过这次 AI 辅助实操，我认识到具体的编码工作正在被 AI 大幅平权，真正的核心竞争力在于“机制设计”与“业务验收能力**”**。
    
-   高并发公链与应用场景的结合**：** 像打卡、任务积分排行榜、轻资产的“增长型逻辑”，在传统的低速、高 Gas 公链上极难有好的用户体验。而像 Monad 这类强调极致性能、高吞吐量（EVM 兼容且具备并行执行能力）和超低手续费的底层基础设施，正是让此类面向大众的 Web3 社交增长与积分系统能够真正落地、达成 Web2 级别流畅体验的土壤。

从我目前做的两个业务来说，比较倾向于做 Ops（运营），因为我所实践的题目（签到、积分、排行榜）天然带有极强的用户增长与运营属性。我又擅长借助 AI 完成底层编码与排障，将个人的核心精力释放到需求定义、功能验证、业务闭环和落地调优上。这种“懂技术实现、能看懂合约逻辑、且精通如何用技术工具达成运营目标”的能力，正是目前 Web3 生态和协议层（如 Monad 生态应用）在获取用户、做真实 DApp 活跃度时最急需的 Tech-Ops 桥梁能力。

今天理解适合 Monad 的高频交互场景，并说明为什么它可能更适合在 Monad 上构建，我选择的是排行榜 / 任务系统。

举个例子：

-   每天签到
    
-   完成任务获得积分
    
-   实时更新排行榜
    
-   前100名获得奖励
    

1.  **为什么需要频繁交互？**
    

-   用户每天签到、完成任务、领取奖励、查看排名，都会不断产生交互。
    

2.  **链慢或 Gas 高有什么问题？**
    

-   排名更新慢，用户不知道自己是不是前100。
    

-   每次签到都要花很多手续费，大家就不愿意参与。
    

3.  **Monad 能带来什么？**
    

-   低延迟让积分和排行榜几乎实时更新。
    

-   高性能支持很多人同时参加活动。
    
-   EVM 兼容，开发者容易开发。
    

4.  **为什么需要上链？**
    

-   排名公开透明，不能后台修改。
    

-   奖励发放公平，所有人都能验证。
    

### 一句话概括

**用户每天签到、完成任务、领取奖励都会产生大量链上交互。如果区块链速度慢或手续费高，排行榜更新不及时、参与成本过高，用户体验会变差。Monad 的高性能和低延迟能够支持大量实时交互，让积分和排行榜快速更新，同时链上记录保证排名和奖励公开透明，避免后台篡改。**
