# 2026-07-22
<!-- DAILY_CHECKIN_2026-07-22_START -->
这场分享主要围绕 **「如何设计一款适合新人的 Web3 钱包」** 展开，以 OneKey 产品经理的实际案例讲解了产品设计思路，而不是技术实现。

### 一、核心观点

**产品设计不是堆功能，而是解决用户意图。**

整个分享反复强调一个流程：

> 用户想做什么 → 用户真正经历什么 → 哪些步骤可以隐藏 → 如何保证安全。

* * *

### 二、第一个案例：钱包创建流程优化

分享者认为，**传统钱包（如 Safe）的创建流程对新人非常不友好。**

主要问题：

-   一上来就要求理解助记词、私钥
    
-   要手抄 12 个单词
    
-   要验证助记词顺序
    
-   新用户根本不知道这些东西是什么
    

因此，大量新人会：

-   截图保存助记词
    
-   发微信收藏
    
-   随便找地方存
    
-   对安全没有任何概念
    

OneKey 的思路是：

> **让用户像注册微信一样创建钱包。**

流程变成：

1.  Google / Apple 登录
    
2.  设置四位 PIN
    
3.  钱包自动创建完成
    

用户几乎不用接触助记词。

* * *

### 三、背后的技术：Keyless Wallet

虽然用户没有看到助记词，

**底层仍然存在私钥。**

只是采用了 **Keyless（免助记词管理）** 方案：

-   私钥本地生成
    
-   PIN 加密
    
-   密钥拆分存储
    
    -   OneKey 后端
        
    -   分布式网络
        
-   登录 Google/Apple 后恢复
    

因此攻击者必须同时攻破：

-   OneKey
    
-   分布式网络
    
-   用户账号
    

才能恢复钱包，提高安全性。

* * *

### 四、第二个案例：链上股票产品设计

分享者举了一个产品设计练习：

> 如果钱包支持购买链上股票，该怎么设计？

他说，大部分人会直接开始画页面。

真正应该先拆需求：

用户到底需要：

-   是否需要 KYC？
    
-   用什么币购买？
    
-   如何报价？
    
-   什么时间可以交易？
    
-   股票拆股怎么办？
    
-   分红怎么办？
    
-   价格如何同步？
    

这些都要先研究。

之后再把复杂流程全部隐藏。

最终用户看到的只有：

-   Buy
    
-   Sell
    
-   输入金额
    
-   买入成功
    

而不是让用户理解：

-   链下询价
    
-   授权
    
-   Swap
    
-   Gas
    
-   Token 映射
    
-   股票托管
    

这些全部由产品处理。

* * *

### 五、产品设计方法论

最后总结了一套比较实用的方法：

1.  **先理解用户真正的目标**
    
    -   例如："我要买苹果股票"
        
2.  **拆解真实业务流程**
    
3.  **删除用户不需要理解的步骤**
    
4.  **保证安全不能妥协**
    

这是他认为设计钱包最重要的四步。

* * *

### 六、QA 环节重点

讨论了几个比较有意思的话题：

-   **Agent Wallet**
    
    -   OKX 在 Agent + Wallet 上布局较早、更完善。
        
    -   OneKey 也支持部分能力，但因主要服务大额资产和硬件钱包用户，更偏向保守，优先考虑安全。
        
-   **钱包遗产继承**
    
    -   非托管钱包无法像交易所那样帮助继承资产。
        
    -   如果助记词彻底丢失，没有任何第三方能够恢复。
        
-   **产品需求从哪里来？**
    
    -   不需要刻意找需求，用户会不断提出。
        
    -   真正困难的是把需求做好，而不是发现需求。
        

* * *

## 一句话总结

这场分享并不是讲钱包开发技术，而是在讲一种 **Web3 产品设计思维**：

> **不要让用户学习区块链，而是把区块链的复杂性隐藏起来；用户只需要完成自己的目标（创建钱包、买股票），产品负责处理背后的所有复杂流程，同时保证安全。**

## 主要内容与结论

-   AI 上下半场判断
    
    -   上半场已基本结束：Swen 观点是“模型层”已成定局（模型能力已达较高成熟度）。
        
    -   下半场开启：核心问题由“AI 能做什么”转为“我用 AI 做什么、如何给 AI 设定边界并将其应用到具体业务场景中”。
        
-   核心观点：业务优先、AI 为工具
    
    -   业务（流程与痛点）才是核心，AI 是“壳”或能力层。必须先把业务流程打通、真实痛点解决后，再用 AI 去增强（复能）。
        
    -   若产品在没有 AI 的情况下也能工作，加入 AI 则更有价值；若没有业务内核，仅靠 AI 的概念或故事容易失败。
        
-   产品方法论与落地建议
    
    -   优先从真实业务场景、个人经历或兴趣出发寻找需求（用户访谈、场景观察）。
        
    -   避免被概念化叙事蒙蔽：比起吹 100 个概念，不如在同一真实业务上跑通并解决痛点。
        
    -   技术并非瓶颈：业界通用框架可以满足实现；更难、更关键的是业务与产品的融合设计。
        
    -   建议创业者远离电脑、走到场景中去寻找需求；把时间花在用户与实际场景上。
        
-   Agent 与 Agent 经济的理解
    
    -   当 agent 能实际工作的那一刻，会形成一种与人类经济并行的新经济形态——“agent 经济”。
        
    -   Agent 经济要求重塑经济关系与交互方式：人与 agent、agent 与 agent 之间的互动会形成新的商业交叉点（例如买方与卖方通过 agent 直接交流并打通中间层）。
        
    -   协议层（如 ERC 等）目前解决的场景较单一或有限，真实业务落地会不断暴露并改变这些模式。

    -   前期策划（proposal、目标人群、活动形式、难度、嘉宾与合作方设计）
        
    -   BD（对外合作）：拓新流量池、联动合作方与平台（例如其他黑客松平台或项目方）

    -   Program 设计：workshop、AMA、co-learning、Mini Demo 到 Hackathon Demo Day 的活动编排
        
    -   报名与招募：表单、报名平台（如 WCB 等）、社群管理
        
    -   活动执行：主持稿、现场与线上运营、评审对接、技术答疑与issue 跟进
        
    -   收尾与交付：评审反馈、优秀项目跟进、对接实习/岗位、写复盘（recap）、提供给合作方的数据与结果
        
    -   行动项与建议（为参与者/运营人）
        
        -   若想转运营/入行：
            
            -   立刻建立并运营个人推特账号，产出成长型内容（学习笔记、活动复盘、宣发文案样例等），并在简历/作品集中链接相关推文或实习成绩。
                
            -   在本次实习计划中争取报名运营组或宣发组岗位，积极在排行榜与群内贡献（笔记、分享、活动参与），以获取对接岗位的优先机会。
                
        -   若负责活动筹办（短期清单）：
            
            -   明确目标与 KPI（报名数、出席率、提交率、项目质量指标）
                
            -   写好 proposal（目标人群、赛道、合作方、预算/资源、时间线）
                
            -   制定宣发计划（中/英文文案、渠道表、频次表、负责人）
                
            -   准备活动流水线（报名表、社群、workshop 时间表、评审机制、技术支持通道）
                
            -   活动结束后立即开始收集数据、跟进高价值项目并在 7–14 天内完成复盘报告与对接名单
                
        -   工具与协作推荐：
            
            -   协作：Notion（流程/素材库）、Google Sheet（数据汇总）、TG（实时沟通）、Twitter（对外形象）
