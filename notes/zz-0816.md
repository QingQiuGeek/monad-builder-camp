# **MOSS：Agentic Protocol 抽象层**

📅 2026-07-17

## **一、MOSS 是什么**

**一句话：**把 Monad 链上协议变成 AI Agent 可调用的 Capability 的中间层框架。

核心流程：`discover → load → action → simulate`。**只构建和验证未签名交易，永不签名、永不发送。**

## **二、MOSS 解决的四个问题**

### **问题 1：Agent 无法直接操作链上协议**

LLM 能理解「帮我用 1 MON 换 USDC」，但不知道合约地址、ABI、calldata。MOSS 把协议封装成标准化的 Protocol 包。

### **问题 2：Agent 直接动钱太危险**

```
Agent 表达意图 → MOSS 构建交易 → 模拟验证 → 返回证据 → 用户签名 → 发送
                    ↑              ↑           ↑
                 AI 不碰钱      人可检查     钱包独立
```

强制 simulate-before-sign，签名方永远在 MOSS 之外。

### **问题 3：缺乏可验证证据**

每次模拟输出 Change + Receipt，人类可逐条检查。

### **问题 4：Protocol 重复造轮子**

一次写好，所有 Agent 复用。

## **三、已支持 Protocol**

| Protocol | 操作 | 查询 |
| --- | --- | --- |
| WMON | wrap / unwrap | balanceOf |
| ERC-20 + native MON | transfer / approve | balanceOf, allowance |
| ERC-721 | transfer | ownerOf, balanceOf |
| Kuru DEX | swap | quote |

## **四、六大应用场景**

1.  **AI 自主理财 Agent**：「30% MON 换 USDC，剩下的 wrap 做 LP」
    
2.  **Intent-based 交易**：说目标不指定路径，Agent 选最优路由
    
3.  **Agent-to-Agent 经济**：Agent 间自动借贷交易
    
4.  **安全 AI 钱包**：MOSS 做钱包的模拟层
    
5.  **DAO 多签自动化**：条件触发 → simulate → 多签确认 → 执行
    
6.  **跨协议组合**：卖 NFT → 换 MON → 借贷 → 跨链
    

## **五、核心价值**

| 维度 | 价值 |
| --- | --- |
| 安全性 | 永不经手私钥，强制 simulate-before-sign |
| 可验证 | Change + Receipt 可逐条检查 |
| 标准化 | Protocol 包一次编写，所有 Agent 复用 |
| Agent 原生 | 专为 AI Agent 调用设计 |
| Monad 原生 | 主网 143，直接对接 Monad 生态 |

> _MOSS 是 Monad 的 Agentic Protocol 抽象层——让协议从「人类可读文档」变成「Agent 可执行指令」。_

# Web3 运营从 0 到 1

🎤 主讲：丽老师（LXDAO）📅 2026-07-15 📺 X Broadcast

## 一、为什么需要运营

> "一个产品再好，没人知道、没人来、没人留下，就等于不存在。"

- 运营 = 让产品/社区/生态**被看到、被理解、被使用、持续增长**
- 不只是"发推、拉群、帮忙"——是从策划到收尾的完整链路
- 运营也有 KPI：**增长、留存、转化**

## 二、运营完整链路

```
策划 → POPOS(提案) → BD(对接合作方) → 营销宣发 → 活动执行 → 收尾反馈
```

### 2.1 策划阶段

- 明确目标：这场活动要达成什么
- 覆盖人群：新手？开发者？研究者？
- 活动形式：Workshop / AMA / Coloring / Hackathon
- 设计难度和主题
- 对接嘉宾、合作伙伴（如 Monad 官方）

### 2.2 BD（商务拓展）

- 写 POPOS 提案给合作方
- 单场活动 vs 多赛道组合（如 AI 赛道 + DeFi 赛道）
- 合作方的目的：**拉新 Builder、扩大生态增长**
- 自己渠道有限 → 必须不断找有流量的合作方

### 2.3 活动执行（最琐碎的部分）

| 任务 | 说明 |
|------|------|
| 拉群 | Telegram / WeChat 社群 |
| 发推 | Twitter 宣发文案 |
| 报名 | 设计报名表 / WCB 平台 |
| 分享会 | 对接嘉宾、主持 |
| 跟进开发者 | 收集问题、解决体验卡点 |
| 录播 & 剪辑 | 发 YouTube / B站 |
| 持续触达 | 不同形式反复提醒（文字→群→直播→Workshop） |

### 2.4 收尾不等于结束

- 联系优秀 Builder、跟进项目
- 对接实习/岗位
- 交付反馈给合作方：这场活动带来了多少 Builder、多少项目提交
- 用户留存到生态长期计划（Builder Program、大使、Intern）

## 三、用户旅程（Builder Pipeline）

```
知道项目 → 加入社群 → 参加 Workshop/AMA/Coloring → 开始 Build → 提交项目 → 获得 Grant/反馈 → 留存生态
```

> "光发一次文字，看到的人不多。来参加直播的人，也不一定是看文字的那批人。所以必须多形式、多渠道、多频率地触达。"

## 四、社交媒体账号运营

### 4.1 推特（Web3 必备）

> "推特相当于你的名片、你的 Profile。"

- 技术向：晒 Hackathon 作品集（官方会转发，等于官方背书）
- 运营向：写经验分享、学习心得
- 研究向：写感兴趣的题材方向、跟大V互动
- 新人最简单的起步：**做成长型账号**，每天写学了什么

### 4.2 小红书

- 适合写个人经验分享
- 不适合深度技术内容（监管严格，易被限流）
- 受众多但 Web3 浓度低

### 4.3 国内平台

- 视频号 / B站：做技术分享类视频 OK
- 公众号：长文案版本

### 4.4 内容分层

| 类型 | 场景 | 平台 |
|------|------|------|
| 短文案 | 社群推送、TG 群通知 | Twitter、TG |
| 长文案 | 主推文、活动介绍 | 公众号、Twitter |
| 中英文 | 有海外受众 | Twitter、Discord |

## 五、AI 工具在运营中的应用

### 5.1 实际使用

- TG 内置 AI Agent（Grog）：搜索 + 写文案
- GPT + Codex：互补使用
- 连接飞书/工作台：AI 知道历史活动数据，写出更精准的文案

### 5.2 使用策略

> "如果对一个新东西完全不懂，让两个 AI 互补，选择最好的结果，根据自己想要的去修改。"

- AI 始终是收集信息 + 辅助写作的工具
- **你自己的思路、过去的经验判断、你想要的东西——这才是最重要的**
- 三个 AI 交叉验证 → 人做最终选择

### 5.3 AI 能帮运营做什么

- 写宣发文案（短 → 社群、长 → 公众号/推文）
- 写 POPOS / 提案
- 调研究方案
- 写主持稿
- 数据分析（如果有接入历史数据）

| 认知 | 说明 |

# **Web3 开发者如何从 0 到 1 使用 AI 编程**

🎤 主讲：扑法📅 2026-07-12📺 X Broadcast

## **一、什么是「从 0 到 1」**

**1** = 产出一个可验证产品核心价值的 Demo，能让目标用户体验和判断你的产品。AI 出现之前需要找到认可你想法的开发伙伴；AI 之后，每个人都可以独立完成。

## **二、AI 编程的两大优势**

| 优势 | 说明 |
| --- | --- |
| 降低门槛 | 零编程经验也能通过对话生成代码 |
| 提升效率 | 功能迭代快、Bug 修复快、重构成本大幅下降 |

> _重构已成为 AI 开发的标准环节——AI 生成代码快但跑偏概率高，需要不断重构逐步接近目标。_

## **三、AI Agent 的四大能力支柱**

🧠 模型更强

推理能力持续提升，脑子越来越好用

📚 上下文更长

读取更多代码、文档、历史信息

🔧 工具更丰富

MCP、Bash、API——Agent 工具箱不断扩充

💾 记忆更完整

Memory + Skills 持久化，Agent 在交互中自我进化

## **四、AGENTS.md — 最重要的实践**

| 层级 | 位置 | 作用 |
| --- | --- | --- |
| 全局 | ~/.codex/AGENTS.md | 个人偏好、长期规则，每次运行加载 |
| 项目级 | 项目根目录 AGENTS.md | 约束边界、编码规范、CI/CD 流程 |

所有 AI Agent 工具（Codex、Claude Code、Hermes）都会读取。写得好比 README 更能传达项目架构。

## **五、两大开发方法论**

### **TDD（测试驱动开发）**

`写测试 → 测试失败 → 最小实现 → 测试通过 → 重构`

Agent 会主动运行测试、自我修正——测试用例越好，AI 开发质量越高。

### **SDD（规范驱动开发）**

| 工具 | 适用场景 | 特点 |
| --- | --- | --- |
| Speed | 全新项目 | 较重、规范、一条龙 |
| OpenSpeed | 现有项目增量 | 轻量、聚焦变更 |

## **六、AI 时代的四项核心能力**

📝 会描述

把想法写成清晰的目标+约束+验收标准

🧩 会拆解

复杂任务拆成可执行、可验证的小任务

✅ 会验证

手动+自动测试+日志，结果由你把关

🏗 会设计

架构边界、数据流、长期质量体系

## 一、两种主流模式（选型第一步）

### 模式 1：代理支付（用户钱包托管，C 端个人 Agent，国内主流）

Agent**没有独立资金账户**，仅作为用户的支付代理人，资金全部走用户本人支付宝 / 微信 / 银行卡，适合个人助手、购物、订票 Agent（支付宝 ACT 协议、微信 AI 开放平台）**凤凰网科技**。

-   逻辑：用户一次性授权 Agent 代理支付，设置消费规则；Agent 发起支付请求，支付平台校验授权范围后扣用户账户资金。
    
-   优点：合规简单、资金安全、不用单独开户；缺点：依赖国内支付服务商开放 AI 代理接口。
    

### 模式 2：自主钱包支付（Agent 自有资金，B 端 / 海外 / 链上 M2M）

Agent 拥有独立托管钱包（法币钱包 / 稳定币钱包 USDC），自有预算自主付费，适合企业采购 Agent、API 付费、跨智能体交易（AWS Bedrock、Stripe Link、x402）**Amazon Web…**。

-   逻辑：企业给 Agent 钱包充值预算，内置 spending policy 消费规则，自动支付服务商、API、云资源。
    
-   优点：高度自治、机器间小额高频支付；缺点：需资金托管资质、海外 / 区块链方案为主。

支付合法前提：**每一笔 Agent 支付必须绑定用户可核验、可撤销、可存证的授权凭证36氪**。

（1）Agent 可信身份

-   国内：支付平台分配独立 AgentID，OAuth2.0/OpenID Connect 机器鉴权，绑定用户实名账户；
    
-   海外 / 链上：W3C DID 去中心化身份，Agent 唯一数字身份，不可伪造。
    

（2）可验证授权凭证（Mandate/VDC）

用户签名生成加密防篡改授权文件，是 Agent 花钱的 “电子许可合同”，包含 5 项强制约束**稀土掘金**：

1.  金额：单笔上限、日 / 月总预算；
    
2.  场景：允许商户 / 品类（仅外卖、禁止转账提现）；
    
3.  时效：授权有效期、仅用户离线可支付；
    
4.  审批阈值：小额自动支付、大额推送人工确认；
    
5.  可撤销：用户一键终止 Agent 支付权限，即时生效。
    

主流标准：

-   国内：支付宝 ACT 2.0 智能体商业信任协议、AHA 互联接口；
    
-   国际通用：AP2 协议（谷歌 + 万事达 + PayPal 共建）、x402 HTTP 微支付协议；
    
-   链上：VC 可验证凭证、EIP-712 签名、ERC4337 账户抽象可编程钱包。
    

授权两种形态

1.  预授权（批量自动）：用户提前设置规则，Agent 小额高频免确认；
    
2.  单次意图授权（高风险）：每笔大额交易推送用户确认，生成单次 Cart Mandate。
    

### 2\. 规则风控引擎（控制层，防止乱花钱）

部署在 Agent 与支付通道中间，**每笔支付请求先过规则校验，不满足直接拦截letagentpa…**：

-   自然语言转规则：用户输入 “每日 API 花费不超过 50 元，禁止夜间消费”，LLM 自动转为可执行 JSON 策略；
    
-   实时校验维度：剩余预算、单笔限额、商户黑白名单、交易频次、时段；
    
-   分级处理：小额自动放行、超阈值人工审批、异常交易直接阻断 + 告警。
    

### 3\. 钱包与签名层（资金载体，密钥安全隔离）

分两类钱包方案，密钥全程隔离，Agent 永远触碰不到私钥：

方案 A：绑定用户现有钱包（国内 C 端）

支付宝 AI 开放平台、微信 AI 助手：Agent 通过授权令牌调用用户钱包，签名由支付平台 TEE 可信执行环境完成，Agent 仅传递交易参数，无密钥权限**Alipay+**。

方案 B：Agent 独立托管钱包（海外 / B 端）

1.  法币托管钱包：Stripe Link、Coinbase CDP、AWS Bedrock Payment Connector，服务商托管私钥；
    
2.  链上稳定币钱包（USDC）：Circle 钱包、ERC4337 智能合约钱包，TEE 硬件加密签名；
    
3.  安全隔离：密钥存储在 TEE / 硬件密钥，Agent 仅发起支付指令，签名由钱包服务独立执行。
    

### 4\. 支付执行通道层（打通资金链路）

根据场景选择对应支付接口，Agent 标准化调用：

1.  **国内互联网支付（个人 Agent 首选）**
    
    支付宝 AI 开放平台 Skill 工具、微信 AI Agent 支付 API，基于 ACT 协议，支持自动下单扣款、账单同步、退款自动处理；
    
    流程：Agent 解析购物意图→组装支付参数→携带 Agent 授权凭证调用支付网关→平台校验用户授权→扣用户余额 / 银行卡→返回支付结果给 Agent。
    
2.  **海外法币通道（企业海外 Agent）**
    
    Stripe Agent Pay、Visa Intelligent Commerce、PayPal Agent API，基于 AP2 协议，支持信用卡自动代扣、企业预算管控。
    
3.  **机器微支付 M2M（API / 模型付费）**
    
    x402 协议：HTTP 原生支付，服务商返回 402 Payment Required，Agent 自动用钱包签名支付凭证放入请求头重试，实现按调用付费，无需预充值套餐。
    
4.  **区块链自主结算（跨 Agent 交易）**
    
    ERC4337 账户抽象钱包、USDC 稳定币，支持批量交易、免 Gas、可编程消费限制，适合纯机器自治场景。
    

### 5\. 审计与对账层（追溯、售后、监管）

全链路日志永久存证，满足监管对账与纠纷举证：

-   每笔交易绑定：AgentID、用户授权凭证、商品明细、时间、预算消耗、签名记录；
    
-   自动对账：Agent 后台实时展示消费账单，支持导出、退款申请、异常交易回溯；
    
-   纠纷举证：加密授权凭证可提交支付机构 / 法院，证明支付为用户授权行为。
    

## 三、端到端完整支付流程（以国内购物 Agent 举例）

1.  用户授权：绑定支付宝，设置规则「单笔≤200 元，日上限 1000 元，仅实物消费，超 500 元需确认」，生成加密 ACT 授权凭证；
    
2.  Agent 任务规划：收到指令 “买 30 元咖啡”，拆解下单 + 支付工具调用；
    
3.  风控校验：引擎检查 30 元未超单笔 / 日预算、商户在白名单，自动放行；
    
4.  发起支付：Agent 携带 AgentID + 授权凭证 + 订单参数调用支付宝 AI 支付网关；
    
5.  平台核验：校验授权有效期、额度、场景合规，TEE 完成用户钱包签名扣款；
    
6.  结果回传：支付成功返回订单号，Agent 同步订单状态；超阈值则推送手机确认，用户同意后再扣款；
    
7.  审计记录：全链路日志存入存证系统，用户可随时查看、撤销授权。
    

## 四、落地三大关键约束：安全 + 合规

### 1\. 安全红线

-   Agent**不可持有用户支付密钥 / 银行卡明文**，所有签名由支付服务商 TEE 隔离执行；
    
-   授权不可无限期、不可无上限，必须强制限额与有效期；
    
-   支持一键撤销全部支付权限，撤销即时生效，未完成交易终止。
    

### 2\. 国内合规硬性要求

1.  实名绑定：Agent 支付必须关联已实名的个人 / 法人支付账户，禁止匿名支付；
    
2.  委托授权书面化：授权规则可视化展示，用户主动确认，禁止静默默认授权；
    
3.  交易留痕：所有 Agent 支付流水至少保存 5 年，满足反洗钱监管；
    
4.  禁止高风险场景：Agent 不可自主转账、提现、大额理财，仅限定消费场景。
    

### 3\. 海外监管要点

-   AP2/x402 需配套 KYA（Know Your Agent）代理身份核验；
    
-   企业 Agent 钱包需资金托管资质，区分自有资金与客户资金隔离。
    

## 五、开发者快速落地两种路径

### 路径 1：国内个人消费 Agent（最简，支付宝 / 微信）

1.  入驻支付宝 AI 开放平台，创建 Agent，开通 AI 支付 Skill；
    
2.  前端实现用户授权弹窗，生成 ACT 标准化支付委托；
    
3.  Agent 内置 MCP 工具调用支付 API，接入风控规则引擎；
    
4.  对接账单、退款、授权撤销接口，完成审计后台。
    

### 路径 2：海外企业 / API 自治 Agent（AWS/Stripe/Circle）

1.  开通 Coinbase CDP / Stripe Link 托管钱包，为 Agent 分配独立预算会话；
    
2.  集成 AP2/x402 支付协议连接器，封装 paidFetch 自动处理 402 付费请求；
    
3.  配置 Spending Policy 预算规则，接入 TEE 签名服务；
    
4.  搭建审计看板，同步每笔支付凭证与结算记录。
