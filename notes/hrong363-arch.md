**1\. 钱包产品设计与安全**

-   **OneKey钱包介绍**：强调其作为基础设施的角色，具备硬件钱包、全开源、安全实验室等核心特点。
    
-   **钱包产品定位**：钱包是加密行业的入口，但容错率极低，安全问题至关重要。其成熟的盈利模式使其能穿越牛熊。
    
-   **核心挑战与风险**：
    
    -   **账户体系**：自托管模式下，用户资产完全由私钥控制，迁移成本低，对用户粘性构成挑战。
        
    -   **安全成本**：钱包安全投入巨大，需进行定期的第三方安全审计。
        
    -   **外部攻击**：主要风险包括地址投毒、假币交易和恶意签名，产品需从自身层面进行防范。
        
-   **产品设计思路**：核心在于简化复杂概念，撮合用户与链上服务。例如，通过传统账号创建钱包，免去助记词的困扰；在链上购股票时，简化交易流程，让用户只关注投资额。
    

**2\. 钱包类型与设计重点**

-   **热钱包与冷钱包**：热钱包（如MetaMask）侧重于日常交易和功能丰富性，安全性次之；冷钱包（如硬件钱包）侧重于资产的长期安全存储。
    
-   **产品线设计**：以OneKey为例，通过不同产品线（如Binance Chain Only、Defi功能丰富版、无电池版）满足囤币、DeFi等不同用户群体的需求。
    

**3\. 遗产继承与AI Agent**

-   **遗产继承**：非托管钱包无法直接处理用户去世后的资产继承问题，OneKey目前不提供此功能，建议用户通过多签或提前规划的方式解决。
    
-   **AI Agent结合**：目前AI Agent钱包多处于内部探索阶段，尚未有成熟的、面向用户的落地应用。
    

**4\. 黑客松准备**

-   **赛道方向**：Web3和Monet生态是确定方向，具体赛道将在后续公布。
    
-   **项目准备**：鼓励基于已有想法进行创新和迭代，可先制作Mini Demo进行预热。最终的黑客松项目需要是完整、闭环的产品。
    
-   **团队组成**：团队可包含前端、后端、产品、设计、运营、AI等多元角色，纯小白也可通过寻找搭子或担任后勤角色参与。

-   **AI落地实操建议**
    
    -   需求挖掘方法：优先从自身经历、生活痛点、行业从业经验中挖掘需求，创始人自用的产品天然具备用户基础，更容易跑通验证。
        
    -   架构选择逻辑：流程可明确量化的场景用 Workflow 即可覆盖，仅处理模糊、非标准化需求的场景才需要搭建 AI Agent 架构。
        
    -   技术认知误区：无需每日跟进最新技术框架，产品核心是解决用户问题，技术架构优劣不直接决定产品的市场接受度。
        
    -   客户选择逻辑：早期创业优先选择共创型客户，不要试图教育认知不匹配的甲方，教育成本极高且无收益保障，遇到不合适的客户直接更换即可。
        
    -   销售AI落地反馈：当前通用大模型无法达到优秀销售 80% 的能力水平，私域销售的信任建立环节暂时无法通过 AI 完成，需人工介入最后一公里。
        
    -   技术转型建议：技术背景转产品岗，要多接触用户、了解付费决策逻辑，多和销售、创始人交流，不要局限于技术框架的优劣判断。
        
-   **AI相关延伸观点**
    
    -   青少年AI教育：青少年作为 AI 原生一代，可快速将创意落地为产品并完成迭代，创造力释放效果远超传统教育模式的预期。
        
    -   认知幻觉应对：无需因无法背诵知识点产生愧疚感，可通过 Agent 每日随机抽查知识点的方式，巩固自身知识掌握程度。
        
    -   核心壁垒判断：AI 无法替代人的团队品质、远见、坚持、审美、信任关系建立能力，这些非标准化要素是商业竞争的核心壁垒。
        
-   **Web3产品运营与推广实操经验**
    
    -   **产品前期调研逻辑**
        
        -   用户群体抽象：Web3 核心用户分为资产合规持有需求者、特殊产品供需方、交易维护者三类，需先明确对应群体特征再设计产品。

## 一、开发者运营 = 从开发者痛点出发

**不要等开发者来找你，而是主动发现问题。**

实践方式：

-   定期查看 GitHub Issues、Discussion、Discord 等社区反馈。
    
-   优先解决高频、影响大的问题。
    
-   主动联系反馈者，提供解决方案或测试资源。
    

**结果：**

-   提高开发体验。
    
-   将抱怨者转化为支持者。
    
-   建立开发者信任。
    

* * *

## 二、开发者是生态传播者

开发者不仅是产品使用者，也是内容生产者。

开发者获得良好体验后，可能会：

-   编写教程
    
-   分享 Demo
    
-   回答社区问题
    
-   推荐项目
    
-   提交 PR
    

因此，开发者运营本质也是**培养生态贡献者**。

* * *

## 三、Hackathon 的目标不是发奖金

重点不是举办活动，而是促进生态连接。

例如设置：

-   最佳互操作性奖
    
-   最佳生态集成奖
    
-   最佳 SDK 应用奖
    

鼓励开发者：

-   集成其他协议
    
-   调用生态基础设施
    
-   提高协议间组合性（Composable）
    

\*\*目的：\*\*让生态形成网络，而不是一批独立项目。

* * *

## 四、开发者运营关注的是开发效率

相比普通社区运营关注活跃度，开发者运营更关注：

-   文档是否清晰
    
-   SDK 是否易用
    
-   API 是否稳定
    
-   Demo 是否可复现
    
-   问题是否快速响应
    

核心指标不是聊天人数，而是**开发成功率**。

* * *

## 五、个人思考

开发者运营更像连接产品、技术与社区的桥梁。

工作重点包括：

1.  收集开发者反馈。
    
2.  推动产品持续优化。
    
3.  降低开发门槛。
    
4.  激励开发者持续贡献。
    

开发者体验越好，生态增长越快；生态增长越快，项目竞争力越强。

今天补上了第一周的任务，即对Monad的认识，

我认为Monad适合构建一个基于链上的Social Quest平台，让用户通过完成社区任务、互动行为以及生态活动获得积分、NFT徽章，并通过排行榜形成持续参与。

这个场景具有明显的高频交互特点。相比传统DeFi用户低频交易，SocialFi和任务系统中的用户每天可能进行签到、评论、任务完成、奖励领取等大量操作，因此需要快速的交易确认和低成本的状态更新。

如果运行在性能较低的区块链上，用户体验会受到影响。此外，实时排行榜和游戏化机制也很难实现。

Monad的高性能、低延迟以及EVM兼容性可以帮助这类产品实现更流畅的用户体验。高吞吐能力支持大量用户同时进行链上互动，降低高频操作成本；EVM兼容则降低开发者迁移和构建应用的成本。

不过，并不是所有数据都需要上链。例如用户资料、界面数据可以存储在传统数据库中。但用户贡献记录、NFT徽章、奖励证明以及链上身份等内容具有公开验证和可组合价值，因此适合记录在链上。

因此，我认为Monad不仅适合作为高性能基础设施，更适合承载类似SocialFi、链游、任务系统等需要大量用户实时互动的Consumer Crypto应用。

# 2026-07-09
<!-- DAILY_CHECKIN_2026-07-09_START -->













AI Agent 最大隐患是**自主、不可审计、会幻觉、会失控**，比传统 Web3 安全风险难防太多。

**真正有效的防护：必须有一套独立于 AI 的底层强制拦截系统**（也就是 Agent Guard）。

### 一、四大真实高危风险

1.  **过度授权**
    

-   权限太高，AI 9 秒删空企业全量数据 + 备份
    
-   无法终止指令、私自读取本地密码
    

2.  **提示词注入**
    

-   网页 / 代码隐藏隐形指令，AI 读取后自动转账、执行恶意操作
    

3.  **恶意插件 / 工具包**
    

-   插件市场大量木马插件，偷私钥、API、接管设备
    
-   刷量高排名插件最容易让人放松警惕
    

4.  **底层与供应链风险**
    

-   开源工具默认无密码，公网可被直接接管
    
-   依赖包投毒、AI 逻辑漏洞可被四步接管账号、泄露隐私
    

### 二、通用安全做法

-   权限：**最小权限原则 + 定期审计 + 敏感操作人工二次确认**
    
-   输入：所有外部内容**零信任**，不盲目执行
    
-   部署：高强度密码、锁定依赖版本
    
-   防护：**不用 AI 自查，用独立安全工具拦截**
    

### 三、官方防护产品 Agent Guard 能力

-   执行前**毫秒级拦截高危操作**
    
-   插件自动扫毒、查杀后门
    
-   全局安全体检、可视化风险报告
    
-   全球威胁情报自动更新、自动处置
    
-   可自定义安全策略、全程操作溯源

# 2026-07-08

基础认知：

1. AI Agent会替代绝大多数重复性执行任务，所有自动化任务几乎都会涉及资金流转，Agent支付是AI经济的底层基础设施。

2. 当前处于早期信任培育阶段，类比早年支付宝：先小额、后大额、先工具、后普及。

3. 行业终局：未来民生、商业、服务类自动化任务，全部由Agent自主完成支付交易。

二、核心架构：

1\. 接入层

兼容全部主流Agent载体（GPT、Cloud、LangChain等），一句话Prompt即可触发全流程支付。

2\. 核心能力层

\- 身份标识：给AI唯一可溯源身份

\- 预算管控：前置锁额、防止超支

\- 风控校验：约束AI幻觉、违规操作

\- 交易审计：全程可查、可追溯

\- 交易可撤销：无需改代码，一键回滚

3\. 核心协议

1. X402协议

互联网原生HTTP支付协议，一次授权、全程自动支付、无需人工干预。

2. 批量结算协议（类Layer2）

预授权额度内，多笔微支付统一结算，解决小额高频低效问题。

3. Policy Engineering风控引擎

金融级约束框架，解决AI乱花钱、误操作，满足银行级合规。

4\. 核心产品：Agent钱包

\- 非AI独占、非人工全控

\- 7×24小时自动执行、规则内自由交易

\- 任意交易支持一键撤销，底层无需改动

5\. 支付链路双通道

\- 链上：Base链 USDC（主力、最成熟）

\- 法币：Stripe充值→平台凭证/积分消费

三、产品矩阵

1\. Agent Card

\- 一次性卡：固定预算、单次使用、用完即废（订票/临时任务）

\- 长期卡：与VISA深度合作，即将上线，可持续自主消费

2\. 三类标准化支付解决方案

1. 开发者货币化：无代码上架API/Skill，自动被Agent付费调用、持续收益

2. 电商变额支付：自动生成支付链接，一键确认完成交易

3. Agent点对点转账：凭唯一Agent ID直接互转

四、已落地真实场景

1. 社交场景：春节「Cloud派」Agent自动抢红包，高用户验证

2. 百度生态合作：AI资产、OPC项目上架、全球分发、付费结算

3. 票务场景：Agent自动订票、额度风控落地

4. 行业调研：付费Agent Skill，自动产出行业报告

五、生态与竞品关系

1. 与Coinbase为深度合作伙伴，非竞品

2. 基于官方X402做自研优化，落地更成熟

3. ERC8004等同类方案完备度不足，无法落地

六、合规体系

现状

\- Agent Card复用B端KYB资质

\- 用户端无需KYC，轻量化落地

未来规划

\- 逐步上线全链路KYC/KYB、反洗钱体系

\- 完善大额支付合规能力

七、一些tips

1. Agent支付是Fintech最新核心赛道（Stripe、各大云厂商均布局）

2. 入行门槛：不需全栈开发，只需懂架构+基础工程能力，AI可写大部分代码

3. 人才缺口：复合型（金融+AI+支付）最稀缺

4. Agent支付的核心：安全风控 + 约束下高效花钱

5. 新产品普及逻辑：用户迁移需要10倍体验优势

6. 未来必然趋势：自动化工作=必然涉及资金=必然需要Agent支付
