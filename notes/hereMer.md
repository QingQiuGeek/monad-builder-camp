**2026/07/16 - 最小合约打卡：AI 生成留言板**

1.  **Prompt**
    

“用 Solidity 0.8.20 写个最简单的留言板合约，记录留言者地址、内容和时间戳。需包含发布和查看函数。”

2.  **AI输出结构**
    

使用 Struct 打包留言数据，Array 存储所有记录，包含一读一写两个函数。

3.  **合约源码**
    

// SPDX-License-Identifier: MIT

pragma solidity ^0.8.20;

contract Guestbook {

struct Message { address sender; string content; uint256 timestamp; }

Message\[\] public messages;

function leaveMessage(string memory \_content) public {

require(bytes(\_content).length > 0, "Empty"); // 人工添加：拦截空消息

messages.push(Message(msg.sender, \_content, block.timestamp));

}

function getAllMessages() public view returns (Message\[\] memory) { return messages; }

}

4.  **人工修改**
    

给leaveMessage函数加了require限制，禁止发空内容，防止产生链上垃圾并浪费Gas。

5.  **AI代码需人工检查的三个点**
    

• 异常输入处理：AI常忽略边界情况（需手动防空值）。

• Gas爆表隐患：getAllMessages会全量读取数组，数据量大时会卡死报错，实操需改分页。

• 恶意防刷机制：AI缺乏安全意识，未限制发送频率，易遭机器刷屏。

**2026/07/11 - 智能合约的搭建与落成**

**合约全链路生命周期拆解**

1.  **合约源码 (Source Code)→编译 (Compile)：** 编译器负责检查逻辑是否跑得通，并将人类可读的代码转化为区块链能执行的底层机器指令，同时生成一份外部交互说明书（ABI）。
    
2.  **连接钱包→部署 (Deploy)→生成合约地址：** 我们需要用钱包授权身份，支付一定的 Gas 作为网络节点的“施工费”，把代码永久刻印在 Monad 链上。成功后生成的“合约地址”，就是这个装置在数字世界里的永久地理坐标。
    
3.  **Read/Write 调用→区块浏览器验证：** 装置落成后，交互正式开始。调用read函数，不改变任何状态，完全免费；而调用write函数，这种改变系统状态的行为必须提交一笔交易(Transaction) 并消耗Gas作为驱动能量。最终，每一次触发都会在区块浏览器这本公开日志上留下不可篡改的Hash凭证。

### 2026/07/10 - 智能合约的“控制中枢”

**概述**

今天的任务进入了智能合约的领域，主要关注在理清底层逻辑上。

**核心概念**

1.  Source Code (源码) 与 ABI (应用二进制接口) 智能合约的 Solidity 源码（Source Code）就像是我们建构系统时编写的 Grasshopper 参数化逻辑，是写给开发者看、用来定义规则体系的。当它被编译并部署到链上后，就会变成机器读取的字节码。此时，ABI 就充当了节点通讯协议的角色（类似于 ROS 2 中的通信接口），它是一份说明书，明确告诉外部的网页或钱包：这里有哪些开放的端口、可以发送什么格式的指令来驱动这个合约。
    
2.  Contract Address (合约地址) 和我们个人钱包地址类似，合约一旦部署，就会在区块链上获得一个永久不变的地址。不同的是，这个地址背后没有私钥，它完全由最初部署进去的那套代码逻辑（规则）来控制。
    
3.  Read (读) vs Write (写) Functions
    
    -   Read (View/Pure 函数): 只是简单地向网络查询当前的状态数据。因为不改变任何物理或链上状态，所以是完全免费的，不需要消耗 Gas。
        
    -   Write (状态改变函数): 当你需要写入新数据、修改变量或转移资产时，就是在改变系统的状态。这需要全网节点重新计算并达成共识，因此必须发起一笔交易 (Transaction)，并支付 Gas 费。
        
4.  Transaction Hash (交易哈希) 任何一次“Write”操作，都会生成一个唯一的Txn Hash。它就像是系统操作日志里的回执单编号，拿着它就可以在区块浏览器中追溯这笔指令是否被成功执行、消耗了多少资源。
    

**总结**

关系链如下。我们编写源码→编译出 ABI→部署生成合约地址→外部应用通过 ABI 调用 Write 函数→生成 Transaction Hash。

2026/07/09 - 链上第二步：完成首笔测试网转账与链上查证

**概述**

核心目的是完成第一次链上交互。流程大致有三步：① 准备测试资产 (MON) 并发起转账；② 获取交易的唯一凭证 (Txn Hash)；③ 利用区块浏览器追踪交易状态，并读懂交易背后的各项参数。

**参数解释**

1.  From / To： From 代表资金或交互指令的发起方（我的主地址），To 代表接收方（我的子地址或目标智能合约）。
    
2.  Value（价值）： 交易中实际转移的代币数量。
    
3.  Gas 与 手续费 (Fee)： Gas 是区块链网络处理这笔请求所消耗的计算资源量，手续费则是为了感谢矿工（验证节点）打包交易而支付的“跑腿费”。
    

**备注**

为什么失败的交易也要扣 Gas？ 因为只要我发起了交易请求，网络节点就已经调动算力去执行和验证了。即使最终因为余额不足或合约报错导致交易 Failure，相当于是进行操作本身需要的能量。
