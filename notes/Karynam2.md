## 一、核心观点

-   **AI 大幅降低了 Web3 入门门槛**：多位零编程基础的学员（英语专业、艺术设计、土木工程等）在 AI 辅助下，一周内即完成了链上留言板、宠物 DApp 等 Demo 的部署。
    
-   **Web3 学习需要"先动手再理解"**：多名分享者指出，直接跟着 Build Anything 实操（部署合约、链上交易）比先读文档更有效，概念在实践中自然串成体系。
    
-   **英语能力是 Web3 关键竞争力**：官方公告、技术文档、社区讨论大多以英文首发，依赖翻译会慢人一步；练好口语能够打开更多机会窗口。
    
-   **AI 角色应从"讲解者"升级为"学习教练"**：有学员提出"输出—诊断—小测—讲解—实操—纠错"的 AI 驱动学习闭环，比单纯让 AI 总结知识更有效。
    
-   **Web3 不只有金融交易，更是一套"可公开验证的数字状态系统"**：链上记录可追溯、可验证，信任由代码和共识而非中心机构维护。
    
-   **Revert 不等于退款**：链上交易一旦确认即不可撤销；Revert 只是回滚状态，钱还在但 Gas 费不退——这是链上操作的基本心智模型，新人极易混淆。
    
-   **"精准提示词 + 空文件"是 AI 开发铁律**：bein 老师强调，让 AI 在原文件上反复修改效果极差，成本可能是"重新开发的十倍以上"；每个特效/模块应单独用空文件生成。
    
-   **Web3 不适合"所有东西都上链"**：yoyo 明确提出，推特登录、复杂任务审核等应先放前端/后端，等产品逻辑成熟再上链——这是产品思维而非纯技术思维。
    
-   **"这还用问吗"是新人学习最大心理障碍**：kiiyasun 指出对初学者而言，很多"常识"其实需要时间消化，社群氛围应包容提问。
    
-   **AI 发展到让零编程基础的人也能做完整项目**：bein 自称"完全不会编程，完全是靠 AI"，但做出了 3D 实时世界杯预测平台，颠覆了传统"必须学编程"的认知。

### **2.10 Zixie — 学术视角下的 Web3 认知重构**

**概念重新定义**（从学术视角）：

-   Transaction ≠ 买卖：而是"所有状态的改变"（上传信息、留言、任何数据变更都是交易）
    
-   Gas = 改变状态需要支付的成本
    
-   用"游戏世界观"类比：通过交易改变公共状态
    

**Web2 vs Web3 信任模型**：

-   Web2：相信中心化机构（银行记录余额、学校认证身份）
    
-   Web3：相信共有规则 + 签名 + 代码 + 公开记录
    

**向 AI 提出的深层问题**：

-   Q："为什么需要不同的链，而不是所有东西都放在一个链上？"
    
-   A：Web3 追求的不是便利，"方便并不是它主要追求的东西"，而是"用更高的个人责任换取更强的控制权和可验证性"
    

**研究向的核心疑问（未解决，待后续探索）**：

-   链上记录测量/代表了什么？交易多次 = 真实参与还是刷交互？
    
-   持有 badge 代表真的学习过还是只完成了任务？
    
-   链上活跃度高 = 生态健康还是激励被套利？
    
-   Web3 的"主体"不是个人（individual），而是一个地址——地址背后可能是人/钱包/合约，"这是一种新的模式"
    
-   未来方向：如何衡量一个地址的信誉？如何确保真的是信誉而不是刷日活？

### **2.12 bein 老师研究赛道详情**

**平台技术细节**：

-   bein："我完全不会编程，我完全是靠AI"
    
-   技术选型方法：给 AI 设定场景（"500 人同时玩，需要实时同步游戏画面"）→ AI 推荐技术栈并做对比表格
    
-   精准提示词关键技巧：加上"每一个重要的细节都要向我确认"这句话，AI 就会每个细节都来问你；不加的话 AI 会略过很多内容
    
-   Codex 计划模式（plan mode）的使用
    
-   模块化开发：整体页面让 AI 一起出效果；特效（粒子/足球动画）单独开发
    
-   服务器部署流程：本地开发完成 → 上传到服务器 → Node.js 编译
    

**游戏机制细节**：

-   核心项目：基于模拟世界杯的预测/量化平台，实时 3D 比赛动画
    
-   每个队伍有不同后台参数（经过大量调整）
    
-   比赛实时数据：控球率、射门、角球
    
-   AI 小人行为：追足球跑 → 靠近后铲球/传球 → 走位通过判断其他人与自己的距离 → 形成机器人式交互
    
-   不是用大模型踢球，而是"基于一套算法"（类似五子棋人机对战逻辑）
    

**区别化激励**：

-   普通路线：每天 100 积分
    
-   研究路线：每天 1000 积分
    
-   预测总分第一：学分奖励

-   不用 Monad 币下注（避免大户影响比分），用平台内部活动积分
    

**项目缘起**：bein 当助教时向组织方提议"以游戏形式完成暑期实习计划"→ 后来变成研究项目

-   **建立固定信息源**：每天花 15-30 分钟浏览深潮、星球日报（中文）或 The Block（英文），避免碎片化刷信息。
    
-   **用 AI 建立学习闭环**：先向 AI 输出自己的理解 → 让 AI 诊断薄弱点 → 小测 → 实操 → 截图反馈 → 纠错复盘。
    
-   **提前准备方向选择**：技术向（前端/合约/全栈）、运营向（社区/内容/增长）、研究向（量化/预测/数据分析），周六前确定意向。
    
-   **合约部署前做三步人工检查**：① 首行是否有 SPDX 开源声明；② Solidity 版本号是否正确；③ 代码逻辑是否合理。不要盲信 AI 生成的代码。
    
-   **Claude Code 中国区用户**：用 winget 安装可避免 curl 被封问题，配合中转站 API 使用；如需多模态，装 OCR 插件或换 Codex。
    
-   **Obsidian 多端同步方案**：内网环境用 fast note sync（有服务器版）；外网用 remotely-save 插件 + OneDrive（5GB 免费够存文稿）。
    
-   **AI 开发时降低 Token 消耗**：每个新模块用空文件重新生成，不要在旧文件上反复修——旧文件修改的成本可达重建的十倍。
    
-   **给 AI 加约束语句**：在 Prompt 末尾加"每一个重要的细节都要向我确认"，AI 就不会跳过关键决策点。

通过摄像头、声音识别、AI 分析猫咪行为和叫声，判断猫咪需求，比如饿了、想吃东西、状态异常等。
