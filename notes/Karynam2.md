## 一、核心观点

### 1\. 运营的本质定义

-   运营不只是"发推、拉群、办活动"，而是一套从策划到收尾的完整体系，是社区的"粘合剂"
    
-   运营的核心目标：**持续拉新 + 维护用户联系 + 留存**，本质是"做增长"
    
-   运营同时面向 B 端（项目方/合作方）和 C 端（开发者/Builder），需要全程设计体验和 Pipeline
    
-   在 Web3 中，运营是需求最大的岗位之一（仅次于开发者），且一人多岗是常态
    

### 2\. 当前 Web3 阶段判断

-   **Hagoo 提出**：目前 Web3 处于"大基建阶段"，运营活动主要面向 Builder 拉新，尚未大规模涉及终端消费者
    
-   Lynn 补充：公链类比"高速公路"，Builder 就是上路的车，消费者（终端用户）通过手续费产生价值，但目前尚未大量进入
    
-   交易所运营 vs 项目方运营是两个方向：交易所更像"银行"（拉存入金/出金），项目方更重社群和 Builder 生态
    

### 3\. 运营的 KPI 视角

-   软指标：推文曝光量（如单条 1.2 万次观看）、活动报名人数（单期可达 1000+ 人）
    
-   硬指标：黑客松项目提交数、Builder 组队数量、后续实习/就业对接成功率
    
-   过往数据：冬季实习计划 2295 人报名，本期数百至小 1000 人，前一 AI Web3 活动 1000+ 人

## 二、Lynn（LXDAO 运营负责人）分享详细内容

### 2.1 运营全流程（零到一方法论）

**阶段一：与项目方对接与 Proposal 撰写**

-   明确活动覆盖人群，按人群设计活动形式和课程难度
    
-   设计分享会形式和嘉宾类型
    
-   如果是多赛道黑客松（如 ZK 赛道 + AI 赛道），需协调多方组合
    
-   Proposal 是 B 端运营的起点，需包含活动形式、预期流量、预期项目提交量
    

**阶段二：推广执行**

-   渠道：Twitter、Telegram、微信群、公众号、B 站/YouTube 录播
    
-   内容分层：长文案（主推文，同时发公众号和 Twitter）→ 短版本（社群分发）→ 中英文版本（涉及海外）
    
-   多形式触达：文字推送 + 直播分享会 + Workshop + AMA，不同渠道覆盖不同人群
    

**阶段三：社群运营与 Builder 跟进**

-   入群后持续触达，不停提醒"快来提项目""快来组队"
    
-   收集社群中 Builder 的问题，及时反馈给技术老师/项目方
    
-   解决开发者 Issue 和 Demo 问题，提升体验感
    
-   灵活调度：根据社群需求实时调整课程内容（如有人提出想要法律方向讲师，下周即可安排）
    

**阶段四：活动执行工具**

-   内部协同：Notion（任务管理）
    
-   社群管理：Telegram 群 + 微信群（多个，按活动/会员分）
    
-   报名平台：WCB 报名页面（可直接后台查看所有注册数据）
    
-   表单设计：报名时可设置"渠道来源"字段，追踪不同渠道转化
    

**阶段五：活动收尾与复盘**

-   联系优秀 Builder，对接实习/工作岗位
    
-   标记高潜力项目和高贡献者，推荐进入项目方孵化/加速器
    
-   复盘数据：报名人数、分享会到场率、项目提交数量与质量、就业对接成功率
    

### 2.2 AI 在运营中的实战应用

-   **TG 内 AI Agent**：连接 Notion 工作平台，可一键生成文案、调研、Proposal，支持 @Agent 直接在 TG 中下达指令
    
-   **多模型互补**：用 GPT + Claude + Grok（Twitter 内置）三选二交叉验证，取综合最优结果
    
-   **写文案**：短文案（分享会通知）→ 长文案（主推文）→ Proposal → 主持稿 → 评审标准，均可用 AI 辅助
    
-   **局限与原则**：AI 是信息收集和文字生成工具，核心还是**自己的思路、经验和判断**——这是 Lynn 反复强调的
    

### 2.3 如何从零入行 Web3 运营

-   Lynn 本人经历：2024 年 8 月通过 LXDAO 实习计划入行，一个月建立认知
    
-   **核心路径**：
    
    1.  先参加实习计划/训练营，一个月建立 Web3 认知
        
    2.  主动表现：积分排行榜靠前、笔记质量高、分享会活跃提问
        
    3.  积累作品集：参加黑客松拿奖（官方会发推背书），或运营自己的 Twitter 账号
        
    4.  实习计划结束后，高分者优先获得内推
        
-   **商科背景完全没问题**：Web3 不太看学历背景，更看重实战经验
    

### 2.4 社交媒体运营建议

-   **Twitter 必做**：在 Web3 里，Twitter 等于你的名片/Profile
    
-   **小红书**：不要发"Web3"这个词，容易没流量或被封；可写"个人经验分享""新人入行"等，有真实流量
    
-   **国内视频号**：可用"区块链技术分享"替代"Web3"，可以数字人不露脸但不稳定
    
-   **起号技巧**：如果没有深度内容可写，就从"成长记录"做起——今天学了什么、参与了什么、有什么心得
    

### 2.5 运营岗位细分

| 类型 | 说明 |
| --- | --- |

| 内容运营 | 推文、公众号、视频脚本 |
| 活动运营 | 策划黑客松、Workshop、分享会全流程 |
| 用户/社群运营 | 群内互动、答疑、留存 |
| 渠道运营 | BD 合作方、跨平台分发 |
| 社媒运营 | Twitter、TG、Discord 等 |
| 研究向运营 | 结合技术背景做深度内容输出 |

核心结论：小公司/Startup 几乎都是一人多岗，大厂才会分得细。

### 2.6 工作节奏与风格

-   时间弹性大，忙时熬夜，闲时可能上午 11 点甚至 12 点才开始工作
    
-   工作碎片化严重：群内 Builder 提问 → 立刻反馈技术 → 策划下周内容 → 对接嘉宾 → 数据统计

## 三、Hagoo（资深运营/增长从业者）分享与提问

### 3.1 提问：为什么 Web3 运营不涉及普通消费者？

-   Hagoo 指出：当前 Web3 运营几乎只面向 Builder 和 VC，终端消费者角色缺失，但资产是需要流动的
    
-   Lynn 回应：公链类比高速公路，开发者是上路车辆；当前阶段 Web3 核心是"修路"（基建），消费者端的运营（如交易所拉新/入金）是另一条线
    

### 3.2 提问：多渠道推广的转化追踪工具？

-   Hagoo 详细描述了 Web2 中"推广者 → 专属链接/邀请码 → 自动追踪拉新数 → 成交转化率"的完整链路
    
-   询问 Web3 是否有类似工具（类似企业微信的活码系统）
    
-   Lynn 回复：目前 Web3 运营体量相对小，最简单方法是在报名表单设计"渠道来源"字段，人工统计即可；复杂数据可视化工具暂未用到
    
-   Hagoo 表示会自行深度调研这方面的工具
    

### 3.3 聊天区内分享的实战经验

-   "我这边打粉团队都关闭了几十个，封了差不多 2W 多个号"（说明国内平台对 Web3 内容管控极严）
    
-   国内平台关键词策略：用"区块链"替代"Web3"，避开审核敏感词
    
-   小红书最严，多位参会者反馈被永久封号
    
-   **求职渠道推荐**：SmartDeer（很多交易所运营岗位）
    
-   运营面试主要看重：文案能力、用户分析能力、拉新推广基础能力
    
-   交易所运营偏"银行"属性，核心围绕 Listing、入金、出金
    

### 3.4 Hagoo 的其他观点

-   国内平台受众不太适合 Web3 方向，如果做国内首选视频号

## **一、钱包安全基础篇（第 1-6 题）**

### **第 1 题｜助记词不可泄漏——客服索要助记词怎么办**

-   **场景**：钱包无法登录，自称官方客服私聊你，要求提供助记词才能恢复钱包。
    
-   **正确答案**：C——立即拒绝并举报。
    
-   **Caro 上麦**：助记词不可以泄漏，这肯定是骗子。
    
-   **yoona 补充核心知识点**：
    
    -   助记词不等于登录验证码，也不是客服验证身份的资料。
        
    -   助记词/私钥是你唯一的身份认证，任何人拿到就能在别处恢复钱包并转走资产。
        
    -   再强调：“虽然也没有很多钱，但一定要注意隐私。”
        

* * *

### **第 2 题｜助记词截图发错群怎么办**

-   **场景**：不小心把助记词截图发进微信群，10 秒后撤回，资产暂未丢失，接下来怎么做？
    
-   **正确答案**：C——立即创建新钱包，把原钱包资产转到新钱包。
    
-   **honora 上麦**：助记词丢了别人就有权限转走资产，一定要建新钱包。
    
-   **yoona 补充**：
    
    -   可能当前没丢，但未来某一刻钱包有钱就会被转，原钱包等于废了。
        
    -   操作：赶快把原钱包的钱转到新钱包。
        
-   **聊天区讨论**：
    
    -   evo swift: “第二题不应该截图呢”
        
    -   CaptainC 回复: “yeah 截图就不安全了”
        

* * *

### **第 3 题｜无限授权（Unlimited Spending Cap）**

-   **场景**：领取空投时网站要求授权 USDT spending cap 为 unlimited（无限授权），怎么选？
    
-   **正确答案**：B——手动修改额度为本次领取所需数量。
    
-   **qwq 上麦**：
    
    -   先核实网站是否合规/正规。
        
    -   如果觉得正规，就只给需要的额度，这样即使被骗也只损失该额度。
        
-   **yoona 补充**：

### **第 8 题｜重入攻击——退款函数被重复调用**

-   **场景**：退款函数先给用户转 ETH，再把用户余额清零。攻击者用合约账户反复调用退款函数。
    
-   **正确答案**：B——先清零余额，再执行转账（检查-生效-交互模式）。
    
-   **Caro 上麦**：顺序反了，应该先清零，这样防止二次退款。
    
-   **yoona 深入讲解（Checks-Effects-Interactions 模式）**：
    
    -   假如用户可退款 10 ETH，函数先把 10 ETH 发出再清零 mapping 余额。
        
    -   恶意合约收到 ETH 时会通过 receive 或 fallback 回调重新触发退款函数。
        
    -   第一次调用后还没执行清零，第二次调用看到的余额仍是 10 ETH，又能提现。
        
    -   **修复方案**：
        
        1.  **CEI 模式**：先检查条件 → 更新内部状态（清零/减余额）→ 最后外部交互（转账）。
            
        2.  可加入 ReentrancyGuard 修饰器防止重入。
            
        3.  **设计建议**：用 Pull 模式（用户主动调用领取），而非 Push 模式（主动转账给用户），更安全。

### **第 9 题｜tx.origin 做权限验证的危险**

-   **场景**：合约使用 require(tx.origin == owner) 判断管理员，开发者认为交易最初确实是管理员发起所以安全。
    
-   **正确答案**：B——改用 msg.sender 并搭配 Ownable / AccessControl 授权体系。
    
-   **gavin 上麦**：
    
    -   tx.origin 不应该用于权限验证。
        
    -   msg.sender 能做更明确的权限控制。
        
    -   tx.origin 只看最开始点击确认交易的地址，会造成混淆。
        
-   **emma 补充上麦**：
    
    -   msg.sender 代表直接调用当前合约的上一级地址。
        
    -   权限方面可以设置管理员白名单，所以选 B。
        
-   **yoona 核心讲解（此题为全场最重要的合约安全知识点之一）**：
    
    -   **攻击链路举例**：管理员钱包 A → 调用恶意合约 M → M 调用金库合约 T。对于 T 来说，tx.origin 仍是管理员 A，msg.sender 则是恶意合约 M。
        
    -   如果用 tx.origin 判断权限，攻击会通过验证，因为无论中间经过多少级恶意调用，最原始的调用者 A 仍是管理员。
        
    -   **正确做法**：
        
        1.  使用 msg.sender（谁当前调用这个函数就判断谁的身份）。
            
        2.  搭配 Ownable / AccessControl 的授权体系。
            
        3.  多签和智能账户普及后交易发起链更复杂，tx.origin 做权限极危险。
            
    -   **记忆口诀**：“只看 msg.sender，不看 tx.origin；谁调谁负责，不追溯老祖宗。”

### **第 10 题｜低级 call 未检查返回值**

-   **场景**：合约使用 [target.call](http://target.call)(data) 发起外部调用，但没有检查 success 返回值，直接把订单标记为完成。
    
-   **正确答案**：B——检查返回的 success 值，失败时做回退或标记失败。
    
-   **bybybyy 上麦**：
    
    -   看返回具体是什么，如果是 success 就不用管。
        
    -   如果是 false，需要回退或标记失败。
        
-   **qin singsing 补充上麦**：
    
    -   D 选项重复调用不行，会产生重复转账。
        
    -   与上次请求无关，盲目重试可能导致重复扣款/重复发货。
        
-   **yoona 核心讲解**：
    
    -   call 会返回 (success, returndata)。
        
    -   若目标合约 revert，低级 call **不会自动让当前函数 revert**，而是把 success 设为 false。
        
    -   忽略 success 检查会导致系统状态与真实结果不一致（显示订单完成但实际失败）。
        
    -   **修复方案**：
        
        1.  根据业务需求：整体回滚 / 记录为 failed / 标记 pending 交补偿或人工处理。
            
        2.  必须解析返回数据（有些代币返回行为不一致）。
            
        3.  ERC20 转账应使用 SafeERC20 / safeTransfer，检查合约地址。
            
        4.  不能盲目重试（第一次成功但响应丢失会造成重复执行）。
            
        5.  可使用 OpenZeppelin 的 Address.functionCall 工具冒泡 revert reason。
            
    -   **核心结论**：“发起调用不代表调用成功了，必须检查 success 和返回数据。”

### **第 11 题｜众筹合约的互斥漏洞**

-   **场景**：众筹达到目标但项目方还没提款，用户仍可调用 refund 退款，导致余额又低于目标。合约出了什么问题？
    
-   **正确答案**：B——给合约增加状态变量（如 Active / Succeeded / Failed），用修饰器限制提款和退款互斥。
    
-   **SHADOWCATHH 上麦**：
    
    -   应使提款和退款互斥，否则项目方要提款时用户又退款，项目方提不够钱。
        
    -   A 选项（页面提示不要退款）不行——万一提示了用户还是要退款怎么办？
        
-   **yoona 核心讲解（众筹合约的经典漏洞）**：
    
    -   开发者脑中知道 withdraw 和 refund 是互斥的（达标→提款 / 未达标→退款），但合约只执 行代码，代码里没写互斥逻辑。
        
    -   **典型攻击**：攻击者专门挑这类漏洞，众筹达标后立刻退款使余额低于目标。
        
    -   **修复方案**：
        
        1.  增加状态变量：Active（筹款中）→ Succeeded（达标，项目方提款）→ Failed（未达标，用户退款）。
            
        2.  在函数中加修饰符限制：withdraw 仅允许在 Succeeded 状态调用；refund 仅允许在 Failed 状态调用。
            
        3.  状态转换后不可逆。
            
        4.  优先用 Pull 模式（用户主动领取）而非 Push 模式（自动打回），更安全。
            
        5.  测试时注意调用顺序。

## **五、可执行建议汇总**

### **钱包安全操作清单**

1.  **助记词/私钥**：绝对不截图、不发给任何人、不输入任何网站；任何索要助记词的“客服”都是骗子。
    
2.  **助记词泄漏后**：立即创建新钱包，转移全部资产，废弃旧钱包。
    
3.  **授权原则**：用多少授权多少，拒绝 Unlimited 授权；授权后用 Token Approval Checker 定期清理不认识的授权。
    
4.  **断开连接 ≠ 撤销授权**：授权在链上，需提交新交易把额度改为 0。
    
5.  **转账安全**：核对完整地址（不只看首尾），大额先小额测试；常用地址存入地址簿。
    
6.  **签名原则**：看不懂的签名内容一律拒绝，不签名。
    
7.  **钱包隔离**：资产钱包和社交/交互钱包分开。
    

### **合约开发安全清单**

1.  **权限控制**：mint/withdraw 等高权限函数必须加 onlyOwner 或 AccessControl 修饰器。
    
2.  **重入防护**：遵循 CEI 模式（Check → Effect → Interaction），搭配 ReentrancyGuard。
    
3.  **权限验证**：用 msg.sender 而非 tx.origin；搭配 Ownable/AccessControl。
    
4.  **返回值检查**：低级 call 必须检查 success 和 returndata；ERC20 转账用 SafeERC20。
    
5.  **状态互斥**：众筹/退款类合约必须加状态变量 + 修饰符限制函数调用时机，代码即法律。
    
6.  **Pull 优于 Push**：让用户主动领取而非合约主动转账，降低风险。
    
7.  **上线前审计**：任何涉及资金的合约必须经过专业审计。

## **第一部分：AI 辅助开发方法论（Cooper 分享）**

### **一、核心观点**

-   AI 时代"初级程序员岗位会越来越少"——一个高级程序员 + AI 能产出远超多个初级程序员的价值；留下来的是高级工程师和架构师级别的人。
    
-   AI 不是替代人，而是加速开发周期。使用 AI 时要主动学习，不能"脑袋放空"只拿结果，要在协作中提升自身架构设计能力。
    
-   AI 开发需要遵循工程规范，不能无脑生成代码。核心是"先对齐意图，再生成代码"。
    
-   所有主流 AI 编程工具（Cursor、Windsurf 等）都兼容 [AGENTS.md](http://AGENTS.md)，这个文件是跨工具的项目"说明书"。
    

### **二、关键事实与方法**

**1\.** [**AGENTS.md**](http://AGENTS.md) **— 项目的 AI 规则文件**

-   在项目根目录放置 [AGENTS.md](http://AGENTS.md)，定义：架构边界、编码约定、允许/禁止的行为、交付标准、评审需求、CI/CD 流水线说明。
    
-   **跨工具兼容**：不管用 Cursor、Windsurf 还是其他 agent 工具，都会读取并遵循这个文件。
    
-   **对人类也有用**：Cooper 个人在看开源项目时，如果项目有 [AGENTS.md](http://AGENTS.md)，他会优先读它而非 README，因为 [AGENTS.md](http://AGENTS.md) 往往比 README 更能清晰展示项目架构和入手方式。
    

**2\. TDD（测试驱动开发）— AI 开发的黄金搭档**

-   **流程**：先写测试（定义 function 的期望输入输出）→ 测试失败（证明有问题）→ 做最小实现 → 测试通过 → 按需重构。
    
-   **为何适配 AI**：
    
    -   给 agent 提供"可执行、可验收的标准"。
        
    -   agent 在生成代码过程中会主动运行测试，形成**快速纠错回路**：写完小功能 → 测试发现问题 → agent 自行修正。
        
    -   防止修复一个问题时破坏其他已有行为，即使破坏了也能立刻发现。
        
-   Cooper 举例：用 Cursor 开发时，agent 会主动运行测试，如果测试用例写得全面，每次新增功能、如果实现有问题，agent 会在开发过程中自动发现并修正。
    

**3\. SDD（规范驱动开发）— 先对齐意图再生成代码**

-   **完整流程**：理解为什么做 → 定义规范（做成什么样）→ 设计方案（怎么实现）→ 任务拆分（拆成小的可执行子任务）→ 逐一实现 → 归档留痕。
    
-   **留痕的目的**：后续开发者或 AI 能溯源，了解当初的设计意图和决策依据。
    
-   **工具推荐**：
    
    -   **Spec Kit**：较重、更规范，适合**全新项目**（从零搭建），一条流水线跑下来效果好。
        
    -   **Open Spec**：轻量，聚焦当前任务，不生成过多文档，适合**已有项目/存量迭代**（修 bug、加小功能）。
        
-   团队也可维护自己的规则和模板，Cooper 认为这完全可行。
    

**4\. 跟着 AI 一起提升的四项能力**

| 能力 | 具体内涵 |
| --- | --- |

| 会描述 | 把想法写成目的、目标、用户场景、约束条件和验收标准；本质是学会写能最大化发挥 AI 能力的 prompt |
| 会拆解 | 把复杂任务拆成可执行、可理解、可验证的小步骤，搞清楚每步的依赖、风险和产出物 |
| 会验证 | 测试（手动 + 自动化 + TDD 测试用例）+ 日志；最终验证结果必须由自己把关，不能交给 AI |
| 会设计 | 在与 AI 协作中逐步学会设计架构边界、数据流、协作规则和长期质量体系 |

**5\. 127.0.0.1 笑话背后的思考**

-   Cooper 分享：有人用 AI 做项目后分享链接，拷贝出来的是 [http://127.0.0.1（本地回环地址），被大家笑话"果然是](http://127.0.0.1（本地回环地址），被大家笑话"果然是) AI 干的"。
    
-   **Cooper 的深层解读**：这不是 AI 的原罪——即使没有 AI，缺乏计算机网络知识的人也会犯同样错误。AI 只是**加速了实现过程**，让人跳过了知识学习环节。好处是：发现这个问题后，可以立即问 AI 搞明白原理，比翻书、问老师同学快得多。

## **第二部分：Web3 技术栈概览（Cooper 分享）**

### **一、核心观点**

-   Cooper 将区块链定义为一句话：**"异步的强一致性的可编程金融平台"**。
    
    -   **异步**：交易发上去不会立刻得到结果，要等打包 → 执行 → 共识完成。
        
    -   **强一致性**：网络中数万个节点通过共识后，同一笔交易的结果在所有节点上完全一致。
        
    -   **可编程**：通过智能合约自定义链上逻辑。
        
    -   **金融平台**：无论是 DeFi、链游、借贷，最终都与金融紧密相关。
        

### **二、Web3 开发的五大方向**

| 方向 | 定位 | 入门门槛 | 直观感受 |
| --- | --- | --- | --- |

| DApp 开发 | 前端页面 + 钱包调用 + 链上数据展示；React/Vue + ethers.js/wagmi + 事件/索引服务 | 最适合新手入门 | 强，面向用户 |
| 智能合约开发 | Solidity（以太坊）/ Rust（Solana/ICP）；Hardhat/Foundry 框架；OpenZeppelin 合约模板 | 中等 | 一般 |
| 钱包开发 | 密钥管理、签名、账户抽象、防钓鱼/防重放/防恶意授权/供应链安全；BIP39/BIP44 协议 | 高，责任极大 | 低 |
| 节点开发 | 客户端工程 + 分布式系统；Go/Rust；P2P 网络、EVM、共识机制（PBFT/PoS）；Docker/K8s/Grafana | 最高，"苦哈哈"但有趣 | 很低 |
| 安全审计 | 审计 DApp 代码、出审计报告；区块链行业的"最后一道红线"；极高行业地位 | 极高，垂直且窄 | 低 |

-   Cooper 强调：DApp 开发与合约开发有耦合（做 DApp 总要写合约），但钱包开发、节点开发、安全审计高度独立。DApp 开发者通常直接用 MetaMask/OKX 等成品钱包，用节点服务商提供的 RPC URL，不自己开发钱包或节点。
    

### **三、各方向的技术栈细节**

**DApp 开发**

-   前端：React / Vue 均可
    
-   链交互框架：ethers.js / wagmi（Cooper 个人常用）
    
-   必须了解：以太坊 ABI 规范（如何调用智能合约）、事件机制（Event）
    
-   数据量大时：需引入链下数据索引服务（如 The Graph 等）
    
-   **开发者视角本质**：你只看到一个 node URL（RPC 端点），按以太坊协议发请求即可，不需要关心背后网络长什么样。
    

**智能合约开发**

-   语言：Solidity（以太坊生态）/ Rust（ICP 等生态）
    
-   框架：Hardhat / **Foundry**（Cooper 特别推荐，简洁易用、体系完善）
    
-   安全模板：OpenZeppelin Contracts（经过广泛验证的高安全合约集合）
    
-   测试：区块链测试框架（如 Foundry 内置测试）+ 可验证性框架
    
-   核心学习路径：Solidity 语法 → EVM 机制（调用/存储/Gas）→ 进阶（预言机、MEV、智能合约升级模式）
    

**钱包开发**

-   必学协议：BIP39（助记词）、BIP44（多链 HD 钱包路径）
    
-   密码学：非对称加密、KDF（密钥派生函数）、安全随机数
    
-   实践必须掌握：签名/验签、地址生成、离线密钥管理
    
-   安全攻防：防钓鱼攻击、防重放攻击、防恶意授权、供应链攻击防御
    
-   **责任极重**：间接接触区块链上所有资金
    

**节点开发**

-   主流语言：Go / Rust（Rust 采用率逐年提升）
    
-   网络层：P2P 网络协议、DHT/Kademlia 节点发现协议
    
-   执行层：EVM、StateDB、MPT（Merkle Patricia Trie）
    
-   共识层：PBFT、PoS 机制、硬分叉、最终性、验证者集合
    
-   运维：Docker、Kubernetes、Prometheus + Grafana 监控面板
    
-   入门路径：先跑测试网节点 → 观察同步流程 → 理解共识/重组 → 尝试给开源仓库修 bug 提 PR
    

**安全审计**

-   定位：区块链行业最后的安全保障，项目方上线前普遍找审计公司 review 代码
    
-   审计报告更多是保护项目方自身资金安全，其次才是给用户信心
    

### **四、共同基础知识**

-   **密码学**：哈希（Keccak256/SHA256）、签名规范、非对称加密
    
-   **钱包协议**：BIP39、BIP44
    
-   **区块链核心原理**：账户与状态 → 交易 → 区块 → 共识 → 执行环境（EVM）→ 开发接口（RPC/WebSocket）
    

### **五、以太坊分层架构（开发者视角）**

```
应用层  ← DApp / 钱包 / 机器人 / 索引器
接口层  ← RPC / WebSocket（与执行引擎交互）
执行层  ← 执行客户端（EVM / Mempool / 状态管理）
共识层  ← 共识客户端（最长合法链 / 最终性）
基础层  ← P2P 网络 / 数据结构（MPT）
```

-   Cooper 画了一张图说明：DApp 开发者只看到 node URL；节点开发者才能看到背后完整的 P2P 网络和共识细节；用户最终看到的是 DApp 的前端页面或 App。
    

### **六、一笔交易在以太坊中的完整生命周期**

1.  **DApp 端编码**：根据 ABI 构造智能合约调用编码
    
2.  **估算 Gas**：发送到节点估算执行费用
    
3.  **构造交易**：填充 gasLimit，完成交易构造
    
4.  **钱包签名**：调用钱包完成签名
    
5.  **RPC 广播**：通过 RPC 发送到节点
    
6.  **交易池 + P2P 广播**：节点放入交易池并广播给全网其他节点
    
7.  **打包 + 执行**：节点选出交易、打包成区块、执行
    
8.  **更新本地状态**：更新 EVM 状态
    
9.  **共识达成最终性**：通过共识使区块达到最终性
    
10.  **DApp 读取结果**：读取交易回执、事件、索引数据

## **第三部分：Q&A（Cooper 回答聊天区问题）**

### **Q1：如何选择开发公链（以太坊 vs Solana）？**

-   **学习阶段**：推荐以太坊——开发者最多、生态最友好、资料最通用最多。
    
-   **有明确项目目标**（如做 DeFi 项目给用户用）：Solana 也是可行选择，不排斥。
    
-   **聊天区 Hagoo 补充**：以太坊有大量共识规范，先学；其他公链大部分是灰色地带，不要乱入。
    

### **Q2：节点开发的性能瓶颈和研究热点？**

-   **TPS 瓶颈已不是主要矛盾**：Monad 等新公链号称万级甚至 5 万 TPS（实验数据），生产环境更复杂但总体 TPS 瓶颈已大幅缓解。
    
-   **共识层面优化**：Solana PoH + Tower BFT、Monad 类 HotStuff 算法等，从不同层面提升共识效率。
    
-   **存储瓶颈**：如何优化存储空间和数据结构，降低个人用户运行节点的门槛，让更多人能参与节点部署和挖矿。
    
-   **当前最热热点**：Cooper 未完整说出但指向**隐私技术**方向（ZK 相关）。
    

### **Q3：Vibe Coding 新手如何学习约束和验证？**

-   **推荐从 TDD 理念入手**：先理解 TDD 的编程哲学，不需要自己手写测试代码（AI 可以承担编码工作）。
    
-   了解 TDD 理念后，在 AI 开发中去实践——这能对 AI 编程的结果产生最大正向效果。
    
-   **路径**：理解理念 → 实践中让 AI 写测试 → 逐步掌握验证和约束的方法。

    ## 一、郑确分享：Web3 职业规划与学习路径
    
    ### 核心观点
    
    -   Web3 行业更偏向实践驱动，工作机会主要通过 Twitter、LinkedIn、GitHub 获取，而非传统招聘渠道
        
    -   与其包装简历，不如通过在网络上持续构建个人作品脉络（GitHub 提交记录、技术文章、推特分享）来证明能力，这种方式比传统简历更直接有效
        
    -   随著 AI 发展，职业边界日益模糊——前端做后端的事、开发者做 Meme 内容运营，交叉型知识人才是更好的方向
        
    -   "两个人的活一个人干"本质是看产出：如果 AI 能帮你更快产出并得到验证，何乐而不为
        
    -   Idea 本身不重要，被抄袭很正常；真正比拼的是产品面向市场的扩张力和基于市场反馈的快速迭代能力
        
    -   开源反而是一种更强大的市场扩张方式，比如 Uniswap 每次更新版本全部开源，BSC 等链都采用其机制，反过来强化了 Uniswap 的产品认可度
        
    
    ### 关键事实
    
    **Weaker 学习系统结构**：
    
    -   日历可见明日、后天、周五均有相关环节
        
    -   通用任务：先选择职业基础方向
        
    -   三个赛道课程：技术（Technical）、运营（Operation）、研究（Research），点击即可查看对应任务
        
    -   挑战者任务：GitLab 开源贡献——认识开源项目、提 PR、提 Issue
        
    -   第三周：团队协作（开发+运营+研究组成团队打磨产品，为黑客松做准备）
        
    -   第五周：按不同方向帮助整理简历
        
    
    **研究方向系统性学习路径**：
    
    -   从底层研究 EIP/NIP 等提案架构
        
    -   做产品层面的分析
        
    -   研究偏向于研究性内容（偏学术/协议层面）
        
    -   如果研究一个产品的用户增长策略，则偏向运营方向——取决于你的切入角度

    -   只需完成对应赛道的任务即可
        
    -   如果还没有明确目标、精力足够，可以三个方向同时学

    -   Web3 从业者建议拥有社交账号并持续输出内容，会加分

    -   即使是纯技术开发者，如以太坊基金会成员，也会定期发推特分享技术内容

    ## 二、Box Chen 分享：Moss 开源项目贡献指南
    
    ### 核心观点
    
    -   Moss 是一个 MCP（Model Context Protocol）项目，定位为 AI Agent 与区块链之间的中间层，核心目标是将交易构建简化为"一键调用"
        
    -   现在是早期窗口期——Moss 目前只对内部全面开放，后续会有严格的贡献者计划，越早参与越容易提交有意义的贡献
        
    -   鼓励使用 AI Agent 辅助开发，但提交前必须自行审计代码、确保所有测试通过，Agent 只是工具，最终兜底的是开发者本人
        
    -   强烈推荐团队合作而非闭门造车——这不仅能建立人际关系，更重要的是提前培养黑客松和职场中的协作意识
        
    -   主力贡献者将获得贡献者计划福利：奖金激励、活动机票报销、生态项目优先考虑、面试机会等
        
    
    ### 关键事实
    
    **Moss 技术原理（四步流程）**：
    
    1.  **发现协议**：扫描区块链上的协议（Uniswap、Neverland、Morpho、Pendle 等 DeFi 协议），将其能力封装为代码
        
    2.  **加载协议**：Agent 获知协议后可加载其具体能力描述（协议能做什么）
        
    3.  **构建交易**：Agent 传递参数，Moss 根据协议暴露的意图能力构建未签名交易结构数据，返回给 Agent 判断
        
    4.  **模拟验证 + 执行**：Agent 确认后将数据返还 Moss 进行模拟验证，告知 Agent 交易具体完成内容和是否符合意图，最后由 Agent 调用任意钱包执行
        
    
    **与传统方式对比**：
    
    -   传统方式：Agent 调用区块链需安装 Viem、Ethers 等多种依赖包，自行构建和提交交易
        
    -   Moss 方式：将构建交易这层抽象为中间层，Agent 只需描述意图即可
        
    
    **仓库结构**：
    
    -   配置目录中有模板包和已有示例包（如 EIP 包），建议先阅读这些包了解开发规范
        
    -   项目有 ADR（Architecture Decision Records）文档，定义了构建时的注意事项和细节
        
    -   提供了 Claude MCP 配置，可直接加载到 Claude Code 中使用
        
    -   Examples 目录包含可在本地完成 fork 并执行模拟交易的完整流程（零资金消耗）
        
    
    **六个贡献方向**：
    
    1.  **主流协议补充**：在 Moss 上接入 DeFi 主流协议（可参考 DeFiLlama 或官方 App 查看热门协议）
        
    2.  **新兴协议接入**：后续新协议出现时帮助做 Agent 接入，这是长期计划，会有奖励任务
        
    3.  **基于 Moss 构建应用**：利用 Moss 构建交易系统、自动收益机器人（遍历所有协议找最高 APY、循环贷、多协议质押等），这也是黑客松赛道之一
        
    4.  **推广运营**：写了好的 Demo 或应用后，在中文社交渠道和官方渠道做推广
        
    5.  **内容创作**：撰写文章、制作视频介绍 Moss 协议
        
    6.  **社区协作**：将以上五点融合——有人做应用、有人做内容推广，在社群内喊队友组队

    -   第一步：复制模板包，让 Agent 先阅读 ADR 文档和 MD 说明

    -   第四步：确认所有 CI 检测通过

    ### 区块链如何保护知识产权？Idea 被抄袭怎么办？
    
    **Box Chen 回答要点**：

### **2.10 Zixie — 学术视角下的 Web3 认知重构**

**概念重新定义**（从学术视角）：

-   Transaction ≠ 买卖：而是"所有状态的改变"（上传信息、留言、任何数据变更都是交易）
    
-   Gas = 改变状态需要支付的成本
    
-   用"游戏世界观"类比：通过交易改变公共状态
    

**Web2 vs Web3 信任模型**：

-   Web2：相信中心化机构（银行记录余额、学校认证身份）
    
-   Web3：相信共有规则 + 签名 + 代码 + 公开记录
    

**向 AI 提出的深层问题**：

-   Q："为什么需要不同的链，而不是所有东西都放在一个链上？"
    
-   A：Web3 追求的不是便利，"方便并不是它主要追求的东西"，而是"用更高的个人责任换取更强的控制权和可验证性"
    

**研究向的核心疑问（未解决，待后续探索）**：

-   链上记录测量/代表了什么？交易多次 = 真实参与还是刷交互？
    
-   持有 badge 代表真的学习过还是只完成了任务？
    
-   链上活跃度高 = 生态健康还是激励被套利？
    
-   Web3 的"主体"不是个人（individual），而是一个地址——地址背后可能是人/钱包/合约，"这是一种新的模式"
    
-   未来方向：如何衡量一个地址的信誉？如何确保真的是信誉而不是刷日活？

### 对参与者

-   **立即动手实操**：跟着 Build Anything 完成一个最小 DApp（留言板即可），不要等"学完理论"再开始。
    
-   **搭建个人知识库**：用 Obsidian/Notion 记录每天学习心得，结合 AI 插件实现标签化管理和检索。
    
-   **建立固定信息源**：每天花 15-30 分钟浏览深潮、星球日报（中文）或 The Block（英文），避免碎片化刷信息。
    
-   **用 AI 建立学习闭环**：先向 AI 输出自己的理解 → 让 AI 诊断薄弱点 → 小测 → 实操 → 截图反馈 → 纠错复盘。
    
-   **提前准备方向选择**：技术向（前端/合约/全栈）、运营向（社区/内容/增长）、研究向（量化/预测/数据分析），周六前确定意向。
    
-   **合约部署前做三步人工检查**：① 首行是否有 SPDX 开源声明；② Solidity 版本号是否正确；③ 代码逻辑是否合理。不要盲信 AI 生成的代码。

-   **Obsidian 多端同步方案**：内网环境用 fast note sync（有服务器版）；外网用 remotely-save 插件 + OneDrive（5GB 免费够存文稿）。
    
-   **AI 开发时降低 Token 消耗**：每个新模块用空文件重新生成，不要在旧文件上反复修——旧文件修改的成本可达重建的十倍。
    
-   **给 AI 加约束语句**：在 Prompt 末尾加"每一个重要的细节都要向我确认"，AI 就不会跳过关键决策点。

6\. Web3 安全意识

新人很容易被钓鱼网站骗，尤其是空投、领福利、钱包授权类网站。

常见风险：

点陌生链接

连接钱包

授权恶意合约

钱包资产被自动转走

U 卡 / 礼品卡被盗

交易所、邮箱、身份信息被关联攻击

所以新人必须注意：

陌生链接不要点，钱包不要乱连，授权一定要看清楚。

Web3 里钱一旦被链上转走，追回非常困难。

7\. 区块链专业和就业现实

即使是区块链专业，最后真正做区块链相关工作的人也不多。

很多人会转向：

普通程序员

AI

销售

主播

考研

考公

其他互联网岗位

原因是区块链行业岗位本身不多，而且对项目经验、圈子资源和实战能力要求比较高。

所以学历或专业不是最关键，真正重要的是：

项目经验、实操能力、社群连接和持续学习能力。

8\. 黑客松项目案例：喵星智能体

他们之前做过一个 AI + Web3 + 宠物硬件 项目，叫“喵星智能体”。

核心想法：

通过摄像头、声音识别、AI 分析猫咪行为和叫声，判断猫咪需求，比如饿了、想吃东西、状态异常等。

项目流程大概是：

采集猫咪声音 / 行为数据

通过摄像头、麦克风等设备收集信息。

AI 分析猫咪意图

把猫叫声转成声纹向量，再结合 RAG 和模型判断猫咪可能想表达什么。

多轮确认需求

如果模型不确定，会继续通过互动判断猫咪需求。

主人审批

当系统判断猫咪想买东西，比如猫粮或零食，会推送给主人确认。

链上支付 / 存证

通过预言机和智能合约，把审批、支付、履约流程连接起来。

9\. 这个项目为什么要用区块链

有人问：这个项目用互联网也能做，为什么要用区块链？

可以理解为，区块链在这里不是为了炫技，而是用于：

支付流程

链上存证

订单记录

权限审批

透明履约

多方共享可信数据

不过这也提醒一个点：

Web3 项目不能为了上链而上链，必须说明区块链解决了什么现实问题。

10\. 对学生最有用的启发

Web3 找机会不能只靠投简历。

更现实的路径是：

先学习基础 → 做一个项目 → 参加社群 / 黑客松 → 输出内容 → 认识人 → 获得内推或合作机会。

对于学生来说，最应该积累的是：

一个能讲清楚的项目

一段持续学习记录

一个内容账号或作品集

一些社群参与经历

一点行业方向判断

基础的安全意识

1️⃣会议总结：普通开发者如何进入以太坊协议层

**主讲人**：JackCC（EPF 第7期华语区唯一入选者，全球仅10人）

**会议时间**：2026-07-07 20:03

###一、核心观点

\- **以太坊协议层 ≠ 智能合约/DApp 开发**：协议层关注的是以太坊底层机制——EIP 提案研究、客户端实现、共识层/执行层规范、扩容（如 Gas 降低、零知识证明）等，而非上层应用。

\- **以太坊生态的核心竞争力是开发者社区**：以太坊是全球开发者数量最多的公链，基金会通过 EPF 等项目吸引新人参与，体现去中心化精神。

\- **EPF 面试不看算法八股，看重热情和研究潜力**：不会考手撕算法或密码学难题，更多聊你对以太坊的了解、对哪些 EIP 感兴趣、是否有持续贡献的意愿。

\- **以太坊技术更新极度保守谨慎**：一个 EIP 提案从提出到上线客户端可能需 1-2 年、多轮 review，作为承载全球大量资金的系统不可能激进跟进新技术（如 AI agent）。

\- **协议层不适合功利求职者**：大部分人是以副业形式参与，经济回报不确定；如果目标是稳定高薪，Web2 路线更确定。

\- **英语是 Web3 的"第一编程语言"**：面试全英文、日常讨论技术英语、国际化协作场景多，但不需要雅思口语级别，能清晰表达想法即可。

\- **多客户端、多语言是以太坊去中心化的基石**：不能让单一语言/客户端占据太大份额，防止单点故障影响整个网络。

\### 二、关键事实

\- EPF（Ethereum Protocol Fellowship）：面向所有对以太坊内部机制感兴趣的人，分为 permissioned 和 permissionless 两种，未获许可也可参加，最终看项目成果。

\- 有学员通过 EPF 项目后以实习生身份进入执行层规范团队，说明存在实习机会，但机会较少。

2️⃣会议总结：Co-Learning 第二次分享

**会议主题**：用 AI Coding 构建 Web3 Mini Demo — 从合约到前端的完整闭环

**分享人**：ETH Coooder / 41

**日期**：2026-07-07 20:05-21:18

\### 一、核心观点

\- **AI Coding 是 Web3 开发的新范式**：用 Codex/Cursor 等工具，无需 Remix 手动操作，即可完成合约编写、编译、部署到测试网的全流程，开发效率大幅提升，甚至非技术人员也可参与。

\- **Remix 仍有学习价值**：初学者应先通过 Remix 走一遍完整流程（编写→编译→部署→获取 ABI），理解各环节后再转向 AI Coding，否则“完成了任务但不知道做了什么”。

\- **私钥安全是 Web3 生存底线**：私钥/助记词代表钱包所有权，泄露即资产归零`.env` 文件绝不能上传 GitHub，中转站 API 有偷私钥风险，开发钱包和真实钱包必须严格隔离。

\- **智能合约 + 前端 = Web3 Demo 的最小闭环**：ABI/JSON 文件是合约与前端之间的接口契约，前端通过它调用链上合约。一个 Mini Demo 0 只需一个合约 + 一个前端即可。

\- **区块链应被理解为后端的补充而非替代**：受 Gas 费和延迟制约，不能将所有传统后端逻辑上链；实际系统通常是传统后端与智能合约互为补充。

\- **Co-Learning 定位是互助交流，不是单向授课**：鼓励自主学习后用 AI 探索，遇到问题来共学时间或群里提问。

\---

\### 二、关键事实

\- 三周 Share 节奏：第一周打理论基础，第二周对应课程设计，第三周请嘉宾分享产品化/商业化思路。

\- 推荐钱包：\*\*Rabby\*\*（测试网模式清晰，比 MetaMask 体验好）；Phantom 兼容 Solana 等多链。

\- AI Coding 工具推荐：Codex（主讲人使用）；Claude Code 也可；Plus 套餐日常够用，Pro 适合专职开发。

\- ABI/JSON 文件由合约编译自动生成，相当于接口文档，前端需据此调用合约函数。

\- 部署合约消耗 Gas（测试网上演示消耗约 0.02 ETH 测试币）。

\- Monad 测试网浏览器地址`testnet.monadscan.com`。

\- BuildAnything 学习平台课程需按解锁顺序学习，可自定节奏。

\- Claude 封号问题：注意节点和付款方式一致性；Codex 有概率因网络环境触发风控；建议开启 2FA 降低风控概率。

\- 曾有黑客松参赛者因 GitHub 上泄露 `.env` 中的私钥，奖金刚到账就被转走。
