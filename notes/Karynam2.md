## **第一部分：AI 辅助开发方法论（Cooper 分享）**

### **一、核心观点**

-   AI 时代"初级程序员岗位会越来越少"——一个高级程序员 + AI 能产出远超多个初级程序员的价值；留下来的是高级工程师和架构师级别的人。
    
-   AI 不是替代人，而是加速开发周期。使用 AI 时要主动学习，不能"脑袋放空"只拿结果，要在协作中提升自身架构设计能力。
    
-   AI 开发需要遵循工程规范，不能无脑生成代码。核心是"先对齐意图，再生成代码"。
    
-   所有主流 AI 编程工具（Cursor、Windsurf 等）都兼容 [AGENTS.md](http://AGENTS.md)，这个文件是跨工具的项目"说明书"。
    

### **二、关键事实与方法**

**1\.** [**AGENTS.md**](http://AGENTS.md) **— 项目的 AI 规则文件**

-   在项目根目录放置 [AGENTS.md](http://AGENTS.md)，定义：架构边界、编码约定、允许/禁止的行为、交付标准、评审需求、CI/CD 流水线说明。
    
-   **跨工具兼容**：不管用 Cursor、Windsurf 还是其他 agent 工具，都会读取并遵循这个文件。
    
-   **对人类也有用**：Cooper 个人在看开源项目时，如果项目有 [AGENTS.md](http://AGENTS.md)，他会优先读它而非 README，因为 [AGENTS.md](http://AGENTS.md) 往往比 README 更能清晰展示项目架构和入手方式。
    

**2\. TDD（测试驱动开发）— AI 开发的黄金搭档**

-   **流程**：先写测试（定义 function 的期望输入输出）→ 测试失败（证明有问题）→ 做最小实现 → 测试通过 → 按需重构。
    
-   **为何适配 AI**：
    
    -   给 agent 提供"可执行、可验收的标准"。
        
    -   agent 在生成代码过程中会主动运行测试，形成**快速纠错回路**：写完小功能 → 测试发现问题 → agent 自行修正。
        
    -   防止修复一个问题时破坏其他已有行为，即使破坏了也能立刻发现。
        
-   Cooper 举例：用 Cursor 开发时，agent 会主动运行测试，如果测试用例写得全面，每次新增功能、如果实现有问题，agent 会在开发过程中自动发现并修正。
    

**3\. SDD（规范驱动开发）— 先对齐意图再生成代码**

-   **完整流程**：理解为什么做 → 定义规范（做成什么样）→ 设计方案（怎么实现）→ 任务拆分（拆成小的可执行子任务）→ 逐一实现 → 归档留痕。
    
-   **留痕的目的**：后续开发者或 AI 能溯源，了解当初的设计意图和决策依据。
    
-   **工具推荐**：
    
    -   **Spec Kit**：较重、更规范，适合**全新项目**（从零搭建），一条流水线跑下来效果好。
        
    -   **Open Spec**：轻量，聚焦当前任务，不生成过多文档，适合**已有项目/存量迭代**（修 bug、加小功能）。
        
-   团队也可维护自己的规则和模板，Cooper 认为这完全可行。
    

**4\. 跟着 AI 一起提升的四项能力**

| 能力 | 具体内涵 |
| --- | --- |

| 会描述 | 把想法写成目的、目标、用户场景、约束条件和验收标准；本质是学会写能最大化发挥 AI 能力的 prompt |
| 会拆解 | 把复杂任务拆成可执行、可理解、可验证的小步骤，搞清楚每步的依赖、风险和产出物 |
| 会验证 | 测试（手动 + 自动化 + TDD 测试用例）+ 日志；最终验证结果必须由自己把关，不能交给 AI |
| 会设计 | 在与 AI 协作中逐步学会设计架构边界、数据流、协作规则和长期质量体系 |

**5\. 127.0.0.1 笑话背后的思考**

-   Cooper 分享：有人用 AI 做项目后分享链接，拷贝出来的是 [http://127.0.0.1（本地回环地址），被大家笑话"果然是](http://127.0.0.1（本地回环地址），被大家笑话"果然是) AI 干的"。

## **第二部分：Web3 技术栈概览（Cooper 分享）**

### **一、核心观点**

-   Cooper 将区块链定义为一句话：**"异步的强一致性的可编程金融平台"**。
    
    -   **异步**：交易发上去不会立刻得到结果，要等打包 → 执行 → 共识完成。
        
    -   **强一致性**：网络中数万个节点通过共识后，同一笔交易的结果在所有节点上完全一致。
        
    -   **可编程**：通过智能合约自定义链上逻辑。
        
    -   **金融平台**：无论是 DeFi、链游、借贷，最终都与金融紧密相关。
        

### **二、Web3 开发的五大方向**

| 方向 | 定位 | 入门门槛 | 直观感受 |
| --- | --- | --- | --- |

| DApp 开发 | 前端页面 + 钱包调用 + 链上数据展示；React/Vue + ethers.js/wagmi + 事件/索引服务 | 最适合新手入门 | 强，面向用户 |
| 智能合约开发 | Solidity（以太坊）/ Rust（Solana/ICP）；Hardhat/Foundry 框架；OpenZeppelin 合约模板 | 中等 | 一般 |
| 钱包开发 | 密钥管理、签名、账户抽象、防钓鱼/防重放/防恶意授权/供应链安全；BIP39/BIP44 协议 | 高，责任极大 | 低 |
| 节点开发 | 客户端工程 + 分布式系统；Go/Rust；P2P 网络、EVM、共识机制（PBFT/PoS）；Docker/K8s/Grafana | 最高，"苦哈哈"但有趣 | 很低 |
| 安全审计 | 审计 DApp 代码、出审计报告；区块链行业的"最后一道红线"；极高行业地位 | 极高，垂直且窄 | 低 |

-   Cooper 强调：DApp 开发与合约开发有耦合（做 DApp 总要写合约），但钱包开发、节点开发、安全审计高度独立。DApp 开发者通常直接用 MetaMask/OKX 等成品钱包，用节点服务商提供的 RPC URL，不自己开发钱包或节点。
    

### **三、各方向的技术栈细节**

**DApp 开发**

-   前端：React / Vue 均可
    
-   链交互框架：ethers.js / wagmi（Cooper 个人常用）
    
-   必须了解：以太坊 ABI 规范（如何调用智能合约）、事件机制（Event）
    
-   数据量大时：需引入链下数据索引服务（如 The Graph 等）
    
-   **开发者视角本质**：你只看到一个 node URL（RPC 端点），按以太坊协议发请求即可，不需要关心背后网络长什么样。
    

**智能合约开发**

-   语言：Solidity（以太坊生态）/ Rust（ICP 等生态）
    
-   框架：Hardhat / **Foundry**（Cooper 特别推荐，简洁易用、体系完善）
    
-   安全模板：OpenZeppelin Contracts（经过广泛验证的高安全合约集合）
    
-   测试：区块链测试框架（如 Foundry 内置测试）+ 可验证性框架
    
-   核心学习路径：Solidity 语法 → EVM 机制（调用/存储/Gas）→ 进阶（预言机、MEV、智能合约升级模式）
    

**钱包开发**

-   必学协议：BIP39（助记词）、BIP44（多链 HD 钱包路径）
    
-   密码学：非对称加密、KDF（密钥派生函数）、安全随机数
    
-   实践必须掌握：签名/验签、地址生成、离线密钥管理
    
-   安全攻防：防钓鱼攻击、防重放攻击、防恶意授权、供应链攻击防御
    
-   **责任极重**：间接接触区块链上所有资金
    

**节点开发**

-   主流语言：Go / Rust（Rust 采用率逐年提升）
    
-   网络层：P2P 网络协议、DHT/Kademlia 节点发现协议
    
-   执行层：EVM、StateDB、MPT（Merkle Patricia Trie）
    
-   共识层：PBFT、PoS 机制、硬分叉、最终性、验证者集合
    
-   运维：Docker、Kubernetes、Prometheus + Grafana 监控面板
    
-   入门路径：先跑测试网节点 → 观察同步流程 → 理解共识/重组 → 尝试给开源仓库修 bug 提 PR
    

**安全审计**

-   定位：区块链行业最后的安全保障，项目方上线前普遍找审计公司 review 代码
    
-   审计报告更多是保护项目方自身资金安全，其次才是给用户信心
    

### **四、共同基础知识**

-   **密码学**：哈希（Keccak256/SHA256）、签名规范、非对称加密
    
-   **钱包协议**：BIP39、BIP44
    
-   **区块链核心原理**：账户与状态 → 交易 → 区块 → 共识 → 执行环境（EVM）→ 开发接口（RPC/WebSocket）
    

### **五、以太坊分层架构（开发者视角）**

```
应用层  ← DApp / 钱包 / 机器人 / 索引器
接口层  ← RPC / WebSocket（与执行引擎交互）
执行层  ← 执行客户端（EVM / Mempool / 状态管理）
共识层  ← 共识客户端（最长合法链 / 最终性）
基础层  ← P2P 网络 / 数据结构（MPT）
```

-   Cooper 画了一张图说明：DApp 开发者只看到 node URL；节点开发者才能看到背后完整的 P2P 网络和共识细节；用户最终看到的是 DApp 的前端页面或 App。
    

### **六、一笔交易在以太坊中的完整生命周期**

1.  **DApp 端编码**：根据 ABI 构造智能合约调用编码
    
2.  **估算 Gas**：发送到节点估算执行费用
    
3.  **构造交易**：填充 gasLimit，完成交易构造
    
4.  **钱包签名**：调用钱包完成签名
    
5.  **RPC 广播**：通过 RPC 发送到节点
    
6.  **交易池 + P2P 广播**：节点放入交易池并广播给全网其他节点
    
7.  **打包 + 执行**：节点选出交易、打包成区块、执行
    
8.  **更新本地状态**：更新 EVM 状态
    
9.  **共识达成最终性**：通过共识使区块达到最终性
    
10.  **DApp 读取结果**：读取交易回执、事件、索引数据

## **第三部分：Q&A（Cooper 回答聊天区问题）**

### **Q1：如何选择开发公链（以太坊 vs Solana）？**

-   **学习阶段**：推荐以太坊——开发者最多、生态最友好、资料最通用最多。
    
-   **有明确项目目标**（如做 DeFi 项目给用户用）：Solana 也是可行选择，不排斥。
    
-   **聊天区 Hagoo 补充**：以太坊有大量共识规范，先学；其他公链大部分是灰色地带，不要乱入。
    

### **Q2：节点开发的性能瓶颈和研究热点？**

-   **TPS 瓶颈已不是主要矛盾**：Monad 等新公链号称万级甚至 5 万 TPS（实验数据），生产环境更复杂但总体 TPS 瓶颈已大幅缓解。
    
-   **共识层面优化**：Solana PoH + Tower BFT、Monad 类 HotStuff 算法等，从不同层面提升共识效率。
    
-   **存储瓶颈**：如何优化存储空间和数据结构，降低个人用户运行节点的门槛，让更多人能参与节点部署和挖矿。
    
-   **当前最热热点**：Cooper 未完整说出但指向**隐私技术**方向（ZK 相关）。
    

### **Q3：Vibe Coding 新手如何学习约束和验证？**

-   **推荐从 TDD 理念入手**：先理解 TDD 的编程哲学，不需要自己手写测试代码（AI 可以承担编码工作）。
    
-   了解 TDD 理念后，在 AI 开发中去实践——这能对 AI 编程的结果产生最大正向效果。
    
-   **路径**：理解理念 → 实践中让 AI 写测试 → 逐步掌握验证和约束的方法。

    ## 一、郑确分享：Web3 职业规划与学习路径
    
    ### 核心观点
    
    -   Web3 行业更偏向实践驱动，工作机会主要通过 Twitter、LinkedIn、GitHub 获取，而非传统招聘渠道
        
    -   与其包装简历，不如通过在网络上持续构建个人作品脉络（GitHub 提交记录、技术文章、推特分享）来证明能力，这种方式比传统简历更直接有效
        
    -   随著 AI 发展，职业边界日益模糊——前端做后端的事、开发者做 Meme 内容运营，交叉型知识人才是更好的方向
        
    -   "两个人的活一个人干"本质是看产出：如果 AI 能帮你更快产出并得到验证，何乐而不为
        
    -   Idea 本身不重要，被抄袭很正常；真正比拼的是产品面向市场的扩张力和基于市场反馈的快速迭代能力
        
    -   开源反而是一种更强大的市场扩张方式，比如 Uniswap 每次更新版本全部开源，BSC 等链都采用其机制，反过来强化了 Uniswap 的产品认可度
        
    
    ### 关键事实
    
    **Weaker 学习系统结构**：
    
    -   日历可见明日、后天、周五均有相关环节
        
    -   通用任务：先选择职业基础方向
        
    -   三个赛道课程：技术（Technical）、运营（Operation）、研究（Research），点击即可查看对应任务
        
    -   挑战者任务：GitLab 开源贡献——认识开源项目、提 PR、提 Issue
        
    -   第三周：团队协作（开发+运营+研究组成团队打磨产品，为黑客松做准备）
        
    -   第五周：按不同方向帮助整理简历
        
    
    **研究方向系统性学习路径**：
    
    -   从底层研究 EIP/NIP 等提案架构

    ## 二、Box Chen 分享：Moss 开源项目贡献指南
    
    ### 核心观点
    
    -   Moss 是一个 MCP（Model Context Protocol）项目，定位为 AI Agent 与区块链之间的中间层，核心目标是将交易构建简化为"一键调用"
        
    -   现在是早期窗口期——Moss 目前只对内部全面开放，后续会有严格的贡献者计划，越早参与越容易提交有意义的贡献
        
    -   鼓励使用 AI Agent 辅助开发，但提交前必须自行审计代码、确保所有测试通过，Agent 只是工具，最终兜底的是开发者本人
        
    -   强烈推荐团队合作而非闭门造车——这不仅能建立人际关系，更重要的是提前培养黑客松和职场中的协作意识
        
    -   主力贡献者将获得贡献者计划福利：奖金激励、活动机票报销、生态项目优先考虑、面试机会等
        
    
    ### 关键事实
    
    **Moss 技术原理（四步流程）**：
    
    1.  **发现协议**：扫描区块链上的协议（Uniswap、Neverland、Morpho、Pendle 等 DeFi 协议），将其能力封装为代码
        
    2.  **加载协议**：Agent 获知协议后可加载其具体能力描述（协议能做什么）
        
    3.  **构建交易**：Agent 传递参数，Moss 根据协议暴露的意图能力构建未签名交易结构数据，返回给 Agent 判断
        
    4.  **模拟验证 + 执行**：Agent 确认后将数据返还 Moss 进行模拟验证，告知 Agent 交易具体完成内容和是否符合意图，最后由 Agent 调用任意钱包执行
        
    
    **与传统方式对比**：
    
    -   传统方式：Agent 调用区块链需安装 Viem、Ethers 等多种依赖包，自行构建和提交交易
        
    -   Moss 方式：将构建交易这层抽象为中间层，Agent 只需描述意图即可
        
    
    **仓库结构**：
    
    -   配置目录中有模板包和已有示例包（如 EIP 包），建议先阅读这些包了解开发规范
        
    -   项目有 ADR（Architecture Decision Records）文档，定义了构建时的注意事项和细节
        
    -   提供了 Claude MCP 配置，可直接加载到 Claude Code 中使用
        
    -   Examples 目录包含可在本地完成 fork 并执行模拟交易的完整流程（零资金消耗）
        
    
    **六个贡献方向**：
    
    1.  **主流协议补充**：在 Moss 上接入 DeFi 主流协议（可参考 DeFiLlama 或官方 App 查看热门协议）
        
    2.  **新兴协议接入**：后续新协议出现时帮助做 Agent 接入，这是长期计划，会有奖励任务
        
    3.  **基于 Moss 构建应用**：利用 Moss 构建交易系统、自动收益机器人（遍历所有协议找最高 APY、循环贷、多协议质押等），这也是黑客松赛道之一
        
    4.  **推广运营**：写了好的 Demo 或应用后，在中文社交渠道和官方渠道做推广
        
    5.  **内容创作**：撰写文章、制作视频介绍 Moss 协议
        
    6.  **社区协作**：将以上五点融合——有人做应用、有人做内容推广，在社群内喊队友组队
        
    
    **使用 AI Agent 辅助开发的推荐流程**：
    
    -   第一步：复制模板包，让 Agent 先阅读 ADR 文档和 MD 说明
        
    -   第二步：给 Agent 详细需求，或让 Agent 拉取 GitHub Issue 中的需求
        
    -   第三步：遵循 TDD（测试驱动开发）模式——让 Agent 参考已有包的 test 写法来完成测试
        
    -   第四步：确认所有 CI 检测通过
        
    -   任何 Issue 都可以自己提交（提出需求或功能建议），也可以认领已有 Issue

    ### 区块链如何保护知识产权？Idea 被抄袭怎么办？

### **2.10 Zixie — 学术视角下的 Web3 认知重构**

**概念重新定义**（从学术视角）：

-   Transaction ≠ 买卖：而是"所有状态的改变"（上传信息、留言、任何数据变更都是交易）
    
-   Gas = 改变状态需要支付的成本
    
-   用"游戏世界观"类比：通过交易改变公共状态
    

**Web2 vs Web3 信任模型**：

-   Web2：相信中心化机构（银行记录余额、学校认证身份）
    
-   Web3：相信共有规则 + 签名 + 代码 + 公开记录
    

**向 AI 提出的深层问题**：

-   Q："为什么需要不同的链，而不是所有东西都放在一个链上？"
    
-   A：Web3 追求的不是便利，"方便并不是它主要追求的东西"，而是"用更高的个人责任换取更强的控制权和可验证性"
    

**研究向的核心疑问（未解决，待后续探索）**：

-   链上记录测量/代表了什么？交易多次 = 真实参与还是刷交互？
    
-   持有 badge 代表真的学习过还是只完成了任务？
    
-   链上活跃度高 = 生态健康还是激励被套利？
    
-   Web3 的"主体"不是个人（individual），而是一个地址——地址背后可能是人/钱包/合约，"这是一种新的模式"
    
-   未来方向：如何衡量一个地址的信誉？如何确保真的是信誉而不是刷日活？

## 三、可执行建议

### 对参与者

-   **立即动手实操**：跟着 Build Anything 完成一个最小 DApp（留言板即可），不要等"学完理论"再开始。
    
-   **搭建个人知识库**：用 Obsidian/Notion 记录每天学习心得，结合 AI 插件实现标签化管理和检索。
    
-   **建立固定信息源**：每天花 15-30 分钟浏览深潮、星球日报（中文）或 The Block（英文），避免碎片化刷信息。

-   **提前准备方向选择**：技术向（前端/合约/全栈）、运营向（社区/内容/增长）、研究向（量化/预测/数据分析），周六前确定意向。
    
-   **合约部署前做三步人工检查**：① 首行是否有 SPDX 开源声明；② Solidity 版本号是否正确；③ 代码逻辑是否合理。不要盲信 AI 生成的代码。

-   **Obsidian 多端同步方案**：内网环境用 fast note sync（有服务器版）；外网用 remotely-save 插件 + OneDrive（5GB 免费够存文稿）。
    
-   **AI 开发时降低 Token 消耗**：每个新模块用空文件重新生成，不要在旧文件上反复修——旧文件修改的成本可达重建的十倍。
    
-   **给 AI 加约束语句**：在 Prompt 末尾加"每一个重要的细节都要向我确认"，AI 就不会跳过关键决策点。

点陌生链接

连接钱包

授权恶意合约

钱包资产被自动转走

U 卡 / 礼品卡被盗

所以新人必须注意：

陌生链接不要点，钱包不要乱连，授权一定要看清楚。

8\. 黑客松项目案例：喵星智能体

他们之前做过一个 AI + Web3 + 宠物硬件 项目，叫“喵星智能体”。

核心想法：

通过摄像头、声音识别、AI 分析猫咪行为和叫声，判断猫咪需求，比如饿了、想吃东西、状态异常等。

项目流程大概是：

采集猫咪声音 / 行为数据

通过摄像头、麦克风等设备收集信息。

AI 分析猫咪意图

把猫叫声转成声纹向量，再结合 RAG 和模型判断猫咪可能想表达什么。

多轮确认需求

如果模型不确定，会继续通过互动判断猫咪需求。

主人审批

当系统判断猫咪想买东西，比如猫粮或零食，会推送给主人确认。

链上支付 / 存证

通过预言机和智能合约，把审批、支付、履约流程连接起来。

9\. 这个项目为什么要用区块链

支付流程

链上存证

订单记录

权限审批

透明履约

多方共享可信数据

Web3 项目不能为了上链而上链，必须说明区块链解决了什么现实问题。

一些社群参与经历

###一、核心观点

\- **以太坊协议层 ≠ 智能合约/DApp 开发**：协议层关注的是以太坊底层机制——EIP 提案研究、客户端实现、共识层/执行层规范、扩容（如 Gas 降低、零知识证明）等，而非上层应用。

\- **以太坊生态的核心竞争力是开发者社区**：以太坊是全球开发者数量最多的公链，基金会通过 EPF 等项目吸引新人参与，体现去中心化精神。

\- **EPF 面试不看算法八股，看重热情和研究潜力**：不会考手撕算法或密码学难题，更多聊你对以太坊的了解、对哪些 EIP 感兴趣、是否有持续贡献的意愿。

\- **以太坊技术更新极度保守谨慎**：一个 EIP 提案从提出到上线客户端可能需 1-2 年、多轮 review，作为承载全球大量资金的系统不可能激进跟进新技术（如 AI agent）。

\- **协议层不适合功利求职者**：大部分人是以副业形式参与，经济回报不确定；如果目标是稳定高薪，Web2 路线更确定。

\- **英语是 Web3 的"第一编程语言"**：面试全英文、日常讨论技术英语、国际化协作场景多，但不需要雅思口语级别，能清晰表达想法即可。

\- **多客户端、多语言是以太坊去中心化的基石**：不能让单一语言/客户端占据太大份额，防止单点故障影响整个网络。

2️⃣会议总结：Co-Learning 第二次分享

**会议主题**：用 AI Coding 构建 Web3 Mini Demo — 从合约到前端的完整闭环

**分享人**：ETH Coooder / 41

**日期**：2026-07-07 20:05-21:18

\### 一、核心观点

\- **AI Coding 是 Web3 开发的新范式**：用 Codex/Cursor 等工具，无需 Remix 手动操作，即可完成合约编写、编译、部署到测试网的全流程，开发效率大幅提升，甚至非技术人员也可参与。

\- **Remix 仍有学习价值**：初学者应先通过 Remix 走一遍完整流程（编写→编译→部署→获取 ABI），理解各环节后再转向 AI Coding，否则“完成了任务但不知道做了什么”。

\- **私钥安全是 Web3 生存底线**：私钥/助记词代表钱包所有权，泄露即资产归零`.env` 文件绝不能上传 GitHub，中转站 API 有偷私钥风险，开发钱包和真实钱包必须严格隔离。

\- **智能合约 + 前端 = Web3 Demo 的最小闭环**：ABI/JSON 文件是合约与前端之间的接口契约，前端通过它调用链上合约。一个 Mini Demo 0 只需一个合约 + 一个前端即可。

\- **区块链应被理解为后端的补充而非替代**：受 Gas 费和延迟制约，不能将所有传统后端逻辑上链；实际系统通常是传统后端与智能合约互为补充。

\- **Co-Learning 定位是互助交流，不是单向授课**：鼓励自主学习后用 AI 探索，遇到问题来共学时间或群里提问。
