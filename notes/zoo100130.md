\### Monad 非同步執行 (Asynchronous Execution) 重點筆記

\#### 1. 核心概念與摘要

\- **定義**：將\*\*共識\*\*（決定交易順序）與\*\*執行\*\*（計算交易結果和狀態）脫鉤，讓兩者可以獨立、並行地進行。

\- **主要目標**：大幅提高執行吞吐量（Throughput），解決傳統區塊鏈執行時間不足的瓶頸。

\- **關鍵結果**：執行工作從原先僅佔用區塊時間的一小部分，擴展到可佔用\*\*完整的區塊時間\*\*。

\#### 2. 背景：傳統「交錯執行」的效率問題 (Interleaved Execution)

\- **傳統模式**（如以太坊）：

1\. 共識前\*\*必須先執行\*\*交易。

2\. 區塊提議者需先執行所有交易，計算出新的狀態根（State Root）後，才能提議區塊。

3\. 驗證節點也必須執行交易，才能對區塊進行投票。

\- **缺點**：

\- **時間預算極度壓縮**：執行必須發生在共識的多輪通訊之間，時間預算非常有限。

\- **Gas 限制保守**：為了確保所有節點在時間內完成，Gas 限制必須非常保守。

\- **效率低落**：在以太坊中，30M 的 Gas 約對應 100 毫秒的執行時間，但區塊時間為 12 秒，執行時間僅佔 **1%**。

\- **總結**：交錯執行導致了巨大的「時間縮減效應」，嚴重限制了區塊鏈的運算能力。

\#### 3. Monad 的解決方案：非同步執行 (Asynchronous Execution)

\- **運作方式**：

\- 節點針對\*\*交易的順序\*\*達成共識，\*\*無需先執行交易\*\*。

\- 提議者僅提議一個交易順序列表，而不包含最終的狀態根。

\- 驗證節點僅對交易順序進行投票，無需先執行以驗證狀態。

\- 當區塊的最終順序確定後，所有節點再獨立地執行這些交易，產生最終狀態。

\- **效益**：

\- 共識過程不再被執行所阻塞。

\- 共識和執行可以在\*\*各自的泳道（Swim Lanes）中並行\*\*，執行可以充分利用整個區塊時間。

\- **圖示比較**：

\- **交錯式**：共識（橘色）與執行（藍色）交替佔用區塊時間，執行時間短。

\- **非同步式**：共識和執行同時進行（並行），執行時間可等同於完整的區塊時間，顯著提升執行預算。

\#### 4. 確定性與狀態

\- **確定的狀態**：一旦共識確定了交易的順序，最終的世界狀態（包含成功與失敗的交易）在數學上就已經被確定了。

\- **執行只是「揭曉」真相**：執行過程是為了計算出這個已經被確定的狀態結果。

\- **交易失敗**：即使交易失敗（如帳戶餘額不足），該「失敗結果」本身也是確定的，是狀態轉換的一部分。

\#### 5. 關鍵技術細節

\##### A. 延遲狀態根 (Delayed Merkle Root)

\- **問題**：由於執行落後於共識，區塊提議時尚無法得知當前的狀態根。

\- **解決方案**：

\- 區塊中不包含當前區塊的狀態根。

\- 改為包含 `D` 個區塊前的狀態根（例如，區塊 `N` 的提議包含區塊 `N-D` 的狀態根）。

\- `D` 是一個系統參數（測試網和主網設定為 `3`）。

\- **作用**：

1\. **確保一致性**：作為安全機制，讓節點可以檢查自己是否與網路其他節點執行的結果同步。

2\. **最終確定性**：當區塊 `N` 被最終確認時，代表網路已對區塊 `N-D` 的狀態根達成共識。

3\. **錯誤復原**：如果節點執行 `N-D` 區塊的結果與提出的狀態根不符，該節點會回退到 `N-D-1` 的狀態，並從 `N-D` 開始重新執行，以修正分歧。

\##### B. 保留餘額 (Reserve Balance)

\- **原因**：共識運行時（區塊 `N`），只知道 `D` 個區塊前的狀態，因此無法百分之百確定當前狀態下某帳戶是否有足夠餘額支付 Gas。

\- **規則**：

\- 為了讓共識能安全地建構區塊，引入了「保留餘額」的規則。

\- 對交易能否被包含在區塊中加入了一些輕量級的限制（例如，確保帳戶在 `D` 個區塊前有足夠的餘額）。

\- 這些規則確保即便狀態落後，共識過程依然能安全進行，不符合規則的交易將在執行時被還原（Revert）。

\##### C. 投機性執行 (Speculative Execution)

\- **定義**：節點在區塊被最終確認前（收到提案後，等待最終化的 `N+2` 個區塊時間內），就可以\*\*提前、在本地先執行\*\*該區塊的交易。

\- **優勢**：

1\. **加速最終確認**：一旦區塊被最終確認，節點已經完成了執行，可以立即更新狀態，無需等待。

2\. **更即時的 RPC 查詢**`eth_call` 或 `eth_estimateGas` 等查詢可以基於這個最新的「投機性」狀態進行，提供更接近最終狀態的結果。

\##### D. 對新創建帳戶的限制

\- **情境**：若一個帳戶 `B` 之前餘額為 0，剛收到一筆來自 `A` 的轉帳。

\- **限制**：因為共識領先執行 `D` 個區塊，帳戶 `B` 必須等到那筆轉帳交易被執行且 `D` 個區塊過去後，才能發送交易。

\- **實務操作**：在看到轉帳收據後，仍需等待約 `D 區塊時間`_（例如_ `3 400毫秒 = 1.2秒`）。

\- **替代方案**：可以撰寫一個智能合約函數，讓帳戶 `A` 在一次呼叫中同時完成轉帳給 `B` 並執行 `B` 想做的操作，以繞過此限制。

\#### 6. 區塊狀態 (Block States)

\- 每個區塊在 Monad 中會經歷一系列狀態，從被提議（Proposed）到最終被確認（Finalized）。詳細狀態轉換可參考「\[區塊狀態\](/monad-arch/consensus/block-states)」文件。

## what the first diagram `depicts ?`

  
An Ethereum node is composed of two closely integrated components: a consensus client and an execution client.The consensus client, commonly referred to as a Beacon Node, is responsible for i_mplementing Ethereum’s proof-of-stake consensus mechanism._

As for the execution Engine,not only validates but also executes transactions where embedded EVM.

The execution client not only validates transactions but also executes them through its embedded EVM.

The Communication between the two clients is handled through the Engine API, which defines methods for building, submitting, and validating execution payloads.

The Beacon API serves a different purpose, as it exposes consensus-layer data to validators, developers, and monitoring applications.  
Although the two clients operate together as a single node, they participate in separate peer-to-peer networks. The execution-layer network primarily propagates transactions, whereas the consensus-layer network distributes Beacon blocks, attestations, and other consensus-related messages. This modular architecture allows both clients to perform distinct functions while collectively maintaining a consistent view of the Ethereum blockchain.

# git & github 協作

專業版本控制指南：使用 Git 和 GitHub 進行策略資產管理

1\. 數位版本控制基礎

分支（實驗）：建立一個“安全空間”，用於嘗試新想法或草稿，而不會影響專案的主穩定版本。

同步（跨裝置存取）：確保您的最新版本可以在多台機器上存取和同步。

理解這些概念是配置專業本地環境的必要基礎。

2\. 本機工作流程：從初始化到不可變快照

「本地倉庫」是版本控制的核心。追蹤不會自動在您的整個電腦上進行；它始於一個經過深思熟慮的策略選擇，即監控特定的專案目錄。

初始化倉庫

若要將標準資料夾轉換為受追蹤的倉庫，請在終端機中導航至該資料夾並執行：

git init

此指令會建立一個名為 .git 的隱藏目錄。此資料夾是專案的​​“大腦”，儲存所有歷史記錄和捕獲過的每個版本。由於 Git 是去中心化的，此資料夾包含完整的專案歷史記錄，即使沒有網路連線也能確保資料安全。重要警告：刪除 .git 資料夾會破壞專案的歷史記錄和所有本機備份。必須極其謹慎地操作。

三層架構：攝影工作室的比喻

Git 的運作過程分為三個階段，可以用攝影工作室的比喻來理解。假設您正在使用 Markdown 檔案（例如 [Specs.md](http://Specs.md) 和 Software\_[Compatibility.md](http://Compatibility.md)）編寫 VR 裝置的產品評測報告：

工作目錄：這是您實際編輯檔案的資料夾（例如，將 VR 裝置的重量更新為 500 克）。這就像工作室的「等候室」。

暫存區：當您執行 git add . 時，您會將所有 c 檔案移至暫存區。

等待室中的待處理文件被移至「舞台」。您正在決定哪些變更已準備好記錄。 「點」指令是提高效率的專業標準，可同時新增所有目前修改。

提交歷史：當您執行 `git commit -m "更新 VR 規格和軟體相容性"` 時，相當於按下相機快門。您已擷取了舞台上目前檔案的「不可變更快照」。

策略性提交管理

堅持使用 `-m` 標誌以提供描述性訊息。清晰的訊息可確保長期可維護性，使您能夠在數月後了解 VR 專案的演變。擷取這些快照後，您現在可以開始管理專案的時間軸。

4\. 導航與復原：管理專案時間軸

版本控制的真正力量在於「安全失敗」。專業人員可以大膽嘗試，因為他們知道專案歷史記錄提供了安全保障。

查看時間軸

若要查看歷史記錄，請使用 `git log`。為了獲得更簡潔的視圖，請使用以下版本：

git log --oneline

在此視圖中，您將看到 HEAD 指針，它指示您當前的位置，通常位於主分支的開頭（這是現代行業標準的主要敘述線）。

診斷驗證和恢復

假設您發現定價錯誤：VR 設備的價格意外地從 499 美元更改為 599 美元。在復原之前，請使用 git diff 比較版本：

git diff \[Commit\_ID\] [Specs.md](http://Specs.md)

這使您可以在提交恢復之前驗證確切的價格變更。當需要恢復時，請根據風險狀況選擇策略：

git checkout \[Commit\_ID\] \[Filename\]：安全、精準的復原。它會將檔案還原到過去的 499 美元版本，而不會更改整個專案的時間軸。

git reset --hard \[Commit\_ID\]：破壞性復原。這會將整個專案回滾到特定時間點，並丟棄所有後續變更。

「那又怎樣？」：專業人士優先處理小的修改，以保持完整的審計追蹤。在共享環境中，通常避免使用 `reset --hard` 指令，因為它會重寫歷史記錄，這可能會擾亂其他團隊成員的工作流程。

5\. 資產衛生與安全`.gitignore` 協議

專業的代碼倉庫應該只包含必要的資產。包含臨時系統檔案、日誌檔案或私有憑證會破壞「專案衛生」。為了管理這一點，請在專案根目錄中建立一個名為 `.gitignore` 的檔案：

`touch .gitignore`

在該檔案中，列出 Git 應該忽略的名稱或副檔名（例如`*.log` 或 `private_config.json`）。

戰略影響`.gitignore` 協議是一項重要的安全措施。它可以防止敏感資料意外上傳到雲端平台，確保您的程式碼倉庫在與雲端同步之前保持專業和安全性。

6\. 雲端整合：GitHub 用於同步和冗餘

從本地管理過渡到「雲端優先」工作流程，將 GitHub 確立為「單一資料來源」。這確保了資料的冗餘性和可存取性。

連結到雲端

在 GitHub Web 介面上建立倉庫後，請依照下列步驟連結您的本機 VR 專案：

連結：git remote add origin \[URL\]

重新命名：git branch -M main（確保使用現代標準命名）。

上傳：git push -u origin main

推播/拉取動態

連結完成後，工作流程主要依賴兩個操作：

git push：將本機更新（例如描述 VR 專案的新 [README.md](http://README.md) 檔案）傳送到雲端。

git pull：從雲端拉取更新到本機。

這有助於實現無縫的跨裝置管理。您可以從辦公室的桌上型電腦推送更改，然後將其拉取到家中的筆記型電腦上，確保您的專案始終保持同步。

7\. 協作架構：分支和拉取請求工作流程

除了備份之外，Git 和 GitHub 還支援非線性開發，允許專案的多個版本或多個貢獻者共存而不相互幹擾。

特性分支工作流程

協作始於在 GitHub 上邀請「協作者」。團隊成員可以透過執行以下命令加入：

git clone \[URL\]

cd \[倉庫名稱\]

專業人士不會讓所有人編輯主分支，而是使用分支來隔離工作。例如，要建立一個新的設定指南，可以執行以下命令：

git checkout -b feature-vr-setup

這將創建一個並行的時間軸。貢獻者可以進行更改（例如添加 [Setup.md](http://Setup.md)），並將該特定分支推送到 GitHub，而不會影響主分支的穩定性。

拉取請求 (PR) 和審查週期

拉取請求是專業協作的基石。它是將特性分支合併到主分支的正式請求。 PR 充當審核工具，用於：

透過 diff 比較更改。

團隊成員會留下評論或「請求變更」以進行改進（例如，「將 Mac 支援更新至 Ventura 13.2」）。

討論在工作範圍內進行。

閉環

審核通過後，PR 將被提交。

合併到 GitHub 主分支後，冗餘的特性分支將被刪除。所有團隊成員隨後執行本機 git pull 操作，以將各自機器上的程式碼同步到最終狀態。採用這些實踐可打造高標準、專業的數位化工作流程，確保專案的完整性、安全性以及卓越的協作水準。
