# Week2 Day3 打卡笔记：路径深入｜吃透 Dev 方向完整 AI Coding 与开源协作 Workflow

## 今日学习目标复盘

本人主攻 Dev 开发方向，今日完成对应学习目标：

1.  系统学习标准化 AI Coding 完整工作流，理清 AI 辅助编码从需求拆解、Prompt 编写、代码生成、人工校验、漏洞修复到合约部署的全流程；
    
2.  吃透开源项目完整协作流程，掌握开源仓库分支管理、PR 提交、代码评审、Issue 沟通等协作规范；
    
3.  同步了解 Research、Ops 赛道标准化工作流，建立三类 Builder 完整工作流程认知，便于后续团队协同。
    

## 一、三大赛道标准化 Workflow 学习收获

### 1\. Research Builder 研究工作流

完整研究流程分为两大并行模块：研读 EIP/MIP/ERC 底层标准 + 拆解头部真实协议产品（Uniswap、Aave、Lido 等 DeFi 头部项目）。

拆解产品时需完整覆盖八大维度：产品解决的核心痛点、底层运行机制、参与用户角色、激励与收益模型、潜在安全风险、竞品对比、可借鉴的产品设计思路；同时锻炼通俗化输出能力，做到能把复杂底层逻辑清晰讲解给行业新人，拒绝单纯摘抄专业概念。

### 2\. Ops Builder 线上活动标准化工作流

核心是把模糊的活动想法落地为可直接执行的 Space 完整执行方案，全套流程包含 6 项标准化产出：

1.  主题定位：拟定清晰标题，明确受众参与价值；
    
2.  活动简介：100-200 字说明背景、讨论主题、适配人群；
    
3.  人员分工：划分开场、控场、嘉宾对接、记录复盘负责人；
    
4.  梯度问题线：设计 3-5 个由浅入深的讨论问题，避免开篇过于晦涩；
    
5.  完整流程脚本（Run of Show）：开场→嘉宾介绍→主题讨论→听众互动→行动引导→收尾；
    
6.  多版本宣传物料：至少产出正式官宣、社群口语两条差异化宣传文案。
    

### 3\. Dev Builder 开发工作流（主攻方向）

今日核心学习两大模块：AI Coding 标准化工作流、开源项目协作规范。

（1）AI Coding 完整 Workflow

1.  需求拆解：明确 MVP 功能、技术栈、开发边界，整理清晰需求清单；
    
2.  精准 Prompt 撰写：区分合约框架、逻辑实现、安全检查、注释文档四类提示词；
    
3.  AI 产出初稿生成：让 AI 输出基础合约、工具脚本代码；
    
4.  人工全量校验：逐行核对语法、业务逻辑、重入 / 权限等安全漏洞；
    
5.  迭代优化：结合 Remix 编译报错、测试网交互问题，反向调整 Prompt 二次生成代码；
    
6.  测试部署：部署至 Monad 测试网，完成读写函数交互，留存交易哈希；
    
7.  成果归档：将 Prompt、代码、报错日志、部署记录同步录入 Role Log。
    

（2）开源项目协作方式

1.  仓库基础操作：Fork 开源仓库、新建独立开发分支，不在 main 主分支直接修改；
    
2.  迭代开发：本地完成代码编写、自测，提交清晰规范的 Commit 注释；
    
3.  沟通协作：通过 Issue 提需求、反馈 bug，和维护者对齐开发思路；
    
4.  提交 PR：整理修改说明、功能演示、测试结果，等待维护者代码评审；
    
5.  版本迭代：根据评审意见修改代码，更新 PR 直至合并入库。

## 文档研读知识收获

1.  研读 Web3 实习手册智能合约开发板块：掌握智能合约基础开发流程、新手高频易错点，了解合约安全审计的基础注意事项，规避基础代码安全漏洞；

4.  参照 Monad 专属 Remix 部署教程：适配 Monad 测试网的部署配置，排查网络、钱包权限类部署报错；借助新手教程了解 DApp 基础架构，明白合约是去中心化应用的链上核心模块。
    

## 实操流程记录

1.  使用 AI 产出极简 Solidity 合约（简易数据存读类合约），逐行校验语法、业务逻辑，删掉存在安全隐患、不符合规范的代码；
    
2.  将定稿代码导入 Remix IDE，选定匹配的编译器版本，完成编译，借助工具做静态漏洞扫描，修复提示的告警问题；
    
3.  钱包切换至 Monad 测试网，配置好环境参数，在 Remix 完成合约部署，拿到专属合约地址与 ABI 文件；
    
4.  分别调用合约里两类函数：执行 read 只读方法，直接读取链上存储的数据，不产生链上交易；调用 write 写入方法提交数据，钱包支付测试 Gas 费完成上链，拿到对应的 transaction hash，跳转区块浏览器核验链上执行结果。
    

## 经验总结与后续规划

### 实操感悟 & 避坑心得

1.  AI 只能充当代码辅助工具，生成内容常会出现版本适配错误、隐性安全漏洞，人工审核校验是必不可少的步骤，不能直接部署未经核查的 AI 代码；
    
2.  ABI 是前端、钱包解析合约接口的凭证，合约地址用来定位链上部署的合约实例，二者缺一不可，要妥善留存；只读接口适合查询数据，写接口会永久改动链上账本，上线前务必反复做测试校验；
    
3.  OpenZeppelin 成熟合约模板能大幅降低重入攻击、权限失控这类经典安全问题的出现概率，后续开发优先复用标准化组件。

1.  微调本次最简合约，新增简易业务逻辑，重复做编译、部署、接口调用实操，强化工具操作熟练度；
    
2.  将本次合约源码、部署交易记录归档进个人 work proof 作品集；
    
3.  补充学习合约基础安全知识，试着排查简单合约漏洞，为后续开发复杂度更高的链上项目铺垫基础。
