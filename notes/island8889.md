# 2026-07-09
<!-- DAILY_CHECKIN_2026-07-09_START -->
以前会觉得 AI Agent 只是“更自动化的 AI”，但真正开始让它调用工具、操作钱包、访问数据库之后，安全问题就完全不一样了。普通聊天机器人答错一句话，影响可能不大；Agent 一旦判断错了，可能会直接转账、删除文件、泄露密钥，甚至连续执行错误操作。

Agent 失控很多时候不是因为 AI 突然变坏，而是权限给得太大、规则写得不清楚，或者它被外部内容误导。比较常见的攻击方式包括提示词注入、恶意网页诱导、工具调用劫持、API Key 泄露，以及让 Agent 把错误信息当成真实指令继续执行。

在 Web3 场景里风险会更高，因为钱包交易一旦确认通常很难撤回。比如 Agent 被诱导连接恶意合约、授权过高额度，或者把助记词和私钥暴露给第三方工具，损失可能是直接且不可逆的。

我觉得更安全的做法，不是完全限制 Agent，而是给它设置清晰边界。比如使用独立钱包、小额预算、白名单地址、单笔限额，高风险操作必须人工确认。API、钱包和数据库权限也要拆开，不要让一个 Agent 同时拥有所有能力。

另外，所有工具调用最好都有日志，方便知道它做过什么、为什么这样做。上线前也要专门测试提示词注入、异常输入和恶意网页，而不是只测试正常流程。

# 2026-07-08
<!-- DAILY_CHECKIN_2026-07-08_START -->

以前用 Codex、Claude Code 这类工具时，我更多把它们当成能写代码、改文件、调用工具的 AI 助手。但现在 Agent 已经开始自己拆任务、调用 API、部署服务，下一步很自然就是自己购买资源和完成支付。

比如 Codex 在执行任务时，可能需要调用付费模型、购买数据、使用云服务器，或者向另一个 Agent 购买服务。现在这些费用大多还是人提前充值和确认，Agent 本身并没有真正的经济能力。

其实 Crypto 里已经出现了一些比较实际的尝试。Coinbase 的 **Agentic Wallet** 可以接入 Codex、Claude 和 Gemini，让 Agent 通过 x402 使用 USDC 自动购买 API 或服务；x402 更像一套机器支付协议，适合按次、小额、自动结算。⁠

Agent 平台方面，目前比较有代表性的包括：

**Virtuals Protocol（VIRTUAL）**：更偏 Agent 创建、发行和商业化，让 Agent 可以拥有身份、资产、工作和收入。⁠（base生态）

[**Fetch.ai**](http://Fetch.ai) **/ ASI（FET）**：提供 uAgents 和 Agentverse，开发者可以创建、部署和连接不同 Agent。⁠

**Olas（OLAS）**：更偏去中心化 Agent 服务和协作市场，Agent 可以提供服务、获取收入，也可以调用其他 Agent。

从目前的产品完整度和市场认知来看，**VIRTUAL、FET 和 OLAS** 可以算 Agent 概念里比较有代表性的项目，但它们方向并不完全一样。VIRTUAL 更偏 Agent 资产化和生态，FET 更偏开发框架和网络，OLAS 更偏自主 Agent 服务市场。

我最大的感受是，AI Agent 的下一步不只是变得更聪明，而是开始拥有钱包、预算和交易能力。未来 Codex 或 Claude Code 可能不只帮人完成任务，还能自己寻找工具、比较价格、购买服务并交付结果。

不过有 Agent 概念不等于代币一定有价值。最后还是要看真实 Agent 数量、实际支付规模、开发者使用量，以及代币能不能真正捕获这些业务价值，而不是只靠市场叙事。

# 2026-07-07
<!-- DAILY_CHECKIN_2026-07-07_START -->


以前我一直觉得，以太坊协议层离普通开发者很远，好像只有很强的底层工程师或者研究员才能参与。通过今天的自学后，感觉它确实有门槛，但没有想象中那么遥不可及。

比较现实的路径，是先把以太坊的基本结构弄懂，比如执行层、共识层、客户端、EIP 分别在做什么。刚开始不用什么都学，可以先选一个自己更感兴趣的方向。会 Go、Rust 这类语言，可以尝试客户端开发；对机制和规则感兴趣，可以看共识和 EIP；有测试或安全经验，也可以从测试、找 Bug、补文档开始。

我觉得最重要的一点是，不要一直停留在“学习资料”的阶段。真正想进入协议层，还是要尽早开始做公开输出，比如运行客户端、整理一篇技术笔记、复现一个问题，或者完成一个小的 GitHub Issue。贡献不一定要很大，但要让别人看到你在持续学习和参与。

EPF 更像是一个加速入口，不是零基础培训班。申请前最好已经有一定的学习记录、代码实践或者社区参与经历。

今天最大的感受是，协议层开发不是等全部学会以后才开始，而是先找到一个小切口，一边做，一边补知识。很多核心贡献者也不是一开始就很厉害，而是在持续参与中慢慢成长起来的。
