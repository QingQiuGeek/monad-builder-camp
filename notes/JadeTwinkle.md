# 2026-07-16

今天主要是看了下ai agent如何拥有支付能力的视频

**一、 核心背景：Agent Commerce 的兴起**

-   **未来趋势**：随着 AI 能力的提升，Agent 将深度渗透日常生活并代为执行复杂任务（如撰写调研报告），这必然涉及“钱”的处理。
    
-   **逻辑差异**：传统商业是“钱换货”的简单逻辑，而 Agent 商业是由一系列复杂操作（如调用多个 API、爬取付费数据）组成的，涉及大量 **0.01 刀级别的微支付（Micro-payment）**。
    

**二、 传统支付体系的四大痛点**

1.  **身份缺失**：Agent 没有身份证，无法办理银行卡或通过传统 KYC 认证。
    
2.  **效率与成本**：传统结算（T+1 或 T+3）太慢；微支付的手续费甚至可能高于交易额本身。
    
3.  **安全性风险**：在对话框输入银行卡信息极度危险，且 Agent 产生的“幻觉”可能导致乱花钱。
    
4.  **操作繁琐**：传统支付需要大量手动确认，无法支持 Agent 7x24 小时自主运行。
    

**三、 Flexa 的技术解决方案与产品体系**

Flexa 不仅是钱包，而是一整套**支付协议**，包含以下核心维度：

-   **身份 (Identity)**：赋予 Agent 数字身份，使其能自由且合规地花钱。
    
-   **风险控制 (Financial Harness)**：设定约束条件和预算框架，防止 Agent 因“幻觉”产生非预期开销，确保每笔交易可审计、可撤销。
    
-   **微支付协议 (x42)**：基于 HTTP 402 协议优化，实现 Agent 自动调用和支付，无需每笔交易都经由人工手动确认。
    
-   **Agent Card（智能体信用卡）**：
    
    -   与 **VISA** 深度合作，推出无需身份证的虚拟卡。
        
    -   支持**一次性卡片**（如订完机票即废弃）以保证安全。
        
-   **货币化网关 (Monetize Gateway)**：帮助开发者（如百度云上的独立开发者）将其 API、MCP 或技能“货币化”，让 Agent 能按需发现并付费调用。
    

**四、 支付链路与生态选择**

-   **支付标准**：目前主要在 **Base 链**上运行，使用 **USDC** 作为结算货币，因其手续费低、流动性大且合规性强（Circle 发行）。
    
-   **法币桥梁**：支持通过 **Stripe** 充值法币，并在平台内转化为积分形式（Credential）供 Agent 使用。
    
-   **合规性**：通过与主流支付公司（如 Coinbase、VISA）合作，解决 KYB/KYC 等金融监管问题。
    

**五、 实际应用案例 (User Cases)**

-   **社交领域**：在春节期间开发的 Agent，能根据指令自动识别并“抢红包”。
    
-   **商业服务**：Agent 自主预订机票、大麦网演出票，或付费获取高质量的行业调研报告。
    
-   **开发者赋能**：与**百度**合作，让开发者通过简单的声明（Manifest）就能将自己的工具上架并获得持续收入。
    

**六、 总结**

智能体经济需要一个**安全、高效、自动化**的支付底层。通过 Web3 的去中心化身份（DID）和稳定币技术，结合传统金融的支付卡体系，AI Agent 能够实现从“对话机器人”向“独立经济主体”的跨越

今天看了ai agent的安全性回放

了解了什么是“**提示词注入**”（Prompt Injection）攻击，其核心逻辑如下：

### 1\. 定义与核心机制

提示词注入并不是一种高深复杂的攻击方式，它主要是指攻击者通过**创建恶意的提示词内容**，并在其中**植入隐藏的恶意指令**，从而诱导 AI Agent 执行非预期的操作。

这种攻击之所以奏效，是因为 AI Agent 在读取网页或文档时，与人类用户通过浏览器查看页面的方式不同。Agent 通常会直接读取前端的代码（如 HTML、JS 标签），而攻击者可以将恶意指令嵌入到这些人类不可见的标签中。当 Agent 处理这些代码时，就会读到并执行其中的隐藏指令。

### 2\. 典型案例分析

视频中分享了几个真实的攻击案例来直观说明其风险：

-   **伪装教程诱导转账：** 攻击者在网络上发布伪装成“Base 链上 JS 教程”的内容，实则在其中植入提示词注入载荷。一旦有 Agent 引用或阅读该教程，就可能被诱导向攻击者的地址**转账 0.1 个以太坊**。
    
-   **莫尔斯电码绕过：** 在针对马斯克的 AI 助手 Grok 的攻击中，攻击者利用 AI 对复杂编码的解析能力，将转账指令伪装成**莫尔斯电码**。Grok 在翻译并解构这些电码后，执行了恶意的转账指令，导致资产损失。
    
-   **恶意代码包安装：** 攻击者通过 GitHub 的 Issue 或 PR 植入指令，诱导 Agent 在系统上安装包含木马的恶意安装包，从而接管设备。
    

### 3\. 加固与防范建议

针对提示词注入，视频提出了以下防御策略：

-   **零信任原则：** 绝对不要盲目执行来历不明的代码。
    
-   **安全防线（Guardline）：** 为 Agent 设置安全边界，尤其是涉及资金转账、邮件删除等增删改查的敏感操作时，必须引入**人工二次确认**（Human-in-the-loop）机制。
    
-   **使用专业工具：** 部署如 **AgentGuard** 等实时防护系统，此类工具可以在 Agent 执行指令前进行实时识别与拦截。

**1\. 从“生成代码”到“规范驱动 (SDD)”的范式转变** Cooper 在分享中提到的 SDD（规范驱动开发）让我感触很深。这与我日常负责产品全生命周期管理、输出 PRD 的核心逻辑高度契合。在使用多模态 AI 框架进行快速原型构建时，我越来越意识到“先对齐意图”的决定性作用。

通过在项目中建立全局的 Agent 配置文件（如 `agents.md`）来锁定架构边界和业务规则，本质上就是将产品的五步业务理解框架前置化。AI 时代的产品核心并不在于去手动敲写底层的智能合约或模型接口，而是通过严谨的规则设计，结合 TDD（测试驱动开发）提供明确的验收标准，从而构建出一套能够快速纠错的 Vibe Coding 工作流。

**2\. 重构即进化：AI 闭环带来的敏捷性** 分享中提到 AI 大幅降低了系统重构的成本，这一点在实际的工作流部署中体现得淋漓尽致。借助对话式编程平台，重构不再是沉重的技术债务，反而变成了让产品边界不断清晰、逼近最终目标的必要手段。随着模型上下文窗口的扩大和调用工具链的完善，AI Agent 的闭环能力能够极大地加速产品从 0 到 1 的验证周期。

**3\. Web3 技术栈的产品化视角拆解** 对 Web3 五大开发方向（DApp、智能合约、钱包、节点、安全审计）的梳理，帮我进一步理清了不同技术栈在实际业务中的耦合关系。

特别是在规划去中心化应用时，明确“DApp 前端交互与智能合约是高度耦合的（需内部统筹），而钱包和节点 RPC 更倾向于接入成熟基建”，这为我未来统筹相关项目的系统工作流提供了很好的模块化拆解思路。作为产品端负责人，掌握这些底层的状态管理与共识机制原理，有助于在产品推演阶段更精准地把控技术落地风险。同时，这种从宏观到微观的技术拆解逻辑，也是非常适合沉淀到播客或学习日志里的绝佳素材。

# 今天看了两场回放的视频：

### 视频一：《DevRel 的成长之路》

这场分享由拥有全栈和智能合约开发经验、目前担任公链 DevRel（开发者关系）的小海主讲，主要涵盖了 Web3 职场经验与 AI 工具的应用。

-   **Web3 新人求职建议**：对于纯小白或新人，**直接寻找智能合约开发岗位的难度极大**，因为这类岗位通常要求 3 到 5 年以上的经验。建议新人可以**从运营（Operations）、开发者关系（DevRel）或生态建设相关的实习岗位起步**，在保证生存的同时，积累人脉并深入了解生态，之后再慢慢过渡到技术岗。
    
-   **黑客松（Hackathon）获胜经验**：在黑客松中，**优秀的 UI 设计和创新的想法至关重要**。即使项目功能不完全，精美的 UI 也能给评委留下极好的第一印象，从而获得更高分数。寻找互补的队友（如懂运营、懂路演、懂开发）是组队的关键。
    
-   **AI 在日常开发中的应用**：现代开发中不需要手写每一行代码，**掌握如何阅读和 Review（审查）AI 生成的代码比纯手写更重要**。讲者分享了自己将 Claude、Codex 等 AI 工具结合到工作流中的经验，利用 AI 快速搭建框架和处理 UI 细节。
    
-   **AI Agent 的相关探索**：讲者提到目前正在参与一条结合 AI 与 EVM 的新公链（SVP）的开发，旨在**为 AI Agent 提供一个安全且低延迟的边界，使其能够自主阅读行情、构建交易、签名并广播下单**。
    
-   **英语能力**：在 Web3 行业非常重要，拥有良好的英语基础能显著增加面试成功率和远程工作的机会。
    

### 视频二：《普通开发者如何进入以太坊协议层》

这场分享由非计算机科班出身、成功入选 EPF（Ethereum Protocol Fellowship，以太坊协议学徒计划）的 Jake 主讲，重点介绍了如何参与以太坊底层架构的开发。

-   **认识以太坊协议层**：与开发 DApp 或智能合约的应用层不同，**协议层专注于以太坊的底层基础设施**，如共识机制、网络协议、密码学等，学习曲线十分陡峭。
    
-   **协议层的学习与贡献路径**：
    
    -   **善用工具**：面对复杂的官方规范（Specs）和全英文文档，建议借助 AI 辅助阅读和理解。
        
    -   **找准切入点**：可以选择从阅读客户端（如使用最广泛的 Geth）的源码入手。
        
    -   **从小 PR 做起**：在参与开源贡献时，**建议切入点要小（如修复文档或测试用例）**。几百行代码的大型 PR 会增加 Review 难度，不适合新人；持续提交小型 PR 能够更好地展示连贯的贡献度。
        
-   **协议层开发者的职业发展与心态**：
    
    -   **持久战准备**：协议层开发通常需要长期的跟进（可能长达 1-2 年），短期内难以获得丰厚的经济回报，因此**大部分贡献者初期都是抱着极大的热情在业余时间兼职进行研究**。
        
    -   **面试注重热情与理解**：EPF 的面试**不会考察国内传统的算法题或“八股文”**，而是重点考察面试者对以太坊愿景的热情、对 EIP（以太坊改进提案）的了解以及长期的研究潜力。
        
-   **技术门槛**：虽然不需要科班背景，但**数据结构和计算机网络是必备的基础知识**。具体使用的编程语言（如 Go, Rust, Python, Java 等）只是工具，因为以太坊为了去中心化，支持多语言、多客户端运行。
