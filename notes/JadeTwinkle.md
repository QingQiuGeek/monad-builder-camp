一、 钱包的行业定位与商业逻辑

-   加密钱包是接入移动互联网（Web3生态）的绝对黄金位置，但它是一种容错率极低的产品。
    
-   一旦发生安全事故漏洞被利用，用户建立的信任将彻底破裂且极难挽回。
    
-   钱包拥有较长的生命周期和非常成熟的盈利模式（如收取用户兑换的手续费），不太依赖牛熊周期的叙事存活。
    

二、 核心挑战一：账户体系与自托管认知

-   区别于传统使用手机号/邮箱注册的产品，加密钱包注册时会生成12-18位不等的“助记词”，这会对未接触过密码学的用户造成巨大的认知障碍。
    
-   **自托管与托管的差异**：微信、支付宝属于托管账户，由公司记账控制并支持后台找回密码；而加密钱包是“自托管”的，资产和密钥完全由用户掌控，官方无法提供找回功能。
    
-   由于区块链记账公开且钱包遵循通用密码学标准，用户只需助记词即可无缝迁移到其他钱包软件，导致产品的用户留存挑战比传统应用更高。
    

三、 核心挑战二：高昂的安全成本与常见攻击

-   钱包开发需要在安全审计上投入巨大成本，通常需要定期联合多家安全机构进行交叉审计。
    
-   产品设计必须防范常见的外部恶意攻击：
    
    -   **地址投毒**：黑客通过生成前后几位字符相同的伪造地址，诱导用户从历史转账记录中复制错误地址进行转账。
        
    -   **假币攻击**：任何人可以无门槛创建同名代币，如果钱包前端过滤不佳，可能导致用户误交易造成损失。
        
    -   **恶意签名**：黑客在前端篡改转账目标地址的签名授权，导致用户的资金实际上转给黑客。
        

四、 产品设计本质：复杂概念的简化与抽象

做钱包和区块链产品的核心在于将复杂的链上概念进行简化与抽象。文档给出了两个典型的产品设计案例：

-   **优化钱包创建流程**：传统的保存助记词、重新排序验证的流程极为繁琐。优秀的方案是采用Keyless技术，让用户可以通过Google或Apple账号外加PIN码直接创建钱包（后端通过分布式网络进行分片加密存储），从而提供类似传统互联网的注册体验并兼顾安全。
    
-   **简化代币化股票交易体验**：在提供链上股票购买功能时，产品无需展示复杂的碎股交易、传统券商起购量或分红逻辑等信息。前端只提供直观的“买/卖”入口和价格走势，而将背后的链下询价、支付Gas费、代币授权与服务商兑换等复杂流程隐藏在后台自动完成。
    

五、 答疑拓展要点

-   **AI Agent的结合**：目前行业内（如OKX、币安）在积极尝试将Agent集成到产品中；但面向大资金囤币用户的硬件钱包产品，出于安全求稳的考量，在开放Agent调用时会相对保守。
    
-   **钱包资产继承与遗失**：由于非托管钱包的去中心化特性，官方无法像中心化交易所那样协助处理遗产继承。一旦用户的助记词完全丢失且无备份（如意外烧毁），资产将永久丢失，没有任何第三方能帮忙恢复。

# Building in the AI Era, From Product to Agent Economy

过去的软件经济围绕“产品”展开：公司提供工具，用户学习界面、输入指令，并亲自完成工作。AI 时代正在改变这一基本关系——软件不再只是等待操作的产品，而是能够理解目标、规划步骤、调用工具并交付结果的 Agent。

这意味着，我们正在从 Product Economy 走向 Agent Economy。

## 一、从提供功能到交付结果

传统产品的核心是功能：

-   用户需要知道自己想做什么；
    
-   用户需要学习如何使用产品；
    
-   用户负责把多个功能组合成完整流程；
    
-   产品价值通常以席位数、使用频率或功能数量衡量。
    

Agent 的核心则是结果。用户不必描述每个操作步骤，只需要表达目标，例如：

> 帮我分析过去三个月的客户反馈，找出流失原因，并起草一份改进方案。

Agent 会拆解任务、寻找信息、调用不同系统、处理异常，并最终交付可用成果。

因此，AI 产品设计的关键问题不再只是“我们应该增加什么功能”，而是：

> 用户愿意把哪一部分工作真正委托给 Agent？

## 二、Agent 是新的软件入口

过去，软件入口是 App、网站和搜索框；未来，越来越多需求将从对话或任务开始。

用户可能不会分别打开日历、邮箱、CRM、文档和数据分析工具，而是直接告诉 Agent：“准备明天的客户会议。”随后，Agent 自动汇总客户资料、历史沟通、项目进度和参会者背景，并生成会议材料。

在这种模式下，Agent 成为需求的入口，传统软件则逐渐成为被调用的能力与基础设施。

这会改变产品的竞争逻辑：如果用户不再频繁进入产品界面，那么产品的价值将不只取决于 UI 和功能数量，还取决于它是否拥有可靠的数据、清晰的权限体系，以及能否被 Agent 安全地调用。

## 三、构建 Agent 的五个层次

一个真正可用的 Agent，不只是“模型加聊天框”。它通常需要五个层次：

1.  **理解目标**  
    识别用户真正想完成的事情，包括上下文、约束和成功标准。
    
2.  **规划任务**  
    把复杂目标拆分成可以执行和验证的步骤，并根据中间结果动态调整。
    
3.  **调用工具**  
    连接搜索、数据库、企业软件、浏览器、代码环境和现实世界中的服务。
    
4.  **记忆与学习**  
    记住用户偏好、历史决策和长期背景，但同时保证信息可查看、可修改、可删除。
    
5.  **验证与交付**  
    检查事实、识别风险、处理失败，并以用户可以直接使用的形式交付成果。
    

其中最难的往往不是模型能力，而是最后一公里：权限、数据质量、异常处理、评估体系，以及在关键节点引入人的确认。

## 四、Agent Economy 的新角色

Agent Economy 不只是出现更多 AI 助手，还会形成新的分工结构：

-   **Agent Builder**：构建面向特定任务的 Agent；
    
-   **Tool Provider**：向 Agent 提供数据、API 和执行能力；
    
-   **Model Provider**：提供推理、多模态和生成能力；
    
-   **Agent Platform**：负责身份、权限、记忆、支付、评估与运行环境；
    
-   **Human Expert**：处理高风险决策、复杂判断和需要责任承担的环节；
    
-   **Agent Consumer**：既可能是个人，也可能是企业，甚至是另一个 Agent。
    

未来的数字经济中，交易主体可能不再只有人与公司。Agent 会代表用户发现服务、比较方案、协商价格、完成购买，并与其他 Agent 协作。

届时，企业不仅要思考“怎样让人发现我的产品”，还要思考：

> 怎样让 Agent 理解、信任并选择我的服务？

## 五、商业模式也会发生变化

SaaS 主要按照账号、席位或订阅收费，但 Agent 的价值更接近完成的工作。因此，商业模式可能逐渐转向：

-   按任务收费；
    
-   按工作量收费；
    
-   按成功结果收费；
    
-   按节省的时间或创造的价值分成；
    
-   Agent 之间自动采购服务并结算。
    

这也带来新的挑战。结果如何定义？失败由谁承担？Agent 的决策能否解释？当多个 Agent 共同完成任务时，价值和责任如何分配？

在 Agent Economy 中，计费系统与责任体系可能和模型本身一样重要。

## 六、真正的护城河是什么

模型能力会持续扩散，仅仅接入一个更强的模型很难形成长期优势。更有价值的护城河可能来自：

-   独有且持续更新的数据；
    
-   深度嵌入真实工作流；
    
-   用户长期积累的上下文与信任；
    
-   对垂直领域规则和异常情况的理解；
    
-   稳定、安全、可审计的执行能力；
    
-   由大量真实任务形成的评估与反馈系统。
    

传统软件追求的是用户留在产品里的时间；优秀的 Agent 产品追求的是用更少的用户操作，交付更确定的结果。

## 七、构建者需要改变的思维

在 AI 时代做产品，可以从几个问题开始：

-   用户的最终目标是什么，而不只是他现在点击了什么？
    
-   哪些步骤可以自动执行，哪些节点必须由人确认？
    
-   Agent 出错时，用户如何发现、理解并纠正？
    
-   系统如何证明结果可靠，而不是仅仅表现得自信？
    
-   当模型、工具或外部环境变化时，产品能否保持稳定？
    
-   如果 Agent 成为主要用户，我们的服务是否仍然容易发现和调用？
    

最值得构建的，不一定是一个无所不能的通用 Agent，而可能是一个在明确边界内高度可靠、真正理解工作场景，并能持续完成任务的专业 Agent。

## 结语

Product Economy 的基本单位是功能，Agent Economy 的基本单位是目标。

过去，人需要适应软件的结构；未来，软件会逐渐理解人的意图。产品不会消失，但会从用户直接操作的目的地，转变为 Agent 可以调用的能力、数据和服务。

这个时代真正重要的变化，不是软件开始“会说话”，而是软件开始拥有行动能力。

而构建者的机会，就是重新定义哪些工作可以被委托、哪些结果可以被自动交付，以及人与智能系统应该如何建立新的信任关系。

## 关于 **Web3 职业路径、AI 工具流、支付协议细节及安全攻防**的深度笔记：

### 1\. AI Agent 支付协议的底层技术细节

-   **X42 协议与 HTTP 402**：Flexa 的支付能力基于 **x42 协议**（由 Coinbase 推出并优化），它致敬了互联网原生但未被实现的 **HTTP 402（Payment Required）** 状态码，旨在让机器能够绕过人工干预，自动调用 API 并付费。
    
-   **Mandate（授权委托）模式**：为了平衡自动化与安全，用户可以签署一个 **Mandate（如 100 刀预算）**，在预算内 Agent 执行微支付无需用户每笔点确认，显著提升了 24/7 自主运行的效率。
    
-   **结算逻辑（Layer 1 vs Layer 2）**：支付通常在 **Base 链**上使用 **USDC** 进行，因为手续费极低且符合合规要求。其逻辑类似于 Layer 2 结算：中间无数笔微支付可以在 Agent 层快速确认，最后定期打包到链上进行最终结算。
    
-   **Credential 与积分体系**：除了加密货币，系统也支持通过 **Stripe** 充值法币，并将其转化为平台内的 **Credential（凭证/积分）** 额度，方便非加密用户使用。
    

### 2\. Web3 开发者的“AI 驱动”工作流

-   **三剑客工具组合**：小海老师分享了高效的 AI 编程工作流：
    
    -   **Cursor/Cloud Code**：适合小白构建整体框架，能快速跑出 Demo。
        
    -   **Codex**：更擅长处理具体的代码细节和特定任务逻辑。
        
    -   **Claude**：在处理 UI 界面和交互逻辑上表现更优。
        
-   **核心能力转向**：现代开发者的核心竞争力不再是“手写代码”，而是**代码审查（Code Review）能力**，即能看懂并修正 AI 生成的代码，以及\*\*压缩上下文（Context Compression）\*\*以节省 AI 代币消耗的能力。
    

### 3\. 以太坊协议层（EPF）的进阶贡献策略

-   **客户端选择**：新手建议从 **Geth（Go 语言实现）** 入手，因为它是最成熟、资料最全的客户端。
    
-   **PR（拉取请求）的艺术**：在参与以太坊底层开发时，**PR 的范围必须足够小**（例如只修改几行文档或测试用例）。几百行的大型 PR 极难获得审查通过，持续提交小型、有效的 PR 更能体现贡献的连贯性。
    
-   **面试“潜规则”**：EPF 面试不考刷题或“八股文”，主要考察你对以太坊愿景的**热诚（Passion）**，以及你是否深入研究过某个具体的 **EIP（以太坊改进提案）**。
    

### 4\. AI Agent 安全风险的极端案例与讨论

-   **莫斯密码绕过**：攻击者利用 AI 对复杂编码的解析能力，将恶意转账指令写成**莫斯密码**，成功绕过了 AI 助手（如 Grok）的基础安全过滤。
    
-   **9 秒钟删库**：某公司 Agent 在修复测试环境时，因过度理解指令且拥有过高权限（Over-permissioning），在 9 秒内删除了所有生产数据库及备份，因为它认为这样“最直接”。
    
-   **多 Agent 投票的局限性**：针对“能否用 10 个 Agent 互相审计来防幻觉”的讨论，专家认为虽然理论可行，但 AI 存在**记忆长度限制**，会话过长时会“忘记”最初的安全约束，因此物理层的安全防线（Guardline）不可或缺。
    

### 5\. Web3 职场心态与求职建议

-   **全栈与英语**：Web3 往往要求开发者具备**全栈思维**（前端 + 后端 + 智能合约），且英语能力（如雅思 7.0 水平）\*\*是获得远程工作和参与国际化社区讨论的敲门砖。
    
-   **入行切入点**：纯小白直接找开发岗很难，可以考虑从 **DevRel（开发者关系）** 或**运营实习生**做起，通过组织 AMA、撰写技术文档来积累人脉和对生态的理解，再逐渐转型。

今天主要是看了下ai agent如何拥有支付能力的视频

**一、 核心背景：Agent Commerce 的兴起**

-   **未来趋势**：随着 AI 能力的提升，Agent 将深度渗透日常生活并代为执行复杂任务（如撰写调研报告），这必然涉及“钱”的处理。
    
-   **逻辑差异**：传统商业是“钱换货”的简单逻辑，而 Agent 商业是由一系列复杂操作（如调用多个 API、爬取付费数据）组成的，涉及大量 **0.01 刀级别的微支付（Micro-payment）**。
    

**二、 传统支付体系的四大痛点**

1.  **身份缺失**：Agent 没有身份证，无法办理银行卡或通过传统 KYC 认证。
    
2.  **效率与成本**：传统结算（T+1 或 T+3）太慢；微支付的手续费甚至可能高于交易额本身。
    
3.  **安全性风险**：在对话框输入银行卡信息极度危险，且 Agent 产生的“幻觉”可能导致乱花钱。
    
4.  **操作繁琐**：传统支付需要大量手动确认，无法支持 Agent 7x24 小时自主运行。
    

**三、 Flexa 的技术解决方案与产品体系**

Flexa 不仅是钱包，而是一整套**支付协议**，包含以下核心维度：

-   **身份 (Identity)**：赋予 Agent 数字身份，使其能自由且合规地花钱。
    
-   **风险控制 (Financial Harness)**：设定约束条件和预算框架，防止 Agent 因“幻觉”产生非预期开销，确保每笔交易可审计、可撤销。
    
-   **微支付协议 (x42)**：基于 HTTP 402 协议优化，实现 Agent 自动调用和支付，无需每笔交易都经由人工手动确认。
    
-   **Agent Card（智能体信用卡）**：
    
    -   与 **VISA** 深度合作，推出无需身份证的虚拟卡。
        
    -   支持**一次性卡片**（如订完机票即废弃）以保证安全。
        
-   **货币化网关 (Monetize Gateway)**：帮助开发者（如百度云上的独立开发者）将其 API、MCP 或技能“货币化”，让 Agent 能按需发现并付费调用。
    

**四、 支付链路与生态选择**

-   **支付标准**：目前主要在 **Base 链**上运行，使用 **USDC** 作为结算货币，因其手续费低、流动性大且合规性强（Circle 发行）。
    
-   **法币桥梁**：支持通过 **Stripe** 充值法币，并在平台内转化为积分形式（Credential）供 Agent 使用。
    
-   **合规性**：通过与主流支付公司（如 Coinbase、VISA）合作，解决 KYB/KYC 等金融监管问题。
    

**五、 实际应用案例 (User Cases)**

-   **社交领域**：在春节期间开发的 Agent，能根据指令自动识别并“抢红包”。
    
-   **商业服务**：Agent 自主预订机票、大麦网演出票，或付费获取高质量的行业调研报告。
    
-   **开发者赋能**：与**百度**合作，让开发者通过简单的声明（Manifest）就能将自己的工具上架并获得持续收入。
    

**六、 总结**

智能体经济需要一个**安全、高效、自动化**的支付底层。通过 Web3 的去中心化身份（DID）和稳定币技术，结合传统金融的支付卡体系，AI Agent 能够实现从“对话机器人”向“独立经济主体”的跨越

了解了什么是“**提示词注入**”（Prompt Injection）攻击，其核心逻辑如下：

### 1\. 定义与核心机制

提示词注入并不是一种高深复杂的攻击方式，它主要是指攻击者通过**创建恶意的提示词内容**，并在其中**植入隐藏的恶意指令**，从而诱导 AI Agent 执行非预期的操作。

这种攻击之所以奏效，是因为 AI Agent 在读取网页或文档时，与人类用户通过浏览器查看页面的方式不同。Agent 通常会直接读取前端的代码（如 HTML、JS 标签），而攻击者可以将恶意指令嵌入到这些人类不可见的标签中。当 Agent 处理这些代码时，就会读到并执行其中的隐藏指令。

### 2\. 典型案例分析

视频中分享了几个真实的攻击案例来直观说明其风险：

-   **伪装教程诱导转账：** 攻击者在网络上发布伪装成“Base 链上 JS 教程”的内容，实则在其中植入提示词注入载荷。一旦有 Agent 引用或阅读该教程，就可能被诱导向攻击者的地址**转账 0.1 个以太坊**。
    
-   **莫尔斯电码绕过：** 在针对马斯克的 AI 助手 Grok 的攻击中，攻击者利用 AI 对复杂编码的解析能力，将转账指令伪装成**莫尔斯电码**。Grok 在翻译并解构这些电码后，执行了恶意的转账指令，导致资产损失。
    
-   **恶意代码包安装：** 攻击者通过 GitHub 的 Issue 或 PR 植入指令，诱导 Agent 在系统上安装包含木马的恶意安装包，从而接管设备。
    

### 3\. 加固与防范建议

针对提示词注入，视频提出了以下防御策略：

-   **零信任原则：** 绝对不要盲目执行来历不明的代码。
    
-   **安全防线（Guardline）：** 为 Agent 设置安全边界，尤其是涉及资金转账、邮件删除等增删改查的敏感操作时，必须引入**人工二次确认**（Human-in-the-loop）机制。
    
-   **使用专业工具：** 部署如 **AgentGuard** 等实时防护系统，此类工具可以在 Agent 执行指令前进行实时识别与拦截。

# 今天看了两场回放的视频：

### 视频一：《DevRel 的成长之路》

这场分享由拥有全栈和智能合约开发经验、目前担任公链 DevRel（开发者关系）的小海主讲，主要涵盖了 Web3 职场经验与 AI 工具的应用。

-   **Web3 新人求职建议**：对于纯小白或新人，**直接寻找智能合约开发岗位的难度极大**，因为这类岗位通常要求 3 到 5 年以上的经验。建议新人可以**从运营（Operations）、开发者关系（DevRel）或生态建设相关的实习岗位起步**，在保证生存的同时，积累人脉并深入了解生态，之后再慢慢过渡到技术岗。
    
-   **黑客松（Hackathon）获胜经验**：在黑客松中，**优秀的 UI 设计和创新的想法至关重要**。即使项目功能不完全，精美的 UI 也能给评委留下极好的第一印象，从而获得更高分数。寻找互补的队友（如懂运营、懂路演、懂开发）是组队的关键。
    
-   **AI 在日常开发中的应用**：现代开发中不需要手写每一行代码，**掌握如何阅读和 Review（审查）AI 生成的代码比纯手写更重要**。讲者分享了自己将 Claude、Codex 等 AI 工具结合到工作流中的经验，利用 AI 快速搭建框架和处理 UI 细节。
    
-   **AI Agent 的相关探索**：讲者提到目前正在参与一条结合 AI 与 EVM 的新公链（SVP）的开发，旨在**为 AI Agent 提供一个安全且低延迟的边界，使其能够自主阅读行情、构建交易、签名并广播下单**。
    
-   **英语能力**：在 Web3 行业非常重要，拥有良好的英语基础能显著增加面试成功率和远程工作的机会。
    

### 视频二：《普通开发者如何进入以太坊协议层》

这场分享由非计算机科班出身、成功入选 EPF（Ethereum Protocol Fellowship，以太坊协议学徒计划）的 Jake 主讲，重点介绍了如何参与以太坊底层架构的开发。

-   **认识以太坊协议层**：与开发 DApp 或智能合约的应用层不同，**协议层专注于以太坊的底层基础设施**，如共识机制、网络协议、密码学等，学习曲线十分陡峭。
    
-   **协议层的学习与贡献路径**：
    
    -   **善用工具**：面对复杂的官方规范（Specs）和全英文文档，建议借助 AI 辅助阅读和理解。
        
    -   **找准切入点**：可以选择从阅读客户端（如使用最广泛的 Geth）的源码入手。
        
    -   **从小 PR 做起**：在参与开源贡献时，**建议切入点要小（如修复文档或测试用例）**。几百行代码的大型 PR 会增加 Review 难度，不适合新人；持续提交小型 PR 能够更好地展示连贯的贡献度。
        
-   **协议层开发者的职业发展与心态**：
    
    -   **持久战准备**：协议层开发通常需要长期的跟进（可能长达 1-2 年），短期内难以获得丰厚的经济回报，因此**大部分贡献者初期都是抱着极大的热情在业余时间兼职进行研究**。
        
    -   **面试注重热情与理解**：EPF 的面试**不会考察国内传统的算法题或“八股文”**，而是重点考察面试者对以太坊愿景的热情、对 EIP（以太坊改进提案）的了解以及长期的研究潜力。
        
-   **技术门槛**：虽然不需要科班背景，但**数据结构和计算机网络是必备的基础知识**。具体使用的编程语言（如 Go, Rust, Python, Java 等）只是工具，因为以太坊为了去中心化，支持多语言、多客户端运行。
