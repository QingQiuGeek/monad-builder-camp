## 核心收获

### 1\. AI Agent 安全防护框架

**为什么需要专门的防护框架？**

-   AI Agent 不仅"生成内容"，还能"自主执行操作"
    
-   传统网络安全防护（WAF、防火墙）已无法覆盖其安全边界
    
-   高自主性带来全新威胁向量
    

**四大威胁向量**：

1.  **过度自治（Excessive Agency）** — 权限过大，可能执行高危操作
    
2.  **间接提示词注入（Indirect Prompt Injection）** — 外部数据隐藏恶意指令
    
3.  **记忆污染（Memory Poisoning）** — 攻击者污染记忆库影响未来决策
    
4.  **沙箱突破（Insecure Tool Execution）** — 恶意代码控制宿主机
    

**四层防护架构**：

```
输入防护层 → 决策规划层 → 工具执行层 → 输出保护层
```

-   **输入防护层**：身份认证、注入检测、数据洗白
    
-   **决策规划层**：策略对齐、记忆安全、状态监控
    
-   **工具执行层**：沙箱隔离、最小特权、人机协同
    
-   **输出保护层**：数据脱敏、内容合规
    

**关键安全原则**：

-   ✅ 明确的人类控制者
    
-   ✅ 最小特权原则
    
-   ✅ 可观察的行为与规划
    
-   ✅ 人机协同（Human-in-the-Loop）
    

### 2\. 哈伯格税 NFT 机制

**核心概念**：

-   **自我定价**：所有者自己设定资产价格
    
-   **强制出售**：任何人可以按定价购买
    
-   **持续纳税**：按定价的一定比例持续缴税
    

**生活类比**：  
想象你有一块土地，你给它定价 100 万。按照哈伯格税制度：

-   你每年要交 10% 的税（10 万）
    
-   但任何人都可以随时按 100 万买走你的土地
    
-   如果你不想被买走，就要报高价
    
-   但报高价意味着交更多税
    
-   → **最终你会报一个"真实愿意出售"的价格**
    

**博弈机制**：

```
报高价 → 被买断风险低，但税金高
报低价 → 税金低，但容易被买走
→ 平衡点 = 真实市场价值
```

**技术实现**：

-   惰性求值模式（节省 Gas）
    
-   Escrow 托管（防逃税）
    
-   禁用标准转移函数（强制通过 buyout 或 claimForeclosed）
    

### 3\. Solidity 基础实战（实验 1.3-1.5）

**学习来源**：[thinkingsolidity.com](http://thinkingsolidity.com) — Solidity 智能合约基础实战篇

实验 1.3：bool 值

**核心概念**：

-   `bool` 类型只有两个值：`true` 和 `false`
    
-   默认值是 `false`
    
-   用于条件判断和状态控制
    

**关键代码**：

```solidity
bool public isActive = true;
bool public isCompleted; // 默认 false

function toggleStatus() public {
    isActive = !isActive; // 取反操作
}
```

**生活类比**：  
就像开关——要么开（true），要么关（false），没有第三种状态。

**踩坑记录**：

-   bool 不能直接参与数值运算
    
-   需要先转换：`uint(isActive)` → 1 或 0
    

* * *

实验 1.4：整型的运算

**核心概念**：

-   `uint`：无符号整数（0, 1, 2, ...）
    
-   `int`：有符号整数（..., -2, -1, 0, 1, 2, ...）
    
-   不同位数：uint8, uint16, uint32, ..., uint256
    

**关键代码**：

```solidity
uint public a = 10;
uint public b = 3;

function calculate() public view returns (uint) {
    uint sum = a + b;        // 加法：13
    uint diff = a - b;       // 减法：7
    uint product = a * b;    // 乘法：30
    uint quotient = a / b;   // 除法：3（整数除法，截断小数）
    uint remainder = a % b;  // 取余：1
    return sum;
}
```

**重要发现**：

-   整数除法会截断小数部分：`10 / 3 = 3`，不是 3.333
    
-   溢出检查：Solidity ^0.8.0 默认检查溢出，会 revert
    
-   除零错误：`a / 0` 会报错
    

**生活类比**：  
就像分蛋糕——10 块蛋糕分给 3 个人，每人得 3 块，剩 1 块。不能说每人得 3.333 块。

* * *

实验 1.5：底层位运算

**核心概念**：

-   位运算直接操作二进制位
    
-   常用操作：AND（&）、OR（|）、XOR（^）、NOT（~）、左移（<<）、右移（>>）
    

**关键代码**：

```solidity
uint8 a = 5;  // 二进制：0101
uint8 b = 3;  // 二进制：0011

function bitOperations() public pure returns (uint8, uint8, uint8, uint8, uint8, uint8) {
    uint8 andResult = a & b;   // AND：0001 = 1
    uint8 orResult = a | b;    // OR：0111 = 7
    uint8 xorResult = a ^ b;   // XOR：0110 = 6
    uint8 notResult = ~a;      // NOT：11111010 = 250（uint8 取反）
    uint8 leftShift = a << 1;  // 左移：1010 = 10
    uint8 rightShift = a >> 1; // 右移：0010 = 2
    return (andResult, orResult, xorResult, notResult, leftShift, rightShift);
}
```

**为什么需要位运算**：

-   Gas 优化：位运算比算术运算更省 Gas
    
-   权限管理：用位掩码表示多个权限
    
-   状态压缩：用一个变量存储多个布尔值
    

**生活类比**：

-   **AND（&）**：两个条件都满足才通过（如：有身份证 AND 有签证）
    
-   **OR（|）**：满足一个条件就通过（如：有学生证 OR 有老年卡）
    
-   **XOR（^）**：只有一个条件满足才通过（如：只能选一个选项）
    
-   **NOT（~）**：取反（如：不是会员 → 是会员）
    

**实际应用场景**：

```solidity
// 权限管理
uint8 public constant CAN_READ = 1;    // 0001
uint8 public constant CAN_WRITE = 2;   // 0010
uint8 public constant CAN_DELETE = 4;  // 0100

uint8 public userPermission = CAN_READ | CAN_WRITE; // 0011

function canDelete() public view returns (bool) {
    return (userPermission & CAN_DELETE) != 0; // 检查是否有删除权限
}
```

## 今日学到什么

### 1\. Solidity 基础语法

**值类型 vs 引用类型**：值类型（uint、bool、address）直接存值，赋值就是复制；引用类型（string、array、mapping）存的是指针，赋值可能只是共享同一个引用。就像——值类型是「拍照」，你有一张独立的照片；引用类型是「共享相册」，大家都指向同一个地方。

**函数修饰符**：pure 不读不写、view 只读不写、external 只能外部调用、public 内外都能调。记忆技巧：pure 是「闭着眼睛算」，view 是「只看不摸」，external 是「只接外卖单」，public 是「堂食外卖都行」。

**memory vs storage vs calldata**：storage 存在链上（贵，但永久）、memory 是临时的（函数结束就消失）、calldata 是只读的外部参数。判断流程：函数参数用 calldata（只读）或 memory（需要修改）；局部变量用 memory；状态变量自动是 storage。

**合约继承**：用 `is` 继承，用 `override` 覆写，用 `virtual` 允许被覆写。接口（interface）只有函数签名，没有实现，用来调用别人的合约——就像「插座标准」，只要接口对得上，任何合约都能接进来。

### 2\. Agent 支付体系

**核心问题**：AI Agent 已经能规划、推理、执行多步任务了，但一到「付钱」环节就撞墙——没有合法身份、需要自主执行、凭证不安全、需要毫秒响应、需要微支付。传统支付系统（T+1 到 T+3）完全跟不上 Agent 的速度。

**四大产品模块**：

-   AgentCard：一次性虚拟卡，金额锁定，用完自毁（给 Agent 一张「临时通行证」）
    
-   Agent Charge：三种模式接收 Agent 付款（商家的「收款终端」）
    
-   Monetize：零代码网关，让 API/MCP 变现（开发者的「收银台」）
    
-   ODP/AEP2：嵌入式授权，先服务后结算（「先用后付」的信用机制）
    

**Intent-Pay（签意图模式）**：传统支付是「每笔交易都签名」，Intent-Pay 是「签一个意图，Agent 自动执行」。用户签一次——比如「本月最多花 500 刀，单笔不超过 50 刀，不能赌博」——之后 Agent 就在这个范围内自主花钱。就像给 Agent 一张「额度卡」，而不是每次都帮你刷卡。

**AEP2 协议**：把签名授权直接附在 HTTP 请求头里（`X-AEP2-Mandate`），先服务后结算。和 x402 的区别：x402 是「先付钱再服务」，AEP2 是「先授权再服务再结算」——对 Agent 来说，后者更高效，因为不用等链上确认。

# 2026-07-07 — Day 2

## 今日学到什么

### 1\. 智能合约部署的完整流程

今天终于动手了——把一个 Solidity 合约真正部署到了 Monad Testnet 上。整个流程：用 Remix IDE 编译 SimpleStorage.sol → 选择 Injected Provider (MetaMask) → 切换到 Monad Testnet → 点击 Deploy → MetaMask 弹窗确认 → 等待交易上链。

**关键洞察**：部署合约没有想象中那么难，难的是理解每一步在做什么。看到合约地址出现在 Deployed Contracts 那一刻，有种"原来我也可以"的感觉。

### 2\. 合约地址 ≠ 交易地址

部署合约会产生一个交易 hash（66 字符），同时生成一个合约地址（42 字符）。之前一直搞混这两个，今天终于分清了：

-   **交易 hash**：记录"我做了一件事"（部署合约）
    
-   **合约地址**：记录"这个合约在哪里"（链上的位置）
    

### 3\. Write 函数要花钱，Read 函数免费

`set(42)` 需要签名 + Gas，`get()` 直接调用就行。这就像——写入是寄快递（要邮费），读取是查快递单号（免费）。这种设计约束其实是在保护网络资源。

### 4\. MetaMask 是"签名器"，不是"执行器"

MetaMask 只负责帮你签名确认，真正的执行是矿工/验证者做的。签名完了还要等区块确认。这让我理解了区块链的去中心化本质——没有单一的服务器在执行，而是全网共识。

### 5\. RPC 直接查询验证

区块浏览器有时候显示不全（比如合约地址被截断），所以用 Monad Testnet RPC 直接查询了交易详情。这种方式拿到的数据比浏览器更完整，也让我理解了"链上数据是公开可查的"这句话的真正含义。

**技术细节**：

```bash
curl -s -X POST https://testnet-rpc.monad.xyz \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"eth_getTransactionReceipt","params":["0xf00f..."],"id":1}'
```

返回的数据是十六进制的，比如 `0x2a` 就是 42。把十六进制转换成十进制的过程，让我第一次直观感受到"链上数据是裸的"——没有中间层帮你翻译，一切都需要自己解读。

### 6\. 区块链上的每一次写入都是有成本的

写入 `set(42)` 的时候，MetaMask 弹窗让我确认 Gas 费。这个体验让我想到——区块链上的每一次写入都是有成本的，这和互联网时代"免费发布"完全不同。这种设计约束其实是在保护网络资源。

### 7\. 合约地址 + ABI：房子的门牌号和钥匙

今天在 Remix 里又学了一个重要概念：**部署合约之后，会同时得到两样东西**——合约地址和 ABI。

**合约地址**就像房子的**门牌号**（比如 `0x321...79c1F`），**ABI** 就像这栋房子的**钥匙**（Application Binary Interface，应用二进制接口）。

ABI 本质上是一份「接口说明书」，告诉外部程序：这个合约有哪些函数、每个函数需要什么参数、返回什么值。没有 ABI，你虽然知道房子在哪（合约地址），但不知道门在哪里、怎么开。

**但为什么平时感觉不到 ABI 的存在？** 因为 Remix 会自动读取 ABI。当你在 Deployed Contracts 里看到已经部署的合约时，Remix 已经帮你把 ABI 加载好了，所以你可以直接点按钮调用函数。

**什么时候需要手动用 ABI？** 当你想在另一个工具（比如 Etherscan、另一个 IDE、或者前端代码）里调用已部署的合约时，就需要手动提供 ABI。这时候你需要：

1.  在 Remix 编译器面板点 **ABI 按钮**，复制 ABI 内容
    
2.  点 **\+ Add Contract**，粘贴合约地址和 ABI
    
3.  就可以重新和这个合约交互了
    

**关键洞察**：你不需要重新部署合约，只要知道门牌号（合约地址）+ 有钥匙（ABI），就可以随时回到原来的地方。这就像你搬了家，但只要还记得老房子的地址，又拿着钥匙，随时可以回去看看。
