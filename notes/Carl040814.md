# Day 5｜访问控制与事件：谁有权动我的合约？

## 🧠 概念微课

### 修饰器（modifier）= 门口的保安

昨天你的 Counter 合约里，任何人都能调 `increment()`。如果是个投票合约，恶意用户写个脚本一秒钟刷 1000 次。

访问控制就是给函数加"保安"——进去之前先检查身份。

```solidity
modifier onlyOwner {
    require(msg.sender == owner, "Not owner");
    _;   // ← 检查通过后，才执行原函数体
}

function increment() public onlyOwner {
    count += 1;
}
```

`msg.sender` 是 Solidity 的内置变量——谁发了这笔交易，`msg.sender` 就是谁。

* * *

-   理解 `msg.sender` 和 `owner` 的关系
    
-   给 Counter 合约加 `onlyOwner`（AI 辅助）
    
-   理解 `event` 的 `indexed` 参数——前端如何筛选特定事件
    
-   跑测试验证访问控制生效 → 13/13 PASS
    
-   部署新版 Counter 到 Monad 测试网（可选）

## 🛠️ AI Prompt 辅助

> Prompt：「给这个 Counter 合约加上 Ownable 访问控制模式，让 increment 和 decrement 只有合约部署者能调用。同时让 CountChanged 事件的 oldCount 和 newCount 都变成 indexed 参数，方便前端按数值范围过滤。请给出完整合约 + 测试代码。」

**AI 回答摘要**：

1.  **Ownable 模式**：`constructor` 中 `owner = msg.sender`；`onlyOwner` modifier 用 `require(msg.sender == owner)` 做门禁；`transferOwnership` 支持所有权转移。
    
2.  `indexed` **事件参数**：`event CountChanged(uint256 indexed oldCount, uint256 indexed newCount)` —— 参数打入 log 的 topic 区，前端可以按 `oldCount=0` 或 `newCount=1` 过滤事件，而不需要拉取所有历史事件再自己筛。
    
3.  **测试新增技巧**：`vm.prank(addr)` 冒充单次调用；`vm.startPrank()/stopPrank()` 批量冒充；`vm.expectEmit(true, true, false, false)` 验证事件——4 个 bool 对应 4 个参数是否 indexed。
    
4.  **AI 额外优化**：`unchecked` 块跳过溢出检查（Day 3 Claude 审计建议）、溢出 require 保护、delete `getCount()`改用自动生成的 `count()`。

# Day 4｜存储：mapping、array、struct

## 🧠 概念微课

### 三种链上存储结构

昨天你用了 `uint256 public count` ——只有一个数字。现实中的合约要存复杂得多的数据，比如一个通讯录、一个投票系统、一个代币余额表。

Solidity 提供了三种结构来组织数据：

| 类型 | 类比 | 用法 |
| --- | --- | --- |
| mapping | 字典 / HashMap | mapping(address => uint256) public balances; |
| array | 数组 / 列表 | address[] public contacts; |
| struct | 自定义类型 / 对象 | struct Person { string name; uint256 age; } |

## 🛠️ AI Prompt 辅助

> Prompt：「请用 mapping + struct + array 写一个 Solidity 通讯录合约，要求：能添加联系人（姓名 + 钱包地址）、能按地址查询、能列出所有联系人。请解释 mapping 和 array 各自在合约中扮演的角色。」

**AI 回答摘要**：

1.  **mapping 的角色**：快速精确查找（O(1)），类似数据库索引。`_contacts[addr]` 直接返回该地址的联系人。但它不能遍历——你无法"列出所有存过的 key"。
    
2.  **array 的角色**：有序列表，可以遍历。`_addresses[]` 存所有联系人地址，`getAllContacts()` 靠它来循环。但 array 不能快速查找——要找一个特定地址得遍历整个数组。
    
3.  **struct 的角色**：把 name + wallet 打包成一个自定义类型。`Contact` 使得 mapping 和 array 都能存复杂数据而不只是单个值。
    
4.  **AI 额外赠送**：`calldata` 优化（比 `memory` 省 Gas）、分页查询 `getContactsPaginated`（防止大量数据时 Gas 超限）、更新逻辑（同一地址多次添加不重复 push）。

| mapping 默认值 | 查不存在的 key → 返回零值（string=""，uint=0） | 不会报错，需要自己判断是否存在 |

# Day 3｜Solidity 第一课：Counter 合约

## 🧠 概念微课

### Solidity 是什么？

Solidity 是以太坊（以及 Monad）上最主流的智能合约语言。它长得像 JavaScript，但运行在区块链上。一旦部署，代码不可修改，所有人都能看到。

### 智能合约 = 自动售货机

你投币 → 售货机吐可乐。没有人手动操作，规则写死在机器里。

智能合约也一样：你发交易 → 合约执行代码 → 结果上链。规则透明、无法篡改、无需信任中间人。

### 今天会接触的 3 个核心概念

| 概念 | 类比 | 代码 |
| --- | --- | --- |
| 状态变量 | 售货机里的可乐库存 | uint256 public count; |
| 函数 | 售货机上的按钮 | function increment() public { ... } |
| 修饰器 | 按钮上的锁（需要钥匙才能按） | public, view, onlyOwner |

-   `require` **失败时 Gas 不退**：这是链上安全的经济基础——攻击者每次尝试都要付出成本
    
-   **Fuzz 测试**是 Foundry 的杀手特性：`testFuzz_Increment(uint256 times)` 自动跑 256 次随机输入，不需要手写循环
    
-   `view` **函数不消耗 Gas**：从链外读取数据是免费的，只有写操作（increment/decrement）需要付 Gas
    
-   **教训**：Agent 一步做完 = 你什么都没学到。下次严格要求"只给指令，你动手"
