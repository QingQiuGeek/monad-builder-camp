# Moss 协议接入实践教程

“Moss 协议”在不同的技术语境下有多种含义。本教程聚焦于目前社区中最活跃、最具实践价值的方向——**基于 MCP（Model Context Protocol，模型上下文协议）的 Moss 智能体协议接入**。

MCP 是一个开源标准，用于将 AI 应用连接到外部系统。它解决了传统自定义插件、函数调用等方式缺乏统一标准的问题，让 AI 模型能够与各类数据源和工具实现无缝交互。

## 一、理解 Moss 协议体系

在开始接入之前，需要先理清 Moss 相关的几个核心概念：

| 概念 | 说明 |
| --- | --- |
| MCP（Model Context Protocol） | 开放标准协议，用于 AI 应用与外部系统的连接 |
| MOSS 协议（GhostOS 语境） | Model-oriented Operating System Simulator，面向模型的操作系统模拟器，核心是“Code As Prompt” |
| Moss 平台（微服务语境） | Spring Cloud 体系的服务治理平台 |
| Moss Agent 运行时 | 提供工具循环、上下文治理、安全钩子等能力的 Agent 框架 |

> **本教程重点**：以 **MCP 协议接入** 为核心，讲解如何将 Moss 智能体与外部工具、数据源进行集成。

## 二、环境准备

### 2.1 基础环境

根据你的接入场景选择对应的环境：

**场景一：接入 Moss Agent（Node.js 生态）**

```bash
# 安装 Moss Agent
npm install @rdk-moss/agent
# 或使用 CLI 工具
npm install -g moss
```

安装后可直接使用内置的 D-Robotics 网关（无需模型密钥，无需强制登录）。

**场景二：接入 Moss 平台（Spring Cloud 微服务）**

-   JDK 8 及以上版本
    
-   Maven 3.5 及以上版本
    
-   Git 工具
    

```bash
git clone <https://gitcode.com/gh_mirrors/moss/Moss>
cd Moss
mvn clean package -Dmaven.test.skip=true
```

### 2.2 启动 Moss 服务（微服务场景）

Moss 提供了 Docker Compose 配置，方便快速启动：

```bash
cd docker-compose
docker-compose up -d
```

服务启动后默认运行在 `http://localhost:8080`。

## 三、MCP 协议接入实践

### 3.1 理解 MCP 的工作方式

MCP 的核心价值在于：**让大模型在运行时动态、安全、高效地获取和使用外部上下文**。

在 Moss 中，MCP 被用于智能体与外部工具交互，支持设备控制、物联网集成和智能家居管理。

典型的工作流程：

1.  **AI 模型**通过 MCP 协议发出请求
    
2.  **MCP Server**接收请求并调用对应的外部工具/数据源
    
3.  **执行结果**通过 MCP 协议返回给 AI 模型
    

### 3.2 接入示例：MCP Server 配置

如果你使用的是 Moss 的 MCP Server，可以在 MCP 客户端中进行如下配置：

```json
{
  "mcpServers": {
    "moss": {
      "command": "node",
      "args": ["/packages/mcp-server/dist/cli.js"],
      "env": {
        "MOSS_RPC_URL": "<https://rpc.monad.xyz>"
      }
    }
  }
}
```

MCP Server 会暴露 `discover`、`load`、`action`、`simulate` 等核心工具供 AI 模型调用。

### 3.3 代码级接入：使用 Moss Agent 运行时

通过 npm 包 `@rdk-moss/agent` 可以将 Moss Agent 嵌入到自己的产品中：

```tsx
import { MossAgent } from '@rdk-moss/agent';

// 初始化 Agent
const agent = new MossAgent({
  // 配置参数
});

// Agent 提供的能力：
// - 工具循环 (tool loop)
// - 上下文治理 (context governance)
// - 安全钩子 (safety hooks)
// - 会话管理 (sessions)
// - 可插拔知识库 (pluggable knowledge)
```

## 四、Moss 智能体的工程实践

在实际的产业应用中，Moss 智能体可以覆盖从需求到运维的全生命周期。

### 5.1 前期：需求理解与方案生成

开发者只需用自然语言描述业务场景（如“我想在小程序里接微信支付和支付宝”），Moss 智能体可在数分钟内输出：

-   接入流程图
    
-   接口清单
    
-   配置项说明
    
-   沙箱参数和调试工具
    

### 5.2 中期：代码生成与智能排错

Moss 智能体可根据选定的语言和场景自动生成核心接口的调用代码，**错误率低至 0.2‰**，质量媲美资深工程师。

当遇到接口报错时，Moss 能：

-   **精准定位**问题根源（如“签名验证失败，请检查商户私钥格式”）
    
-   **提供修复建议**
    
-   **关联知识库**推送相关文档
    

### 5.3 后期：智能运维

Moss 平台集成了智能异常检测系统，对交易链路进行 24 小时实时监控，能在 5-8 分钟内预警或自动处置常见故障。

## 五、实践总结

以下是 Moss 协议接入的几条核心建议：

### 6.1 协议选择

-   **AI Agent 与工具交互**：优先使用 MCP 协议
    
-   **微服务治理**：使用 Moss 平台的 Spring Cloud 集成方案
    
-   **代码驱动型 Agent**：参考 GhostOS 的 MOSS 协议（Code As Prompt）
    

### 6.2 接入步骤

1.  **明确场景**：确定是设备控制、数据集成还是微服务治理
    
2.  **选择协议**：根据场景选择 MCP 或 Moss 平台接入方式
    
3.  **配置环境**：安装对应的依赖包或启动服务
    
4.  **注册能力**：将外部工具/数据源通过 MCP 注册为可调用能力
    
5.  **测试验证**：在沙箱环境中验证接入是否正常
    

### 6.3 避坑指南

-   确保 Moss 服务正常启动后再接入应用
    
-   检查网络连通性和服务地址配置是否正确
    
-   在微服务场景中，确认已正确引入 Moss 客户端依赖
    
-   Moss 是未经审计的 Alpha 软件，**请勿用于生产资金**

# GitHub 开源项目探索日志：Solidity

### 项目简介

**Solidity** 是面向以太坊平台的智能合约编程语言，一种静态类型、支持继承的合约导向高级语言。

### 项目目录结构

Solidity 编译器采用 **CMake** 构建系统，是一个典型的 **C++ 项目**，按功能模块划分清晰的子目录：

```
solidity/
├── .circleci/              # CI/CD 配置（CircleCI）
├── .github/                # GitHub 相关配置（Issue/PR 模板）
├── cmake/                  # CMake 构建配置
├── deps/                   # 依赖管理（如 google/fuzztest）
├── docs/                   # 文档源文件
├── libevmasm/              # EVM 汇编层（EVM 字节码生成）
├── liblangutil/            # 语言工具库（词法/语法分析基础）
├── libsmtutil/             # SMT 求解器工具（形式化验证）
├── libsolc/                # solc 编译器接口
├── libsolidity/            # Solidity 核心编译器（解析、分析、代码生成）
├── libsolutil/             # 通用工具库
├── libstdlib/              # 标准库实现
├── libyul/                 # Yul 中间表示层（优化器）
├── scripts/                # 构建和测试脚本
├── snap/                   # Snap 打包配置
├── solc/                   # 命令行编译器入口（solc 可执行文件）
├── test/                   # 测试套件
├── tools/                  # 开发工具
├── .clang-format           # 代码格式化配置
├── .codespellrc            # 拼写检查配置
├── .editorconfig           # 编辑器配置
├── .gitignore              # Git 忽略文件
├── .readthedocs.yml        # Read the Docs 文档托管配置
├── CMakeLists.txt          # 顶层 CMake 构建文件
├── CODE_OF_CONDUCT.md      # 行为准则
├── CODING_STYLE.md         # 编码风格规范
├── CONTRIBUTING.md         # 贡献指南
├── Changelog.md            # 版本变更日志
├── LICENSE.txt             # GPL v3.0 许可证
├── README.md               # 项目说明
├── ReleaseChecklist.md     # 发布检查清单
├── ReviewChecklist.md      # PR 审查检查清单
└── SECURITY.md             # 安全政策
```

**目录特点分析：**

-   `lib*` 系列目录是核心，按编译流程分层：`liblangutil`（基础工具）→ `libsolidity`（核心编译）→ `libevmasm`/`libyul`（代码生成与优化）→ `solc`（入口）
    
-   `test/` 目录包含完整的测试套件，确保编译器质量
    
-   项目根目录有大量 **Checklist 文档**（`ReleaseChecklist.md`、`ReviewChecklist.md`），说明发布和审查流程极其规范
    
-   `deps/` 目录管理外部依赖（如 Google 的 FuzzTest 模糊测试框架）
    

### GitHub 各模块的作用

1\. README

-   项目定位：面向以太坊的智能合约编程语言
    
-   快速上手：推荐使用 Remix 浏览器 IDE

3\. Issues

Solidity 使用 Issues 追踪 Bug 报告和功能请求。项目有一个活跃的 Issue 列表（最新可见 #16835、#16833、#16810、#16762 等），并配合 **GitHub Projects** 进行功能优先级管理。

4\. Pull Requests

PR 是代码贡献的核心渠道。Solidity 有严格的 PR 审查流程：

# Moss：为 AI Agent 搭建通往区块链的安全桥梁

> 当 AI Agent 开始真正操作链上资产时，我们需要的不只是“更聪明的 Agent”，更是“更安全的 Agent 基础设施”。

## 一、Moss 是什么？

Moss 是一个为 AI Agent 提供链上交互能力的开源框架，专注于 **Monad** 区块链生态。它把复杂的 DApp 和协议交互抽象为一套统一的、Agent 可调用的“能力层”（capability layer），遵循 `discover → load → action → simulate` 的标准流程。

一句话概括：**Moss 让 AI Agent 用自然语义操作链上协议，并在每一笔交易签名前增加一道机械化的安全闸门。**

Moss 目前处于 **Alpha 阶段**，已在 Monad 主网（chain id 143）上构建和模拟交易。但有一个关键原则贯穿始终：**Moss 永不签名、永不发送交易**。它只负责“构建和验证”，私钥留在钱包里，最终决定权留在用户手里。

## 二、Moss 解决了什么问题？

### 2.1 一个“几乎正确”的交易，就是资金丢失的方式

考虑一个 DEX 上的“简单”换币操作——背后涉及：路由地址选择、exact-in 与 exact-out 变体、原生代币的包装与解包装、退款和清扫等清理调用、带正确小数位的滑点计算。

**一个靠读 ABI 来组装这些的 Agent，迟早会把其中某一步搞错——而一个“几乎正确”的 Agent 构建交易，正是资金丢失的方式**。

### 2.2 AI 的“概率性正确”与区块链的“不可逆性”之间的矛盾

当前 AI Agent 的能力边界正在迅速扩展，但链上操作的高风险性（不可逆、资金安全）与 AI 的“概率性正确”本质之间存在根本矛盾。Moss 的洞察在于：**不要让 Agent 去处理 ABI、地址、精度这些底层细节，而是把这一切封装成“能力”，让 Agent 只用自然语义调用；同时用一套机械化的模拟验证机制，确保任何错误在签名前就被拦截**。

## 三、Moss 的设计理念

### 3.1 四步标准流程

Moss 把链上操作抽象为四个清晰的步骤：

| 步骤 | 工具 | 作用 |
| --- | --- | --- |
| discover | discover(verb?, category?, protocol?) | 发现各协议注册的能力和查询，返回坐标列表 |
| load | load(items) | 加载坐标对应的调用合约、意图模板、参数语义和风险标签 |
| action | action(protocol, method, account, params) | 调用能力，返回 Plan（未签名交易 + 声明的预期效果） |
| simulate | simulate(plans[]) | 按顺序模拟执行，比对实际效果与声明预期，返回警告 |

这套流程的设计哲学非常清晰：**把复杂性后移到协议维护的能力层**，Agent 只需要用人类可读的参数调用能力，剩下的由系统负责。

### 3.2 Plan：声明式安全模型

Plan 是 Moss 的核心数据结构，它包含：

-   `intent`：人类可读的意图描述（如“用 1 MON 在 Kuru 上换 USDC”）
    
-   `expects`：声明的预期效果——资金流出/流入、授权限额等
    
-   `txs`：实际未签名的交易数据
    
-   `planHash`：防篡改哈希
    

任何对 Plan 内容的篡改都会被 `simulate` 检测并标记为 `PLAN_TAMPERED` 警告。

### 3.3 双重安全检查

Moss 设计了**两道防线**：

**第一道：效果对账（Effects Reconciliation）—— 机械化，服务端执行**

-   模拟交易提取实际发生的链上效果——资产进出、授权、收款方，包括不发 Transfer 事件的原生 MON 流动和 wrapped-token 铸造
    
-   与 Plan 的 `expects` 比对，任何未声明的差异都会产生警告
    
-   **任何警告 → 立即停止**
    

**第二道：意图对齐（Intent Alignment）—— Agent 负责**

-   警告清零只说明“Plan 做了它声明要做的事”，不代表“它做了用户要求的事”
    
-   只有 Agent 掌握用户的原始指令，需要自行比对效果摘要与用户意图
    

这两道防线的设计非常精妙：**机械化的效果对账保证“执行正确”，Agent 侧的意图对齐保证“方向正确”**。两者缺一不可。

### 3.4 信任最小化（Trust Minimization）

Moss 有几个关键的设计选择体现了区块链精神的正向延续：

-   **不签名、不发送、不持有私钥**——它只构建和验证
    
-   **注册表显式组装**——没有任何东西通过 import 自动注册，开发者明确选择自己要用的协议
    
-   **Token 身份通过 curated catalog 解析**——绝不通过链上 `symbol()` 查询（同名 scam token 可轻易伪造）
    

## 四、为什么 AI Agent 需要 Moss？

### 4.1 降低 AI 的门槛

没有 Moss 时，一个 AI Agent 要执行链上操作需要：

-   理解 ABI 和合约地址
    
-   处理 multicall 和 cleanup 逻辑
    
-   做 decimals 算术
    
-   组装正确的 calldata
    

有了 Moss 之后，Agent 只需要：

-   用 `discover` 找到合适的能力
    
-   用 `action` 传入人类可读的参数（如 `"1.5"`，而非 `"1500000000000000000"`）
    
-   用 `simulate` 验证
    
-   把验证通过的未签名交易交给用户
    

**Agent 不再手搓 calldata。不碰 ABI、合约地址、multicall 扫尾、decimals 换算**。

### 4.2 安全是流程的一部分，而非附加项

`simulate` 不是可选的“安全检查”，而是标准流程的强制环节。安全被**设计进了工作流**，而非事后补救。

Agent Skill Guide 中明确列出了硬规则：

-   绝不签名、绝不发送、绝不索取私钥
    
-   绝不跳过 `simulate`
    
-   绝不用“重新模拟直到警告消失”的方式来满足警告——**警告是发现，不是噪音**
    

### 4.3 支持复杂多步流程

`simulate` 接受有序的 Plan 列表，并在它们之间链式传递状态——Plan B 可以花掉 Plan A 产生的资产。这是复杂多步流程（如“领取奖励 → 兑换 → 提供流动性”）的基础。

## 五、当前支持的协议

Moss 目前已在 Monad 主网上支持：

-   **WMON**（canonical wrapped MON）：wrap、unwrap、balanceOf
    
-   **ERC20 通用协议**：任意代币转账、余额查询、授权查询（含原生 MON）
    
-   **ERC721 通用协议**：任意 NFT 转账、归属查询
    
-   **Kuru**：链上订单簿 DEX——市价单 swap、报价、市场列表
    

设计上暂不支持：Permit 类签名流、跨链桥、闪电贷原子组合。

# 认识一个项目——MOSS

Moss 是一个为 AI Agent 提供链上交互能力的框架，专注于 **Monad** 区块链生态。它将复杂的 DApp 和协议交互抽象为统一的、可被 Agent 调用的“能力层”（capability layer），遵循 `discover → load → action → simulate` 的标准流程。

Moss 目前处于 **Alpha 阶段**，已在 Monad 主网（chain id 143）上构建和模拟交易，但 **Moss 本身永不签名、永不发送交易**。

## 核心问题

Moss 要解决的根本问题是：**AI Agent 直接操作区块链时，因复杂性过高而极易出错，且错误代价极大——直接导致资金损失。**

具体来说，一个看似"简单"的 DEX 换币操作，背后涉及：

-   路由地址选择
    
-   exact-in 与 exact-out 变体
    
-   原生代币的包装与解包装（wrap/unwrap）
    
-   退款和清扫（refund/sweep）等清理调用
    
-   带正确小数位的滑点计算
    

README 中直言："An agent that assembles this by reading ABIs will eventually get one of those wrong — and an agent-built transaction that is _almost_ right is how funds get lost."—— 一个"几乎正确"的 Agent 构建交易，正是资金丢失的方式。

Moss 的解法是把这些复杂性**后移到协议维护的能力层**，并在每一次签名前增加一道**机械化的安全闸门**。

## 核心能力

### 1\. 四步标准流程（Four-Step Flow）

| 步骤 | 工具 | 作用 |
| --- | --- | --- |
| discover | discover(verb?, category?, protocol?) | 发现各协议注册的能力（写操作）和查询（读操作），返回坐标列表 |
| load | load(items) | 加载坐标对应的调用合约、意图模板、参数语义和风险标签 |
| action | action(protocol, method, account, params) | 调用能力，返回 Plan（一组未签名的交易 + 声明的预期效果 expects） |
| simulate | simulate(plans[]) | 按顺序模拟执行 Plan，比对实际效果与声明预期，返回警告 |

### 2\. Plan 声明式安全模型

Plan 是 Moss 的核心数据结构，包含：

-   `intent`：人类可读的意图描述
    
-   `expects`：声明的预期效果（资金流出/流入、授权限额等）
    
-   `txs`：实际未签名的交易数据
    
-   `planHash`：防篡改哈希
    

任何对 Plan 内容的篡改都会被 `simulate` 检测并标记为 `PLAN_TAMPERED` 警告。

### 3\. 双重安全检查

**第一道：效果对账（Effects Reconciliation）—— 机械化，服务端执行**

-   模拟交易提取实际发生的链上效果——资产进出、授权、收款方，包括原生 MON 流动和 wrapped-token 铸造
    
-   与 Plan 的 `expects` 比对，任何未声明的差异都会产生警告
    
-   **任何警告 → 立即停止**
    

**第二道：意图对齐（Intent Alignment）—— Agent 负责**

-   警告清零只说明"Plan 做了它声明要做的事"，不代表"它做了用户要求的事"
    
-   只有 Agent 掌握用户的原始指令，需要自行比对效果摘要与用户意图
    

### 4\. 跨 Plan 状态链（Multi-step Composition）

`simulate` 接受有序的 Plan 列表，并在它们之间链式传递状态——Plan B 可以花费 Plan A 产生的资产。这是复杂多步流程（如"领取 → 兑换 → 提供流动性"）的基础。

### 5\. 协议适配器架构

-   一个协议 = 一个 npm 包（如 `@themoss/protocol-kuru`）
    
-   注册表通过包清单显式组装，**没有任何东西通过 import 自动注册**
    
-   目前已支持：WMON、通用 ERC20/ERC721、Kuru（链上订单簿 DEX）
    

## 可能的应用场景

### 1\. AI 驱动的 DeFi 交易助手

用户用自然语言说"用 1 个 MON 换成 USDC"，Agent 自动完成发现协议、构建交易、模拟验证的全流程，最后将已验证的未签名交易交给用户钱包审核签名。

### 2\. 多步 DeFi 策略自动化

"从 A 协议领取奖励 → 在 B 协议兑换成稳定币 → 在 C 协议提供流动性"——Moss 的 `simulate(plans[])` 支持跨 Plan 状态链式传递，天然适合这类复杂流程。

### 3\. 链上操作的安全审计层

任何 AI Agent 在签名前都必须经过 Moss 的 simulate 验证。Moss 可以作为 **AI Agent 的安全中间件**，为所有链上操作提供一道机械化的、可审计的安全闸门。

### 4\. 协议接入的标准化层

协议方只需维护一个适配器包（从 `_template` 复制并填写清单），即可让所有接入 Moss 的 AI Agent 自动获得与该协议交互的能力。这降低了 AI 与 DeFi 协议之间的集成成本。

### 5\. 本地开发与测试环境

Moss 支持在本地 Anvil 分叉上运行完整流程，使用真实订单簿状态、零真实资金。开发者可以用它来测试 AI Agent 的链上行为，无需承担资金风险。

## 我的理解

Moss 解决的是一个非常具体但又极其关键的问题：**AI Agent 与区块链之间的"最后一公里"信任鸿沟**。

当前 AI Agent 的能力边界正在迅速扩展，但链上操作的高风险性（不可逆、资金安全）与 AI 的"概率性正确"本质之间存在根本矛盾。Moss 的洞察在于：**不要让 Agent 去处理 ABI、地址、精度这些底层细节，而是把这一切封装成"能力"，让 Agent 只用自然语义调用；同时用一套机械化的模拟验证机制，确保任何错误在签名前就被拦截。**

这种设计有几个值得注意的哲学选择：

1.  **信任最小化（Trust Minimization）**：Moss 不签名、不发送、不持有私钥。它只负责"构建和验证"，最终决定权在用户手中。这是区块链精神的正向延续。
    
2.  **声明式 vs 命令式**：Plan 不是"执行指令"，而是"声明预期"。这种声明式模型让模拟验证成为可能——你可以先"承诺"会发生什么，再验证实际是否如此。
    
3.  **能力（Capability）而非合约（Contract）**：Moss 对 Agent 暴露的是"能力"（wrap、swap、supply），而非合约方法（deposit、swapExactTokensForTokens）。这大幅降低了 Agent 的理解负担。
    
4.  **安全是流程的一部分，而非附加项**：`simulate` 不是可选的"安全检查"，而是标准流程的强制环节。安全被设计进了工作流，而非事后补救。
    

Moss 目前仍是 Alpha 软件，尚未经过审计。但它指向了一个清晰的未来方向：**当 AI Agent 开始真正操作链上资产时，我们需要的不只是"更聪明的 Agent"，而是"更安全的 Agent 基础设施"**。Moss 正是在这个方向上的一次务实尝试。

一个基于 Monad的轻量级去中心化留言板智能合约。

# 为什么Monad体验不同

### 方向选择：链上订单簿交易所（On-chain CLOB）

### 一、Research：为什么 Monad 更适合这个高频交互应用

1\. 产品场景为何天然需要“高频”交互？

在订单簿模型中，市场由**做市商（Maker）和吃单者（Taker）共同驱动。做市商的核心职责不是“持有资产”，而是通过连续挂单和撤单来赚取买卖价差**。这是一场围绕“队列优先级”的毫秒级竞赛：

-   **价格发现的连续性**：在中心化交易所（CEX）中，顶级做市商的订单更新频率通常在**每秒 50~100 次**以上。每当外部市场价格发生 0.01% 的波动，链上做市商必须立即撤掉旧单并挂出新单，否则其挂单会被套利者迅速吃掉。
    
-   **订单与交易比极高**：传统金融中，挂单/撤单次数与实际成交次数的比值常高达**100:1**。这意味着一个每日成交 1 万笔的活跃交易对，背后隐藏着**每日超过 100 万次的链上状态写入**。如此高频的状态变更，决定了 CLOB 是区块链性能的“极限测试场”。
    

2\. 拥堵链和高 Gas 费带来的“死亡螺旋”困境

在 Solidity 生态（以太坊主网或低性能 L2）上尝试 CLOB 会陷入三重困境，这也解释了为何过去 DeFi 只能退而求其次选择 AMM：

-   **延迟导致的滑点灾难**：以太坊 12 秒的出块时间意味着，当用户提交限价单时，指令可能在多个区块后才被确认，此时市场价格早已变化，订单要么无法成交要么以不利价格成交。对于做市商，这种延迟等同于“蒙眼接飞刀”。
    
-   **Gas 费吞噬做市利润**：在以太坊网络拥堵时，一笔简单的 `cancelOrder` 操作可能耗费数十美元 Gas 费。假设一个做市商每天需要调整 1,000 次订单，其单日运营成本就高达数万美元。这直接导致链上 CLOB 的买卖点差必须极大才能覆盖成本，流动性陷入“越差越薄，越薄越差”的负循环。
    

3\. Monad 如何从底层架构颠覆这一痛点？

Monad 的并行执行架构和超高 TPS 不仅带来了“数字上的快”，更重要的是带来了**经济可行性与策略精确性**的质变。

Monad 是一个高性能的 EVM 兼容 Layer 1 区块链，**于 2025 年 11 月 24 日正式上线主网**。其核心性能指标包括：

-   **每秒 10,000 笔交易（10,000 TPS）** 的吞吐量
    
-   **0.4 秒出块时间**
    
-   **约 0.8 秒的交易最终确认（finality）**
    

Monad 由前 Jump Trading 高频交易团队成员创立，将高频交易的优化原则直接应用到了区块链架构中。其核心技术包括**乐观并行执行（Optimistic Parallel Execution）** ——独立交易可并行处理，仅在发生状态冲突时回退到顺序执行——以及**MonadBFT 流水线共识机制**。

这让**完全链上的 CLOB 在 EVM 生态中首次变得可行**：

-   **0.8 秒终局性与 0.4 秒区块时间**：做市商可以在**1 秒内完成撤单+重挂**，将价格风险敞口压缩到极低水平。
    
-   **极低的交易成本**：Monad 的高吞吐量使得单笔挂撤单的 Gas 费维持在美分级甚至更低。做市商可以频繁更新报价而不会产生显著成本负担。
    
-   **EVM 完全兼容，无需改代码**：Monad 100% 兼容 EVM 字节码，开发者可以使用熟悉的 Solidity、Hardhat、Foundry 等工具直接部署。现有的 DeFi 做市商和 MEV 机器人无需学习新语言即可迁移策略。
    

4\. 为什么“普通数据库”无法替代？

很多人会问：既然订单簿需要高频更新，为何不采用“链下撮合 + 链上结算”的混合模式？这种模式虽然可行，但牺牲了 DeFi 最核心的 **“可组合性”与“透明度”**：

-   **原子性与可组合性**：在纯链上 CLOB 中，**“订单撮合”与“资产交割”是一个不可分割的原子事务**。用户可以将 CLOB 中的限价单直接作为借贷清算的触发条件，或与其他 DeFi 协议组合成复杂策略。这是任何中心化数据库永远无法提供的终极护城河。
    
-   **抗审查与可验证性**：传统数据库由项目方中心化控制，存在“插队”或定向撤单的风险。而 Monad 上的 CLOB 将撮合逻辑运行在开放的虚拟机中，任何第三方节点都可以独立验证每一笔挂单是否按照规则公平执行。
    

Monad 生态中已涌现出一批原生 CLOB 项目。**Kuru** 是 Monad 生态中首个完全链上的中央限价订单簿（CLOB）DEX，已获得 Electric Capital 领投的 200 万美元种子轮融资，目前已完成 1000 万美元融资并在 X 上新增 12 万粉丝。**Clober** 是 Monad 上的链上订单簿基础设施协议，引入了 LOBSTER（Limit Order Book with Segment Tree for Efficient oRder-matching）机制来优化 Gas 效率。**DrakeExchange** 则是完全链上的永续合约 DEX，采用 CLOB+vAMM 混合模式。这些项目的集中涌现，本身就说明了 Monad 在 CLOB 方向上的独特吸引力。

## 二、Monad 的优势与劣势

### ✅ 核心优势

**1\. 100% EVM 兼容（字节码级别）**

现有以太坊智能合约和生态工具可以直接迁移至 Monad，无需大幅改动。开发者可以使用 Solidity、MetaMask、Foundry 等熟悉的工具，无需学习新语言。“尽可能少动 EVM 规则”的设计哲学，使其成为“EVM 世界的并行加速器”。

**2\. 高性能与低费用**

目标性能为 **10,000 TPS**、**1 秒最终确认**、**低于 1 美分的平均 Gas 费**。Monad 通过四大技术创新实现这一目标：**乐观并行执行**（假设无冲突，同时运行所有交易，碰撞时再重新执行）、**MonadBFT 共识**（约 1 秒完成最终确认）、**异步执行**（共识与执行解耦）以及 **MonadDB**（专为高性能状态管理设计）。

**3\. 消费级硬件可运行节点**

相比 Solana 的数据中心级硬件要求，Monad 优化了节点性能，降低了硬件门槛。虽然目前仍推荐较高配置，但设计目标之一是让更多用户参与网络验证。

**4\. 顶级资本背书与生态冷启动能力**

超 **4.3 亿美元**融资储备，生态已聚集超 **280 个项目**，累计生态融资达 **13.2 亿美元**。Aave V3 上线 48 小时内存款突破 1 亿美元，Uniswap、Curve 等头部协议已部署。

### ❌ 核心劣势与挑战

**1\. 性能尚未被大规模验证**

主网上线仅半年多，**10,000 TPS 在复杂真实场景下的稳定性仍需时间验证**。市场对“性能叙事”的敏感度已明显降低——**高 TPS 早已不再是稀缺指标**。

**2\. 激励驱动的增长能否持续**

当前 TVL 和用户活跃度很大程度上由激励计划驱动。**激励退潮后能否留住用户和流动性**，是 Monad 面临的最大考验。

**3\. 代币经济学的长期压力**

MON 总供应量 **1000 亿枚**，团队（27%）和早期投资者（19.7%）持有大量代币。流通供应量仍远低于总供应量，**未来的解锁、验证者奖励、生态排放对价格表现至关重要**。MON 价格从公售价 $0.025 短暂冲高后有所回落。

**4\. 竞争格局极其激烈**

| Sui | 以对象为中心（Object-Centric） | 将资产视为独立对象，拥有对象交易可绕过全局共识 |
| Monad | 乐观并行执行（Speculative EVM） | 默认无冲突，先并行执行；冲突时再按序重新执行 |

### 🎯 各公链核心定位

**以太坊：“守成之王”**

拥有最大的开发者网络、最深的流动性和最强的网络效应。EVM 是加密领域最重要的开发者环境。但底层性能瓶颈明显，扩容依赖 L2。

**Solana：“性能之王”**

经过多轮牛熊考验的蓝筹高性能公链，在高吞吐和低费用场景中优势明显。但开发者需学习 Rust，节点硬件门槛高。

**Sui：“架构创新者”**

基于 Move 语言，以对象为中心的并行模型提供了独特优势。拥有对象交易可实现亚秒级最终确认。但生态仍在建设，Move 学习曲线对以太坊开发者不够友好。

**Monad：“EVM 的挑战者”**

核心定位可概括为 **“拥有 Solana 速度的以太坊”** 。100% EVM 兼容是最大王牌——以太坊开发者和现有应用可以**无缝迁移**，获得性能的巨大提升。Monad 选择了兼容路径，更像是 **“性能版以太坊”** ，是 EVM 世界的并行加速器。

## 💎 总结

Monad 试图在 **“性能、兼容性、去中心化”** 三者之间找到最优平衡点。它既不像 Solana 那样需要开发者学习新语言，也不像以太坊 L2 那样受限于主网的最终结算延迟。它提供了一个 **“熟悉但更快”** 的开发环境。

但挑战同样明确：**“更快”的叙事已经不够稀缺**。Monad 需要在激励退潮后证明自己拥有**不可替代的真实需求**——真实的 Gas 消耗、真实的 DeFi 活动、真实的用户留存。正如其空投后的表现所示：**空投驱动的关注度并不等同于产品与市场契合**。真正的考验，才刚刚开始。

Monad 的理解：**一个从底层软件架构出发，旨在解决“去中心化”与“高性能”对立关系的高性能 Layer-1 区块链。它试图在完全兼容以太坊生态的前提下，解决主流区块链的性能瓶颈。**

**核心技术：**  
一、**MonadBFT 共识机制**：这是Monad的共识核心，是一个前沿的拜占庭容错（BFT）协议。

-   **高性能**：它能实现**单轮（400毫秒）的推测性最终确认**和**两轮（800毫秒）的完全最终确认**。
    
-   **大规模节点支持**：在正常（无故障）路径下，其消息复杂度是线性的，允许共识节点集扩展到很大的规模。
    
-   **关键特性**：它具备**乐观响应性**（网络有多快，共识就有多快）、**领导容错**（单个失败领导者只带来一次超时延迟）以及对**尾分叉（tail-forking）攻击的抵抗能力**。
    

**二、RaptorCast**：一种高效的区块传输协议，用于优化网络层的数据分发。

**三、异步执行（Asynchronous Execution）**：这是Monad架构中非常关键的一环。它将**共识（交易排序）** 和**执行（交易计算）** 解耦。

-   **传统模式的问题**：在以太坊等链上，执行是共识的前提，导致执行时间被严重压缩，仅占区块时间的很小一部分。
    
-   **Monad的方案**：节点先对交易的**顺序**达成共识，然后再去执行它们。这使得执行可以占用**整个区块时间**，极大地提高了执行预算。
    

**四、并行执行（Parallel Execution）**：Monad 采用**乐观执行**策略。

-   **工作原理**：它会**假设交易之间没有冲突，同时开始执行一个区块内的所有交易**。
    
-   **冲突处理**：如果发生冲突（如两笔交易操作了同一个账户），系统会**重新执行**受影响的交易。由于输入数据已在缓存中，重新执行的速度远快于首次执行。最终，Monad 的执行结果与以太坊**完全相同**。
    

**五、MonadDb**：专为以太坊状态存储设计的定制化**键值数据库**。

-   **原生优化**：它原生实现了以太坊使用的**默克尔帕特里夏树（Merkle Patricia Trie）** 数据结构，避免了“一种数据结构嵌入另一种”的低效问题。
    
-   **异步I/O**：充分利用了 Linux 最新的 `io_uring` 异步I/O接口，避免因I/O操作阻塞执行。
    
-   **极致性能**：为追求极致性能，MonadDb 甚至可以**绕过文件系统，直接在块设备上运行**，以完全解锁SSD的性能潜力。
