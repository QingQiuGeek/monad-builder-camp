# 二、Monad 技术学习记录

## 2.1 什么是 Monad？

Monad 是一个高性能 EVM 兼容区块链，目标是在保持 Ethereum 开发体验的基础上，大幅提升交易处理能力。

传统 Ethereum 虽然具有较高的安全性和去中心化程度，但由于执行模型主要采用串行处理方式，当大量用户同时交互时，会出现：

-   TPS 较低；
    
-   Gas 费用上涨；
    
-   高频应用体验下降。
    

Monad 的核心理念：

> 在保持 EVM 兼容性的同时，通过底层架构优化，提高区块链执行效率。

简单理解：

Ethereum 更像一台安全可靠的服务器。

Monad 希望成为一台高性能分布式计算机。

* * *

# 三、Monad 与 Ethereum 的核心区别

通过阅读 Monad 官方文档，我总结了两者主要区别：

| 对比项 | Ethereum | Monad |
| --- | --- | --- |
| 虚拟机 | EVM | 优化版 EVM |
| 执行方式 | 单线程串行执行 | 并行执行 |
| TPS | 较低 | 高吞吐 |
| Gas成本 | 网络拥堵时较高 | 更低、更稳定 |
| 开发体验 | Solidity + EVM工具链 | Solidity + EVM工具链 |
| 目标 | 安全稳定 | 高性能应用 |

* * *

## 3.1 并行执行（Parallel Execution）

传统区块链执行交易：

```
交易A
 ↓
交易B
 ↓
交易C
 ↓
交易D
```

所有交易依次执行。

Monad 通过并行执行思想：

```
交易A   交易B   交易C
 ↓       ↓       ↓
并行计算处理
 ↓
结果合并
```

如果多个交易之间不存在数据冲突，就可以同时处理。

例如：

用户A购买NFT

用户B进行Swap

用户C存储数据

这些操作互相独立，可以同时执行。

二、智能合约基础理解

在前三天学习中，主要接触的是普通区块链交易。

普通交易模型：

钱包A
    ↓
发送ETH
    ↓

用户钱包
    ↓

    ↓

四、智能合约代码分析


1. Solidity版本声明

pragma solidity ^0.8.20;


作用：

指定合约编译所使用的 Solidity 版本。

避免不同版本之间产生兼容问题。


2. 状态变量


string private message;


该变量存储在区块链 Storage 中。

与传统Web开发中的数据库类似。

区别：

传统数据库：

服务器维护数据。

区块链：

所有节点共同维护数据。


3. constructor构造函数


constructor(){

message = "Hello Web3";

}


特点：

部署合约时自动执行一次。

用于初始化合约状态。


4. write function


function setMessage()


该函数会修改链上数据。

调用流程：

用户点击按钮

↓

钱包签名

↓

生成交易

↓

消耗gas

↓

区块确认

↓

状态改变


因此会产生：

Transaction Hash。


5. read function


function getMessage()
view


view表示：

函数只能读取数据，不能修改状态。


调用特点：

不产生交易。

不需要支付gas。

不会生成transaction hash。


五、Solidity修饰符学习


今日重点学习了Solidity中的常用修饰符。


1. public

表示函数或者变量可以被外部访问。


2. private

表示只能当前合约内部访问。


3. internal

表示当前合约和继承合约可以访问。


4. external

表示只能通过外部调用。


5. view

表示函数只能读取数据。


6. pure

表示函数不能读取和修改链上状态，只能进行计算。


7. payable

表示函数可以接收ETH。


8. modifier

用于增加函数执行条件。


例如：

onlyOwner

表示只有合约管理员可以执行。


modifier本质类似Web2中的权限控制中间件。

七、理解ABI和Bytecode


Solidity源码不能直接被区块链执行。

Solidity源码

↓

Solidity Compiler

↓

Bytecode + ABI

智能合约真正运行的机器代码。

Application Binary Interface

即合约接口说明。


ABI描述：

- 有哪些函数
- 函数参数
- 返回值类型
- 如何调用


前端DApp通过ABI知道如何和智能合约交互。

六、核心知识总结

### 1\. Nonce

Nonce 是钱包地址发送交易时使用的编号。

作用：

防止交易重复执行，同时保证交易顺序。

例如：

第一次交易：

Nonce = 12

第二次交易：

Nonce = 13

区块链通过 Nonce 判断交易执行顺序。

* * *

### 2\. Gas

Gas 是区块链执行操作所需要支付的计算资源费用。

简单理解：

Gas = 区块链运行程序所需要的手续费。

普通 ETH 转账通常消耗约：

21000 Gas

智能合约交互由于计算复杂度更高，会消耗更多 Gas。

Gas 机制可以防止恶意程序无限占用区块链计算资源。

* * *

### 3\. Block

Block（区块）是区块链存储交易数据的基本单位。

多个交易会被验证者打包进入一个区块。

区块包含：

-   交易记录
    
-   时间戳
    
-   前一区块信息
    
-   区块哈希
    

多个区块按照时间顺序连接，形成不可篡改的区块链。

# 二、理论知识学习

## 1\. 区块链钱包基本概念

在传统互联网中，用户身份通常由账号和密码进行管理，而在 Web3 世界中，用户身份由区块链钱包地址表示。

钱包本身并不存储数字资产，而是保存用户访问区块链账户所需要的密钥信息。

区块链钱包主要包含：

-   私钥（Private Key）
    
-   公钥（Public Key）
    
-   钱包地址（Address）
    
-   助记词（Mnemonic Phrase）
    

其中：

私钥是用户控制资产的唯一凭证，通过密码学算法生成公钥，再由公钥经过哈希计算得到钱包地址。

整个过程：

```
随机数
 ↓
私钥 Private Key
 ↓
公钥 Public Key
 ↓
钱包地址 Address
```

钱包地址类似于传统银行账户号码，而私钥则类似于银行卡密码，但区块链中的私钥权限更高，一旦泄露，资产将无法恢复。

**三、核心概念理解（重点）**

**1\. 钱包的本质：不是“存钱工具”，而是“身份系统”**

在传统互联网中，用户身份依赖中心化账号体系（如手机号、邮箱、用户名）。

而在 Web3 中：

钱包 = 去中心化身份（Decentralized Identity）

钱包的核心作用不是“存储资产”，而是：

-   生成唯一身份（Public Address）
    
-   提供签名能力（Signature）
    
-   控制链上状态变更权限
    

换句话说：

区块链并不“认识人”，只认识“能签名的密钥”

* * *

**2\. 地址（Address）与私钥（Private Key）的关系**

当前理解如下：

-   私钥：唯一控制权，类似“终极密码”
    
-   公钥：由私钥推导生成
    
-   地址：公钥进一步哈希后的结果（用户可公开）
    

关键认知：

地址是公开身份，但控制权永远只存在于私钥

因此：

-   地址可以公开（用于收款 / 交互）
    
-   私钥必须绝对保密（资产安全核心）
    

* * *

**3\. 签名机制：Web3的信任基础**

Web3中不存在“账号密码登录”，取而代之的是：

Sign (签名) = 用私钥证明“我是这个地址的拥有者”

签名的作用：

-   证明身份
    
-   授权交易
    
-   防止篡改
    

可以理解为：

区块链世界的“身份验证 = 数学证明”，而不是中心化验证

* * *

**4\. 钱包在区块链系统中的位置**

当前形成的系统认知：

用户 → 钱包 → 签名 → 交易 → 区块链网络 → 状态更新

可以理解为：

-   钱包是用户与区块链之间的“唯一接口”
    
-   所有链上行为必须经过钱包签名授权
    
-   区块链本身不执行“用户意图”，只执行“已签名交易”
