# AI从产品到Agent经济路径

## 一、核心行业判断

1.  AI 行业阶段划分
    
    1.  上半场落幕：模型层竞争格局已定，行业不再需要验证 AI 能力上限
        
    2.  下半场核心：聚焦行业场景落地，把 AI 嵌入真实业务流程
        
2.  产品落地优先级：**业务＞流程＞规则＞AI**
    
    1.  合格 AI 产品标准：剥离 AI 功能后，产品仍可独立解决用户真实需求，无刚需的纯 AI 概念产品属于伪需求
        
    2.  避坑提醒：仅头部团队适合做概念项目，普通从业者深耕业务痛点才可持续
        

## 二、Agent 经济发展方向

1.  新型商业交互模式 打破传统 “人 - 软件 - 服务” 链路，未来形成「人 - Agent - 多 Agent」协同经济体系；现有链上 Agent 协议仅能解决单一问题，真实业务落地存在大量边界缺陷。
    
2.  产品与底层协议关系：落地产品反向驱动底层协议迭代优化。
    
3.  项目孵化资源（Virtuous 平台）
    
    1.  平台定位：链上 Agent 孵化 Launchpad，覆盖虚拟智能体、具身智能赛道
        
    2.  扶持资源：实验室、数十台智能机器人、技术与资金扶持
        
    3.  对接对象：拥有可落地真实场景、解决实际问题的 Agent / 具身智能项目团队
        
4.  主讲人自身业务布局
    
    1.  垂直赛道：大健康、心理疗愈 AI 产品（配备合伙人长期深耕）
        
    2.  线下业务：青少年 AI 创意教育营，青少年使用 AI 可快速完成创意落地与产品迭代
        
    3.  内容输出：社交账号持续更新 AI、Web3 交叉领域行业思考
        

## 三、落地实操方法论

### （1）需求挖掘思路

1.  三大需求来源：自身行业经验、日常生活痛点、个人未被满足需求
    
2.  案例参考：ADHD 从业者基于自身记录需求，开发语音记录智能硬件并开启预售
    

### （2）技术选型区分

-   标准化、流程可固定场景：Workflow / 脚本即可实现，无需 Agent
    
-   复杂、无固定规则非标场景：引入 AI Agent 处理
    

### （3）AI 销售落地痛点与方案

1.  现存问题：通用大模型对接销售场景转化率低、对话成本高、知识库搭建繁琐
    
2.  落地建议：先做最小可行性验证，小范围跑通业务，逐步迭代，前期控制投入成本
    
3.  核心难点：人与人线下建立的信任链路，AI 难以完全替代
    

### （4）AI 原生世代认知观点

1.  AI 定位：如同互联网，是基础工具，而非单纯检索工具，无需强行背诵全部知识
    
2.  应对认知幻觉：定期抽查知识点、输出内容核验，巩固自身知识储备
    
3.  Agent 信任赛道具备长期探索价值
    

## 四、从业者成长与创业建议

### 1\. AI Agent 零基础学习路线

提示词工程 → LLM 基础 → RAG 检索增强 → 模型微调 → 记忆架构 → 主流开发框架

-   建议：非技术研发无需追逐每日前沿论文，优先选用成熟方案解决业务问题；记忆架构是行业普遍难点
    

### 2\. 产品 / ToB 业务从业经验

1.  工作难度取决行业熟悉度：不懂行业时，梳理甲方混乱业务流程是最大难点；熟悉行业则原型开发、对接开发更耗时
    
2.  客户筛选准则：拒绝需求模糊、频繁变更需求的甲方，宁可少接单，不消耗核心团队精力
    
3.  岗位护城河：跳出纯技术视角，研究用户付费逻辑，多对接销售、创始人理解商业本质
    

### 3\. 创业避坑经验

不要花费高成本教育认知不符的客户 / 老板，沟通成本高且收益不确定，不合适及时更换合作对象；好产品核心是解决真实问题，技术融合难度远低于业务梳理。

### 4\. 长期个人不可替代性

想象力、独立判断与责任担当、行业远见、审美、人际链接、长期坚持，均是 AI 无法替代的核心能力。

### 5\. 行业社交建议

线下交流以解决问题、深度链接为目标，摒弃只为拍照、加好友的无效流量社交。

# **Moss 项目分析**

## **1\. 项目简介**

Moss 是一个面向 Monad 区块链的协议交互框架。它把链上协议（如 WMON、ERC-20/721、Kuru 等）封装成 **Agent 可调用的 Capability（能力单元）**，并通过统一的四步流程对外服务：

```
 discover（发现） → load（加载） → action（执行） → simulate（模拟）
```

核心定位：**Moss 只构建和验证「未签名交易」，永不签名、永不发送。** 它是一个处于 Alpha 阶段、未经审计的开源项目（MIT 协议），当前只支持 Monad 主网（chain ID `143`）。

## **2\. 核心问题**

传统上让 AI Agent 操作链上协议时，存在三个痛点：

-   **协议知识散落**：地址、ABI、calldata 构造、参数规则、回执解析都混在调用方代码里，Agent 难以安全、一致地复用。
    
-   **缺少可验证证据**：Agent 生成的交易是否真的做了用户要求的事，往往只能"盲信"，没有结构性证据。
    
-   **签名与意图易失控**：Agent 一旦能直接签名发交易，就绕过了用户确认，风险极高。
    

Moss 的设计正是为了把「协议操作」与「签名授权」彻底解耦。

## **3\. 核心能力**

| 能力 | 说明 |
| --- | --- |
| 协议即能力（Protocol-owned） | 协议包自持地址、ABI、calldata、参数规则与 Receipt 解析，Agent 只调用统一接口 |
| 模拟产出证据 | 每笔成功交易产出有序原始 Change 与结构化 Receipt，且 Receipt 必须完整、按序覆盖每个 Change |
| 签名独立 | MCP Agent 先用每条有序 Receipt 文本对照用户请求；SDK 可用结构化 Outcome，钱包最后才接触未签名交易 |
| 四步工具契约 | MCP server 仅暴露 discover、load、action、simulate 四个工具 |
| 严格验证 | 回滚、trace 失败、Receipt 失败、覆盖不一致都会触发终止性 Warning，库会暴露完整 Receipt 树 |
| 多协议支持 | 已支持 WMON（wrap/unwrap）、ERC-20/native MON（transfer/approve）、ERC-721（transfer）、Kuru（swap） |

## **4\. 可能应用场景**

-   **AI 钱包助手**：用户用自然语言下指令（"把 1 MON 换成 USDC"），Agent 调用 Moss 生成并模拟交易，用户确认后再由钱包签名。
    
-   **链上操作自动化**：把复杂的多步协议交互封装为可组合 Capability，供 Agent 编排执行。
    
-   **交易前安全审计**：在签名前用模拟输出验证"这笔交易到底改了什么状态"，防止误操作或恶意 calldata。
    
-   **协议接入中台**：新协议按模板（`packages/protocols/_template`）提交 Protocol 包，即可被所有 Agent 复用。
    
-   **本地 fork 上的 Agent/签名方分离**：在隔离环境验证 Agent 行为，再接入真实签名方。
    

## **5\. 理解**

Moss 的本质是给链上协议做了一层「语义化 + 可验证」的封装接口。它不解决"怎么签名发交易"，而是把更危险、更模糊的环节——**让 Agent 理解协议、构造正确交易、证明交易效果符合意图**——做到了可验证、可审计。

它把"信任"从"相信 Agent 写了正确的 calldata"转移到了"模拟器产出的结构化证据"。这对 AI 与区块链结合非常关键：模型擅长理解和决策，但不擅长保证字节级正确；Moss 用模拟证据补上了这块短板。

## **总结**

Moss 解决的是 **AI Agent 安全操作区块链协议** 的问题：把各类链上协议封装成 Agent 可调用的标准能力，并通过"构建未签名交易 + 模拟验证 + 结构化回执"流程，在不触碰私钥的前提下，证明交易效果与用户意图一致，从而把签名授权与 Agent 决策彻底解耦。未来可应用于自然语言链上助手、交易前安全审计、多协议自动化编排、协议接入中台等场景，尤其适合需要"AI 决策、人/钱包最终确认"的 Web3 交互产品。

# **Web3 基础概念复习**

* * *

## **一、Web 的演进：从读到写再到拥有**

| 阶段 | 名称 | 核心特征 | 数据控制权 | 典型代表 |
| --- | --- | --- | --- | --- |
| Web1.0 | 静态只读网 | 单向展示，用户只能看 | 平台（中心服务器） | 早期门户、HTML 静态页 |
| Web2.0 | 交互社交网 | 用户可创作，但平台垄断 | 平台（中心化公司） | 微信、抖音、Facebook |
| Web3.0 | 价值互联网 | 用户拥有数据与资产 | 用户（去中心化） | 以太坊、钱包、DeFi |

**一句话**：Web2 让你"免费使用但被平台收割数据"，Web3 让你"自己掌握身份、资产与规则"。

* * *

## **二、Web3 的四大核心特征**

1.  **去中心化（Decentralization）**：没有单一公司说了算，网络由全球节点共同维护。
    
2.  **用户主权（Self-Sovereignty）**：身份、资产、数据归用户自己，靠私钥控制。
    
3.  **无需许可（Permissionless）**：任何人都能接入、开发、转账，不需要注册审批。
    
4.  **价值互联（Value-native）**：信息和钱在互联网上原生传递，代码即契约。
    

* * *

## **三、底层技术基石**

### **1\. 区块链（Blockchain）**

-   本质：一个**只能追加、不可篡改**的公开账本。
    
-   结构：区块按时间顺序用哈希首尾相连，改一个块会让后续全部失效。
    
-   作用：解决"陌生人之间如何互不信任也能达成共识"的问题。
    

### **2\. 密码学三件套**

| 技术 | 作用 | 类比 |
| --- | --- | --- |
| 哈希函数（SHA-256等） | 把任意数据变成固定长度指纹，不可逆 | 数据"身份证号" |
| 非对称加密 | 公钥能看、私钥能签，成对出现 | 公钥=锁，私钥=钥匙 |
| 数字签名 | 用私钥签名，别人用公钥验真 | 电子盖章，防抵赖 |

### **3\. 共识机制（Consensus）**

节点如何就"账本长什么样"达成一致：

-   **PoW（工作量证明）**：比算力，比特币用，安全但耗能。
    
-   **PoS（权益证明）**：比质押，以太坊用，节能高效。
    

### **4\. 智能合约（Smart Contract）**

-   写在链上的自动执行程序，条件触发即运行，无需中介。
    
-   类比：自动售货机——投币（条件满足）就掉货（执行结果），不靠人。
    
-   平台代表：以太坊（EVM）、Solana、Monad 等。
    

### **5\. 钱包与密钥**

-   **私钥**：你资产的唯一凭证，**泄露即归零**，不可找回。
    
-   **公钥/地址**：给别人转账用的"账号"。
    
-   **助记词**：私钥的人类可读备份（12/24 个英文单词）。
    

* * *

## **四、核心组成模块**

| 模块 | 是什么 | 解决什么 |
| --- | --- | --- |
| 公链 | 以太坊、比特币等底层网络 | 提供信任与结算层 |
| Token | 链上可流通的价值单位 | 激励、支付、治理 |
| NFT | 非同质化代币，唯一不可拆分 | 确权数字资产/收藏品 |
| DeFi | 去中心化金融（借贷/交易/衍生品） | 无银行中介的金融服务 |
| DAO | 代码治理的分布式组织 | 透明、社区共治 |
| 稳定币 | 锚定法币（如 USDT）的币 | 降低波动，便于支付 |

**同质化 vs 非同质化**：

-   同质化（BTC、ETH）：你的一枚和我的一枚等价，可互换。
    
-   非同质化（NFT）：每个都独一无二，不能等价互换。
    

* * *

## **五、典型应用场景**

-   **支付与跨境汇款**：无需银行，几分钟到账，费用低。
    
-   **去中心化金融（DeFi）**：存币生息、抵押借贷、链上交易。
    
-   **数字身份（DID）**：自我主权身份，一次认证多处使用。
    
-   **创作者经济**：作品直接上链确权，绕过平台抽成。
    
-   **链游（GameFi）/ 元宇宙**：资产归玩家所有，可自由交易。
    

* * *

## **六、风险与争议（必须知道）**

1.  **私钥即一切**：丢失/泄露无法找回，无"忘记密码"通道。
    
2.  **智能合约漏洞**：代码即法律，Bug 可被利用导致资产归零。
    
3.  **价格波动**：加密资产剧烈波动，非稳健储值。
    
4.  **监管不确定性**：各国政策差异大，合法性随地域变化。
    
5.  **诈骗与 Rug Pull**：匿名环境下项目方跑路、假币泛滥。
    
6.  **Gas 费与拥堵**：网络繁忙时手续费高、确认慢。
    

* * *

## **七、关键术语速查**

| 术语 | 含义 |
| --- | --- |
| 区块 / Block | 打包一段时间交易的账本单元 |
| 节点 / Node | 参与维护网络的计算机 |
| Gas | 执行交易/合约支付的手续费 |
| 主网 / Mainnet | 正式运行的链 |
| 测试网 / Testnet | 免费演练用的链（如 Sepolia） |
| 桥 / Bridge | 连接不同链的跨链通道 |
| 空投 / Airdrop | 项目免费发放代币做推广 |
| Rollup | 二层扩容方案（链下算、链上存） |

* * *

## **八、给初学者的行动建议**

1.  先装一个非托管钱包（如 MetaMask），理解"私钥自己保管"的含义。
    
2.  在测试网领测试币，亲手发一笔交易，看 Gas 和确认过程。
    
3.  读一份最小智能合约（如本仓库的 Solidity 示例），理解合约结构。
    
4.  只投入"亏得起"的资金，绝不把私钥/助记词告诉任何人。

# Web3事故急救室

## 一、第一模块：钱包与用户安全（6 大典型风险）

核心底层逻辑：**助记词 = 钱包所有权；授权 = 链上永久花钱权限；签名 = 授权凭证；断开连接≠撤销授权；链上转账不可逆**。

### 事故 01：假客服索要助记词

1.  正确操作：拒绝提供任何助记词，关闭私聊，通过官网 / 官方应用入口核验客服身份
    
2.  风险原理：完整助记词可直接恢复钱包、转走全部资产，不存在 “只给部分单词安全”
    
3.  踩坑后果：资产瞬间被盗，无法追回
    
4.  记忆口诀：谁拿到助记词，谁就是钱包主人
    
5.  预防：官方绝不会私聊索要助记词、私钥
    

### 事故 02：助记词截图误发群聊（已撤回）

1.  正确操作：安全设备新建钱包，立刻转移全部资产，旧钱包永久弃用
    
2.  风险原理：撤回消息、修改钱包登录密码都无法作废助记词，黑客仍可跨设备恢复钱包
    
3.  踩坑后果：后续存入资产持续暴露被盗风险
    
4.  记忆口诀：助记词泄露后，不是改密码，而是换钱包
    
5.  预防：绝不截图、线上传输助记词，离线纸笔单独保存
    

### 事故 03：陌生空投要求无限代币授权

1.  正确操作：核验域名、合约地址；确需授权仅开放最小必要额度
    
2.  风险原理：授权是合约永久动用你代币的权限，断开钱包连接不会自动取消授权；当前无代币不代表未来安全
    
3.  踩坑后果：后续转入的对应代币会被合约无限制转走
    
4.  记忆口诀：授权的是未来的花钱权限，不只是当前一次交易
    
5.  预防：陌生空投、零成本福利一律拒绝无限授权
    

### 事故 04：无 Gas 链下签名钓鱼

1.  正确操作：看不懂签名内容直接拒绝；核对网站、签名权限、合约对象
    
2.  风险原理：无 Gas 仅代表不产生即时链上交易，签名可被攻击者后续提交链上盗币
    
3.  踩坑后果：悄无声息被盗，无即时交易提醒难以察觉
    
4.  记忆口诀：没有 Gas，不等于没有资产权限
    
5.  预防：任何签名弹窗完整阅读权限内容，不明签名一律拒绝
    

### 事故 05：地址投毒（首尾相似伪造地址）

1.  正确操作：从可信渠道重新复制收款地址，完整核对所有字符，大额转账先小额测试
    
2.  风险原理：攻击者伪造首尾字符一致的地址混入交易记录，诱导用户复制错误地址
    
3.  踩坑后果：链上转账不可逆，资金几乎无法找回
    
4.  记忆口诀：交易记录里的地址，不一定是你以为的那个人
    
5.  预防：固定保存常用地址到钱包通讯录，不依赖历史交易复制地址
    

### 事故 06：DApp 断开连接≠安全

1.  正确操作：区块浏览器 / 钱包授权管理页面，主动撤销可疑合约的代币授权额度
    
2.  风险原理：断开连接仅关闭网页交互入口，链上授权记录永久生效
    
3.  踩坑后果：合约仍可在授权额度内划转代币
    
4.  记忆口诀：断开的是网站连接，撤销的才是花钱权限
    
5.  预防：定期批量清理无用 DApp 授权
    

## 三、第二模块：智能合约安全（5 大开发典型漏洞）

核心底层逻辑：**前端无法保障链上安全；合约执行顺序、权限、状态、外部调用均需代码强制约束**。

### 事故 07：无权限限制，任何人可增发代币 mint

1.  正确操作：合约内部增加管理员角色权限控制，设置代币总量、增发规则限制
    
2.  风险原理：链上函数可绕过前端直接调用，前端隐藏按钮无法阻止外部调用
    
3.  踩坑后果：无限增发代币，代币价值归零、经济模型彻底崩盘
    
4.  记忆口诀：前端没有按钮，不代表链上不能调用
    
5.  预防：所有铸币、提款、管理类函数强制添加访问控制
    

### 事故 08：退款函数重入攻击（先转账、再清零余额）

1.  正确操作：遵循**先更新状态、再外部转账**标准流程，搭配重入保护锁 ReentrancyGuard
    
2.  风险原理：先转账会把合约控制权交给恶意攻击合约，可重复递归调用提款函数
    
3.  踩坑后果：合约资金被反复提取，余额快速掏空
    
4.  记忆口诀：先记账，再打钱；不要先打钱，再记账
    
5.  预防：所有提款、退款逻辑严格遵循检查 - 状态更新 - 外部交互顺序
    

### 事故 09：使用 tx.origin 判断合约管理员

1.  正确操作：采用 msg.sender 判断直接调用者，搭配角色权限体系
    
2.  风险原理：tx.origin 代表交易发起账户，管理员被诱导调用恶意合约后，恶意合约调用目标合约时 tx.origin 仍为管理员，权限校验被绕过
    
3.  踩坑后果：攻击者冒用管理员身份执行合约敏感操作
    
4.  记忆口诀：tx.origin 看交易从谁开始，msg.sender 看现在谁在调用
    
5.  预防：禁止用 tx.origin 做权限校验
    

### 事故 10：外部 call 调用不校验返回结果

1.  正确操作：捕获 call 返回 success 布尔值，调用失败则回滚交易或标记业务失败
    
2.  风险原理：call 执行失败不会自动回滚业务状态，代码会默认标记订单完成
    
3.  踩坑后果：链上转账失败，但业务系统显示交易成功，造成对账、资金差错
    
4.  记忆口诀：发起调用，不代表调用成功
    
5.  预防：所有低层级外部调用必须校验执行结果
    

### 事故 11：众筹成功后仍可退款（无状态机约束）

1.  正确操作：合约设计完整状态机（进行中 / 成功 / 失败 / 结束），不同状态限制可执行函数
    
2.  风险原理：仅靠页面提示约束用户行为无效，链上业务规则必须硬编码进合约
    
3.  踩坑后果：众筹达标后用户批量退款，项目资金不足无法提款，双方权益受损
    
4.  记忆口诀：业务规则只有写进合约，才是真正的规则
    
5.  预防：任何多阶段业务（众筹、质押、挖矿）强制设计互斥状态
    

## 四、总结

1.  **用户侧安全核心**：守住助记词、严控授权、谨慎签名、核对转账地址、定期清理授权；所有官方客服不会私索要私钥 / 助记词。
    
2.  **开发侧合约安全核心**：严控函数访问权限、规范代码执行顺序、正确区分 tx.origin 与 msg.sender、校验外部调用结果、用状态机约束业务流程，不依赖前端做安全防护。
    
3.  **通用应急思维**：出现泄露 / 授权异常优先隔离资产；合约漏洞优先修复权限、状态逻辑，而非仅做前端限制；区块链交易不可逆，所有操作事前核验远重于事后补救。
    
4.  课程核心方法论：处理 Web3 安全事故优先理清**权限归属、执行逻辑、不可逆后果**，再选择处置动作，而非凭直觉操作。

# AI开发与Web3技术栈学习

## 一、AI闭环开发核心体系（重点实操）

### 1\. 核心概念：0到1与1到100开发边界

\- **0到1阶段**：产出可验证核心价值的Demo，可直接面向C端用户、投资人展示核心功能与产品价值，核心是验证想法可行性。

\- **1到100阶段**：将Demo迭代为稳定、安全、可持续迭代的生产级系统，完成产品商业化、体验优化与体系化运维。

\- **AI带来的变革**：彻底降低0到1开发门槛，无编程基础也可通过对话生成可运行Demo；大幅提升迭代、Bug修复效率，降低系统重构成本。

### 2\. AI开发核心价值与底层逻辑

\- 两大核心优势：**降门槛**（零基础可落地想法）、**提效率**（快速迭代、低成本重构）。

\- 关键认知：AI生成代码易出现方向偏差，**重构是AI开发的必要环节**，通过持续重构提升系统模块化、边界清晰度与可控性。

\- 行业趋势：AI将替代大量初级开发岗位，未来留存的核心是高级工程师、架构师，开发者需从“搬砖编码”转向“需求拆解、架构设计、结果验证”。

### 3\. AI Agent能力体系

\- 核心定位：将大模型从“问答工具”升级为**自主任务执行助手**，可自主拆解目标、感知运行环境、调用工具、落地任务并输出可验证结果。

\- 能力跃升四大核心原因：模型推理能力升级、上下文窗口扩容、工具生态完善、记忆与指令持久化（可自我进化）。

### 4\. 三大核心实操开发方法（必学）

（1）[agent.md](http://agent.md) 配置规则（AI开发核心文件）

\- 分级配置：全局配置（工具用户目录，存储个人开发偏好、长期规则）、项目级配置（项目根目录，约定架构边界、编码规范、交付标准、CI/CD流程）。

\- 核心价值：通用性强，所有AI开发工具均可识别，内容比README更精准，是项目规范化开发的核心依据。

（2）TDD测试驱动开发（AI纠错核心）

\- 执行流程：先定义测试用例与预期结果 → 测试失败后最小化功能实现 → 测试通过后按需重构。

\- 适配AI的核心作用：搭建快速纠错回路，避免迭代中破坏原有功能，解决AI代码不严谨、逻辑跑偏的核心问题，保障代码健壮性。

（3）SDD规范驱动开发（项目溯源核心）

\- 执行流程：对齐需求背景与开发意图 → 定义开发规范与交付标准 → 拆分细分任务 → 落地开发实现 → 归档留痕。

\- 工具选型：全新规范项目用**Space Kit**（重规范、体系完整）；存量迭代项目用**Open Space**（轻量、聚焦需求），团队可自定义内部模板。

### 5\. AI协作四大核心能力（开发者必备）

1\. **精准描述**：清晰输出需求场景、约束条件、验收标准，撰写高质量提示词；

2\. **任务拆解**：将复杂需求拆分为可执行、可验证、低耦合的小任务，明确依赖与风险；

3\. **结果验证**：通过手动测试、自动化测试、日志校验等多维度把关AI产出，不盲目依赖；

4\. **架构设计**：自主设计系统边界、数据流与协作规则，掌控项目顶层逻辑。

\- 认知修正：AI产出低级错误，根源是开发者领域知识缺失，而非AI不可靠，可借助AI快速补全知识短板。

## 二、Web3技术栈完整入门体系

### 1\. 区块链核心底层认知

\- 专业定义：多节点共同维护、依托密码学与共识机制的分布式账本，无需第三方信任即可达成全网交易共识。

\- 通俗核心属性：**异步、强一致性、可编程金融平台**

1\. 异步：交易提交后需等待打包、共识确认，才能获取最终结果，无即时反馈；

2\. 强一致性：全网所有节点共识后，交易、账户状态完全统一；

3\. 可编程：通过智能合约自定义链上逻辑；

4\. 金融属性：公链核心场景围绕资产流通、金融衍生玩法展开。

### 2\. Web3五大开发赛道（入门选型核心）

1\. **DApp开发（新手首选）**

\- 核心工作：Web前端工程 + 钱包对接 + 链上数据展示与交互；

\- 技术栈：TypeScript、React/Vue、Ethers.js/Web3.js；

\- 优势：可视化效果强、入门门槛低、成就感高。

2\. **智能合约开发**

\- 核心工作：编写链上资产、状态管理、协议规则的智能合约；

\- 技术栈：以太坊生态Solidity、开发框架Foundry、安全模板OpenZeppelin；

\- 特点：与DApp开发高度耦合，是Web3核心底层能力。

3\. **钱包开发（高安全要求）**

\- 核心工作：密钥管理、账户签名、防钓鱼、防重放、防恶意授权；

\- 特点：独立赛道，安全优先级极高，普通DApp项目方不会自研钱包。

4\. **节点开发（底层硬核方向）**

\- 核心工作：维护链底层P2P网络、共识机制、EVM运行、区块打包；

\- 技术栈：Go、Rust、libp2p协议、共识机制（POS/PBFT）；

\- 特点：偏底层、门槛高，适合深耕底层技术的开发者，多为公链项目方深耕方向。

5\. **安全审计（高价值垂类）**

\- 核心工作：智能合约代码审计、漏洞排查，保障项目资金安全；

\- 特点：行业刚需、价值极高，是项目上线前的最后安全防线，独立垂类赛道。

### 3\. 所有Web3赛道通用基础能力

\- 密码学体系：哈希算法（KECCAK256、SHA256）、签名规范、非对称加密、安全随机数；

\- 密钥与钱包协议：BIP39/BIP32/BIP44协议、离线密钥管理、验签、地址派生；

\- 区块链原理：账户状态、交易、区块、共识机制、EVM执行环境、RPC开发接口。

### 4\. 以太坊系统架构与交易全流程

（1）五层分层架构

从上至下：应用层（DeFi、钱包、索引器）→ 接口层（JSON RPC、WebSocket）→ 执行层（EVM运行、状态更新）→ 共识层（合法链判定、区块共识）→ 基础协议层（P2P网络、区块数据结构）。

\- 新手开发者仅需对接接口层，无需感知底层网络。

（2）交易完整生命周期

DApp编码构造交易 → 估算Gas、构建交易 → 钱包签名 → RPC广播至节点 → 交易池缓存、P2P全网扩散 → 节点打包区块、执行交易 → 更新EVM状态 → 全网共识确认最终性 → DApp读取交易回执与结果。

## **二、智能合约（Solidity）知识点**

### **2.1 合约骨架**

```
 // SPDX-License-Identifier: MIT
 pragma solidity ^0.8.20;
 ​
 struct Todo {
     uint256 id;
     string content;
     uint256 deadline;   // Unix 时间戳（秒）
     bool completed;
 }
 ​
 mapping(address => Todo[]) private userTodos;
```

-   `pragma solidity ^0.8.20`：限定编译器版本，避免新版破坏行为。
    
-   `SPDX-License-Identifier`：源码许可声明，方便他人复用。
    

### **2.2 四个公开函数**

| 函数 | 关键逻辑 | 学习点 |
| --- | --- | --- |
| addTodo(string content, uint256 deadline) | require(deadline >= block.timestamp)；newId = myTodos.length；push | 时间戳校验防止过去时间；用数组长度做自增 id |
| getMyTodos() external view | return userTodos[msg.sender] | view 不消耗 gas；只返回调用者自己的数据 |
| completeTodo(uint256 todoId) | 越界校验后 completed = true | 只能改自己的数据 |
| deleteTodo(uint256 todoId) | 末位覆盖 + pop() | gas 优化删除 |

### **2.3 设计要点（核心价值）**

NaN. **隐私隔离靠** `msg.sender`：所有写操作都以 `msg.sender` 为身份，用户只能操作自己地址下的数据，看不到别人的待办。

NaN. `private` **变量的语义**：`userTodos` 标为 `private` 仅表示「外部合约/直接读取不可见」，但通过 `getMyTodos()` 仍可读取——它与「保密」无关，是访问控制手段。

NaN. **删除为何用「末位覆盖 + pop」**：Solidity 动态数组若从中间删除并前移元素，gas 成本随数组长度线性增长；用最后一项覆盖目标项再 `pop()`，只需一次写操作，省 gas。代价是：被移动的末项 `id` 会变为被删项的下标，需在覆盖后修正 `id` 以保持 `id == 下标`。

NaN. **时间用 Unix 时间戳**：`block.timestamp` 与前端 `Date.now()` 单位均为秒（前端注意 `Date.now()` 是毫秒，需 `/1000`）。

## **三、前端 DApp（ethers.js v6）知识点**

### **3.1 单文件架构**

-   一个 `index.html` 内含 HTML + CSS + JS，无构建步骤，双击即可在浏览器打开。
    
-   ethers.js v6 通过 CDN 引入；v6 与 v5 API 差异较大（`BrowserProvider` 替代 `Web3Provider`，`ethers.isAddress` 等）。
    
-   合约地址与链 ID 硬编码于文件顶部常量，普通用户连接钱包即用，无需输入地址。
    

```
 const CONTRACT_ADDRESS = "0x3bEd71932240266608Ef0e84bD9f96796261E7C9";

```

### **3.2 钱包连接（MetaMask）**

```
 const provider = new ethers.BrowserProvider(window.ethereum);
 const signer = await provider.getSigner();
```

-   `window.ethereum` 是 MetaMask 注入的对象；未安装时需引导用户安装。
    
-   监听 `accountsChanged` / `chainChanged` 事件以刷新界面状态。
    

### **3.3 读操作与写操作**

-   **读（view）**：`await contract.getMyTodos()` 直接返回数据，不弹钱包、不消耗 gas。
    
-   **写（state-changing）**：`const tx = await contract.addTodo(...)` 返回交易对象，`await tx.wait()` 等待上链确认后再刷新界面。
    

### **3.4 状态展示逻辑**

-   **已逾期**：`deadline < 当前时间` 且未完成。
    
-   **即将到期 / 已完成**：分别按时间窗口与 `completed` 标记。
    
-   统计卡片（全部 / 进行中 / 已完成 / 已逾期）由前端计算，不依赖链上聚合。

1\. **基础操作规范**：链上交互、钱包连接必须选择**测试网**而非主网，避免操作失误；所有链上转账、合约部署行为均可通过区块浏览器查询交易哈希，验证执行结果。

### （二）链上产品设计思维

1\. **核心设计逻辑**：链上产品的核心竞争力不是简单的数据上链，而是**场景化包装、降低用户认知门槛**，提升普通用户交互意愿。

2\. **数据分层存储**：核心用户数据（宠物状态、打卡记录、成长等级）上链存证、公开可溯源；图片、动画、前端特效等非核心资源存放于本地assets文件夹，大幅降低链上存储成本。

3\. **功能取舍原则**：优先将用户身份、链上行为、成长状态等核心数据上链；复杂审核、社交、特效等功能先在前端迭代，避免前期开发复杂度过高。

4\. **优质案例参考**：Monad宠物喂养Demo，将枯燥的链上打卡操作包装为宠物喂养、升级、连续打卡激励、排行榜等趣味场景，适配新人入门，可用于社区用户增长运营。

3\. **开发避坑技巧**：采用**空文件迭代、分模块开发**，不在原有问题代码上反复修改；跨模块特效、独立功能单独开发后再整合，大幅降低调试成本。

2\. **四重防控机制**：有限授权、预算上限管控、关键操作人工确认、交易全程可审计。

# AI Agent安全+Web3线上共学分享会总结

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/monad-builder-camp/main/assets/Tuskiand/images/2026-07-09-1783604121362-image.png)

### （一）AI Agent核心安全痛点

AI Agent安全攻防已从传统自动化转向自主化，其核心风险被定义为“无法审计的自主性”，也是AI时代的核心安全隐患。区别于常规工具，AI Agent具备自主决策能力，风险无明显前兆、难以提前干预，同时模型概率性输出的特性易产生内容幻觉、脱离用户指令自主执行操作，极易引发失控事故。AI在大幅提升工作效率的同时，也会成倍放大操作失误与安全风险。

### （二）四大类典型高风险场景及真实案例

1\. **过度授权风险**：为AI Agent开放过高权限是最常见风险。包括公共群聊Agent被他人诱导窃取本地密码文件、用户终止指令无效导致Agent持续误删邮件、海外SaaS平台Agent9秒越权删除全部生产及备份数据等事故，凸显无管控高权限的致命危害。

2\. **提示词注入风险**：攻击者通过隐藏指令、编码绕过、伪装教程等方式诱导Agent执行恶意操作，包括诱导链上转账、植入恶意代码包、摩斯密码绕过校验窃取加密资产等，隐蔽性极强，普通用户难以识别。

3\. **恶意工具风险**：主流AI工具平台存在大量风险插件，抽检显示超两成工具存在高危、严重漏洞，部分攻击者通过刷量登顶榜单，以高信任背书诱导用户下载，窃取私钥、API密钥、服务器凭证等核心数据。

4\. **底层不可控风险**：涵盖Agent网关高危漏洞、AI幻觉引发的DOS攻击、第三方依赖包供应链投毒、平台业务逻辑漏洞等底层问题，波及范围广，包括OpenAI、Meta等主流平台均曾受相关风险影响。

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/monad-builder-camp/main/assets/Tuskiand/images/2026-07-09-1783604009743-image.png)

### （三）通用安全防护规范与产品能力

1\. **基础防护规范**：严格遵循最小权限原则，动态梳理精简Agent权限；资金转账、数据删除等高危操作强制人工二次校验；对所有外部输入、工具保持零信任，仅使用官方认证工具；锁定第三方依赖版本，及时修复系统漏洞，规避供应链攻击。

2\. **专业防护产品（Agent Guard）**：端到端AI安全防护平台，具备实时高危操作拦截、恶意工具自动扫描、全维度安全体检、全球威胁情报匹配、操作全链路溯源等能力，支持自定义安全策略，可形成风险发现、验证、防护、治理的闭环体系。

3\. **行业方案局限认知**：多Agent交叉校验、投票风控的方案存在落地瓶颈，模型上下文遗忘、指令权重不足会导致校验机制失效，无法实现实时防护，底层模型特性优化才是核心解决方向。

## 二、Web3线上共学分享核心内容总结

本次Web3主题分享聚焦行业实战与学生发展刚需，从就业求职、链上安全、本校专业优势、产学研落地项目四个维度展开，打破了行业认知误区，明确了区块链+AI复合发展方向。

### （一）Web3行业求职与就业实战经验

1\. **求职渠道优先级**：内推 > 线下峰会/黑客松人脉 > 线上社群/平台投递。TG求职群可获取运营岗岗位，但诈骗高发需警惕；自主搭建推特、抖音等Web3垂直自媒体，可作为简历核心加分项。线下行业活动是当前低迷行情下获取实习、内推机会的最优路径。

2\. **各岗位求职特点**：技术岗需熟练背诵区块链八股文，行业裁员波动大，开发岗稳定性较弱；运营岗无大量背诵压力，可从社区志愿者起步积累实战经历；研究岗门槛较高，后续将由老师专项分享。

3\. **就业现状与简历重点**：本校区块链专业应届毕业生对口就业率极低，多数学生选择转行、考公；行业求职不看重论文，重点考察自媒体运营、线下活动、项目实战、实习经历。行业实习薪资可观，远程实习是主流形式，人脉内推是后期求职核心渠道。

### （二）Web3链上安全与反诈科普

行业链上被盗、诈骗案例频发，新人安全意识薄弱是主要问题。核心诈骗套路为钓鱼网站仿制官方空投、活动页面，诱导用户授权钱包，直接盗取全部链上资产。高危风险行为包括点击陌生私信链接、访问未知空投网站，同时个人人脸、邮箱、交易所身份信息存在AI伪造、泄露风险，需时刻保持警惕，杜绝陌生操作。

### （三）AI+Web3落地项目：喵星智能体实战解析

该项目是AI技术与区块链结合的典型落地案例，分工清晰、技术链路完整、风控体系完善，具备极强的学习和复用价值。

1\. **团队分工**：由师生联合开发，核心分工明确，苏生、花花负责智能合约与前端开发，守琼负责项目路演，叶老师担任项目经理统筹整体。

2\. **核心功能与技术链路**：通过设备采集猫咪体态、叫声、心率数据，经降噪切片、梅尔图谱转换、声纹特征提取后，依托RAG向量数据库和LLM大模型判定宠物情绪与需求，设置75分置信阈值，达标后可自动触发链上资金履约、宠物用品下单，并推送人工审批通知。

3\. **区块链价值与风控设计**：通过链上资金存证、多层合约校验、预言机对接，实现交易透明可溯源、自动履约，同时优化链上Gas费用；内置消费上限、健康防沉迷机制，规避不合理消费，且该AI识别逻辑可迁移至老人、儿童行为监测场景，拓展性极强。

得到代码以后：

-   不急着部署
    
-   先阅读
    

这一阶段只需要保存：

-   AI 输出的完整 Solidity 合约
    

### 第三阶段：理解 AI 输出

不要只复制代码，而是让 AI 再解释一遍。

可以继续提问：

```
请逐行解释这个 Solidity 合约。

1. pragma 的作用

4. Todo[] 为什么使用数组

6. toggleCompleted 做了什么
7. getTodo 返回什么
8. memory、storage 的区别
9. view 函数为什么不能修改状态
```

检查1：是否可以编译

检查2：函数是否符合需求

检查3：权限问题

检查4：是否存在复杂逻辑

检查5：命名是否清晰
