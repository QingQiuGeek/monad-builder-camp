# 2026-07-16

今天主要学习了 AI Agent 生态中的几个基础概念，包括 Agent、Skill、MCP 以及 [AGENTS.md](http://AGENTS.md)。随着 AI 能力不断增强，未来很多工作可能不再是人与多个工具之间的交互，而是通过一个 Agent 自动调用不同能力完成任务。因此理解这些基础概念对于后续学习 Web3 + AI 方向非常重要。

**Agent（智能体）**：相比传统 AI 只能进行问答，Agent 更强调自主完成任务的能力。用户提出目标后，Agent 可以主动拆解任务、调用工具、执行操作并输出结果。例如，一个 Web3 研究 Agent 可以自动获取市场数据、分析项目动态、整理资料并生成研究报告。简单来说，Agent 就像一个能够执行工作的数字员工。

**Skill（技能模块）**：Skill 可以理解为赋予 Agent 专业能力的组件。一个基础 Agent 可能只能完成简单交流，但通过不同 Skill 可以具备更多能力，例如 Crypto Research Skill 可以帮助完成项目研究，Data Analysis Skill 可以进行数据分析，Solidity Skill 可以辅助智能合约开发。Skill 解决的是“Agent 能做什么”的问题。

**MCP（Model Context Protocol）**：MCP 是一种帮助 AI 连接外部工具和数据的标准协议，可以理解为 AI 与外部世界之间的连接接口。通过 MCP，Agent 可以更加方便地调用外部资源，例如连接 Dune 获取链上数据，连接 Notion 管理资料，连接 Github 查看代码等。MCP 解决的是“Agent 如何获取外部信息和使用工具”的问题。

此外，还了解了 [**AGENTS.md**](http://AGENTS.md) 文件。它本质上是一个 Markdown 格式的配置文件，用于告诉 Agent 在某个项目中应该如何工作，包括项目背景、技术栈、开发规范以及注意事项。例如在 Solidity 项目中，可以通过 [AGENTS.md](http://AGENTS.md) 规定合约版本、安全要求以及代码规范，让 Agent 在后续协作过程中保持一致。

通过学习发现，这几个概念之间存在明显区别：

-   Agent 是负责执行任务的主体；
    
-   Skill 是 Agent 掌握的能力；
    
-   MCP 是连接外部工具和数据的方法；
    
-   [AGENTS.md](http://AGENTS.md) 是指导 Agent 工作的规则说明。
    

未来如果构建一个 Web3 Agent，例如自动进行市场研究、追踪项目动态或者辅助开发智能合约，就需要结合这些模块：通过 [AGENTS.md](http://AGENTS.md) 定义工作流程，通过 Skill 提供专业能力，通过 MCP 连接链上数据和外部工具，最终让 Agent 完成完整的任务闭环。

# 2026-07-15
<!-- DAILY_CHECKIN_2026-07-15_START -->




今天主要学习了 Web3 中一些常见的安全问题。相比传统互联网，Web3 很多操作都需要通过钱包授权或者签名完成，而且链上交易通常不可逆，所以在点击确认之前理解自己到底授权了什么非常重要。

首先是代币授权问题。有些网站在领取空投或者连接 DApp 时，会要求用户授权 USDT 等代币的 Spending Cap。如果直接选择 Unlimited，就意味着这个合约可能拥有长期、无限额度使用代币的权限。因此不能因为钱包里暂时没有 USDT 就随意授权。比较安全的方式是先检查网站和合约，只授权实际需要的额度。如果已经对可疑合约进行了授权，即使断开钱包连接也不能真正取消权限，还需要检查链上授权并主动撤销。

今天还了解到，“免费签名”不代表没有风险。有些签名确实不需要 Gas，也不会立即产生链上交易，但签名本身可能包含授权、Permit 等信息。因此在钱包弹出签名请求时，需要检查签名内容、涉及的代币、额度和对象。如果完全看不懂签名内容，最好先拒绝，而不是因为“不花 Gas”就直接确认。

另一个比较有意思的风险是地址投毒。攻击者可能生成一个开头和结尾与常用地址非常相似的钱包地址，并通过小额交易让这个地址出现在交易记录中。如果转账时只检查地址前几位和后几位，很容易复制到错误地址。因此大额转账前应该从可信来源重新复制完整地址，并核对完整地址，必要时先进行小额测试。

智能合约本身也可能存在安全问题。例如代币合约中的 `mint(to, amount)` 函数，如果任何地址都能调用，就意味着所有人都可以增发代币。解决这种问题不能只是在前端隐藏按钮，而应该直接在合约层增加权限控制和供应量限制。这让我更直观地理解了一个概念：前端限制不等于链上安全，真正的权限必须写进智能合约。

此外还学习了重入攻击。如果退款函数先向用户转 ETH，再修改用户余额，攻击者可能通过恶意合约在余额清零之前重复调用退款函数。比较基础的防护思路是遵循“Checks-Effects-Interactions”，先检查条件、更新状态，再进行外部转账，同时可以加入重入保护。

# 2026-07-14

今天学习了 Web3 中一个比较新的应用方向——**DeSci（Decentralized Science，去中心化科学）**。DeSci 是区块链技术与科研领域结合产生的新模式，希望通过 Web3 的去中心化机制，改善传统科研体系中存在的资金获取困难、数据共享不足、科研贡献难以衡量等问题。

传统科研体系通常由政府、机构或企业提供资金，科研人员在实验室开展研究，最终通过论文发表和机构认证获得认可。这种模式虽然推动了科学发展，但也存在一些限制。例如，科研资金往往集中在少数机构，年轻研究者或者小型团队较难获得资源；科研数据通常被封闭管理，不同团队之间共享困难；同时，科研贡献主要通过论文和期刊评价，一些数据贡献、代码贡献或者实验支持等价值难以被准确记录。

DeSci 希望利用区块链、智能合约和 DAO 等技术，建立更加开放透明的科研协作体系。

其中，**DAO（去中心化自治组织）** 是 DeSci 的重要组成部分。传统科研中，研究方向和资金分配通常由少数机构决定，而 DeSci 可以通过 DAO 让研究者、投资者、数据贡献者等共同参与决策。例如，一个科研项目可以通过社区投票确定研究方向，并由社区共同支持科研资金。

在科研融资方面，DeSci 希望改变传统基金申请模式。通过链上机制，科研项目可以直接连接全球支持者，让更多有价值的研究获得资金支持，提高科研资源配置效率。

在科研数据方面，DeSci 关注数据的确权和价值分配。传统科研数据可能存储在机构内部，贡献者很难获得长期收益。而通过区块链，可以记录数据来源、使用权限以及贡献者信息，让科研数据更加透明，同时探索数据贡献者获得激励的可能性。

此外，DeSci 也关注科研成果和知识产权管理。传统模式下，科研成果通常经过专利申请、企业转化等流程，贡献者之间的收益分配较为复杂。DeSci 可以利用智能合约记录科研成果归属，并通过链上机制实现更加透明的权益分配。

DeSci 与 RWA（Real World Assets）有一定相似之处，但两者关注点不同。RWA 主要是将现实资产，例如房地产、债券、艺术品等映射到区块链，提高资产流动性；而 DeSci 更关注科研体系本身，希望将科研数据、知识产权和协作过程进行链上化。

同时，DeSci 也可能与 AI 产生更深结合。未来 AI 可以帮助科研人员分析数据、设计实验方案、发现新的研究方向，而 DeSci 可以通过区块链记录数据来源、贡献关系和收益分配，形成 AI + Web3 + 科研的新生态。

### Web3 不同赛道的理解与区别

###   
1\. 基础设施（Infrastructure）

基础设施是 Web3 的底层支撑，类似传统互联网中的云计算、服务器和网络协议。

这一赛道主要解决的是区块链本身的问题，例如：

-   更高的交易速度；
    
-   更低的 Gas 成本；
    
-   更好的开发体验；
    
-   不同链之间的数据和资产互通。
    

代表方向包括：

-   Layer1 公链：提供独立的区块链网络，例如 Ethereum、Monad 等；
    
-   Layer2：通过扩展方案提高主链效率；
    
-   跨链协议：帮助不同区块链之间进行通信。
    

基础设施赛道更偏技术，需要关注共识机制、虚拟机、开发工具以及生态建设。

### 2\. DeFi（去中心化金融）

DeFi 是目前 Web3 中最成熟的应用方向之一，目标是利用智能合约重新构建传统金融服务。

主要包括：

-   去中心化交易所（DEX）：用户无需中心化平台即可交易资产；
    
-   借贷协议：用户可以抵押资产获得贷款；
    
-   收益协议：通过流动性提供获得收益。
    

相比传统金融，DeFi 的核心特点是：

-   无需银行等中介；
    
-   规则由智能合约自动执行；
    
-   所有交易公开透明。
    

但同时也面临智能合约风险、流动性风险以及监管问题。

### 3\. NFT 与数字资产

NFT 最初主要用于数字收藏品，但目前正在向更多应用场景发展。

除了艺术品之外，还包括：

-   游戏资产；
    
-   身份凭证；
    
-   会员权益；
    
-   数字门票。
    

NFT 的核心价值不只是图片本身，而是利用区块链实现：

-   数字资产所有权确认；
    
-   可验证的真实性；
    
-   用户资产跨平台流通。
    

未来 NFT 可能更多作为数字身份和权益系统的一部分，而不仅仅是收藏品。

### 4\. GameFi（区块链游戏）

GameFi 是游戏与金融结合的方向。

传统游戏中，玩家获得的装备、角色等资产通常属于游戏平台，而 Web3 游戏希望让玩家真正拥有游戏资产。

核心特点：

-   游戏资产链上化；
    
-   玩家可以交易自己的资产；
    
-   通过游戏行为获得奖励。
    

但目前 GameFi 仍面临：

-   游戏体验不足；
    
-   经济模型容易失衡；
    
-   用户增长依赖激励。
    

未来更重要的方向可能是“游戏体验优先”，而不是单纯 Play to Earn。

### 5\. RWA（Real World Assets，现实资产上链）

RWA 是近年来受到关注的新方向，核心是将现实世界资产通过区块链进行数字化。

例如：

-   国债；
    
-   房地产；
    
-   黄金；
    
-   艺术品；
    
-   企业债券。
    

RWA 的价值在于连接传统金融和 Web3：

传统资产拥有稳定价值，但流动性较低；

区块链可以提供：

-   更高效率的交易；
    
-   全球化流通；
    
-   自动化结算。
    

因此 RWA 被认为可能是未来 Web3 与传统金融结合的重要方向。

### 6\. AI + Web3

AI 与 Web3 是目前新的融合方向。

主要探索：

-   AI Agent 自主执行任务；
    
-   Agent 之间进行链上支付；
    
-   去中心化计算资源；
    
-   AI 模型和数据资产确权。
    

例如未来 AI Agent 可以：

-   自动管理资金；
    
-   自动购买服务；
    
-   自动完成链上交易。
    

Web3 可以提供 AI 所需要的身份、支付和资产管理能力。

今天主要完成了第一个 Solidity 智能合约从编写、编译到部署以及链上交互的完整流程。相比之前学习钱包创建、测试网络连接等基础操作，今天真正体验了一个 Web3 应用从代码到链上的过程。

首先，我使用 Remix 作为 Solidity 合约开发工具，将之前生成的 Onchain Todo 合约代码进行编译。在编译过程中，需要选择对应的 Solidity 版本，并检查代码是否存在语法错误。通过这一步，我理解了智能合约在部署之前需要先经过编译，将 Solidity 代码转换成区块链能够执行的字节码，同时生成 ABI（应用程序二进制接口），方便前端或者其他程序调用合约。

完成编译后，我连接 MetaMask 钱包，并切换到 Monad Testnet 测试网络。随后通过 Remix 的 Deploy & Run 模块进行合约部署。在部署过程中，需要确认钱包连接、网络环境以及 Gas 费用，最终通过钱包确认交易，将合约发布到区块链上。

部署完成后，我学习了如何查看链上部署结果，包括：

-   合约地址（Contract Address）：代表已经部署在链上的智能合约位置，后续所有交互都会通过这个地址进行；
    
-   部署交易 Hash（Transaction Hash）：代表这一次部署行为在区块链上的唯一交易记录，可以通过区块浏览器查询详细信息；
    
-   合约交互 Transaction Hash：当调用合约函数修改链上状态时，会产生新的交易记录，与部署交易 Hash 不同。
    

之后，我进一步尝试调用合约功能，例如读取 Todo 数据以及执行写入操作。通过实际操作，我理解了智能合约中的 Read Function 和 Write Function 的区别：

-   Read Function 只读取链上数据，不会改变状态，因此通常不需要消耗 Gas；
    
-   Write Function 会修改链上状态，需要发送交易并支付 Gas 费用。
    

通过今天的实践，我第一次完整经历了一个智能合约生命周期：

**编写 Solidity 合约 → Remix 编译 → 连接钱包 → 部署到测试网络 → 获取合约地址 → 调用合约函数 → 查看链上交易记录**

在检查过程中，我也对 AI 输出进行了自己的判断。例如，对于一些命名不够清晰的变量，我进行了调整，使代码更加容易阅读；同时确认没有加入当前阶段暂时不需要的复杂功能，保持整个合约结构尽量简单，方便以后继续扩展更多功能。

在学习方法方面，分享中提到了利用 AI 学习 Web3 的方式。相比直接让 AI 输出一份完整教程，更有效的方法是让 AI 帮助建立学习机制，例如先分析自己的知识短板，再根据实际情况调整学习路径。AI 不只是一个回答问题的工具，更像一个辅助学习的伙伴，可以帮助拆解概念、提出问题、检查理解程度。通过这种方式，学习者可以从实际问题出发，而不是停留在理论阅读。例如在学习链上知识时，可以结合具体操作理解 Gas、钱包权限、交易签名、智能合约等概念。在实践过程中，也能够发现自己真正不理解的地方，而不是单纯接受 AI 给出的答案。

今天分享中重点介绍了一个从想法到 Demo 的完整 Vibe Coding 流程。整个过程大致包括：

**想法构思 → 明确项目方向 → AI 辅助生成界面和代码 → 测试修改 → 链上部署 → 完成 Demo**

其中，AI 在开发过程中承担了辅助角色，例如帮助生成前端页面、优化交互设计、编写部分代码以及解决开发过程中遇到的问题。但 AI 并不是简单地“一键生成产品”，开发者仍然需要负责方向判断、需求设计以及最终验证。

在 Web3 Demo 开发案例中，分享者展示了一个基于 Monad Testnet 的链上 Buddy 项目。这个项目通过“养成宠物”的形式降低 Web3 使用门槛，让用户通过完成链上任务获得成长值和状态变化。从产品设计角度来看，这个项目比较有启发性。传统 Web3 应用往往直接展示钱包、交易、Gas、智能合约等技术元素，对于新用户来说学习成本较高。而通过游戏化设计，将复杂的链上操作转化为“培养一个 Buddy”的简单行为，可以提升用户参与意愿。

在技术实现方面，该 Demo 不只是一个普通前端页面，而是真正连接到了 Monad Testnet：

-   使用 MetaMask 连接钱包；
    
-   通过 Solidity 编写并部署智能合约；
    
-   使用 Remix 完成合约测试和部署；
    
-   前端通过 wagmi 等工具调用合约；
    
-   用户操作会产生真实链上交易，并同步更新状态。
    

通过这个案例，我第一次比较完整地理解了一个 Web3 产品从设计到落地的流程。一个链上应用不仅包含前端页面，还需要考虑钱包连接、智能合约逻辑、链上数据存储、用户交互以及交易反馈等多个环节。

# 2026-07-09
<!-- DAILY_CHECKIN_2026-07-09_START -->











在 AI Agent 部分，随着 AI 从简单的信息生成工具发展到可以自主调用工具、执行任务后，带来的新的安全挑战。例如 Agent 可能因为权限配置过高，执行超出预期范围的操作；也可能受到 Prompt Injection（提示词注入）攻击，通过隐藏指令诱导 AI 执行危险行为。因此，在设计 AI Agent 产品时，需要关注权限控制、人工审核机制以及运行环境安全。

分享会中提到了一些重要的安全设计原则，包括最小权限原则（只给予 Agent 完成任务所需的权限）、Human-in-the-Loop（关键操作需要人工确认）以及安全检测工具对 Agent、Skills 和插件进行风险扫描。

在 Bein 助教分享的黑客松项目中，我对项目整体流程图的设计产生了很大兴趣。相比传统互联网产品，链上产品需要同时考虑用户层、前端交互、链下服务、链上智能合约以及区块链网络之间的协作。例如用户发起操作后，需要经过参数处理、交易签名、调用智能合约、链上执行、状态更新以及事件监听等多个环节。整个流程体现了链上产品“用户行为 → 链下逻辑 → 链上验证 → 状态同步”的完整闭环。

Web3 产品设计不仅仅是在页面上增加一个功能，而是需要理解整个链上业务流程，包括钱包连接、交易构造、智能合约逻辑、链上数据存储以及用户体验之间的关系。未来如果参与 Web3 产品开发或项目分析，需要从完整架构角度理解产品，而不仅仅关注前端展示。

# 2026-07-08
<!-- DAILY_CHECKIN_2026-07-08_START -->












今天主要学习了一个 AI Agent 支付项目——FluxA，对 AI Agent 为什么需要新的支付基础设施有了初步认识。之前一直觉得 AI Agent 只是一个聊天工具，但今天了解到，未来 AI Agent 不仅可以回答问题，还能够自主完成订酒店、调用付费 API、购买服务等任务。当 AI Agent 开始拥有执行能力后，支付就会成为它必须解决的问题，而传统支付方式需要人工确认、输入银行卡信息等流程，并不适合 AI Agent 自主完成任务，因此也催生了专门面向 AI Agent 的支付基础设施。

FluxA 希望解决的就是这个问题，它定位为 AI-native payment infrastructure，让 AI Agent 能够安全、自主地完成支付，而不用直接接触用户真实的银行卡信息。我了解到，它支持 Agent 与 Agent、Agent 与商家、Agent 与工具之间的支付场景，比如 AI 自动调用付费 API、订阅 SaaS 服务，甚至未来 Agent 之间也可以完成协作和价值交换。这让我意识到，未来 AI Agent 不只是工具，也可能成为真正参与经济活动的主体。

产品设计中让我印象比较深的是 Co-wallet 的概念。它不是完全把钱包交给 AI，而是由用户先设定预算、权限和规则，AI Agent 只能在这些范围内执行支付任务，用户依然保留最终控制权。相比传统钱包，这种方式既提高了自动化程度，也兼顾了资金安全，让我觉得这种设计思路比较符合未来 AI Agent 的发展方向。

另外还了解到 AgentCard 的设计，它可以为 AI Agent 生成一次性的虚拟银行卡，在需要使用传统支付网络时完成支付，交易结束后卡片自动失效，这样既能够兼容现有的 Visa、MasterCard 等支付体系，也减少了银行卡信息泄露的风险。这让我感受到，很多 Web3 项目并不是完全推翻现有系统，而是在原有基础上寻找更好的连接方式。

今天最大的收获是对 AI Agent 和 Web3 结合的应用场景有了更具体的认识。以前更多关注区块链和加密资产本身，而今天开始意识到，未来 Web3 也可能成为 AI Agent 的基础设施之一，让 AI 能够真正拥有身份、钱包和支付能力，并独立完成更多复杂任务。后续也希望继续学习 AI Agent、支付协议以及 Web3 基础设施相关的内容，进一步理解整个生态是如何运作的。

# 2026-07-07
<!-- DAILY_CHECKIN_2026-07-07_START -->













**智能合约**：部署在区块链上的代码，部署需消耗 Gas（测试网可用测试代币），部署操作本质上是一笔链上交易

**钱包私钥 / 助记词**：代表钱包所有权，泄露即等同于失去资产控制权，与登录密码无关

**ABI 文件**：智能合约编译产物，相当于接口文档，前端通过它调用合约方法

**环境变量文件（.env）**：存放私钥、RPC 地址等敏感配置

**钱包与网络：**使用支持"开发者模式"的钱包，开启后所有链自动切换为测试网，避免误操作主网；钱包不绑定特定区块链，可手动切换主网 / 二层网络（如 OP、Base、Arbitrum）；开发钱包与存储真实资产的钱包必须隔离

**AI 编程部署合约流程：**

1.  用自然语言描述需求 → AI 自动生成合约代码
    
2.  AI 安装依赖、编写部署脚本
    
3.  提供 `.env`（含私钥 + 测试网 RPC 地址）→ AI 执行部署
    
4.  在区块链浏览器中验证部署交易
