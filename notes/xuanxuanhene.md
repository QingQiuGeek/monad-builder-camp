**学习主题：** 了解 GitHub 开源协作流程，学习 Moss 项目，并完成第一次真实开源贡献。

## 一、今日学习内容

今天主要围绕 GitHub 开源项目 **Moss** 展开学习。我首先阅读了项目的 README 和官方文档，了解了 Moss 的定位和设计目标。Moss 是一个面向 AI Agent 的开源框架，它不仅能够调用大语言模型，还能够结合工具、技能（Skills）和工作流（Workflow）完成复杂任务，让 AI 从单纯的对话助手发展为能够执行实际工作的智能代理。

随后，我继续学习了一个成熟 GitHub 项目的组织方式，包括 README、Docs、Issues、Pull Requests 和 Discussions 等模块。通过浏览项目结构，我了解到 README 用于帮助开发者快速了解项目，Docs 提供详细文档，Issues 用于反馈 Bug 或提出功能建议，Pull Requests 用于提交代码修改，而 Discussions 更适合进行社区交流和经验分享。一个优秀的开源项目不仅需要高质量的代码，也需要完善的文档和良好的社区协作机制。

## 二、第一次参与 GitHub 开源协作

今天最大的收获是完成了自己的第一次 GitHub 开源贡献。

-   Fork Moss 项目到自己的 GitHub 仓库；
    
-   阅读 README 和相关文档；
    
-   分析项目可以改进的地方；
    
-   编写并提交 Documentation Improvement Issue；
    
-   成功在原项目提交了 **GitHub Issue #16**。
    

虽然这次贡献没有修改代码，但我了解到文档完善、教程编写、问题反馈等同样属于开源贡献的重要组成部分。作为一名 Ops Builder，这种贡献方式更符合自己的角色定位，也能够真正帮助项目改善开发者体验。

## 三、学习收获

通过今天的学习，我对 GitHub 的开源协作流程有了更加直观的认识。以前认为参与开源必须具备较强的编程能力，而现在发现，社区运营、文档维护、教程编写、用户反馈整理等工作同样是开源生态的重要组成部分。

## 一、学习背景

在 Web3 项目中，并不是只有开发者才能参与项目建设。一个完整的 Web3 产品通常需要研究（Research）、运营（Ops）和开发（Dev）三类成员共同合作，每个岗位都承担着不同的职责。通过今天的学习，我对三种职业方向有了更加清晰的认识，也开始思考自己未来更适合的发展方向。

* * *

## 二、今日学习内容

### （一）Research Builder——项目的分析者

Research Builder 的主要任务是对 Web3 项目进行深入研究，为团队提供决策依据。他们需要阅读大量官方文档、白皮书以及治理提案，分析项目的发展现状、技术特点和市场竞争力。同时还需要借助 DeFiLlama、Dune 等工具分析链上数据，撰写研究报告，为团队提供有价值的信息。

我认为 Research 岗位最大的特点是需要较强的信息整理能力和逻辑分析能力，对行业保持持续关注。

* * *

### （二）Ops Builder——社区的连接者

Ops Builder 更多负责项目与用户之间的沟通。他们不仅需要维护 Discord、Telegram 等社区，还需要策划线上活动、AMA、X Space 等内容，提高项目的影响力和用户活跃度。

优秀的运营人员不仅要具备良好的沟通能力，还要能够分析社区反馈，不断优化运营策略，为项目带来持续增长。

* * *

### （三）Dev Builder——产品的创造者

Dev Builder 是将想法真正落地的人。他们负责智能合约开发、DApp 编写、前后端开发以及自动化工具实现，将产品需求转换成能够运行的软件。

随着 AI Coding 的快速发展，开发者已经不仅仅依赖传统编程，而是逐渐形成“AI + 人工”的协同开发模式，提高开发效率。

* * *

## 三、我的职业方向分析

综合自己的兴趣、学习经历以及未来规划，我决定选择 **Dev Builder** 作为本阶段重点发展的方向。

选择 Dev 的主要原因有以下几点：

-   对编程开发比较感兴趣，希望学习更多 Web3 技术。
    
-   希望掌握 Solidity 智能合约开发，为以后参与真实项目打下基础。
    
-   AI 编程工具的发展能够帮助开发者提高效率，也符合未来的发展趋势。
    
-   团队项目中开发岗位具有较强的实践性，可以快速积累经验。
    

当然，我也认识到，开发并不是只会写代码，还需要理解业务需求，与 Research 和 Ops 成员保持良好的沟通协作。

* * *

## 四、今天最大的收获

今天最大的收获是理解了 Web3 团队并不是单纯依靠程序员完成所有工作，而是多个岗位共同协作。

Research 提供方向，Ops 负责连接用户，Dev 实现产品功能，这三者缺一不可。一个优秀的 Web3 Builder 不仅要具备自己的专业能力，还要了解其他岗位的工作内容，才能更好地进行团队合作。

此外，我还了解到学习 Web3 不应盲目追求技术，而应该先明确自己的职业定位，再制定针对性的学习路线，这样能够提高学习效率。

* * *

## 五、AI 协作心得

今天在学习过程中，我借助 AI 帮助整理学习资料、归纳三类岗位的职责，并生成学习计划，大大节省了查找资料和整理内容的时间。

不过，我也发现 AI 提供的内容仍需要结合官方文档进行验证。例如，不同项目的开发流程、技术细节以及协议设计可能会有所不同，因此最终的判断仍然需要自己完成。AI 更适合作为学习助手，而不是完全代替自己的思考。

活动首先介绍了 AI Agent 面临的几类典型高危案例。例如，**Prompt Injection（提示词注入攻击）会诱导 Agent 忽略原有指令，从而执行攻击者指定的恶意任务；工具滥用可能导致 Agent 被诱导调用邮件、数据库或支付接口，造成信息泄露或资金损失；此外，还有越权访问、数据泄露、恶意代码执行以及供应链攻击**等风险，这些问题都可能影响 AI Agent 的可靠性和安全性。

随后，讲师分析了这些风险产生的原因，指出 AI Agent 与传统 AI 最大的区别在于其具有"自主决策+工具调用+长期记忆"三种能力。当 Agent 能够自主规划任务并调用多个外部系统时，一旦安全边界设计不合理，就可能产生连锁风险。因此，安全设计不能仅依赖模型本身，而需要贯穿整个 Agent 系统架构。

在安全构建方面，活动介绍了多种防护措施，包括最小权限原则、权限隔离、工具调用白名单、输入输出过滤、安全审计、风险监控、人机确认机制以及敏感操作审批等。其中让我印象最深的是"Human in the Loop（人类参与决策）"机制，即对于支付、删除数据、发送邮件等高风险操作，需要经过人工确认后才能执行，这能够有效降低 AI Agent 被攻击或误操作带来的风险。

此外，分享还强调了构建 AI Agent 时应遵循"默认不可信（Zero Trust）"的安全理念，对所有外部输入、第三方工具以及网络资源进行验证和权限控制，并建立完整的日志记录和异常检测机制，以便及时发现和处理安全问题。

同时，我也进一步了解了 Prompt Injection、权限控制、工具安全调用、人机协同、安全审计等核心概念，对 AI Agent 的整体运行机制和安全防护思路有了更加全面的认识。这些知识不仅拓宽了我的技术视野，也为今后学习大模型应用开发、智能体构建以及 Web3 AI Agent 等相关内容奠定了基础。

本次分享内容丰富、案例真实，既分析了 AI Agent 在实际应用中的高危风险，也介绍了较为完整的安全构建方案。我深刻认识到，未来 AI Agent 的竞争不仅体现在模型能力上，更体现在系统安全、权限管理和可信执行能力上。今后我将继续学习 AI Agent 的相关技术，不断提升自己的安全意识和工程实践能力，为开发更加安全、可靠、智能的 AI 应用打下坚实基础。
