# Week 3 学习笔记（三）｜从协作到执行：一次团队 Build 的真实体验

## 本周学习内容

Week 3 的主题是 **Builder Collaboration**，与前两周最大的区别是开始真正进入团队协作阶段。前两周更多是在积累个人能力，而这一周则要求不同方向的成员共同完成一个可以展示的 Mini Demo。

在这一过程中，我学习了如何介绍自己的能力、如何寻找合适的队友、如何拆解任务，以及如何围绕一个共同目标推进项目。

* * *

## Builder Profile 带来的思考

本周首先完成的是 Builder Profile。

以前介绍自己时，我通常会强调学过什么知识、会使用哪些工具。但这次更多思考的是：

**我能够为团队带来什么价值？**

作为 Ops Builder，我能够承担的工作包括：

-   整理用户需求和项目背景；
    
-   撰写项目介绍和 Demo 展示内容；
    
-   设计用户测试流程；
    
-   收集并整理反馈；
    
-   协调团队沟通和任务推进。
    

通过整理这些内容，我更加明确了自己的定位，也方便其他 Builder 判断是否适合一起合作。

* * *

## 团队分工的重要性

项目开始后，我们首先确定了每位成员负责的内容，而不是直接开始开发。

团队按照不同方向进行了任务划分：

Research 负责验证用户需求和竞品分析；

Ops 负责项目介绍、用户测试和展示材料；

Dev 负责技术方案和 Demo 实现。

这种分工让我意识到，一个团队真正需要的是清晰的交付，而不是简单地分配角色名称。

例如，与其写“负责研究”，不如明确写“完成用户调研和竞品分析”；与其写“负责开发”，不如写“完成钱包连接页面”或“完成资产查询功能”。

只有交付目标明确，团队协作才能更加顺畅。

* * *

## Scope 控制比增加功能更重要

在讨论项目时，我们提出了不少想法，包括交易助手、资产管理助手、学习助手等。

如果全部都做，时间肯定不够。

最后，我们决定只保留一个最核心的用户流程，把项目范围控制在一周内能够完成的程度。

这个过程让我理解了 Scope 的重要性。

很多项目失败，并不是因为技术能力不足，而是因为目标过大，没有在有限时间内聚焦最关键的问题。

对于 Mini Demo 来说，一个能够顺利演示的核心功能，比多个没有完成的功能更有价值。

* * *

## 决策需要依据，而不是感觉

本周团队还记录了 Decision Log。

每次调整项目方向时，我们都会思考几个问题：

-   为什么要这样修改？
    
-   有没有用户反馈支持？
    
-   是否符合当前时间安排？
    
-   是否超出了团队能力范围？
    

这种记录方式让我感觉整个项目更加有条理。

以后即使增加新成员，也能够快速了解项目的发展过程，而不是重新讨论已经决定过的问题。

* * *

## 用户反馈带来的改变

完成初版 Demo 后，我们邀请其他同学进行了体验。

大家提出了几个比较一致的问题：

第一，项目介绍中技术术语偏多，新用户理解成本较高。

第二，不清楚 AI 与 Moss 分别负责什么。

第三，希望增加更多关于钱包安全的说明。

这些反馈帮助我们重新调整了 Demo 的展示方式。

例如，我们增加了流程图，解释 Agent、Moss 和 Protocol 的关系；同时也强调 Demo 只展示查询功能，不涉及自动交易。

这让我认识到，真正影响用户体验的，很多时候不是功能本身，而是产品是否表达得足够清楚。

* * *

## AI 在协作中的作用

整个 Week 3，我们大量使用了 AI 协助完成工作。

例如：

-   整理会议记录；
    
-   生成项目文档；
    
-   优化介绍文案；
    
-   梳理用户测试问题；
    
-   总结反馈内容。
    

AI 提高了工作效率，但最终所有内容仍然需要团队结合实际情况进行修改。

尤其是在产品方向、功能范围和用户需求判断上，不能完全依赖 AI，而应该结合团队讨论和真实反馈进行决策。

* * *

## 本周收获

这一周让我最大的收获，是理解了团队协作不仅仅是“分工”，更重要的是“协同”。

每位成员完成自己的任务后，还需要让成果能够帮助下一位成员继续推进工作。

Research 的结论会影响产品设计；

Ops 收集的反馈会影响开发方向；

Dev 的实现情况也会影响展示方式和测试安排。

只有不断沟通和调整，项目才能顺利推进。

* * *

## 下一步计划

进入 Week 4 Hackathon 后，我希望继续承担 Ops Builder 的工作，进一步完善项目介绍、优化用户体验，并参与更多真实用户测试。

同时，我也希望学习更多关于 Web3 产品运营、社区增长和 AI Agent 应用的知识，让自己不仅能够完成运营工作，也能更好地理解产品和技术之间的关系。

通过 Week 3 的实践，我更加坚定了继续学习 Web3 和参与团队项目的想法，希望在接下来的 Hackathon 中，把这次 Mini Demo 继续完善，做出一个更有实际价值的产品。

# Week 3 学习笔记（二）｜一次 Mini Demo 协作的实践与思考

## 学习背景

Week 3 与前两周最大的不同，是开始真正以团队的形式完成一个项目。

前两周更多是个人学习，包括了解 Web3、学习 Moss、完成调研和开源实践。而这一周，我第一次站在团队协作的角度思考：如果一个产品只有想法，没有分工、没有沟通、没有反馈，它很难真正落地。

因此，本周的重点不是做出一个功能复杂的产品，而是体验一次完整的 Build 流程。

* * *

## 从“想做什么”到“用户需要什么”

刚开始讨论项目时，大家提出了很多想法，例如交易助手、资产管理、学习助手等。

一开始，我们更容易从技术出发，比如“AI 能做什么”“Moss 能调用什么能力”。但经过讨论后，我们逐渐意识到，一个好的项目应该先回答一个问题：

**用户真正遇到了什么问题？**

于是，我们重新调整思路，把重点放在用户身上。

最终，我们选择了帮助 Web3 新用户理解链上资产信息这个方向，因为这是一个比较真实、明确，而且能够在一周内完成验证的场景。

这让我认识到，产品设计不是不断增加功能，而是不断缩小问题，找到最值得验证的核心价值。

* * *

## Mini Demo 的价值

以前我总觉得 Demo 应该尽可能完整，功能越多越好。

这一周学习后，我改变了这个想法。

Mini Demo 的意义不是展示所有功能，而是验证一个最重要的假设。

我们的 Demo 只围绕一个流程展开：

用户提出查询需求 → AI Agent 理解 → 调用链上能力 → 返回资产信息。

虽然很多功能还是 Mock，但整个流程已经能够让别人理解产品想解决的问题。

这种“小步验证、快速反馈”的方式，比一开始就追求完整产品更加高效。

* * *

## Ops Builder 的职责

作为 Ops Builder，我主要负责项目介绍、测试设计、反馈整理以及 Demo 展示。

刚开始我认为运营更多是宣传项目，但随着项目推进，我发现运营其实贯穿整个产品流程。

例如，在项目早期，需要帮助团队明确目标用户；在 Demo 阶段，需要设计用户测试任务；收到反馈后，还需要分析哪些问题最值得优先修改。

因此，Ops 不只是负责“讲故事”，更重要的是帮助团队理解用户，并推动产品持续优化。

* * *

## 用户反馈的重要性

完成 Demo 后，我们邀请其他同学体验项目。

反馈中让我印象最深的是：

有用户表示，他们不知道 Moss 在整个流程中发挥了什么作用。

还有用户担心 AI 是否会自动操作钱包。

这些问题在团队内部讨论时并没有出现，但真实用户却第一时间提出了。

于是，我们修改了项目介绍，增加了流程说明，并补充了安全提示。

这让我认识到：

团队熟悉项目，并不代表用户也熟悉项目。

真正的产品，需要不断通过用户反馈来修正表达方式和交互流程。

* * *

## AI 在团队中的作用

本周我们也大量使用了 AI 工具。

AI 帮助我们整理资料、生成初稿、优化文案和梳理流程，提高了工作效率。

不过，我也意识到 AI 并不能代替团队决策。

例如，在确定项目范围、分析用户需求和选择功能时，最终还是需要团队结合实际情况进行判断。

AI 更适合作为辅助工具，而不是最终决策者。

* * *

## 本周最大的收获

这一周让我更加理解了团队协作的重要性。

一个 Mini Demo 的完成，并不是某一个成员单独努力的结果，而是不同角色不断沟通、不断交付、不断修改的结果。

Research 提供事实依据，Ops 负责连接用户和产品，Dev 完成技术实现，每个人都缺一不可。

同时，我也更加明确了自己的发展方向，希望继续提升 Web3 产品运营能力，学习更多关于用户研究、社区运营和产品增长的方法。

* * *

## 总结

Week 3 不仅让我完成了一次团队项目，也让我体验了真实产品从想法到 Demo 的全过程。

相比最终完成了多少功能，我认为更重要的是学会了如何围绕用户思考问题、如何与不同角色合作，以及如何通过反馈不断改进产品。

这些经验也将成为我继续参加 Week 4 Hackathon 的基础，希望能够在下一阶段把 Demo 做得更加完整，并继续积累 Web3 产品实践经验。

# 二、Builder Profile：明确自己的角色定位

## 我的方向：

Ops Builder（Web3 社区运营）

通过 Builder Profile，我重新梳理了自己的能力和优势。

作为 Ops Builder，我不需要完成所有技术开发，而是帮助团队连接：

用户需求 → 产品价值 → 项目表达 → 用户反馈

* * *

## 我的主要贡献：

### 1\. 用户需求理解

通过 Research 和用户反馈，帮助团队确认：

-   谁会使用这个产品；
    
-   用户遇到了什么问题；
    
-   当前解决方式有什么不足。
    

### 2\. 产品叙事

将技术能力转换成用户能够理解的语言。

例如：

技术表达：  
“AI Agent 调用 Moss Protocol 获取链上数据。”

用户表达：  
“用户只需要一句话，就能让 AI 帮助理解自己的钱包资产。”

### 3\. 用户测试与反馈

设计测试任务，观察：

-   用户是否理解项目；
    
-   是否能够完成核心操作；
    
-   哪些地方产生困惑。
    

* * *

一个好的团队不是简单地寻找相同兴趣的人，而是寻找能够互补的人。

一个完整 Web3 项目需要：

Research：  
验证问题是否真实。

Ops：  
让用户理解产品，并获得反馈。

Dev：  
将想法实现为可运行 Demo。

* * *

## 团队形成过程：

团队首先讨论：

-   想解决什么问题；
    
-   面向什么用户；
    
-   一周能完成什么。
    

最终确定：

## Moss-powered Web3 Portfolio Assistant

目标：

帮助 Web3 新用户通过 AI Agent 理解自己的链上资产。

* * *

# 四、Problem Validation：从想法转向用户问题

在项目设计阶段，最大的学习是：

不要从“我们想做什么”开始，而要从：

“用户为什么需要它？”

开始。

完成：

用户输入需求 → AI Agent 理解 → Moss 调用能力 → 返回资产解释

优秀产品不是功能越多越好，而是在有限时间内验证一个核心假设。

# 五、团队协作中的重要方法

以前容易写：

Research 提供用户洞察；

Ops 转化为用户语言；

Dev 根据范围实现功能；

用户反馈再次影响方向。

# AI Chain Assistant MVP Prototype 开发记录

## 一、项目目标

本次任务目标是设计并实现一个最小化的 Web3 AI 助手原型（AI Chain Assistant）。

核心想法：

> 用户通过自然语言描述链上操作，由 AI 助手解析意图，然后调用钱包完成真实链上交易。

例如用户输入：

```
Send 0.001 ETH to 0x08827101CA4e49610aFB6aCE73A73B315814fE94
```

系统完成：

```
自然语言输入
        ↓
意图解析
        ↓
生成结构化交易参数
        ↓
调用钱包
        ↓
用户签名
        ↓
Sepolia 测试网广播交易
        ↓
返回交易结果
```

# 三、我让 AI 帮我理解了什么

本次开发过程中，我使用 AI 辅助理解 Viem 和 Web3 钱包交互流程，主要学习了以下内容：

## 1\. Viem 的作用

Viem 是一个用于以太坊交互的 TypeScript 库，可以帮助开发者：

-   连接区块链网络
    
-   创建钱包客户端
    
-   读取链上数据
    
-   发送交易
    

其中：

```
createWalletClient()
```

用于创建钱包操作客户端。

# 学习笔记：参与 Moss 开源项目并完成 Chainlink Oracle Adapter 开发实践

## 一、学习背景

在 Web3 开源生态中，开发者不仅需要掌握智能合约和区块链基础知识，也需要了解如何参与真实开源项目，包括阅读项目结构、理解贡献规范、提交代码以及通过社区协作完成功能迭代。

本次学习围绕 Moss AI Agent Framework 展开，通过实际参与开源贡献，体验完整的 GitHub 协作流程，并尝试为 Moss 添加一个新的协议适配器（Protocol Adapter）。

目标是开发一个 Chainlink Oracle Adapter，使 Moss Agent 能够访问链上预言机提供的数据，实现 AI Agent 与 Web3 数据基础设施之间的连接。

* * *

# 二、Moss 项目结构学习

在开始开发之前，首先对 Moss 仓库结构进行了分析。

项目主要目录包括：

```
moss
├── packages
│   ├── core
│   ├── protocols
│   ├── simulator
│   └── system
├── docs
├── examples
└── README.md
```

其中：

-   `packages/core`
    
    负责 Moss 的核心框架，包括 Protocol、Capability、Query、Plan 等基础能力。
    
-   `packages/protocols`
    
    存放不同 Web3 协议适配器，例如：
    
    -   DEX
        
    -   Lending
        
    -   Staking
        
    -   Oracle
        
-   `packages/simulator`
    
    用于模拟交易执行过程，验证 Agent 生成的操作计划是否符合预期。
    

通过阅读已有协议 Adapter，例如 Kuru Adapter，理解了 Moss 中协议接入的基本模式。

* * *

# 三、理解 Moss Protocol Adapter 设计

Moss 中的 Protocol Adapter 并不是简单封装合约调用，而是将区块链能力转换为 AI Agent 可以理解的语义能力。

核心流程：

```
用户需求
   ↓
AI Agent Intent
   ↓
Protocol Adapter
   ↓
智能合约调用
   ↓
链上数据返回
```

例如：

用户：

> 查询 ETH/USD 价格

Agent：

```
query price
```

Adapter：

```
Chainlink Aggregator
        ↓
latestRoundData()
        ↓
返回价格
```

最终 Agent 获取结构化数据。

# 四、Chainlink Oracle Adapter 开发实践

## 1\. 创建协议目录

```
│   └── tokens.ts
├── test

├── README.md
└── package.json
```

* * *

## 2\. 添加 Chainlink ABI

通过 ABI 文件定义 Chainlink Aggregator 合约接口。

主要使用：

```
latestRoundData()
```

获取：

-   roundId
    
-   price
    
-   startedAt
    
-   updatedAt
    
-   answeredInRound
    

ABI 的作用是让 Moss 能够正确解析智能合约方法。

* * *

## 3\. 实现 Protocol Adapter

Adapter 中主要包含：

-   Protocol 定义
    
-   Query 查询能力
    
-   Manifest 注册
    

核心思想：

将底层合约方法转换为 Agent 能理解的能力。

例如：

链上方法：

```
latestRoundData()
```

转换为：

```
latestPrice()
```

让 Agent 不需要理解复杂合约细节。

# 五、测试与问题解决过程

开发完成后，执行：

```
pnpm install
```

安装项目依赖。

第一次测试过程中遇到：

```
use() takes a ProtocolPackage
```

错误。

原因：

Adapter 没有正确导出 ProtocolPackage。

解决：

检查：

```
defineProtocolPackage()
```

并确保：

```
export const chainlinkManifest
```

正确注册。

修改后：

运行：

```
pnpm test
```

测试结果：

```
✓ chainlink adapter
✓ loads price query

Test Files 1 passed
Tests 2 passed
```

说明 Adapter 已成功接入 Moss。

Research Builder 的主要任务是对 Web3 项目进行深入研究，为团队提供决策依据。他们需要阅读大量官方文档、白皮书以及治理提案，分析项目的发展现状、技术特点和市场竞争力。同时还需要借助 DeFiLlama、Dune 等工具分析链上数据，撰写研究报告，为团队提供有价值的信息。

## 学习主题

AI Agent 高危案例解析与安全构建

## 学习内容

今天参加了关于**AI Agent 高危案例解析与安全构建**的线上分享活动，对 AI Agent 的安全风险和防护措施有了更加系统的认识。随着 AI Agent 从简单的聊天助手逐渐发展为能够自主调用工具、访问网络、执行任务甚至进行支付的智能体，其拥有的权限越来越高，因此安全问题也成为 AI Agent 落地过程中最重要的挑战之一。

活动首先介绍了 AI Agent 面临的几类典型高危案例。例如，**Prompt Injection（提示词注入攻击）会诱导 Agent 忽略原有指令，从而执行攻击者指定的恶意任务；工具滥用可能导致 Agent 被诱导调用邮件、数据库或支付接口，造成信息泄露或资金损失；此外，还有越权访问、数据泄露、恶意代码执行以及供应链攻击**等风险，这些问题都可能影响 AI Agent 的可靠性和安全性。

随后，讲师分析了这些风险产生的原因，指出 AI Agent 与传统 AI 最大的区别在于其具有"自主决策+工具调用+长期记忆"三种能力。当 Agent 能够自主规划任务并调用多个外部系统时，一旦安全边界设计不合理，就可能产生连锁风险。因此，安全设计不能仅依赖模型本身，而需要贯穿整个 Agent 系统架构。

在安全构建方面，活动介绍了多种防护措施，包括最小权限原则、权限隔离、工具调用白名单、输入输出过滤、安全审计、风险监控、人机确认机制以及敏感操作审批等。其中让我印象最深的是"Human in the Loop（人类参与决策）"机制，即对于支付、删除数据、发送邮件等高风险操作，需要经过人工确认后才能执行，这能够有效降低 AI Agent 被攻击或误操作带来的风险。

此外，分享还强调了构建 AI Agent 时应遵循"默认不可信（Zero Trust）"的安全理念，对所有外部输入、第三方工具以及网络资源进行验证和权限控制，并建立完整的日志记录和异常检测机制，以便及时发现和处理安全问题。

## 学习收获

通过本次学习，我认识到 AI Agent 不仅需要具备强大的任务执行能力，更需要完善的安全体系作为保障。未来 AI Agent 将广泛应用于办公自动化、金融、医疗、智能制造等多个领域，如果缺乏安全设计，可能会造成严重的业务风险和数据安全问题。因此，在开发 AI Agent 时，应将安全设计作为系统架构的重要组成部分，而不是开发完成后的补充工作。

同时，我也进一步了解了 Prompt Injection、权限控制、工具安全调用、人机协同、安全审计等核心概念，对 AI Agent 的整体运行机制和安全防护思路有了更加全面的认识。这些知识不仅拓宽了我的技术视野，也为今后学习大模型应用开发、智能体构建以及 Web3 AI Agent 等相关内容奠定了基础。

## 总结

本次分享内容丰富、案例真实，既分析了 AI Agent 在实际应用中的高危风险，也介绍了较为完整的安全构建方案。我深刻认识到，未来 AI Agent 的竞争不仅体现在模型能力上，更体现在系统安全、权限管理和可信执行能力上。今后我将继续学习 AI Agent 的相关技术，不断提升自己的安全意识和工程实践能力，为开发更加安全、可靠、智能的 AI 应用打下坚实基础。
