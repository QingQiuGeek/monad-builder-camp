1.  **需求分析与规划**
    
    在开发 Dapp 之前，首先需要进行需求分析和规划，明确应用的目标和功能。此阶段包括：
    
    -   **确定功能需求**：需要定义用户可以进行的操作，比如转账、查询余额、创建投票等。
        
    -   **选择区块链平台**：决定在哪个平台上构建 Dapp（如以太坊、Solana、Polygon 等），这通常取决于目标用户群、交易成本、可扩展性等因素。
        
    -   **设计用户体验（UX）**：定义 Dapp 的界面设计和交互流程，确保用户能够轻松使用应用并与区块链交互。
        
2.  **智能合约开发**
    
    智能合约是 Dapp 的核心，负责执行去中心化的业务逻辑和存储重要的数据。在这一阶段，开发者需要：
    
    -   **编写智能合约**：使用 **Solidity** 或其他智能合约语言编写合约，确保合约的功能满足需求分析中定义的要求。
        
    -   **编写测试用例**：为智能合约编写单元测试，确保合约逻辑正确、无漏洞。
        
    -   **审计和优化**：对合约进行安全审计，确保合约的安全性，避免常见漏洞（如重入攻击、整数溢出等）。
        
3.  **检索器开发**
    
    检索器是获取链上数据的核心，负责捕获智能合约释放的事件并以合理的方式将其存入数据库的不同的表内部。在这一阶段，开发者需要:
    
    -   **确定功能需要的数据内容**: 前端使用的数据大部份都直接来自检索器，所以开发者需要确定前端工程师所需要的数据
        
    -   **编写检索器程序**: 目前主流的检索器框架，如 ponder 和 subgraph 都是用了 TypeScript 语言作为检索器的程序编写语言，开发者主要编写事件数据清理以及事件数据写入数据库的代码
        
    -   **部署和运维**: 编写程序完成后，一般使用 Docker 部署到云服务器中，当然目前很多检索器框架也提供 SaaS 服务，同时检索器作为一个常规的数据库应用需要运维
        
4.  **前端开发**
    
    前端是用户与 Dapp 交互的主要界面，因此开发前端时需要：
    
    -   **选择前端框架**：可以使用现代前端框架（如 **React**、**Vue**）来构建 UI。前端将通过 JavaScript 与智能合约进行交互。
        
    -   **连接钱包**：通过集成 **MetaMask** 等 Web3 钱包，用户可以连接到 Dapp，并授权其与智能合约交互。
        
    -   **显示区块链数据**：前端需要从区块链和检索器内获取数据（如账户余额、交易记录），并通过用户界面展示。
        
    -   **处理交易签名与确认**：当用户发起交易时，前端需要与钱包进行交互，获取用户的签名并将交易发送到区块链。
        
5.  **与区块链交互**
    
    前端和智能合约通过 **Viem**（推荐）、**Ethers.js** 或 **Wagmi** 等现代化库进行交互。这些库提供更好的 TypeScript 支持和性能优化：
    
    -   **读取数据**：前端通过智能合约的公共函数读取区块链上的状态数据（如余额、合约信息）。
        
    -   **发送交易**：当用户发起交易时，前端需要通过钱包签署交易并发送到区块链，执行合约中的某个功能（如转账）。
        
6.  **部署与上线**
    
    一旦开发完成，Dapp 进入部署阶段。具体步骤包括：
    
    -   **部署智能合约**：推荐使用 **Hardhat** 或 **Foundry**（现代化开发工具）将智能合约部署到测试网（如 **Sepolia**、**Holesky**）或主网。
        
    -   **前端部署**：将前端应用部署到去中心化平台（如 **IPFS**）或传统的 Web 服务（如Vercel）。
        
    -   **发布和维护**：将 Dapp 上线，进行用户反馈收集，定期更新合约和前端，修复潜在问题。

DeFi 核心概念

概念 说明 代表项目

AMM（自动做市商） 算法定价，无需订单簿 Uniswap

流动性池 用户存入代币对，赚取交易费 Uniswap V2/V3

借贷协议 超额抵押借贷，存款赚利息 Aave / Compound

预言机 链上获取链下数据（价格） Chainlink

质押 锁仓代币获取收益 Lido

稳定币 锚定法币的代币 USDT / USDC / DAI

Uniswap V2 AMM 原理

恒定乘积公式：x × y = k

x = 代币 A 储备量，y = 代币 B 储备量，k = 常数

交易改变 x 和 y，但 k 不变 → 价格自动调整

流动性提供者（LP）存入代币对，获得 LP Token

交易手续费（0.3%）分配给 LP

预言机（Oracle）

智能合约无法直接获取链下数据

Chainlink：去中心化预言机网络，提供价格 feeds

使用方式：AggregatorV3Interface 获取最新价格

⚠️ 预言机攻击是 DeFi 重大风险源（闪贷攻击操纵价格）

IPFS 去中心化存储

NFT 元数据（图片、属性）通常存储在 IPFS 而非链上

IPFS = 内容寻址存储，CID 标识文件唯一性

Pinata / NFT.Storage 提供免费 IPFS pinning 服务

完全去中心化方案：Arweave（永久存储）

合约安全基础

重入攻击（Reentrancy）：最经典的漏洞，The DAO 事件

整数溢出：Solidity 0.8+ 默认检查溢出

权限控制：onlyOwner / AccessControl / 多签

Checks-Effects-Interactions 模式：防止重入

⚠️ 先验证条件 → 再更新状态 → 最后外部调用

今天学了vibecoding和agent的一些知识，web3链上相关的知识还需提升

从第一性原理出发，Loop Engine（循环引擎）本质上是一种让AI代理（Agent）自主重复“思考-行动-观察-评估”周期，直到满足停止条件的工作系统。 它不是什么新发明，而是把计算机和控制论里的经典反馈循环（feedback loop）应用到LLM（大型语言模型）上。 第一性原理拆解：为什么需要Loop Engine？LLM单次推理的局限（原子能力） LLM本质是一个“下一词预测器”：给定上下文，输出最可能的token序列。它擅长单次生成（one-shot），但不擅长长时间、多步、需要纠错的复杂任务。一次prompt最多就是“想一下、做一点、输出”。现实任务（如写代码、修复bug、处理PR、监控系统）需要多次迭代、验证、调整。 反馈循环是智能的核心 任何智能系统（人、动物、自动控制系统）都依赖感知 → 处理 → 行动 → 观察结果 → 调整的闭环。 没有反馈 = 开环控制，容易漂移、失败。 有反馈 = 闭环，能纠错、收敛到目标。 这就是ReAct（Reason + Act）、Agent Loop、OODA循环（Observe-Orient-Decide-Act）的本质。 Loop Engine = 把反馈循环“工程化” 把上面这个自然闭环变成可控、可重复、可组合、可长时间运行的系统。它包括：触发器（Trigger）：什么启动循环？（用户prompt、定时、外部事件） 行动体（Actor）：谁/什么执行？（Claude、子agent、工具调用） 观察/验证机制（Verifier）：怎么知道做得好不好？（运行测试、检查指标、第二个评审模型） 停止条件（Stop Condition）：什么时候结束？（目标达成、最大轮次、超时） 状态管理：记住历史、持久化上下文（避免无限循环或遗忘） 这样，AI就从“一次性回答工具”变成“能自主干活的工人”。 trueventures.com Claude Code里的具体Loop类型（对应不同控制粒度）根据你发的那个ClaudeDevs帖子，他们把Loop分成了几类，从人类干预最多到最少：Turn-based loops（手动/代理循环）：最基础。每发一条prompt就是一个turn。用户手动检查、给下一步指令。像人指挥AI一步步走。适合短任务。 改进：把你的检查标准写成SKILL.md，让AI自己验证（量化检查越好越容易自验）。 Goal-based loops（/goal）：把“停止判断权”交给系统。定义清晰的成功标准（e.g. “通过所有测试，得分>90”），AI迭代直到满足或达到最大轮次。减少“AI自己觉得够了就停”的主观错误。 @ClaudeDevs Time-based loops（/loop 和 /schedule）：定时触发。适合重复性工作（每天总结Slack、定期检查PR、CI失败处理）。把“什么时候干”也自动化了。 Proactive loops（主动循环）：完全无人值守。结合schedule + goal + 多agent工作流 + auto mode。AI自己监控事件、处理、修复、review。适合bug triage、依赖升级等持续流程。 这些可以组合成更复杂的“Loop Engine”：一个持久运行的系统，能 spawning 多个子agent（swarm），处理不同子任务。更高层次：Loop Engineering（循环工程）从第一性看，这已经不是简单prompt工程，而是系统设计：状态与持久化：循环要记住历史，不能每次从零开始（memory、向量检索、外部数据库）。 纠错与鲁棒性：加评审agent（second model as judge）、重试机制、guardrails。 资源控制：token使用、成本、停止边界（不然容易无限烧钱）。 可组合性：小loop组成大workflow，像乐高一样搭“代理工厂”。 演化：每次循环后，把学到的“什么算好”编码回系统（skills、docs、patterns），让未来循环更强。 youtube.com 这就接近“ compounding loop engine”：一个能自我改进、累积进步的持久系统，而非每次都从头开始。 trueventures.com 实际意义对开发者：别再one-shot prompt coding agent了，而是设计一个“好用的循环”——清晰目标 + 自验证 + 边界控制。代码质量和效率会大幅提升。 对未来：真正的Agentic AI不是更聪明的单模型，而是高效的Loop Engine + Swarm。模型负责智能，引擎负责可靠运行。 简单说，Loop Engine 就是把AI的“思考-做事”能力，用工程化的反馈循环包装起来，让它能可靠地、长时间地、自主地完成人类想干但不想反复盯着的活。想深入的话，可以从你手头的一个重复瓶颈任务开始：定义Done的标准、写验证skill、加停止条件，先跑一个小loop试试。效果会比纯prompt强很多。

\## 💡 学习建议

1\. **每天写一段学习笔记** — 哪怕是 3 行，积少成多

2\. **善用 AI 但不依赖** — 先自己思考，再用 AI 验证和补充

3\. **出错是常态** — 把每次报错当作学习机会，积累排错经验

4\. **公开你的学习过程** — 发 Twitter/X Thread 或 Mirror 文章，既是记录也是宣传

5\. **加入社区提问** — ETHPanda 和 LXDAO 的 TG 群非常活跃，大胆提问
