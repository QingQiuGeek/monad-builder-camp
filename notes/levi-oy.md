# 2026-07-11
<!-- DAILY_CHECKIN_2026-07-11_START -->
Monad 提速的核心机制:传统 EVM 链交易是串行执行,Monad 用并行执行，先同时跑多笔交易，最后再检测冲突、回滚有问题的部分,这才是 10,000 TPS 的关键,而不只是堆硬件。配套的 MonadDB 把状态存取做了定制优化,减少了并行执行时的读写瓶颈。

Monad 提速的三个关键设计:一是交易乐观并行执行,先同时跑再处理冲突;二是共识和执行拆开做成流水线,不用互相等待;三是自研的存储引擎优化了高并发读写。三者配合起来才撑起真正的高性能,而不是单点优化。

# 2026-07-10
<!-- DAILY_CHECKIN_2026-07-10_START -->

Monad 不是"更快的以太坊"，而是想让区块链第一次能像数据库一样，能承受更高量级的操作。

以前区块链慢，所以很多东西"不敢"放到链上，只能放在链下处理，比如：

-   订单簿撮合（交易所后台那种毫秒级报价）
    
-   实时游戏里的状态变化
    
-   小额高频支付
    

现在速度上来了，这些场景有可能搬到链上，而不用绕一圈靠链下服务器处理完再"补记账"。

以太坊高峰期堵车时，Gas 费会突然飙升。Monad 想让GAS价格可预测。

# 2026-07-09
<!-- DAILY_CHECKIN_2026-07-09_START -->


**合约源码**：开发者用 Solidity 编写的智能合约代码，定义了合约的功能和逻辑。

**ABI**：合约的接口说明书，告诉外部有哪些函数可以调用以及如何调用**。**

**合约地址**：智能合约部署到区块链后获得的唯一地址，用于定位和访问合约**。**

**Read Function**：用于查询合约中的数据，不会修改区块链状态，也无需支付 Gas。

**Write Function**：用于修改合约状态，需要钱包签名、支付 Gas，并记录到区块链。

**Transaction Hash**：每笔链上交易生成的唯一编号，可用于查询交易状态和执行结果。

**开发者编写合约源码，编译生成 ABI，部署后获得合约地址；用户通过 ABI 和合约地址调用 Read 或 Write Function，其中 Write 会产生一笔交易，并生成唯一的 Transaction Hash。**

# 2026-07-08
<!-- DAILY_CHECKIN_2026-07-08_START -->



DApp 与传统 App 区别

最大区别：

-   Web2 的核心逻辑运行在中心化服务器中；
    
-   Web3 将核心业务逻辑放入智能合约，由区块链网络执行。
    

DApp 主要组成

-   **前端**：负责用户交互。
    
-   **钱包**：管理身份、签名交易。
    
-   **智能合约**：实现业务逻辑和链上数据管理。
    
-   **区块链**：存储数据并执行交易。
    
-   **数据检索器**：提高链上数据查询效率。
    

DApp开发流程

-   需求分析 → Solidity 合约开发 → 测试 → 部署到区块链 → 前端连接合约 → 上线运行。

# 2026-07-07
<!-- DAILY_CHECKIN_2026-07-07_START -->




**交易结构**

-   **From/To**:转账双方地址
    
-   **Value**:转账金额
    
-   **Transaction Hash**:每笔交易唯一的"身份证号",凭它可查到完整细节
    
-   **Method**:操作类型标注 (MonTransfer=原生代币转账)
    

**Gas 与手续费**

-   Gas = 计算资源消耗单位,Transaction Fee = Gas Used × Gas Price
    
-   手续费付给维护网络的验证者,不是某个公司
    
-   **失败交易也扣gas**:gas买的是"计算资源已被使用"这个事实,不是"结果保证";执行到失败之前的算力已经产生,必须付费
