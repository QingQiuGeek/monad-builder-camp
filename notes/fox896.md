**智能合约开发**

[一、Dapp 架构和开发流程](https://web3intern.xyz/zh/smart-contract-development/#%E4%B8%80%E3%80%81dapp-%E6%9E%B6%E6%9E%84%E5%92%8C%E5%BC%80%E5%8F%91%E6%B5%81%E7%A8%8B)

去中心化应用（Dapp）是与传统集中式应用不同的全新应用模式，通常运行在区块链或分布式网络上。与传统应用相比，Dapp 的核心特点在于去中心化，意味着应用的逻辑和数据不由单一实体控制，而是由多个参与者共同维护。因此，开发 Dapp 需要理解和掌握去中心化技术栈、智能合约编程以及前端与区块链的交互方式。

[1\. Dapp 架构](https://web3intern.xyz/zh/smart-contract-development/#_1-dapp-%E6%9E%B6%E6%9E%84)

Dapp 的架构主要由三个核心部分组成：

**1. 前端（User Interface）：**

-   前端是 DApp 与用户交互的界面，通常由 HTML、CSS 和 JavaScript（如 React、Vue 等框架）构建。与传统 Web 应用不同，DApp 前端不会直接连接区块链网络，而是通过 钱包注入的 Provider 或第三方 RPC 节点 与区块链交互：
    
    -   通过 RPC 节点对 智能合约发起只读调用（如 eth\_call），获取合约状态、事件日志等链上数据
        
    -   对需要修改状态的操作，由前端构造对 智能合约的交易调用，交由钱包完成签名后，再通过 RPC 节点广播到区块链网络并最终上链执行
        
-   前端还需要集成区块链钱包（如 MetaMask）来进行身份验证和签署交易，确保用户的隐私和安全。
    

**2. 智能合约（Smart Contracts）：**

-   智能合约是 Dapp 的核心，它定义了应用的业务逻辑，并部署在区块链上。智能合约通过执行自动化的规则来确保交易和操作的透明性与不可篡改性。
    
-   在以太坊平台上，智能合约通常使用 **Solidity** 编程语言编写，并通过 **Ethereum Virtual Machine (EVM)** 执行。
    

**3. 数据检索器（Indexer）：**

-   智能合约通常以 Event 形式释放日志事件，比如释放代表 NFT 转移的 Transfer 事件，数据检索器会检索这些数据并将其写入到 PostgreSQL 等传统数据库中
    
-   Dapp 在前端进行数据展示时需要检索器内的数据。一个简单的示例是某 NFT 项目需要展示用户持有的所有 NFT，但是 NFT 合约并不会提供通过输入地址参数返回该地址下的所有 NFT 的函数，此时我们可以运行数据检索器将 Transfer 事件读取后写入传统数据库内，前端可以在传统数据库内检索用户持有的 NFT 数据
    

**4. 区块链和去中心化存储（Blockchain & Decentralized Storage）：**

-   区块链用于存储智能合约的状态数据及交易记录。去中心化存储如 **IPFS**（InterPlanetary File System）或 **Arweave**，用于存储大规模的非结构化数据（如图片、文档等），确保数据不易丢失和篡改。
    
-   通过使用去中心化存储，Dapp 确保所有数据在多个节点上备份，保证数据的持久性和去中心化特性。
    

[2\. Dapp 开发流程](https://web3intern.xyz/zh/smart-contract-development/#_2-dapp-%E5%BC%80%E5%8F%91%E6%B5%81%E7%A8%8B)

Dapp 的开发流程可以分为以下几个阶段：

Dapp 开发流程图

1.  需求分析与规划
    

在开发 Dapp 之前，首先需要进行需求分析和规划，明确应用的目标和功能。此阶段包括：

-   确定功能需求：需要定义用户可以进行的操作，比如转账、查询余额、创建投票等。
    
-   选择区块链平台：决定在哪个平台上构建 Dapp（如以太坊、Solana、Polygon 等），这通常取决于目标用户群、交易成本、可扩展性等因素。
    
-   设计用户体验（UX）：定义 Dapp 的界面设计和交互流程，确保用户能够轻松使用应用并与区块链交互。
    

2.  智能合约开发
    

智能合约是 Dapp 的核心，负责执行去中心化的业务逻辑和存储重要的数据。在这一阶段，开发者需要：

-   编写智能合约：使用 **Solidity** 或其他智能合约语言编写合约，确保合约的功能满足需求分析中定义的要求。
    
-   编写测试用例：为智能合约编写单元测试，确保合约逻辑正确、无漏洞。
    
-   审计和优化：对合约进行安全审计，确保合约的安全性，避免常见漏洞（如重入攻击、整数溢出等）。
    

3.  检索器开发
    

检索器是获取链上数据的核心，负责捕获智能合约释放的事件并以合理的方式将其存入数据库的不同的表内部。在这一阶段，开发者需要:

-   确定功能需要的数据内容: 前端使用的数据大部份都直接来自检索器，所以开发者需要确定前端工程师所需要的数据
    
-   编写检索器程序: 目前主流的检索器框架，如 ponder 和 subgraph 都是用了 TypeScript 语言作为检索器的程序编写语言，开发者主要编写事件数据清理以及事件数据写入数据库的代码
    
-   部署和运维: 编写程序完成后，一般使用 Docker 部署到云服务器中，当然目前很多检索器框架也提供 SaaS 服务，同时检索器作为一个常规的数据库应用需要运维
    

4.  前端开发
    

前端是用户与 Dapp 交互的主要界面，因此开发前端时需要：

-   选择前端框架：可以使用现代前端框架（如 **React**、**Vue**）来构建 UI。前端将通过 JavaScript 与智能合约进行交互。
    
-   连接钱包：通过集成 **MetaMask** 等 Web3 钱包，用户可以连接到 Dapp，并授权其与智能合约交互。
    
-   显示区块链数据：前端需要从区块链和检索器内获取数据（如账户余额、交易记录），并通过用户界面展示。
    
-   处理交易签名与确认：当用户发起交易时，前端需要与钱包进行交互，获取用户的签名并将交易发送到区块链。
    

5.  与区块链交互
    

前端和智能合约通过 **Viem**（推荐）、**Ethers.js** 或 **Wagmi** 等现代化库进行交互。这些库提供更好的 TypeScript 支持和性能优化：

-   读取数据：前端通过智能合约的公共函数读取区块链上的状态数据（如余额、合约信息）。
    
-   发送交易：当用户发起交易时，前端需要通过钱包签署交易并发送到区块链，执行合约中的某个功能（如转账）。
    

6.  部署与上线
    

一旦开发完成，Dapp 进入部署阶段。具体步骤包括：

-   部署智能合约：推荐使用 **Hardhat** 或 **Foundry**（现代化开发工具）将智能合约部署到测试网（如 **Sepolia**、**Holesky**）或主网。
    
-   前端部署：将前端应用部署到去中心化平台（如 **IPFS**）或传统的 Web 服务（如Vercel）。
    
-   发布和维护：将 Dapp 上线，进行用户反馈收集，定期更新合约和前端，修复潜在问题。
    

[3\. 小结](https://web3intern.xyz/zh/smart-contract-development/#_3-%E5%B0%8F%E7%BB%93)

Dapp 的开发流程从需求分析、智能合约编写、前端开发，到最终部署，涵盖了多个技术栈的综合应用。开发 Dapp 时，区块链技术的透明性、不可篡改性以及智能合约的自动执行能力为应用提供了去中心化的基础。但与此同时，开发者需要应对与传统 Web 应用不同的挑战，如用户体验、交易处理以及安全性问题。

## 1. 项目概述

### 1.1 一句话描述

**残响远征**（Echo March）是一款部署在 Monad Testnet 上的链上领土策略游戏。旧日之主的残响仍在大陆上回荡——玩家派遣军队穿越格网地形，夺取可驻守的城池，与 AI 操控的「旧日之主」及其他玩家展开长期对抗。区块归属、驻军、行军、积分流转与战斗结算均在链上记录与执行。

### 1.2 核心体验

- **扩张**：派出行军，跨地形向目标区块推进
- **防守**：在可驻守地形上布置驻军、触发邻格协防
- **对抗**：与旧日之主争夺危险区，与其他玩家相互攻伐
- **策略**：在积分稀缺、增援延迟、战斗消融的多重约束下做长期决策

### 1.3 与 Week 1 的关系

Week 1 已实现 `PointsLedger` 积分账本合约。Week 2 采用 **内置积分（方案 A）**：

- 积分全部在 `EchoMarch.sol` 内管理（`Player.liquidPoints`）
- 不依赖 Week 1 合约，游戏自成一体，降低开发复杂度
- 后续可通过 `depositFromLedger()` / `withdrawToLedger()` 桥接 PointsLedger（非 MVP 范围）

## 2. 设计目标与链上必要性

### 2.1 设计目标

| 目标 | 说明 |
|------|------|
| 可信领土 | 区块归属不可被平台暗改 |
| 透明经济 | 积分补给、消耗规则公开可验证 |
| 公正战斗 | 交战消融、增援延迟由合约统一结算 |
| AI 可约束 | 旧日之主的优势与限制均链上 enforce |
| Monad 适配 | 利用高吞吐支持多区块、多玩家并发操作 |

### 2.2 为什么需要链上记录

| 数据/行为 | 链上理由 |
|-----------|----------|
| 区块归属 | 领土是核心资产，需抗篡改 |
| 区块驻军 | PvP/PvE 争议时的唯一可信来源 |
| 玩家积分 | 经济规则透明，防止运营暗改 |
| 战斗状态 | 多方同时操作需统一仲裁 |
| 旧日约束 | 危险区兵力上限等规则需可验证 |
| 邻格增援 | 涉及多格状态联动，需原子性结算 |

### 2.3 什么不需要上链

| 内容 | 处理方式 |
|------|----------|
| 地图渲染、动画 | 纯前端 |
| 旧日 AI 决策逻辑 | 链下 Bot，调用链上接口 |
| 战报推送、通知 | 链下索引 + 事件监听 |
| 音效、特效 | 纯前端 |

### 2.4 核心架构原则

> **链上存世界状态与结算规则，链下跑 AI 和 UI，时间推进采用 timestamp 惰性结算（Lazy Settlement），不依赖链上定时器。**

## 3. 世界观与核心玩法

### 3.1 背景设定

远古时代，一块大陆被分割为若干**区块（Tile）**。大部分区块仍被 **旧日之主（The Old Lords）** 占据——它们是被 AI 操控的远古势力，每日从深渊获得巨量积分补给，不断向玩家领土施压。

玩家作为新兴领主，以 **1000 初始积分** 起家，每日仅获 **100 积分补给**，必须精打细算地扩张、设防、结盟（或攻伐），逐步从旧日手中夺回大陆。

### 3.2 核心玩法循环

```
每日领取补给 → 规划路径 → 派出行军（5min/格）
      ↑                              ↓
  积分管理 ← 战斗结算/占领 ← 途经旧日城池 / 抵达目标
```

### 3.3 游戏阶段

| 阶段 | 特征 | 玩家目标 |
|------|------|----------|
| 早期 | 旧日控制大部分区块 | 在危险区边缘建立据点 |
| 中期 | 玩家领土扩张，PvP 出现 | 夺取危险区，限制旧日出生点 |
| 后期 | 危险区被玩家占满 | 旧日只能在危险区内活动，玩家互相争霸 |

## 4. 地图与区块系统

### 4.1 地图结构

- 大陆由 **10 × 10 网格** 组成，共 **100 个区块**
- 每个格子是一个 **区块（Tile）**，拥有唯一 `tileId`（0 至 99）
- 坐标映射：`tileId = y * 10 + x`（x, y 均为 0~9）
- 邻接关系：**四方向邻格**（上、下、左、右），不支持对角

### 4.2 区块属性（链上）

```solidity
enum TerrainType {
    Plain,      // 平原：可通行，可驻守
    Mountain,   // 山地：不可通行，不可驻守
    Water,      // 水域：不可通行，不可驻守
    Road,       // 通路：可通行，不可驻守（无法建城）
    Ruins       // 遗迹：可通行，不可驻守（无法建城）
}

struct Tile {
    address owner;          // 所属玩家；address(0)=中立；OLD_LORD=旧日固定地址
    TerrainType terrain;    // 地形类型
    uint256 garrison;       // 当前驻军（仅可驻守地形有效）
    bool isDangerZone;      // 是否为危险区（旧日核心区域）
    uint256 maxGarrison;    // 该格驻军上限（默认 1000，仅可驻守地形）
}
```

### 4.3 地形类型

每个区块拥有固定地形（部署时写入，不可更改）：

| 地形 | 可通行 | 可驻守（建城） | 说明 |
|------|--------|----------------|------|
| **平原 Plain** | ✅ | ✅ | 主要争夺区，可占领、驻军 |
| **山地 Mountain** | ❌ | ❌ | 天然屏障，行军不可穿越 |
| **水域 Water** | ❌ | ❌ | 河流/湖泊，行军不可穿越 |
| **通路 Road** | ✅ | ❌ | 可快速通过，但无法建立城池或驻军 |
| **遗迹 Ruins** | ✅ | ❌ | 可通行，历史废墟，不可建城 |

**关键规则：**

- 士兵行军只能经过 **可通行（passable）** 地形
- 只有 **可驻守（canGarrison）** 地形（即平原）才能占领并驻扎兵力
- 通路/遗迹/山地/水域 **永远不能** 拥有 `garrison > 0`

### 4.4 区块类型（归属维度）

| 类型 | 初始归属 | 地形要求 | 说明 |
|------|----------|----------|------|
| 危险区 | 旧日之主 | 平原 | 旧日核心领土，有特殊规则约束 |
| 普通区 | 旧日/中立 | 平原 | 可被任意方占领 |
| 中立区 | 无归属 | 平原 | 无驻军，首占成本低 |
| 通路/遗迹 | 无归属 | Road/Ruins | 不可占领，仅作为行军通道 |
| 障碍区 | 无归属 | Mountain/Water | 不可进入 |

| 平原 P | ~72 | 可占领、驻军、战斗主战场 |
| 山地 M | ~12 | 地图边界屏障，分割区域 |
| 水域 W | ~4 | 内陆河流，迫使绕行 |
| 通路 R | ~8 | 快速行军通道，不可设防 |
| 遗迹 U | ~4 | 中心障碍，不可设防 |

> 危险区标记（`isDangerZone`）叠加在平原格上，与地形独立。外圈 + 中心 2×2 平原格同时标记为危险区。

### 4.7 领土相连判定

- 两格 **属同一玩家** 且 **四方向相邻** → 称「相连」
- 相连关系用于：邻格协防、领土完整性展示
- **注意**：通路/遗迹/山地/水域不参与「相连」判定（不可驻守）

### 4.8 行军与途经规则

> 跨格派遣士兵不再即时到达，必须通过 **行军（March）** 在地图上移动，**每 5 分钟推进一格**。

#### 4.8.1 行军基本规则

| 项 | 值 |
|----|-----|
| 移动速度 | **1 格 / 5 分钟** |
| 移动方向 | 四方向，每步必须是相邻格 |
| 路径限制 | 只能经过 **可通行** 地形 |
| 终点限制 | 若目标为进攻/驻防，终点必须是 **可驻守（平原）** 地形 |
| 路径提交 | 发起行军时提交完整路径 `path[]`，链上校验合法性 |
| 时间结算 | 惰性结算：`(now - lastMoveTime) / 300` = 可推进格数 |

#### 4.8.2 途经遭遇规则

行军推进到新的一格时，根据该格归属判定：

| 途经格类型 | 处理方式 |
|------------|----------|
| **中立格**（无归属） | 直接通过，行军继续 |
| **己方城池** | 直接通过，不停留 |
| **其他玩家城池** | **直接通过**，不触发战斗 |
| **通路 / 遗迹** | 直接通过（无归属，不可驻守） |
| **旧日之城**（旧日所属 + 平原） | **立即进入战斗**，行军暂停并转为攻击方 |
| **山地 / 水域** | 路径非法，不允许进入 |

**「旧日之城」判定：**

```solidity
bool isOldLordCity = (tile.owner == OLD_LORD && _canGarrison(tile.terrain));
```

**「玩家城池」判定：**

```solidity
bool isPlayerCity = (tile.owner != address(0) && tile.owner != OLD_LORD && _canGarrison(tile.terrain));
```

#### 4.8.3 行军 vs 本格驻防

| 行为 | 方式 | 速度 | 适用 |
|------|------|------|------|
| **跨格派兵** | `startMarch(path, amount)` | 5 min/格 | 进攻、远程增援、占领 |
| **本格增兵** | `dispatchDefend(tileId, amount)` | 即时 | 已在己方城池内追加驻军 |
| **邻格协防** | `reinforceFromNeighbor` | 5 min 到场 | 被攻击格 ≤ 500 时 |

> 跨格操作统一走行军系统；`dispatchDefend` 仅用于 **当前已是己方城池** 的格子上即时增兵。

#### 4.8.4 行军状态（链上）

```solidity
struct March {
    uint256 id;
    address commander;          // 行军所属玩家
    uint256 troops;             // 行军兵力
    uint256 originTile;         // 出发格
    uint256 destinationTile;    // 目标格
    uint256[] path;             // 完整路径（含起点和终点）
    uint256 currentStep;        // 当前所在 path 索引
    uint64 lastMoveTime;        // 上次推进时间
    bool active;                // 是否行军中
    bool isAttack;              // true=进攻意图, false=驻防/增援意图
}
```

**惰性推进公式：**

```
elapsed     = block.timestamp - march.lastMoveTime
steps       = elapsed / 300          // 每 5 分钟一步
for each step in steps:
    currentStep++
    检查 path[currentStep] 遭遇 → 旧日之城则创建 Battle，行军暂停
    若 currentStep == path.length - 1 → 抵达目标，执行占领/驻防/开战
lastMoveTime += steps * 300
```

#### 4.8.5 行军流程图

```
startMarch(path, amount)
      │
      ▼
校验路径：全部可通行？终点可驻守？
      │
      ▼
扣除 liquidPoints，创建 March
      │
      ▼
advanceMarch(marchId)  ← 任何人可调用 / 操作前自动推进
      │
      ▼
推进到新格 → 检查遭遇
      │
      ├── 旧日之城 → 创建 Battle，行军暂停（兵力转入 atkTroops）
      ├── 玩家之城 → 直接通过
      ├── 通路/遗迹/中立 → 直接通过
      │
      ▼
抵达终点？
      │
     是
      ▼
  进攻意图 → 有驻军则 Battle，无则占领
  驻防意图 → 加入 garrison（≤ 1000）
```

## 6. 旧日之主（AI）系统

### 6.1 旧日之主身份

- 链上使用 **固定地址** `OLD_LORD_ADDRESS`（部署时设定或由合约自身持有）
- 旧日操作通过 **链下 AI Bot** 调用与玩家相同的合约接口
- 所有旧日专属限制在合约层 **modifier / 校验** 中 enforce

### 6.2 旧日属性

| 属性 | 数值 | 与玩家差异 |
|------|------|------------|
| 初始积分 | 按地图配置 | 远高于玩家 |
| 每日补给 | 10000 | 玩家为 100 |
| 单格驻军上限 | 1000 | 相同 |
| 操作逻辑 | 与玩家一致 | 无特殊作弊能力 |

### 6.3 旧日专属约束

#### 约束 1：危险区兵力上限

> 旧日在 **危险区** 内的总驻军，不得超过 **所有玩家总驻军之和**。

```
sum(oldLord.garrison for tile in dangerZone)
    <= sum(player.garrison for all tiles owned by any player)
```

- 防止旧日无限堆兵碾压玩家
- 玩家扩张有效驻军 → 旧日危险区可用兵力池也增大 → 形成动态平衡

#### 约束 2：出生点限制

> 当 **所有危险区区块均被玩家占领** 后，旧日只能将部队派往 **危险区范围内** 的区块（即使这些区块当前属于玩家）。

- 旧日失去全图自由出兵能力
- 形成明确的「阶段性胜利条件」

#### 约束 3：单格驻军上限（建议新增）

> 旧日在任意单格的驻军不得超过 **500**（可配置）。

- 防止旧日单格堆 1000 兵拖战斗
- 与玩家 1000 上限形成非对称但合理的差异

### 6.4 旧日 AI 策略（链下 Bot）

Bot 周期性（如每 30 秒）扫描链上状态，按优先级决策：

| 优先级 | 策略 | 条件 |
|--------|------|------|
| P0 | 回防 | 己方区块正在被攻击且驻军不足 |
| P1 | 反击 | 玩家边境区块驻军薄弱 |
| P2 | 扩张 | 中立区块或低防区块 |
| P3 | 压制 | 危险区边界增兵 |

Bot 调用合约前，合约会校验旧日约束；不合法的操作 revert。

## 7. 战斗系统

### 7.1 战斗触发

战斗可通过以下三种方式触发：

#### 方式 A：行军途经旧日之城（新增）

行军推进时进入 **旧日所属的可驻守区块**，立即触发战斗：

1. 行军暂停（`march.active = false`）
2. 行军兵力转入 `Battle.atkTroops`
3. 目标格驻军作为 `Battle.defTroops`
4. 若旧日格无驻军 → 直接占领，不创建战斗

#### 方式 B：行军抵达进攻目标

行军抵达终点且 `isAttack = true`：

1. 目标格有敌军驻军 → 创建 Battle
2. 目标格无驻军/中立 → 直接占领

#### 方式 C：本格/邻格增援触发（已有）

被攻击格已有 active Battle 时，增援加入 `defTroops`（见第 8 章）。

**公共校验：**

1. 调用者有足够流动积分（行军发起时扣除）
2. 目标格为 **可驻守地形**
3. 该格同一时间只允许一场 active 战斗
4. 操作前自动 tick 当前战斗/推进行军

### 7.2 战斗状态（链上）

```solidity
struct Battle {
    address attacker;           // 攻击方
    address defender;           // 防守方（格主）
    uint256 tileId;             // 战斗发生格
    uint256 atkTroops;          // 攻击方投入兵力
    uint256 defTroops;          // 防守方当前兵力（含初始驻军+已到增援）
    uint64 startTime;           // 战斗开始时间
    uint64 lastTickTime;        // 上次结算时间
    bool active;                // 是否进行中
}
```

### 7.3 战斗规则：相互消融

> 两军交战时，**每分钟**双方各损失 **50 积分等价兵力**。

**惰性结算公式：**

```
elapsed     = block.timestamp - battle.lastTickTime
ticks       = elapsed / 60
totalLoss   = ticks * 50

atkTroops  -= totalLoss
defTroops  -= totalLoss
lastTickTime = lastTickTime + ticks * 60
```

- 任何玩家可调用 `tickBattle(tileId)` 触发结算
- 玩家操作（增援、新攻击）前，合约 **自动** 先 tick 当前战斗
- 不需要 keeper 或定时交易

### 7.4 战斗结束条件

| 条件 | 结果 |
|------|------|
| `atkTroops <= 0` 且 `defTroops > 0` | 防守成功，归属不变 |
| `defTroops <= 0` 且 `atkTroops > 0` | 攻击成功，归属变为攻击方，剩余 `atkTroops` 成为新驻军 |
| 双方均 <= 0 | 归属不变，驻军归零（平局） |

### 7.5 战斗锁定

- 一个区块 **同一时间只允许一场 active 战斗**
- 战斗进行中，第三方不可对该格发起新攻击
- 增援操作不受此限制（增援进入 `defTroops`）

### 7.6 战斗流程图

```
startMarch（进攻路径）
      │
      ▼
行军推进（5min/格）
      │
      ├── 途经旧日之城 ──→ 立即 Battle
      ├── 途经玩家之城 ──→ 直接通过
      ├── 途经通路/中立 ──→ 直接通过
      │
      ▼
抵达终点
      │
      ├── 有驻军 → Battle
      └── 无驻军 → 直接占领
      │
      ▼
[tickBattle：每分钟双方 -50]
      │
      ▼
resolveBattle → 更新 owner + garrison
```

## 8. 增援与协防系统

### 8.0 增援类型总览

| 类型 | 函数 | 来源 | 到场 | 触发条件 |
|------|------|------|------|----------|
| **直接驻防** | `dispatchDefend` | 流动积分 | 即时 | 目标格当前兵力 < 1000 |
| **邻格协防** | `reinforceFromNeighbor` | 邻格驻军 | 5 分钟 | 被攻击格兵力 ≤ 500 |
| **紧急增援** | `emergencyReinforce` | 流动积分 | 5 分钟 | 目标格当前兵力 < 1000 |

**全局硬约束：任意区块的兵力总量永远不得超过 1000，不允许累积超限。**

### 8.1 直接驻防（玩家派遣）

> 玩家用 **流动积分** 向己方区块增兵，**即时生效**。

**规则：**

| 项 | 值 |
|----|-----|
| 来源 | `Player.liquidPoints` |
| 到场 | 即时（无延迟） |
| 前置条件 | 目标格当前 `garrison`（或战斗中 `defTroops`）**严格小于 1000** |
| 数量限制 | `当前兵力 + 派遣量 ≤ 1000`，超出则 revert |
| 适用场景 | 和平时期在本方城池内追加驻军 |

> **注意**：向非相邻或跨格增援/进攻一律使用 `startMarch`，不再支持即时跨格传送。

**校验逻辑：**

```solidity
uint256 currentTroops = battle.active ? battle.defTroops : tile.garrison;
require(currentTroops < MAX_GARRISON_PER_TILE, "Already at cap");
require(currentTroops + amount <= MAX_GARRISON_PER_TILE, "Would exceed 1000 cap");
```

### 8.2 邻格协防

> 被攻击格兵力 **≤ 500** 时，**相邻格** 可每 **5 分钟** 从 **自身驻军** 抽调 **100** 积分协防。

**规则：**

| 项 | 值 |
|----|-----|
| 触发条件 | ① 目标格正在被攻击；② 目标格当前 `defTroops ≤ 500`；③ 邻格与目标格属同一玩家 |
| 增援量 | 固定 100 积分/次 |
| 冷却 | 5 分钟/邻格（不累积，到期仅可触发 1 次） |
| 来源 | **从邻格 `garrison` 扣除 100**（消耗自身驻军，非凭空生成） |
| 到场 | 5 分钟后加入战斗 `defTroops` |
| 上限约束 | 增援后目标格 `defTroops ≤ 1000`，否则 revert |
| 邻格前置 | 邻格 `garrison ≥ 100`（自身兵不够则无法协防） |

**设计意图：**

- 兵力 > 500 时不需要邻格支援 → 避免过度协防
- 消耗邻格自身驻军 → 协防有代价，不能无限支援
- 不累积 → 防止长时间不上线后一次性爆发协防

**校验逻辑：**

```solidity
require(battle.active, "Target not under attack");
require(battle.defTroops <= 500, "Garrison above 500, no neighbor reinforce");
require(tiles[fromTile].garrison >= 100, "Neighbor garrison insufficient");
require(block.timestamp >= lastNeighborReinforce[fromTile][toTile] + 300, "Cooldown");
require(battle.defTroops + 100 <= 1000, "Would exceed 1000 cap");

tiles[fromTile].garrison -= 100;   // 消耗邻格驻军
// 5 分钟后加入 battle.defTroops
```

**链上记录：**

```solidity
mapping(uint256 => mapping(uint256 => uint64)) neighborLastReinforce; // fromTile → toTile → timestamp
```

### 8.3 在线紧急增援

> 玩家用 **流动积分** 向正在被攻击的己方区块增援，**5 分钟到场**。

**规则：**

| 项 | 值 |
|----|-----|
| 来源 | `Player.liquidPoints` |
| 到场时间 | 5 分钟 |
| 前置条件 | 目标格当前 `defTroops < 1000` |
| 数量限制 | `defTroops + amount ≤ 1000`，超出则 revert |
| 与协防区别 | **无 500 兵力门槛**，任何时候只要 < 1000 即可派遣 |

**链上结构：**

```solidity
struct PendingReinforcement {
    address from;           // 增援发起者
    uint256 tileId;         // 目标格
    uint256 amount;         // 增援数量
    uint64 arriveTime;      // 到场时间
    bool consumed;          // 是否已生效
}
```

### 8.4 增援上限与累积规则（已确认）

| 规则 | 说明 |
|------|------|
| **硬上限 1000** | 任意区块兵力（garrison 或 defTroops）永远 ≤ 1000 |
| **不允许超限累积** | 协防/增援不允许排队累积到超过 1000；每次操作实时校验 |
| **协防 500 门槛** | 仅 `defTroops ≤ 500` 时邻格才可协防 |
| **直接驻防无门槛** | 只要当前 < 1000，玩家可直接派遣任意合法数量 |
| **邻格消耗自身** | 协防必须从邻格 garrison 扣除，不能从流动积分扣 |

### 8.5 增援决策流程图

```
玩家选择增援方式
      │
      ├── 直接驻防 (dispatchDefend)
      │     当前兵力 < 1000？
      │       是 → 扣除流动积分 → 即时增加兵力
      │       否 → revert
      │
      ├── 邻格协防 (reinforceFromNeighbor)
      │     目标格被攻击 且 defTroops ≤ 500？
      │       是 → 邻格 garrison ≥ 100 且冷却已过？
      │              是 → 扣除邻格 100 驻军 → 5min 后加入 defTroops
      │              否 → revert
      │       否 → revert（兵力充足，不需要协防）
      │
      └── 紧急增援 (emergencyReinforce)
            目标格被攻击 且 defTroops < 1000？
              是 → 扣除流动积分 → 5min 后加入 defTroops（≤ 1000）
              否 → revert
```

## 9. 经济系统

### 9.1 积分类型

| 类型 | 存储位置 | 说明 |
|------|----------|------|
| 流动积分 | `Player.liquidPoints` | 可自由使用：进攻、驻防、紧急增援 |
| 驻军（冻结积分） | `Tile.garrison` | 已派遣到区块，不可直接使用 |
| 战斗中兵力 | `Battle.atkTroops / defTroops` | 战斗中的临时锁定 |

### 9.2 积分流转

```
流动积分 ──startMarch──→ 行军兵力（March.troops）──推进──→ 占领/战斗
流动积分 ──dispatchDefend──→ 驻军（Tile.garrison，仅可驻守地形）
邻格驻军 ──reinforceFromNeighbor──→ 战斗防守（Battle.defTroops）
流动积分 ──emergencyReinforce──→ 待到场增援 → 战斗防守
战斗损耗 ──每分钟-50──→ 永久销毁
```

### 9.3 积分来源与 sink

| 来源 | 数量 |
|------|------|
| 初始分配 | 1000（玩家） |
| 每日补给 | 100/天（玩家），10000/天（旧日） |

| Sink | 说明 |
|------|------|
| 战斗消融 | 每分钟双方各 -50，永久销毁 |
| 占领空区 | 派遣量转为驻军（非销毁，是转换） |

### 9.4 积分体系（已确认：内置）

Week 2 使用 **内置积分**，所有积分操作在 `EchoMarch.sol` 内完成：

## 10. PvP 规则

### 10.1 玩家间攻伐

- 玩家可对 **其他玩家** 的区块派发行军进攻，规则与 PvE 相同
- **途经其他玩家城池不触发战斗**，仅抵达目标时开战
- 旧日约束 **不适用** 于玩家间战斗
- 单格驻军上限 1000 对所有人一致

### 10.2 并发冲突处理

| 场景 | 处理方式 |
|------|----------|
| 两人同时攻击同一格 | 先到先得（区块交易排序），第二笔 revert |
| A 攻 B 格，B 攻 A 格 | 两格各自独立战斗，互不影响 |
| 行军途中被第三方攻击 | 行军部队不在格上驻留，不受协防影响；抵达后才参与战斗 |
| 行军途中目标格归属变更 | 抵达时按 **当前** 归属判定，不按出发时快照 |

### 10.3 新玩家保护（可选，后续版本）

- 注册 **24 小时内** 不可被其他玩家攻击
- 或出生区 3×3 为保护区

MVP 阶段 **不实现**，但预留 `Player.registeredAt` 字段。

### 11.2 惰性结算（Lazy Settlement）策略

所有时间相关逻辑 **不依赖定时交易**：

| 机制 | 触发时机 | 计算方式 |
|------|----------|----------|
| 行军推进 | advanceMarch / 任意操作前 | `(now - lastMove) / 300` 格 |
| 战斗消融 | tickBattle / 任意操作前 | `(now - lastTick) / 60 * 50` |
| 邻格协防冷却 | reinforceFromNeighbor | `(now - lastReinforce) >= 300` |
| 紧急增援到场 | tickBattle / resolveBattle | `now >= arriveTime` |
| 每日补给 | 任意玩家操作 | `now - lastClaim >= 86400` |

## 15. 数值平衡与约束

### 15.1 关键数值一览

| 参数 | 值 | 设计意图 |
|------|-----|----------|
| 玩家初始积分 | 1000 | 够初期扩张 1-2 格 |
| 玩家日补给 | 100 | 稀缺，需策略分配 |
| 旧日日补给 | 10000 | 压倒性但受约束 |
| 单格驻军上限 | 1000 | 防止无限堆兵 |
| 旧日单格上限 | 500 | 非对称但可攻破 |
| 战斗消融 | 50/min/方 | 100 兵力 = 2 分钟战斗 |
| 邻格协防 | 100/5min，消耗邻格驻军 | 仅 defTroops ≤ 500 时可触发 |
| 行军速度 | 1 格 / 5 min | 跨格派遣基础速度 |
| 途经旧日 | 立即开战 | 行军的 PvE 核心触发 |
| 途经玩家 | 直接通过 | 不触发战斗，鼓励绕路奇袭 |

### 15.2 平衡性分析

**旧日 vs 玩家：**

- 旧日积分补给是玩家 100 倍，但危险区兵力受玩家总驻军约束
- 玩家占格越多 → 旧日可用危险区兵力越多 → 旧日反扑越强
- 形成「此消彼长」的动态平衡

**协防 vs 攻击：**

- 协防仅在 `defTroops ≤ 500` 时触发 → 前半段战斗靠自身，后半段才可求援
- 协防消耗邻格驻军 → 多线作战时邻格兵力被掏空，无法无限支援
- 4 邻格 × 100/5min = 理论最多 400/5min，但邻格自身需有兵
- 战斗消融 50/min/方，协防可延缓但难以逆转强势进攻

**地形 vs 行军：**

- 山地/水域形成天然 choke point，迫使行军绕路
- 通路提供快速通道但不可设防 → 「有捷径但无据点」
- 攻击旧日必须途经其城池或正面抵达，无法「传送」

**行军 vs 战斗：**

**PvP：**

- 同等规则，纯策略对抗
- 多线作战分散兵力是核心策略

### 15.3 建议模拟场景

开发过程中应手动或脚本模拟：

1. 1 玩家 vs 旧日，20 天（游戏日）走势
2. 2 玩家互相攻伐 + 旧日搅局
3. 危险区全丢后旧日反扑能力

DAY2\_WEB3\_笔记

**持有加密货币**

**加密货币**

**概念简介**

定义：加密货币是用密码学保障交易安全的数字货币，基于区块链在去中心化网络中运行，不依赖银行或政府

起点：比特币 2009 年由中本聪创建，解决了"双花问题"——同一笔数字资产不会被重复使用

现状：截至 2024 年全球数万种加密货币、总市值数万亿美元，已催生 DeFi、NFT、DAO 等应用，覆盖金融、艺术、游戏

**核心特性**

**去中心化**

加密货币的发行和管理不受任何中央机构控制，交易通过点对点网络进行验证和记录。没有单一实体能够任意增发货币、冻结账户或审查交易。这种去中心化特性赋予了用户真正的财务自主权。

**透明性与匿名性**

区块链上的所有交易都是公开透明的，任何人都可以查询和验证。但交易地址通常是匿名的（如比特币地址是哈希值），提供了一定程度的隐私保护。部分加密货币（如 Monero、Zcash）使用高级密码学技术提供更强的隐私保护。

**稀缺性与可编程性**

多数加密货币有固定或可预测的供应上限，如比特币总量限制在 2100 万枚，通过算法保证稀缺性。智能合约平台（如以太坊）使得加密货币具有可编程性，可以实现复杂的金融逻辑和自动化执行。

**全球性与 7x24 交易**

加密货币不受地理和时区限制，可以在全球范围内 7×24 小时交易，无需依赖传统银行的营业时间和跨境转账系统。转账速度从几分钟到几秒不等，远快于传统银行的跨境汇款（通常需要数天）。

**核心优势**

**安全性**：密码学算法和区块链的不可篡改性保证了交易安全和历史记录的可信。

**低成本**：跨境转账无需中介机构，手续费远低于传统金融系统（尤其在 Layer2 网络上）。

**金融普惠**：任何有互联网连接的人都可以使用加密货币，无需银行账户或信用记录，为全球 17 亿无银行账户人口提供了金融服务。

**可编程性**：智能合约使得加密货币可以承载复杂的金融逻辑，催生了 DeFi、NFT 等创新应用。

**抗审查**：去中心化网络使得加密货币难以被单一权威机构禁止或冻结。

**Token**

在区块链技术中，Token（代币）是一个非常重要的概念。Token 是基于区块链技术创建的可交换单位，可以代表价值、资产、权益、投票权等各种数字化资产或权限。

**分类**

1.  功能（Functional）代币：
    

这些代币被广泛用于特定的应用中，例如作为访问某个服务的通行证。

例子：以太坊上的 Golem（GNT），用于分布式计算的支付。

2.  证券（Security）代币
    

这些代币代表了对公司或资产的所有权或权益

3.  稳定（Stable）代币：
    

这些代币的价值与某种外部资产挂钩，如法定货币（美元、欧元）或其他形式的资产。

例子：Tether（USDT），其价值与美元挂钩。

4.  非同质化代币（NFTs）：
    

这些代币是独一无二的，用于代表独特的资产，如艺术作品、游戏内物品等。

例子：CryptoKitties、数字艺术品等。

**发行和标准**

1.  ICO 和 IEO
    

ICO（Initial Coin Offering）和 IEO（Initial Exchange Offering）是项目通过发行代币来筹集资金的方法。ICO 通常是在自己的平台上直接发行；IEO 则是通过交易所进行的。

2.  标准
    

Token 通常通过智能合约创建，智能合约定义了代币的规则和行为。以太坊的 ERC-20 和 ERC-721 是常见的智能合约标准。

**用途**

1.  支付：
    

-   Token 可以用作支付手段，类似于加密货币。
    

2.  资产表示：
    

-   代表股票、债券、房地产等各种资产。
    

3.  激励机制：
    

-   用于激励用户参与和贡献，如在去中心化应用（DApps）中用于奖励用户的活动。
    

4.  治理：
    

-   用于投票和项目治理，持有代币的用户可以对提案和更新进行投票。
    

总之，Token 是区块链生态系统中不可或缺的组成部分，发挥着多种多样的功能。它们不仅仅是金融资产，还在越来越多的应用场景中展现出其灵活性和多功能性。

**创建钱包**

**钱包**

**区块链钱包**

区块链钱包是用户安全地管理他们的数字资产的硬件设备、软件程序或在线服务。

钱包本身并不存储资产，资产是存在于分布式的区块链网络上。钱包更像是一个密钥管理器。

**钱包分类**

1.  软件钱包（Software Wallets）：
    

 这些是安装在计算机、智能手机或平板电脑上的应用程序，用于管理数字资产。形式可以为桌面钱包、移动钱包和插件钱包等。如：MetaMask、Trust Wallet 、ImToken 、Electrum 等

2.  硬件钱包（Hardware Wallets）：
    

 这是一种物理设备，类似于 USB 驱动器，用于存储比特币的私钥（或助记词），并处理签名。硬件钱包通常被认为是一种更加安全的选项，因为它们通常是离线存储，不容易受到网络攻击。

3.  纸钱包（Paper Wallets）：
    

 纸钱包是将私钥（或助记词）以纸质形式印刷出来的方式。完全离线、冷存储，但如果纸钱包丢失或损坏，资金将无法恢复。

4.  Web 钱包（Web Wallets）
    

：通过网络浏览器访问，尽管它们提供了易用性，但也有一定的安全风险，容易丢失。

**钱包常见功能**

1.  管理**私钥**（或助记词）：提供备份和恢复功能，私钥是数字资产的所有权证明。
    
2.  签署交易：使用私钥签署交易，确保交易的真实性和合法性。
    
3.  查看余额和交易历史：钱包允许用户查看其加密货币的余额和交易历史。
    
4.  发送和接收加密货币：用户可以通过钱包地址发送和接收比特币、以太坊等加密货币。
    
5.  多币种支持：许多钱包支持多种加密货币，使用户能够在同一钱包中管理不同类型的数字资产。
    

**DAPP**

**去中心化应用 - DApp（Decentralized Application）**

去中心化应用是指一类运行在分布式网络上的应用程序，而不是依赖于单个中央服务器。通常DApp 的核心逻辑由区块链上的[智能合约](https://learnblockchain.cn/tags/%E6%99%BA%E8%83%BD%E5%90%88%E7%BA%A6)负责，理想情况下 DApp 的前端也托管在去中心化存储网络中。

DApp 前端界面需要集成[钱包](https://learnblockchain.cn/tags/%E9%92%B1%E5%8C%85)（或与钱包链接），当用户需要调用合约程序时，用用户钱包获取程序调用的签名，实现真正有用户自己控制自己的数据所有权。

部分DApp 也有提供自己的后端程序，但通常是用来做链上数据的缓存，方便用户快速访问链上数据。

**DApp 的主要特征：**

-   去中心化: 与依赖集中式服务器的传统应用程序不同，DApp 在去中心化网络上运行，确保没有单点故障，降低了审查风险。
    
-   开源: DApp的源代码通常对任何人开放，以检查，验证和贡献，促进社区内的信任和协作。
    
-   区块链集成: DApp通常与区块链上的智能合约交互。智能合约是自动执行的合约，协议条款直接写入代码，促进无需信任和自动化的交易。
    
-   代币化: 许多DApp使用加密货币或代币(tokens) 作为应用程序内价值转移的手段。这些代币可用于如治理、访问服务或奖励用户等各种目的。
    
-   不可变性: 区块链上的数据是不可变的，一旦交易或数据记录下来，就不能被更改或删除。这确保了数据的完整性和透明性。
    

**常见应用领域**

由于 DApp 具有去中心化、抗审查、透明、安全和全球可访问等优势，展示了巨大的潜力和发展前景：

1.  去中心化金融（DeFi）：包括去中心化交易所（DEX）、借贷平台、稳定币、衍生品交易等。DeFi dApp 通过智能合约提供金融服务，消除了对传统金融中介的依赖。例如[Uniswap](https://uniswap.org/)[Aave](https://aave.com/)[Compound](https://compound.finance/)
    
2.  游戏和娱乐：去中心化游戏和虚拟世界（如 [Decentraland](https://decentraland.org/)[Axie Infinity](https://axieinfinity.com/)
    
3.  社交媒体: 去中心化社交平台旨在让用户控制他们的数据和内容，例如[friend.tech](https://www.friend.tech/)[Mastodon](https://joinmastodon.org/)
    
4.  去中心化自治组织（DAO）：DAO 是通过智能合约运行的组织，成员可以通过代币投票来决策组织的运营和发展。
    
5.  数字身份和认证：利用区块链的不可篡改性和透明性，dApps 可以用于管理和验证数字身份，提供去中心化的身份认证服务。
    
6.  供应链管理：通过区块链记录产品的生产、运输和销售过程，dApp 可以提供更透明和可追溯的供应链管理系统，例如[VeChain](https://vechain.org/)
    
7.  内容版权和分发：创作者可以通过 dApps 直接发布和出售内容，确保版权保护，并直接获得收益。
    

DApp 在大规模采用上，也面临一些问题，dApps 的用户体验通常不如传统应用，主要因为钱包交互不够便利及区块链交易的等待时间较长，一些应用还面临法律和监管上的挑战。

不过相信随着技术的进步和法律框架的完善，DApp 有望在更多领域得到广泛应用和普及。

**DAO**

**DAO (去中心化自治组织)**

**概念简介**

DAO (Decentralized Autonomous Organization,去中心化自治组织)是一种基于区块链技术的组织形式,通过智能合约编码规则和决策流程,实现无需传统中心化管理层即可运作的组织结构。DAO 的核心理念是将组织的控制权和所有权分散给社区成员,通过代币持有者投票来做出集体决策。

DAO 的概念最早由以太坊联合创始人 Vitalik Buterin 在 2013 年提出,并在 2016 年随着"The DAO"项目的推出而引起广泛关注。虽然第一个 DAO 因为智能合约漏洞遭受攻击而失败,但这一理念在随后几年中不断演进完善。到 2024 年,DAO 已成为 Web3 生态系统中不可或缺的组织形式,广泛应用于 DeFi 协议治理、NFT 社区管理、投资基金运作等多个领域。

DAO 代表了一种全新的组织范式,它挑战了传统的公司治理模式,通过透明的链上治理机制、自动化的执行流程和去中心化的决策权,为全球协作提供了新的可能性。目前,全球数千个 DAO 管理着数百亿美元的资产,参与者遍布世界各地。

**核心特性**

**智能合约驱动**

DAO 的规则和运作逻辑完全编码在智能合约中,这些合约自动执行预定义的操作,无需人工干预。例如,当提案获得足够的投票支持时,智能合约会自动执行相应的资金转移或参数修改。这种自动化机制消除了对中介和管理层的依赖,确保了规则的客观执行。智能合约的透明性使得所有成员都能够审计组织的运作流程,任何人都无法单方面篡改规则。

**代币化治理**

DAO 通常发行治理代币,持有这些代币的成员可以对组织的重大决策进行投票。投票权重通常与代币持有量成正比,但也有采用二次方投票等更复杂机制的 DAO。代币化治理使得组织的控制权分散给社区,避免了权力集中。成员可以通过购买、赚取或获赠代币来参与治理,这种开放性是传统组织难以实现的。

**提案与投票机制**

DAO 的决策通过提案和投票流程进行。任何成员(或满足一定条件的成员)都可以提交提案,建议组织采取某项行动或修改某项规则。其他成员通过投票表达支持或反对,当提案达到预设的通过条件(如超过 50% 的赞成票或达到法定人数)时,智能合约会自动执行提案内容。这种民主化的决策机制确保了所有利益相关者都有发言权。

**国库管理**

DAO 通常拥有一个由智能合约控制的国库(Treasury),用于存放组织的资产。国库资金的使用需要经过治理提案和投票,确保资金按照社区意愿分配。一些 DAO 使用多签钱包或时间锁机制来进一步保护国库安全,防止恶意提案或快速攻击。国库的透明性使得所有成员都能实时查看组织的财务状况。

**无需许可的参与**

大多数 DAO 是开放和无需许可的,任何人都可以通过获取治理代币来加入组织并参与治理。这种全球化的参与模式打破了地理、国籍和身份的限制,使得来自世界各地的人可以为共同目标而协作。无需许可的特性也降低了参与门槛,促进了组织的多样性和包容性。

**技术优势**

**透明度与可审计性**

DAO 的所有活动,包括提案、投票、资金流动等,都记录在区块链上,完全透明且可审计。任何人都可以查看组织的历史决策和财务记录,这种透明度大大降低了腐败和暗箱操作的可能性。相比传统组织的不透明治理,DAO 提供了前所未有的问责机制。

**抗审查性**

由于 DAO 运行在去中心化的区块链网络上,没有任何单一实体能够关闭或审查组织的运作。即使某些成员或节点被禁止,DAO 仍可继续运行。这种抗审查性对于那些在传统体系中受到限制的组织尤为重要,例如跨国协作项目或政治敏感领域的组织。

**全球协作**

DAO 使得全球范围内的陌生人可以为共同目标协作,无需建立传统的信任关系。智能合约的自动执行确保了承诺的履行,代币激励机制协调了参与者的利益。这种新型协作模式已经催生了众多创新项目,从开源软件开发到艺术品收藏,从投资基金到慈善事业。

**灵活性与可编程性**

DAO 的治理规则可以根据需要进行调整和优化。通过提案和投票,组织可以修改投票规则、添加新功能或调整激励机制。智能合约的可编程性使得 DAO 可以实现复杂的治理逻辑,如二次方投票、委托投票、多阶段投票等,以适应不同的治理需求。

**发展历程**

**2013-2015 年:概念萌芽**

Vitalik Buterin 提出 DAO 概念,认为区块链和智能合约技术可以创建无需中心化管理的自治组织。以太坊的推出为 DAO 的实现提供了技术基础。

**2016 年:The DAO 事件**

德国创业公司 [Slock.it](http://Slock.it) 推出了"The DAO"项目,通过众筹募集了价值约 1.5 亿美元的以太坊,成为当时最大的众筹项目。然而,由于智能合约存在重入攻击漏洞,黑客窃取了约三分之一的资金。这一事件导致以太坊社区进行了有争议的硬分叉来恢复被盗资金,也促使行业更加重视智能合约安全。

**2017-2019 年:低谷与反思**

The DAO 事件后,DAO 概念进入低谷期。开发者和研究者专注于改进智能合约安全、治理机制和投票系统。这一时期诞生了 Aragon、DAOstack 等 DAO 基础设施项目,为后来的 DAO 复兴奠定了基础。

**2020-2021 年:DeFi Summer 与 DAO 复兴**

DeFi 的爆发式增长推动了 DAO 的复兴。Uniswap、Compound、Aave 等 DeFi 协议通过发行治理代币实现了去中心化治理。这些协议 DAO 管理着数十亿美元的资产,证明了 DAO 治理的可行性。同时,Constitution DAO、PleasrDAO 等社会实验性 DAO 也引起了广泛关注。

**2022-2024 年:多样化发展**

DAO 的应用场景不断扩展,从 DeFi 协议治理延伸到投资基金(BitDAO、Krause House DAO)、内容创作(Mirror、Rabbithole)、公共物品资助(Gitcoin、MolochDAO)、社交网络(Friends with Benefits)等多个领域。DAO 工具和基础设施也日益成熟,Snapshot、Tally、Gnosis Safe 等工具被广泛使用。

**主要类型**

**协议 DAO**

管理 DeFi 协议或区块链基础设施的 DAO,如 Uniswap DAO、Aave DAO、MakerDAO 等。这类 DAO 负责决定协议的参数调整、新功能开发、资金分配等重大事项,通常拥有庞大的国库和活跃的治理社区。

**投资 DAO**

成员共同决策投资方向的 DAO,如 The LAO、MetaCartel Ventures、BitDAO 等。投资 DAO 允许成员集体管理投资组合,共享投资收益和风险,适合希望分散投资或参与早期项目的投资者。

**收藏 DAO**

专注于收藏和管理 NFT 或其他数字资产的 DAO,如 PleasrDAO、FlamingoDAO、Constitution DAO 等。这类 DAO 通过集体资金购买高价值的艺术品或历史文物,成员共享所有权。

**社交 DAO**

围绕共同兴趣或社交目的组织的 DAO,如 Friends with Benefits(FWB)、Bright Moments 等。社交 DAO 通过代币门槛筛选成员,为成员提供独家内容、活动和社交网络。

**服务 DAO**

提供专业服务的去中心化组织,如 LexDAO(法律服务)、DeveloperDAO(技术服务)、Raid Guild(开发服务)等。服务 DAO 汇聚专业人才,为 Web3 项目提供咨询、开发、设计等服务。

**公共物品 DAO**

致力于资助公共物品和开源项目的 DAO,如 Gitcoin DAO、MolochDAO、Protocol Guild 等。这类 DAO 通过二次方资助等创新机制,支持对生态系统有益但难以盈利的项目。

**治理工具**

**Snapshot**

最流行的链下投票平台,允许 DAO 进行无 Gas 费的投票。Snapshot 支持多种投票策略和提案类型,被数千个 DAO 使用。

**Tally**

链上治理平台,支持提案创建、投票和执行。Tally 提供友好的用户界面和治理分析工具,帮助社区参与链上治理。

**Gnosis Safe**

多签钱包和资产管理工具,被广泛用于 DAO 国库管理。Gnosis Safe 支持多人共同管理资金,提高了资产安全性。

**Aragon**

完整的 DAO 创建和管理平台,提供模块化的 DAO 框架和治理工具。Aragon 允许用户无需编程即可创建和定制 DAO。

**DAOhaus**

基于 Moloch DAO 框架的 DAO 平台,专注于简洁高效的治理机制。DAOhaus 适合小型协作组织和资助项目。

**Colony**

专注于任务管理和贡献奖励的 DAO 平台,适合面向项目和工作流的组织。

**应用场景**

**DeFi 协议治理**

主流 DeFi 协议如 Uniswap、Aave、Compound 都采用 DAO 治理模式,社区决定协议的发展方向、参数调整和资金使用。

**NFT 社区管理**

NFT 项目如 Nouns DAO、ApeCoin DAO 通过 DAO 管理社区资金,决定品牌发展和生态建设。

**风险投资**

The LAO、MetaCartel Ventures 等投资 DAO 为早期 Web3 项目提供资金,成员共同决策投资方向。

**内容创作与媒体**

Mirror、Rabbithole 等平台支持创作者通过 DAO 与粉丝互动,共同决定内容方向和收益分配。

**公益与慈善**

Big Green DAO、Ukraine DAO 等组织通过 DAO 进行透明的慈善募捐和资金分配。

**社区治理**

城市 DAO、社区 DAO 等尝试将 DAO 应用于现实世界的社区治理和资源协调。

**挑战与风险**

**投票率低**

许多 DAO 面临投票参与率低的问题,大量代币持有者不参与治理,导致少数大户或团队控制决策。改进投票激励机制和降低参与门槛是重要课题。

**治理攻击**

拥有大量代币的个人或组织可能操纵投票结果,通过恶意提案损害 DAO 利益。时间锁、多签审核、二次方投票等机制可以缓解这一问题。

**智能合约风险**

DAO 的运作依赖智能合约,合约漏洞可能导致资金损失。The DAO 事件就是典型案例。严格的安全审计和漏洞赏金计划是必要的防范措施。

**法律地位不明**

DAO 的法律地位在大多数司法管辖区尚不明确,成员的责任、税务处理、知识产权归属等问题缺乏清晰的法律框架。一些 DAO 尝试在怀俄明州等地注册为有限责任公司(DAO LLC)以获得法律保护。

**协调难题**

大规模 DAO 面临协调挑战,成员的利益和观点可能存在冲突,达成共识需要漫长的讨论和投票过程。这可能导致决策效率低下,难以应对快速变化的市场环境。

**中心化风险**

虽然 DAO 追求去中心化,但实践中许多 DAO 仍存在中心化倾向,如核心团队掌握过大权力、代币分配过于集中、关键基础设施由单一实体控制等。

**最佳实践**

**渐进式去中心化**

新项目可以采用渐进式去中心化策略,早期由核心团队主导快速迭代,随着项目成熟逐步将控制权转移给社区。

**多层治理**

设计多层次的治理结构,对日常运营和重大决策采用不同的投票机制和门槛,提高治理效率。

**透明沟通**

保持与社区的频繁沟通,使用论坛、Discord、推特等渠道讨论提案,在正式投票前形成共识。

**专业委员会**

设立专业委员会(如财务委员会、技术委员会)处理专业性强的事务,再向社区报告和寻求批准。

**代币经济学**

合理设计代币分配和释放计划,避免过度集中,同时为长期贡献者和参与者提供激励。

**安全优先**

对所有智能合约进行专业审计,使用时间锁和多签机制保护国库,建立漏洞赏金计划鼓励安全研究。

**法律合规**

咨询法律专业人士,了解所在司法管辖区的法律要求,必要时注册法律实体以保护成员利益。

**著名 DAO 案例**

**MakerDAO**

管理 DAI 稳定币协议的 DAO,拥有数十亿美元资产,是 DeFi 领域最成功的 DAO 之一。

**Uniswap DAO**

去中心化交易所 Uniswap 的治理组织,管理着数十亿美元的国库和协议发展方向。

每天拍卖一个 Nouns NFT,拍卖收入进入国库,持有者投票决定资金用途,是 NFT 与 DAO 结合的创新实验。

2021 年为竞拍美国宪法原件而成立的 DAO,虽然竞拍失败,但展示了 DAO 快速动员资源的能力。

**Gitcoin DAO**

资助以太坊生态公共物品的 DAO,通过二次方资助机制支持开源项目。

管理以太坊域名服务的 DAO,决定 ENS 协议的发展和.eth 域名的规则。

**区块链**

**什么是区块链**

区块链起源于比特币。比特币的诞生是基于P2P网络技术，加密技术，时间戳技术，区块链技术等的电子现金系统的构架理念。2009年1月出现了序号为1的区块，并与序号0的创世区块相连接形成了链，标志着区块链的诞生。

区块链就是区块与区块之间相连组成的链条。每个区块包含了一组交易信息，并与前一个区块的通过哈希（Hash）链接在一起。这个链条被保存在所有的分布式网络的节点（节点可以理解为是一台服务器）中。只要整个系统中有一台节点可以工作，整条区块链数据就是安全的。区块链系统中的节点为整个区块链系统提供存储1空间的算力支持。

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/monad-builder-camp/main/assets/fox896/images/2026-07-06-1783328636670-image.png)

如果要修改区块链上的信息，必须征得半数以上节点的同意并修改所有节点中的信息，而这些节点通常掌握在不同的主体手中，因此篡改区块链的信息是及其困难的。

相比与传统网络，区块链的核心特点：数据难以篡改，去中心化。

**区块链如何工作的**

**核心定位**

区块链是一个有计算能力的分布式数据库，类比传统数据库：

-   用户发起数据修改请求 → 创建交易
    
-   服务器验证请求 → 交易验证
    
-   写入数据库 → 交易打包成区块
    
-   同步给其他服务器 → 区块广播
    

**四大步骤详解**

**一、创建交易**

用户创建交易，指定接收方、发送方和交易金额（有时包括指定运行的链上代码）。使用钱包对交易信息进行签名，然后将交易及签名信息发送给连接的节点。

**二、交易验证**

节点验证交易的合法性，确保发送方有足够余额，并且没有**双花**问题（同一笔钱不能被花两次）。通过验证的交易，节点会将其广播给其他节点。

**三、交易打包成区块**

节点将一批经过验证的交易打包成区块，区块包含：

-   一组交易记录
    
-   前一个区块的哈希值
    
-   时间戳
    

打包区块有奖励，多个节点竞争出块，由**共识机制**决定谁打包的区块有效并延长区块链。

**四、区块广播**

节点将新添加的区块广播到网络中其他节点。其他节点验证区块的合法性，并延长自己的区块链。每个节点都有一个完整的区块链副本，确保数据的分布式和去中心化。

* * *

**共识机制**

共识机制解决分布式节点之间如何达成共识的问题，各个区块链平台通过节点共同遵守的算法协议来实现。

**工作量证明（PoW）**

矿工节点通过解决复杂的数学问题来竞争区块的添加权。解决问题最快的节点将区块添加到区块链中，并获得奖励（如比特币）。

**权益证明（PoS）**

质押一定数量的代币可成为验证者。网络通过随机算法从验证者中选出出块的验证者（以太坊使用此机制）。
