## Agent Bounty Escrow Demo

今天重点推进的是 `Agent 任务托管` 这个 demo。这个项目的想法是：用户发布一个小任务，并把 MON 赏金锁定到智能合约里；执行者接单、提交结果后，发布者确认完成，合约自动把赏金释放给执行者。

我把它理解成一个适合 AI Agent 场景的小额任务托管产品。第一版可以先服务人类执行者，后面再扩展到 AI Agent 自动接单、提交结果和收款。

今天整理出来的核心流程是：

```text
发布任务 -> 锁定赏金 -> 接单 -> 提交结果 -> 确认完成 -> 释放付款
```

已经完成的 demo 内容包括：

- 写了 Solidity 赏金托管合约 `AgentBountyEscrow`。
- 设计了任务状态：Open、Accepted、Submitted、Approved、Cancelled、Refunded。
- 支持发布任务并锁定赏金。
- 支持执行者接单。
- 支持执行者提交结果链接或说明。
- 支持发布者确认完成并释放赏金。
- 支持取消任务和超时退款的基础逻辑。
- 用 React + ethers.js 做了一个毛坯前端。
- 前端支持连接 MetaMask。
- 支持 Monad Testnet 网络识别和切换提示。
- 没有合约地址时可以使用演示模式先跑流程。
- 配置了 GitHub Pages 部署相关内容。

## 今天学习到的内容

### 1. DApp 的完整结构

之前我对 DApp 的理解比较碎片化，今天通过 demo 更清楚地看到它至少包含几层：

- 合约层：定义资产托管、任务状态和结算规则。
- 钱包层：用户用钱包地址作为身份，并用签名确认操作。
- 前端层：展示任务、按钮、状态和交易反馈。
- 网络层：需要确认当前是否在 Monad Testnet。
- 验证层：通过区块浏览器查询交易和合约状态。

### 2. 为什么这个 demo 需要上链

如果只是普通任务列表，其实数据库就够了。但 `Agent Bounty Escrow` 需要上链的原因在于：

- 赏金托管需要公开可信。
- 任务状态变化需要可验证。
- 放款规则需要由合约执行。
- 钱包地址可以积累任务历史和信誉。
- 未来 AI Agent 可以直接用钱包参与任务和收款。

这也是我今天对 Web3 产品更深的一点理解：不是所有功能都应该上链，只有那些需要可信结算、公开验证、资产流转或身份信誉沉淀的部分，才更适合上链。

### 3. Agent 支付方向

晚上读了 7 月 8 日的 AI Agent 支付分享纪要，里面对我启发比较大的是：Agent 如果要真正执行任务，就不可避免会涉及支付。

会议里提到的重点包括：

- Agent 支付目前更适合从小额微支付开始。
- 用户可以先授权预算，Agent 在预算范围内执行支付。
- 一次性 Agent Card 可以降低 Agent 乱花钱的风险。
- X402 这类协议可以让 Agent 直接调用付费 API。
- Agent 支付最核心的问题不是按钮怎么做，而是安全、风控、授权和约束。
- 当前阶段 Base + USDC 是比较现实的支付路径，但不同场景未来可能会适配不同链。

这和我做的 `Agent 任务托管` demo 有一定关联。我的 demo 更偏任务赏金托管，分享会讲的是更大的 Agent 支付生态。它们共同指向一个方向：未来 Agent 不只是聊天或生成内容，而是可以完成任务、调用服务、产生收入和付款。

### 4. AI 写代码可以提速，但安全判断不能外包

今天继续体会到，AI 很适合帮我搭项目骨架、解释报错、整理 README、生成前端和合约的基础版本。但涉及链上资产、合约权限、退款、放款、重入保护、私钥安全时，最后必须自己检查。

尤其是合约类项目，不能因为代码能跑就认为安全。至少要检查：

- 谁能调用这个函数。
- 钱会不会被锁死。
- 退款和放款有没有重复转账风险。
- 状态是不是先更新再转账。
- 有没有把私钥、助记词、`.env` 提交到 GitHub。

## 今天遇到的问题

1. 一开始会把“能跑的前端 demo”和“真的链上 DApp”混在一起。后来理解到，演示模式只是帮助先体验流程，真正链上版本还需要部署合约、配置合约地址、连接正确网络并发起交易。
2. GitHub Pages 部署时需要注意 Vite 的 `base` 配置和环境变量配置，否则页面可能能打开但资源路径或合约地址不对。
3. 对任务数据放链上还是放链下还需要继续判断。第一版为了简单可以把简短说明写入链上，但后续长文本、附件、Agent 输出更适合放 IPFS、URL 或 hash。
4. 当前合约还没有争议处理机制。如果执行者提交了错误结果，第一版只能由发布者决定是否确认，后续可能需要仲裁、多签或评价机制。
