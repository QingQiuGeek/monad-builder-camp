## 二、Vibe Coding（氛围编程）分级体系与AI开发方法论（配套文档+X推文观点）

### 1\. 基础概念科普

Vibe Coding是Karpathy提出的AI驱动新型编程范式：开发者以自然语言描述需求，依靠大模型生成代码，弱化逐行手写、侧重快速验证想法；分享基于`vibe-coding-levels`文档拆解完整能力分级模型，划分从入门到专家的6个成长层级，明确每一层的行为、工具、适用场景与短板。

### 2\. 六级能力分层核心要点

1.  **Level 0：纯手动编码** 完全不使用AI工具，全程手写代码、查阅原生文档，适合底层协议、高安全要求合约（Web3智能合约、资金逻辑），缺点是原型迭代极慢。
    
2.  **Level 1：基础AI补全** 仅用Copilot做单行代码补全，不依赖大模型完整生成文件；适合日常业务增量开发，对代码有完整掌控，但AI赋能程度低。
    
3.  **Level 2：基础Vibe Coding（大众主流层级）** 使用Cursor、[Bolt.new](http://Bolt.new)等AI编辑器，自然语言描述需求让AI生成完整功能，不深度阅读代码、仅测试功能能否运行；适合原型、hackathon、验证创意，**不适合Web3合约、支付等高风险代码**，缺陷是代码不可控、存在安全漏洞隐患。
    
4.  **Level 3：先规范再开发（Spec-First）** 先撰写完整技术需求文档、接口规范，再交给AI生成代码；可追溯、可控，适合中小型正式DApp开发，解决Level2代码混乱、逻辑丢失问题。
    
5.  **Level 4：工具扩展Vibe开发** 接入MCP、数据库、链上API等外部数据源，让AI读取链上真实数据、合约源码再生成代码；适配Web3场景，AI可以结合以太坊EIP、合约源码输出合规链上交互逻辑。
    
6.  **Level 5：自动化工作流与多智能体** 封装固定Prompt、搭建n8n自动化流水线，使用多AI Agent分工协作，无需手动编写基础代码；适合团队规模化开发、协议批量工具开发，是AI开发成熟形态。
    

### 3\. X推文补充的落地建议（Web3开发者专属）

1.  **场景取舍原则** 低风险场景（活动页面、数据看板、概念原型）可用高等级Vibe Coding提速；高风险场景（代币合约、AA账户、资金逻辑）最多到Level1/Level3，必须逐行审计AI生成代码。
    
2.  **Web3开发者结合方案** 前端页面、链下工具、学习平台（如主讲人ETHCorePlay）可用Vibe Coding快速搭建；以太坊底层协议、智能合约核心逻辑坚持手动+逐行校验，规避重入、权限、支付漏洞。
    
3.  **能力成长路线** 建议开发者先熟练手动编码（Level0打底，吃透Solidity、EIP底层），再逐级升级AI开发能力，避免只会AI生成代码、看不懂底层逻辑。
    

## 三、分享会整体核心结论

1.  **职业层面**：Web3开发者具备多元发展路径，不止局限于合约/前端开发，DevRel、协议研究、DAO治理、技术内容创作都是可持续赛道，技术+社区复合能力是核心竞争力；
    
2.  **AI工具层面**：Vibe Coding是高效原型工具，但存在明确边界，Web3区块链、金融类高安全开发不能完全依赖AI；需要遵循分级模型循序渐进，底层技术功底是驾驭AI代码的基础；
    
3.  **实践指引**：华语以太坊开发者可以一边深耕底层协议技术，一边用AI开发工具降低产品、学习平台的开发成本，同步做社区内容输出，打通技术研究与生态传播。

AI 可以帮助完成资料检索、内容整理和初稿生成，但真正的价值仍来自研究者对信息的判断、交叉验证和独立思考。对于我来说，即使主方向是 **Developer（Tech）**，Research 能力依然是必备能力。未来我希望坚持将每一次学习 Monad、Ethereum、Moss 等协议的过程整理成 Build Log、GitHub README 或 Obsidian 知识库，逐步积累自己的 **Public Knowledge** 和 **Proof of Work**。

第二场分享让我更加理解了 **GitHub Contribution** 的意义。在 Web3 生态中，团队更关注的是开发者是否拥有 **GitHub、Pull Request、Demo、技术文章、社区贡献** 等可验证成果，而不仅是学历或证书。

-   阅读 README 和 CONTRIBUTING  
    
-   Fork → Clone → 创建分支  
    
-   提交 Commit 与 Pull Request  
    
-   接受 Code Review 并持续优化  
    

分享还以 **Moss** 开源项目为实践案例，鼓励大家通过真实项目参与 GitHub 协作，积累属于自己的 Web3 Portfolio。

# 📝 今日核心知识整理

## 1\. 钱包（Wallet）

钱包不仅仅是存放资产的工具，更是进入 Web3 世界的身份入口。

它负责：

-   管理账户
    
-   管理私钥
    
-   发起交易
    
-   与 DApp 交互
    

* * *

## 2\. 私钥、助记词、钱包地址

这是今天最重要的知识点。

### 私钥（Private Key）

代表钱包的控制权。

任何拥有私钥的人，都可以控制钱包里的资产，因此绝不能泄露。

### 助记词（Mnemonic）

助记词本质上是私钥的人类可读形式，用于恢复钱包。

同样需要妥善保管。

### 钱包地址（Address）

钱包地址可以公开，相当于银行卡号。

别人可以通过地址向你转账，也可以查询你的链上记录。

* * *

## 3\. 测试网（Testnet）

测试网用于开发和学习。

特点：

-   与主网环境基本一致
    
-   测试币没有真实价值
    
-   可以免费领取
    
-   用来验证应用是否正常运行
    

今天所有操作都是在测试网上完成的。

* * *

## 4\. Faucet（水龙头）

Faucet 是测试币领取平台。

作用：

-   免费发放测试币
    
-   为开发者提供 Gas
    
-   通常每天有领取次数限制
    

没有测试币，就无法完成部署合约、转账等链上操作。

* * *

## 5\. Gas Fee

Gas 可以理解为区块链运行所消耗的燃料。

简单来说：

> **Gas Fee = Gas 消耗量 × Gas 单价**

不同操作消耗不同：

-   普通转账：Gas 较低
    
-   部署智能合约：Gas 较高
    
-   写入链上数据：需要 Gas
    
-   查询链上数据：通常无需支付 Gas
    

今天也第一次体会到，链上每一次「写入」都是有成本的。

* * *

## 6\. 智能合约（Smart Contract）

智能合约就是部署在区块链上的程序。

今天部署的留言板虽然简单，但已经包含了智能合约最基础的几个能力：

-   写入链上数据
    
-   查询链上数据
    
-   记录操作地址
    
-   记录时间戳
    
-   触发事件（Event）
    
-   基础数据校验
    

这让我理解到：

> 智能合约并不是一个网站，而是一段运行在区块链上的程序。

* * *

## 7\. 区块浏览器（Block Explorer）

区块浏览器相当于区块链世界的搜索引擎。

可以查询：

-   钱包地址
    
-   交易记录
    
-   交易 Hash
    
-   合约地址
    
-   Gas 消耗
    
-   区块高度
    
-   时间戳
    

今天也学会了通过交易 Hash 查询一笔交易的完整信息。

* * *

# 💡 今日最大的收获

相比记住定义，我觉得今天最大的收获是理解了 Web3 的整体流程：

**钱包 → 测试网 → 测试币 → Gas → 交易 → 智能合约 → 区块浏览器**

这些概念并不是独立存在，而是串联成了一套完整的链上交互闭环。

只有真正动手完成一次流程，才能理解它们之间的关系。

* * *

# ⚠️ 今日避坑总结

今天也记录了几个新手最容易犯的错误：

-   私钥、助记词绝不能泄露。
    
-   测试币仅用于开发测试，没有真实价值。
    
-   区块浏览器负责查询，不负责交易。
    
-   操作前需要确认钱包已切换到正确网络。
    
-   部署合约和写入数据都需要消耗 Gas。
    
-   智能合约部署后通常不能直接修改，因此正式上线前必须充分测试。
