# 👇 笔记

# 🎄 区块链上有什么：

区块链上除了智能合约，还有以下几类与合约**同等级别**的核心组件：

* * *

## 一、账户（Account）

| 类型 | 说明 |
| --- | --- |
| 外部账户（EOA） | 用户钱包，由私钥控制，能发起交易 |
| 合约账户 | 部署后的合约，由代码逻辑控制 |

**与合约同等级别** — 账户是状态的载体，合约也得挂在账户下运行。

* * *

## 二、交易（Transaction）

-   账户之间的价值转移
    
-   调用合约的入口
    
-   改变链上状态的唯一方式
    

**没有交易，合约永远不会自动执行。**

* * *

## 三、区块（Block）

| 内容 | 作用 |
| --- | --- |
| 打包多笔交易 | 确定执行顺序 |
| 包含区块头信息（时间戳、哈希、Gas限制等） | 保证链的一致性和安全性 |

**区块是共识的单位，合约执行结果必须被打包进区块才最终确认。**

* * *

## 四、状态（State）

-   所有账户余额、合约存储、Nonce 等的**全局快照**
    
-   合约运行就是读写状态
    

**状态 = 链的"数据库"，合约是操作数据库的逻辑。**

* * *

## 五、虚拟机 / 执行环境（如 EVM）

| 对比 | 说明 |
| --- | --- |
| 合约 | 跑在虚拟机里的字节码 |
| EVM | 执行合约的运行时环境 |

**没有 EVM，合约只是一堆死代码。**

* * *

## 六、共识机制

-   PoW、PoS、Monad 的 **MonadBFT** 等
    
-   决定谁出块、交易怎么被确认
    

**共识是链的"操作系统"，合约跑在它之上。**

* * *

## 七、Gas / 费用机制

-   防止滥用计算资源
    
-   合约执行的计价单位
    

**没有 Gas 经济模型，合约可能被无限循环攻击。**

* * *

## 层级关系图

```
┌─────────────────────────────┐
│        共识机制（底层安全）      │
├─────────────────────────────┤
│        区块 / 区块链结构        │
├─────────────────────────────┤
│    虚拟机（EVM）+ Gas 机制     │
├─────────────────────────────┤
│   账户（EOA + 合约账户）        │
├─────────────────────────────┤
│   交易（触发状态变更的唯一方式）   │
├─────────────────────────────┤
│   状态数据库（最终数据落点）      │
├─────────────────────────────┤
│        智能合约（业务逻辑）       │
└─────────────────────────────┘
```

* * *

## 一句话总结

> **合约是"业务逻辑层"，但离开账户、交易、区块、状态、虚拟机、共识、Gas 中的任何一个，它都无法独立存在。**

它们不是合约的附属，而是**平行且相互依赖的核心模块**。

# 💰 链上钱包

链上支付就是“用钱包发起一笔交易，把某个**链上资产**的**控制权**从一个地址转移到另一个地址”。

钱存在链上的账本状态里，表现为某个地址拥有多少余额或多少代币；

> 钱包: 本身不装钱，钱包保存的是私钥，  
> \*\*私钥: 用于证明“我有权动用这个地址上的资产”。

区块链里的“交易”是状态修改请求:

1.  转账只是最常见的一类；
    
2.  也可以是调用智能合约、铸造 NFT、兑换代币、抵押借贷、授权扣款等操作。
    

# **1\. 钱在哪里：钱存在链的状态里，不存在钱包软件里**

⚠️ 注意： 区块链上的“钱”记录在**链的公共账本**中，不在钱包里

钱的形式：

```text
区块链资产
├─ 原生币
│  ├─ 例子：BTC、ETH、SOL、BNB
│  ├─ 用途：支付手续费、转账、参与链上应用
│  └─ 记录方式：链的基础账本直接记录余额或未花费输出
├─ 代币
│  ├─ 例子：USDT、USDC、UNI、LINK
│  ├─ 用途：稳定币支付、治理、交易、质押
│  └─ 记录方式：智能合约内部记录地址余额
└─ NFT
   ├─ 例子：头像、游戏道具、会员凭证
   ├─ 用途：表示唯一权益或凭证
   └─ 记录方式：智能合约记录某个编号归哪个地址所有
```

# ➡️ 关于原生币、代币

|   | 原生币 | 代币 |
| --- | --- | --- |
| 谁发行的 | 链自己"长"出来的 | 别人在链上"造"的 |
| 记在哪 | 链的底层账本 | 智能合约里 |
| 能不能独立活 | 能，没它链就死了 | 不能，寄生在链上 |

|   |   |
| --- | --- |
| 原生币 | 像 国家法定货币（人民币、美元）——国家信用背书，交税、办手续必须用 |
| 代币 | 像 商场积分、游戏点券、公司股票——某个机构发的，只能在特定场景用 |

* * *

## **ETH 是原生币**

-   以太坊这条链自己产生的
    
-   转账 ETH、执行智能合约，**手续费必须用 ETH 付**
    
-   没有 ETH，以太坊这条链就转不动
    

## USDT 是代币

-   Tether 公司在以太坊上发的一个智能合约
    
-   合约里写：“地址 0x123… 有 500 USDT”
    
-   它**依赖以太坊存在**，以太坊死了 USDT 也活不了
    
-   但 USDT 本身不是以太坊"长"出来的，是外人插进来的
    

> **原生币是链的"血液"，代币是链上跑的"应用"。**

没血液人活不了，但应用可以有很多个。

每条链都有一套当前状态，例如某个地址有 1 ETH、某个地址有 200 USDT、某个**合约里锁着多少资产**。

节点共同保存并更新这套状态，所有有效交易都会改变状态。

钱包 App、浏览器插件、硬件钱包只是读取链上状态并帮助用户签名，它们本身不保存资产。

手机丢了，只要私钥或助记词还在，用户可以在另一个钱包里恢复同一个地址并继续控制资产。

# ➡️ 关于合约里锁着资产

"合约里锁着多少资产"指的是

**智能合约作为资金托管方**的常见场景

* * *

🚩 典型场景

| 场景 | 钱怎么"锁"在合约里 |
| --- | --- |
| DEX（去中心化交易所） | 你把 USDT 存进 Uniswap 的流动性池合约，合约记录"这个地址存了 1000 USDT"——此时你的钱包里 USDT 余额变少，合约里多了 1000 |
| 质押（Staking） | 把 ETH 存进 Lido 合约，换出 stETH。你的 ETH 被合约锁定，合约给你发一个凭证代币 |
| 跨链桥 | 把 1 BTC 存进跨链桥合约，合约锁定后在另一条链给你铸造 1 WBTC |
| NFT 交易 | 出价时把 WETH 授权给合约，合约托管这笔资金直到交易完成或取消 |
| 借贷协议 | 抵押 ETH 进 Aave 合约，借出 USDC。ETH 被合约锁定作为担保 |

* * *

🚩 核心区别

|   | 钱包里的余额 | 合约里锁着的资产 |
| --- | --- | --- |
| 记录位置 | 地址本身的 balance | 合约的存储变量中 |
| 控制权 | 私钥直接控制 | 合约代码规则控制 |
| 用户操作 | 直接转账 | 需调用合约函数（deposit/withdraw） |
| 风险点 | 私钥泄露 | 合约漏洞/项目方跑路 |

> **“锁在合约里” = 你把钱交给了智能合约这个"机器人管家"，它按代码规则替你保管，不再直接躺在你钱包地址上。**

# **2\. 钱的形式：链上资产是“账本状态 + 控制权”**

链上资产没有纸币形态，也不是支付宝账户里的平台余额。

它的形式是链上数据：某个地址对应多少资产，某个地址是否拥有某个代币或 NFT。

用户拥有资产的含义：是用户掌握能控制该地址的私钥。私钥可以生成签名，签名可以证明“这笔操作由资产控制者发起”，全网节点验证签名和规则后才会接受这笔交易。

可以把关系压缩成一句话：地址是账户编号，私钥是控制权，钱包是操作工具，链上状态是资产记录。

# **3\. 钱包怎么交互：钱包负责连接、展示、签名、广播**

用户与链上交互通常经过钱包。以 MetaMask、Phantom、OKX Wallet、Rabby 这类钱包为例，它们做四件事：读取地址余额，连接去中心化应用，展示待签名操作，使用私钥签名并把交易发送到链上网络。钱包不会替用户“托管”资产，除非用户使用的是交易所账户或托管钱包。

典型交互链路如下：

```text
用户
└─ 打开钱包或 DApp
   └─ 连接钱包
      └─ DApp 生成交易请求
         └─ 钱包展示交易内容
            └─ 用户确认
               └─ 钱包用私钥签名
                  └─ 交易广播到节点网络
                     └─ 验证者/矿工打包进区块
                        └─ 链上状态更新
                           └─ 收款方余额增加或合约状态改变
```

这里最关键的动作是“签名”。签名相当于不可伪造的授权凭证。用户点击确认后，钱包把“我要做什么”变成一笔带签名的链上交易。节点验证通过后，交易才有法律意义上的链上效果，表现为余额变化、合约记录变化或权限变化。

# **4\. 链上支付怎么发生：转账是把状态从 A 改到 B**

以 ETH 转账为例，用户从地址 A 给地址 B 转 0.1 ETH。钱包会构造一笔交易，其中包含发送方、接收方、金额、手续费参数、随机序号 nonce、链 ID 等信息。用户确认后，钱包用 A 地址对应的私钥签名。交易发到网络后，节点验证 A 的签名是否正确、余额是否足够、nonce 是否正确、手续费是否满足要求。验证通过并被打包后，链上状态发生变化：A 的余额减少 0.1 ETH 和手续费，B 的余额增加 0.1 ETH，出块者或验证者获得手续费。

时间线可以这样理解：

```text
支付前
A 地址：1.0 ETH
B 地址：0 ETH

用户签名并广播
交易内容：A 向 B 转 0.1 ETH，愿意支付手续费

节点验证
检查签名、余额、nonce、手续费、链规则

打包上链
交易进入区块，区块被网络接受

支付后
A 地址：约 0.9 ETH 减手续费
B 地址：0.1 ETH
```

这就是链上支付。它和银行卡、支付宝的最大差异在于，支付状态由区块链网络共同更新，用户通过私钥直接授权，不依赖单一支付平台替用户记账。

# 5\. 交易是什么：任何改变链上状态的操作，不只是买卖

**交易是一个技术术语**，指**用户**或**合约**向区块链提交的一条**状态变更指令**：

> 这是理解区块链账户模型的关键

⚠️ **所以！！链上交易是 “请求区块链修改状态” 的 数据包！！！！**

区块链里的“交易”不是日常语境里的买卖，也不是支付宝贷款。

转账、授权、兑换、借贷、领取空投、铸造 NFT、参与治理投票，都可以通过交易完成。

🚩 **只要这条操作需要改变链上状态，通常就要发交易并支付手续费。**

交易的常见类型如下：

| 交易类型 | 用户在做什么 | 链上发生什么 |
| --- | --- | --- |
| 原生币转账 | 给别人转 ETH、BTC、SOL | 发送方余额减少，接收方余额增加 |
| 代币转账 | 转 USDT、USDC 等代币 | 代币合约修改两个地址的余额 |
| 授权 approve | 允许某个合约使用自己的代币 | 合约记录一条额度许可 |
| 兑换 swap | 用 ETH 换 USDC | 去中心化交易所合约按规则交换资产 |
| 借贷 deposit/borrow | 存入资产或借出资产 | 借贷合约记录抵押、债务和利率 |
| NFT 铸造 mint | 创建或领取 NFT | NFT 合约记录编号归属 |
| 合约交互 | 调用某个链上程序 | 合约状态按代码规则变化 |

所以，“交易”这个词在链上更接近数据库里的“状态变更事务”。买卖只是其中一种业务场景，贷款也是其中一种合约应用场景。

# ➡️ 关于 合约账户 / EOA(用户)

🚩 **EOA 就是"用户账户"** = 由**人**控制的账户

| 日常说法 | 对应链上账户 |
| --- | --- |
| “我的钱包” | EOA |
| “用户” | EOA |
| “你的地址” | EOA |
| “合约地址” | 合约账户 |

\*\*🚩 合约账户 = 地址 + 余额 + 代码 + 存储状态

> 它是"活的程序"，但**没有主人**，按代码逻辑自动运行。

|   | EOA | 合约账户 |
| --- | --- | --- |
| 能主动发起交易？ | ✅ 能 | ❌ 不能 |
| 怎么动起来？ | 私钥签名 → 广播交易 | 被 EOA 或其他合约调用 |

> 合约账户永远"被动"——它不会自己突然执行，必须有人（EOA）或另一个合约触发它。

## 一、“用户” 和 “合约” 都能 交易

|   | 用户（外部账户，EOA） | 合约（智能合约） |
| --- | --- | --- |
| 怎么"发起"交易 | 用户用钱包签名，主动提交到链上 | 合约不能自己主动发起，只能被触发后执行 |
| 举例 | 你点击"转账"，MetaMask 弹出让你确认 | Uniswap 的合约被你的交易调用后，自动去换币、记账 |
|   |   |   |

**区块链只有两种账户**：

|   | EOA（外部账户） | 合约账户 |
| --- | --- | --- |
| 创建方式 | 生成私钥 → 算出地址 | 部署合约代码 → 生成地址 |
| 有无代码 | ❌ 没有 | ✅ 有，且不可改 |
| 有无私钥 | ✅ 有 | ❌ 没有，无人直接控制 |
| 存什么 | 余额（ETH/MON 等原生币） | 余额 + 存储变量（状态） |
| 例子 | 0x71C…A3f（你的钱包） | 0xA0b…9E2（Uniswap 合约） |
|   |   |   |

|   |   |
| --- | --- |
| 人的代理，有私钥，能主动发起交易 | EOA |
| 程序的代理，有代码，只能被动响应 | 合约账户 |

所以"用户" ≈ EOA，  
但严格说：**用户是一个角色， EOA 是这个角色在链上的账户载体**

* * *

**交易的起点必须是人（EOA），但链上状态变更的执行者可以是人写的合约代码。**

## 二、关键区分：交易 vs 内部调用

```
你（EOA）── 发起交易──→ 调用 Uniswap 合约
                              ↓
                        Uniswap 合约 ── 内部调用──→ USDC 合约（转账给你）
                        ↑ 这不是新交易，是同笔交易里的连锁反应
```

| 概念 | 说明 |
| --- | --- |
| 交易（Transaction） | 必须由 EOA 签名发起，消耗 gas，上链可查 |
| 内部调用（Internal Tx） | 合约被触发后，代码里再去调其他合约，不另开新交易 |

**即 合约在用户的那笔交易里被动执行，并产生后续调用**。

* * *

# **6\. 手续费是什么：给网络执行和记录交易的成本**

链上支付通常需要手续费。手续费支付给打包交易、维护网络安全的验证者或矿工。以以太坊为例，手续费常被称为 gas fee，因为链上每一步计算和存储都消耗资源。普通转账消耗少，复杂合约交互消耗多，所以兑换、跨链、铸造 NFT、链上借贷通常比普通转账贵。

手续费有两个作用：补偿网络资源消耗，防止垃圾交易攻击。没有手续费，攻击者可以用海量无意义交易堵塞网络。手续费机制让每次写入链上状态都带有成本。

# **7\. 链上支付和支付宝支付的差异**

支付宝支付通常是平台内部账户系统更新余额，用户把操作授权给支付宝，支付宝根据自己的账本和银行清算体系完成支付。

链上支付是用户用私钥直接签名，区块链网络根据共识规则更新公开账本。两者都能表现为“余额从 A 到 B”，但信任结构和资产控制方式不同。

对比可以简化如下：

| 维度 | 链上支付 | 支付宝支付 |
| --- | --- | --- |
| 账户形式 | 地址 | 平台账户 |
| 控制方式 | 私钥签名 | 登录、密码、风控、平台规则 |
| 账本位置 | 区块链公共账本 | 平台和银行系统账本 |
| 交易确认 | 网络共识确认 | 平台确认与清算系统处理 |
| 失败原因 | 余额不足、手续费不足、签名错误、合约报错 | 余额不足、风控、限额、系统或银行原因 |
| 是否可撤回 | 通常不可逆 | 平台可冻结、退款、争议处理 |
| 钱包作用 | 管理私钥并签名 | 类似账户入口，但平台实际控制账户体系 |

# **8\. 智能合约让“支付”扩展成复杂金融操作**

普通转账只涉及两个地址。

**智能合约**出现后，链上交易可以触发程序逻辑。用户向合约发交易，合约按照代码规则处理资产。例如去中心化交易所会根据池子价格完成兑换，借贷协议会根据抵押率允许用户借款，NFT 合约会根据规则发放编号。

链上贷款的基本结构是：用户先抵押一种资产，合约按抵押率允许借出另一种资产。如果抵押品价格下跌到危险线，合约允许清算人还债并拿走部分抵押品。这类操作看起来像金融业务，但在链上表达为一组交易和合约状态变化。

```text
链上借贷
├─ 用户存入抵押品
│  └─ 合约记录用户抵押余额
├─ 用户借出资产
│  └─ 合约记录用户债务
├─ 价格变化
│  └─ 预言机提供资产价格
└─ 抵押率不足
   └─ 清算交易触发，抵押品被出售或转移
```

# ➡️ 链/合约关系 合约是代码

链和合约的关系，用一句话说：

> **链是"土地"，合约是"地上的建筑"。**

* * *

**链是底层状态机，合约是状态机里可编程的"子账户+逻辑单元"。** 合约的状态变化，就是链的状态变化的一种。

|   | 链（Layer 1） | 合约（Smart Contract） |
| --- | --- | --- |
| 本质 | 全局状态数据库 + 共识机制 | 部署在链上的程序代码 |
| 保存什么 | 所有地址余额、所有合约存储、全部状态 | 自己的代码 + 自己的存储空间 |
| 谁维护 | 全网节点共同维护 | 由链的节点"代执行"，但逻辑自治 |
| 能否独立存在 | 能，空链也能跑 | 不能，必须部署在某条链上 |

* * *

```
链的全局状态
├── 地址 A: 1 ETH
├── 地址 B: 200 USDT  
└── 合约 X: ← 也是状态的一部分
    ├── 代码: swap/质押/借贷逻辑
    ├── 存储: 谁存了多少钱、利率多少
    └── 余额: 合约自己持有的资产  ← "锁着的资产"
```

**“合约里锁着多少资产”**

-   **\*链的全局账本**里记录了：合约这个地址名下有多少 Token
    
-   这些 Token 的转移权 被 **合约代码的逻辑** 约束着
    

* * *

### 核心关系总结

| 关系 | 说明 |
| --- | --- |
| 包含 | 合约状态是链状态的一个子集 |
| 依赖 | 合约靠链的节点执行、靠链的共识确认结果 |
| 寄生/共生 | 合约没有自己的节点网络，"租用"链的基础设施 |
| 多对一 | 一条链可以有 N 个合约，一个合约只能活在一条链上（同态部署除外） |

* * *

-   **链** = 政府 + 土地局 + 公证处（提供登记、执行、公信力）
    
-   **合约** = 你在这块地上盖的房、签的协议（自动执行的规则）
    
-   房子不能飘在空中，必须落在某块地上；但地可以空着没房子。

# ➡️ 关于 节点、链、合约

> **链是账本，节点是运行账本的计算机，合约是账本上的程序。**

区块链： 是【链】这个账本的数据结构\*\*，是\*\*专门用来防篡改的链表结构

**链： 是账本**  
\*\*节点：是运行账本的计算机，节点们运行着相同的软件，各自保存链的副本，  
合约： 是代码写在链上，

节点执行合约改变链的状态，再通过共识让所有节点账本保持一致。

**合约不拥有任何节点**，是无状态的代码片段，**散布在 所有全节点 的存储里**

| 概念 | 是什么 | 作用 |
| --- | --- | --- |
| 链（Chain） | 按顺序链接的数据账本 | 记录所有交易和合约状态，不可篡改 |
| 节点（Node） | 运行区块链软件的计算机/服务器 | 存储账本副本、验证交易、执行合约、广播区块 |
| 合约（Contract） | 部署在链上的程序代码 | 自动执行规则（如"转账X给Y"） |

|   |   |
| --- | --- |
| 节点 | 专门安装了客户端软件（如 Geth、Monad 客户端），参与网络共识或同步数据 |
| 全节点 | 存完整账本，独立验证一切 |
| 轻节点 | 只存区块头，依赖全节点验证 |
| 验证者节点（Validator） | 有质押权重，能出块/投票（PoS）或挖矿（PoW） |

> 你的笔记本电脑**可以**成为节点，但默认**不是**。必须主动下载、配置、运行节点软件。

## 执行流程

```
用户发起交易 → 某节点收到 → 节点本地 EVM 执行合约代码
                                    ↓
                           产生状态变更（临时结果）
                                    ↓
                           广播给全网节点
                                    ↓
                           共识机制（如 Monad 的 MonadBFT）投票/验证
                                    ↓
                           超多数节点认可 → 区块敲定 → 结果永久生效
```

* * *

| 要点 | 解释 |
| --- | --- |
| 节点不"忠于"某个合约 | 节点按规则执行所有合法合约，一视同仁 |
| 单节点执行 ≠ 最终结果 | 恶意节点可能乱执行，共识机制就是防这个的 |
| Gas 费的本质 | 用户为占用节点计算/存储资源付费，防止合约无限占用 |

* * *

## Monad 的特殊点——**并行执行 + 乐观执行 + MonadBFT 共识**

Monad 用 **并行执行 + 乐观执行 + MonadBFT 共识**，但核心关系不变：

> 节点先各自乐观执行 → 共识层最终确认顺序和结果 → 状态回滚处理冲突

**合约永远依赖节点执行，依赖共识背书，自己不能独立存在。**

### ⚠️ 深度解析

| 概念 | 本质 |
| --- | --- |
| 区块链（账本） | 一份大家共同认可的"记录本" |
| 节点 | 运行这份账本的所有计算机 |
| P2P网络 | 节点之间互相传话、对账的机制 |

三者合起来 ➡️ 一个**没有中心服务器、但所有人数据一致**的系统。

```
第1步：有人要记账
        ↓
    "张三转100块给李四"（交易）

第2步：广播到微信群（P2P网络）
        ↓
    节点A收到 → 转发给节点B、C
    节点B收到 → 转发给节点A、C
    节点C收到 → 转发给节点A、B

第3步：各节点验证并记录
        ↓
    每个节点都检查：张三余额够不够？
    够的话，各自把这笔账写进自己的账本

第4步：打包成区块，链上去
        ↓
    节点A生成"区块3" → 广播出去
    节点B、C验证区块3没问题 → 接在自己的账本后面

第5步：达成一致
        ↓
    现在A、B、C三家的账本完全一致：
    ... → 区块2 → 区块3（含"张三转李四100"）
```

## 区块链

**区块链： 是【链】这个账本的数据结构**，是**专门用来防篡改的链表结构**

如"Excel表格"是一种数据结构：

-   每个区块有个"指纹"（哈希值）
    
-   下一个区块里**记着上一个区块的指纹**
    
-   改中间任何一个区块 → 指纹变 → 后面全崩 → 所有人都能发现
    

```
区块1 [数据: 创世]  [指纹: abc123]
   ↓
区块2 [数据: 交易A] [指纹: def456] [上一块指纹: abc123]
   ↓
区块3 [数据: 交易B] [指纹: ghi789] [上一块指纹: def456]
   ↓
...
```

# **9\. 一笔链上交易内部包含什么**

以账户模型链为例，一笔交易通常包含这些字段：发送方地址、接收方地址、金额、要调用的合约函数、参数、nonce、手续费设置、链 ID、签名。普通用户不需要手动填写这些字段，钱包和 DApp 会自动生成。用户需要看的重点是：收款地址是否正确，金额是否正确，调用的合约是否可信，授权额度是否过大，手续费是否可接受。

更具体地说，用户点击“确认”前应重点检查三类信息：第一，资产流向，谁拿走什么资产；第二，权限变化，某个合约是否获得代币使用权；第三，成本，手续费和滑点是否异常。很多链上风险来自用户签了自己看不懂的授权或合约调用。

# **10\. 两种核心账本模型：UTXO 和账户模型**

比特币主要使用 UTXO 模型。它的思路像现金找零：你拥有若干个未花费输出，每次支付时拿出一个或多个输出，生成新的输出给收款方和自己。以太坊主要使用账户模型。它的思路像账户余额：地址有余额，交易直接让账户余额或合约状态发生变化。

```text
账本模型
├─ UTXO 模型
│  ├─ 代表：Bitcoin
│  ├─ 类比：现金和找零
│  └─ 支付逻辑：花掉旧输出，生成新输出
└─ 账户模型
   ├─ 代表：Ethereum、BSC、Polygon、Solana 的账户式设计
   ├─ 类比：账户余额
   └─ 支付逻辑：修改账户或合约状态
```

理解这两种模型后，“钱在哪里”的问题会更清晰。比特币里的钱表现为你能花费哪些未花费输出；以太坊里的钱表现为地址余额和合约内部余额。

# **11\. 最小完整例子：用钱包给别人转 USDT**

用户要给朋友转 100 USDT。用户打开钱包，选择 USDT，输入朋友地址和金额。钱包生成一笔调用 USDT 合约的交易，意思是“把我的 100 USDT 记到对方地址”。用户确认后，钱包用私钥签名并广播。链上节点验证签名和手续费，USDT 合约检查用户余额是否足够。交易成功后，USDT 合约内部的余额表更新：用户地址减少 100 USDT，朋友地址增加 100 USDT。链上浏览器可以查到这笔交易哈希、区块高度、发送方、接收方、金额、手续费和状态。

这里的“支付”并没有发生实体货币移动。发生的是一条全网认可的账本更新：某个合约里的余额映射被改写，控制权随之转移。

# **12\. 学习时抓住四个核心概念**

区块链支付可以压缩成一个因果链条：私钥控制地址，地址拥有链上资产，钱包用私钥签名交易，交易被网络验证并打包，链上状态随之更新。所有复杂概念都围绕这条链展开。

```text
私钥
└─ 控制地址
   └─ 地址持有资产
      └─ 钱包发起签名
         └─ 交易进入网络
            └─ 节点验证
               └─ 区块确认
                  └─ 状态更新
```

最重要的判断标准是：链上资产的存在形式是账本状态，用户的控制方式是私钥签名，交易的技术含义是状态修改请求，支付只是交易的一种结果。

# 🟡 AI Agent “运行在链上”

不是指 AI 模型本身部署在区块链上（链上算力和存储都支撑不了大模型），而是指 **AI Agent 的决策结果、资产所有权、执行权限被锚定在链上**。具体实现方式有几种：

* * *

## 1\. AI 本体在链下，链上持有"身份 + 钱包 + 权限"

这是最主流的模式：

| 组件 | 链上 / 链下 | 功能 |
| --- | --- | --- |
| AI 推理引擎（LLM/模型） | 链下（服务器/去中心化算力网络） | 做决策、分析数据、生成交易指令 |
| Agent 钱包/地址 | 链上 | 持有资产、签名交易、参与治理 |
| 智能合约 | 链上 | 定义 Agent 的操作规则、权限边界、资金限额 |
| 执行记录 | 链上 | 所有交易可追溯、可审计 |

**工作流程**：

```
AI Agent（链下）分析市场 → 生成"买入 100 USDC 的 ETH"指令 
    → 用链上钱包签名 → 提交到智能合约 
    → 合约检查"是否超单日限额？是否白名单代币？" 
    → 通过则执行，否则拒绝
```

* * *

## 2\. 账户抽象（ERC-4337）：Agent 的"智能合约钱包"

传统钱包 = 私钥控制，Agent 需要保管私钥，风险高。

**账户抽象**让 Agent 拥有"智能合约钱包"：

-   钱包本身是智能合约，可编程设定规则
    
-   例如：单日转账上限 1 ETH、只能与特定协议交互、多签授权等
    
-   Agent 不需要直接掌握私钥，而是通过合约权限触发操作
    

> 引用：_“AI 代理作为智能合约钱包（ERC-4337 / Move AA）独立管理资产与权限”_ 5^

* * *

## 3\. 链上可验证的决策：ZK 证明 + 预言机

AI 决策是"黑箱"，区块链需要确定性。怎么让链上信任 Agent 的决策？

| 方案 | 原理 |
| --- | --- |
| ZK 证明 | Agent 生成"我按规则做了推理"的零知识证明，链上合约验证证明而非原始数据 |
| 预言机网络 | Chainlink 等将 Agent 决策结果以去中心化方式喂到链上 |
| TEE（可信执行环境） | Agent 在硬件隔离区运行，输出带硬件签名的可信结果 |

> 引用：_“通过零知识证明（ZK）验证智能体决策的合规性与透明度”_ 2^

* * *

## 4\. 全链上 Agent：极端案例

少数项目尝试把 Agent 逻辑也上链：

-   **自治世界（Autonomous Worlds）**：游戏 NPC 的行为规则写在智能合约里
    
-   **简单规则引擎**：if-then 逻辑直接合约化，复杂推理仍靠链下
    

但 LLM 级别的推理目前不可能全链上，Gas 和延迟都不支持。

* * *

## 一句话总结

> **AI Agent “运行在链上” = 链下大脑做决策 + 链上身份管资产 + 智能合约控权限 + 密码学保可信**

不是 AI 真的在区块链节点上跑，而是 **Agent 的经济行为、所有权、执行规则被区块链固化**，使其成为可信的自主经济参与者。

## ❕ 关于 DeFi

解释 **一个 DeFi 协议的流动性可以被其他应用直接调用**，就像搭积木

即 DeFi 协议"像积木可拼接"这个核心创新：

* * *

### 1️⃣ DeFi 是怎么形成的？

DeFi（去中心化金融）诞生于**区块链 + 智能合约**的结合：

| 传统金融 | DeFi |
| --- | --- |
| 银行、券商等中心化机构控制 | 代码（智能合约）自动执行 |
| 各自封闭的系统 | 公开透明的协议 |
| 需要许可才能接入 | 无许可（Permissionless） |

**关键转折点**：

以太坊在 2015 年引入**智能合约**，让开发者可以在链上编写自动执行的金融程序。

2018-2020 年，Compound（借贷）、Uniswap（交易）等基础协议出现，**DeFi 生态**开始形成。

* * *

### 2️⃣ "积木拼接"是什么意思？ **可组合性（Composability）**

♥️ 核心机制是：

### 1\. 标准化接口

就像 USB 接口通用一样，DeFi 协议使用统一标准：

-   **ERC-20**：代币标准，任何钱包/交易所都能识别
    
-   **ERC-721**：NFT 标准
    

### 2\. 无许可调用

任何开发者可以直接调用已部署的合约功能，**不需要对方同意**2^

### 3\. 原子性交易

多个操作打包在一个交易中：**要么全部成功，要么全部失败**1^

* * *

### 3️⃣ 什么时候需要"拼接"？

典型场景举例：

| 场景 | 拼接方式 | 效果 |
| --- | --- | --- |
| 收益聚合 | Yearn 自动把你的资金在 Aave、Compound 等协议间切换，找最高收益 | 用户一键操作，背后拼接多个借贷协议 |
| 闪电贷 | 借 → 套利（跨 Uniswap/SushiSwap）→ 还，全在一笔交易内完成1^ | 零本金套利 |
| 流动性质押 | 存入 ETH → 获得 stETH（Lido）→ 把 stETH 抵押到 Aave 借出更多资金5^ | 同一笔资金多处生息 |
| DEX 聚合 | 1inch 同时查 Uniswap、SushiSwap、Curve 的价格，拆分订单找最优路径 | 自动选最优交易路径 |

* * *

对普通用户：

-   **更高效的资本利用**：一份钱同时做几件事（上面 stETH 的例子）
    
-   **更低门槛**：不需要银行审批，有网就能用
    

对开发者：

-   **站在巨人肩上**：不用自己从头写借贷逻辑，直接调用 Compound 的合约1^
    
-   **创新加速**：像搭乐高一样快速组合出新产品
    

类比理解：

> 传统金融像**苹果手机**——封闭、好用但受限  
> DeFi 像**安卓 + 开源硬件**——模块公开，任何人能拼装出新设备

* * *

### 这种"拼接"也带来了**系统性风险**：

-   复杂性增加，普通用户看不懂背后的风险链条

DeFi 的积木拼接 = 公开的智能合约 + 统一标准 + 无许可调用，让金融协议像开源软件一样可以被自由组合、复用、创新。

> **部署在区块链上的自动执行程序**
> 
> 满足条件 → 自动触发 → 结果不可篡改

* * *

“虚拟”

| 层面 | 答案 |
| --- | --- |
| 代码本身 | 是虚拟的，一行行程序代码 |
| 运行结果 | 非常真实：转账、发行代币、锁定资产都是链上实际发生的 |
| 存在形式 | 部署在区块链每个节点的副本上，全球成千上万台电脑同时执行 |

> 像云端的自动售货机——你看不见内部，但投币（触发条件）真的掉出饮料（执行结果）。

🔵 智能合约 由什么组成？

```
┌─────────────────────────┐
│  1. 状态数据（Storage）   │  ← 永久存储在链上，如"谁有多少余额"
├─────────────────────────┤
│  2. 执行逻辑（Code）      │  ← 函数规则，如"转账时检查余额是否够"
├─────────────────────────┤
│  3. 触发条件（Trigger）   │  ← 外部调用、时间到达、事件等
├─────────────────────────┤
│  4. 事件日志（Events）    │  ← 执行记录，供外部监听
└─────────────────────────┘
```

**以太坊智能合约具体用 Solidity 语言编写**，编译成字节码后部署到链上。

🔵 结构详解：

智能合约 = 区块链上的"自动售货机"

| 类比 | 说明 |
| --- | --- |
| 自动售货机 | 投入硬币 → 按规则出货 → 记录交易 |
| 智能合约 | 收到交易 → 执行代码 → 修改状态 → 记录日志 |

🚩 四层结构详解

1\. 状态数据（Storage）

```
┌─────────────────────────────┐
│  永久存在区块链上的数据        │
│  ─────────────────────────    │
│  mapping(address => uint)     │
│    balances;                  │
│  // 记录每个地址的余额         │
│                               │
│  address public owner;        │
│  // 合约创建者地址             │
│                               │
│  uint public totalSupply;     │
│  // 代币总发行量               │
└─────────────────────────────┘
```

| 特性 | 说明 |
| --- | --- |
| 永久存储 | 写入后永久保存在链上，删除合约也保留历史 |
| 昂贵 | 修改Storage消耗大量Gas（约20,000 gas/槽位） |
| 公开透明 | 任何人都能读取，即使没写getter函数 |

> 💡 类比：区块链上的"硬盘"，但写入成本极高

* * *

2\. 执行逻辑（Code）

```solidity
// 转账函数：核心执行逻辑
function transfer(address to, uint amount) public {
    // 规则1：检查余额是否足够
    require(balances[msg.sender] >= amount, "余额不足");
    
    // 规则2：防止溢出
    require(balances[to] + amount >= balances[to], "溢出错误");
    
    // 执行转账
    balances[msg.sender] -= amount;  // 扣款
    balances[to] += amount;          // 到账
    
    // 触发事件通知外部
    emit Transfer(msg.sender, to, amount);
}
```

| 特性 | 说明 |
| --- | --- |
| 不可篡改 | 部署后代码无法修改（除非设计升级机制） |
| 确定性 | 相同输入永远产生相同输出 |
| 公开可审计 | 字节码反编译后可验证逻辑 |

> 💡 类比：自动售货机的"内部机械结构"，规则固定透明

* * *

3\. 触发条件（Trigger）

```
┌─────────────────────────────────┐
│  谁/什么能启动合约执行？          │
├─────────────────────────────────┤
│                                 │
│  ① 外部账户调用                  │
│     EOAs (Externally Owned      │
│     Accounts)                   │
│     → 用户发送交易调用函数        │
│                                 │
│  ② 其他合约调用                  │
│     → 合约A调用合约B的函数        │
│                                 │
│  ③ 预定义条件自动触发             │
│     → 时间锁：区块高度达到X时执行  │
│     → 预言机：ETH价格跌破Y时清算   │
│                                 │
│  ④ 事件回调（较少见）             │
│     → 特定日志触发监听者响应       │
│                                 │
└─────────────────────────────────┘
```

| 触发方式 | 示例 |
| --- | --- |
| user.transfer(0.1 ETH) | 用户主动发起转账 |
| contractA.callContractB() | DeFi协议组合调用 |
| block.number >= unlockTime | 代币解锁时间到 |
| Chainlink price < liquidationPrice | 借贷协议自动清算 |

> ⚠️ **关键限制**：合约不能"自己运行"，必须由外部触发

```solidity
// 定义事件

);

```

```
区块链存储结构：
┌────────────────────────────────────────┐

```

| 用途 | 说明 |
| --- | --- |
| 前端监听 | Dapp实时显示"转账成功" |
| 历史索引 | TheGraph等工具按事件构建数据库 |
| 低成本存储 | 比Storage便宜约10倍 |
| 链下审计 | 分析所有Transfer记录追踪资金流向 |

> 💡 类比：自动售货机的"交易小票"，不存机器内部但可留底查询

具体例子：最简单的"存钱罐"合约

```solidity
// 任何人可以存钱，但只有指定人能在特定时间后取钱
contract TimeLock {
    address owner;      // 存钱的人
    uint unlockTime;    // 解锁时间
    
    function deposit() public payable {
        // 存钱逻辑，自动记录
    }
    
    function withdraw() public {
        require(msg.sender == owner, "不是本人");      // 条件1：身份
        require(block.timestamp > unlockTime, "时间未到"); // 条件2：时间
        // 自动转账，无需任何人审批
    }
}
```

-   到时间前，连开发者都改不了"提前取钱"
    
-   到时间后，owner 调用函数，**代码自动打款**
    
-   全过程无银行、无客服、无人工审核

## ✝️ 二、web3 组件、关系 🌟【重要】

```
用户 → 钱包（如MetaMask）→ 签名授权 → Dapp（去中心化应用）→ 调用智能合约 → 读写区块链数据
```

| 组件 | 作用 |
| --- | --- |
| 钱包 | 管理你的私钥和资产，相当于"数字身份证+银行卡" |
| 签名 | 用私钥确认"这个操作是我发起的"，防止冒充 |
| Dapp | 去中心化应用，界面类似传统App，但后端接区块链 |
| 智能合约 | 自动执行的代码规则，比如"转账X元则自动发货" |
| 区块链 | 底层账本，永久记录所有交易 |

## 🟡 二、"自动做市、无人工干预"什么意思？

### 传统做市（有人）

> 庄家挂买单"1万买"、挂卖单"1.1万卖"，赚差价。需要人 constantly 调整价格。

### Uniswap 自动做市（没人）

```

- 小明存入 10 ETH + 20000 USDC
- 小红存入 5 ETH + 10000 USDC
- 池子现在有 15 ETH + 30000 USDC

你想用 USDC 买 1 ETH？
→ 公式自动算出价格：约 2000 USDC
→ 你确认 → 智能合约自动：
  1. 扣你 2000 USDC（+0.3%手续费）
  2. 给你 1 ETH
  3. 更新池子：14 ETH / 32000 USDC
  4. 手续费分给小明、小红
```

**全程没有客服、没有审核、没有"系统维护"——代码就是规则。**

## 🟡 核心公式（看懂这个=懂了一半）

Uniswap V2 用 **x\*y = k**（恒定乘积）

```

交易前：10 ETH × 20000 USDC = 200000
买走 1 ETH 后：9 ETH × ? USDC = 200000

所以你付的 USDC：22222 - 20000 = 2222
实际价格：~2222 USDC/ETH（比2000贵，因为滑点）
```

**买得越多 → 价格越贵 → 防止池子被掏空**

## 🔵 以太坊 ERC-20/ ERC-721

| 缩写 | 全称 |
| --- | --- |
| ERC-20 | Ethereum Request for Comments 20 |
| ERC-721 | Ethereum Request for Comments 721 |

> “ERC” 是以太坊社区的技术提案编号，类似互联网 RFC（Request for Comments）。

## 🔵 ERC-20：同质化代币标准

一套**接口规范**，规定了代币合约必须实现的基本功能。

**🚩 干嘛用**

| 场景 | 说明 |
| --- | --- |
| 发币 | 项目方一键发币（USDT、UNI、LINK 都是 ERC-20） |
| 互操作性 | 任何钱包、交易所、DeFi 协议"即插即用"识别 |
| 标准化 | 不用为每个新代币写定制代码 |

特点

> **同质化** = 每个代币完全相同，1个 USDT = 另1个 USDT，可互换、可分割。

* * *

## 🔵 ERC-721：非同质化代币标准

一套**独一无二的资产**接口规范，每个代币有独立 ID。

🚩**核心功能**

```solidity
ownerOf(tokenId)         // 查某 NFT 归谁
transferFrom(from, to, tokenId)  // 转让特定 NFT
approve(to, tokenId)     // 授权他人转移某 NFT
```

**🚩 干嘛用**

| 场景 | 说明 |
| --- | --- |
| NFT 头像 | CryptoPunks、BAYC 等 |
| 游戏道具 | 武器、皮肤，每个属性不同 |
| 数字藏品 | 艺术品、门票、证书、房产证 |
| 域名 | ENS（.eth）、Unstoppable Domains |

特点

> **非同质化** = 每个代币**唯一**，不可互换。Token ID #1 ≠ Token ID # 2。

* * *

🚩 核心对比

|   | ERC-20 | ERC-721 |   |
| --- | --- | --- | --- |
| 资产类型 | 同质化代币 | 非同质化代币（NFT） |   |
| 互换性 | 可互换，1:1等价 | 不可互换，每个唯一 |   |
| 分割性 | 可分割（0.5个） | 不可分割（只能整个转让） |   |
| 典型代表 | USDT、DAI、UNI | BAYC、CryptoKitties、ENS |   |
| 用途 | 货币、积分、股权 | 收藏品、身份、物权证明 |   |
| 转账参数 | transfer(to, amount) | transferFrom(from, to, tokenId) |   |

* * *

> **ERC-20 = 发"钱"的标准**（同质化，可互换）  
> **ERC-721 = 发"物"的标准**（非同质化，独一无二）

### [Web 3.0（语义网）](https://web3intern.xyz/zh/blockchain-basic/#_2-web-3-0-%E8%AF%AD%E4%B9%89%E7%BD%91)

**Web 3.0 = 让机器理解数据含义的技术；  
Web3 = 让用户拥有数据所有权的技术。**  
两者名字像，但解决的问题完全不同——前者解决"机器怎么读懂信息"，后者解决"信息归谁控制"。

**Web 3.0核心特征**：

-   **语义标记**：使用 RDF、OWL 等标准描述数据关系
    
-   **结构化数据**：信息按照标准格式组织，便于机器理解
    
-   **知识图谱**：构建实体间的语义关系网络
    
-   **典型技术**：本体工程、语义查询语言（SPARQL）、链接数据
    

**关键区别**：

-   ❌ **不是区块链技术**，而是传统互联网的数据组织升级
    
-   ❌ **主要不依赖 AI**，而是通过标准化数据格式实现
    
-   ✅ 与 Web3 可结合（语义标记 + 区块链存储）
    

**比喻**：

像把图书馆的每本书都贴上详细标签（作者、主题、关联书籍），让图书管理员能快速找到相关资料。

**RDF** 和 **OWL** 是语义网的核心数据格式，本质是给信息加上"机器能读懂的结构化标签"。

* * *

### 1\. 核心特征（三个层次）

| 特征 | 通俗理解 | 技术对应 |
| --- | --- | --- |
| 语义标记 | 给数据加"标签"，让机器知道"这是什么" | RDF、OWL 等标准格式 |
| 结构化数据 | 信息按统一模板整理，不是杂乱无章的文本 | 数据库 schema、JSON-LD |
| 知识图谱 | 把不同信息点连成网络，发现关联 | 谷歌搜索背后的技术 |

### 2\. 关键区别（三个易混淆点）

-   **❌ 不是区块链** → Web 3.0 是**传统互联网的数据层升级**，不需要链、代币、矿工
    
-   **❌ 不依赖 AI** → 重点在**数据格式标准化**，让机器能"读懂"；AI 是另一套技术
    
-   **✅ 可与 Web3 结合** → 语义标记的数据 + 区块链存储 = 既智能又可信
    

### 3\. 比喻的本质

| 搜索靠关键词匹配 | 机器能推理：“找鲁迅 → 关联到周树人、呐喊、同时代作家” |

## 语义网（Semantic Web）的核心技术栈：

## 1️⃣ RDF（资源描述框架）

Resource Description Framework

  
**作用**：描述"某个东西是什么属性、有什么值"

**基本结构**：三元组 —— **主语-谓语-宾语**

```
<周杰伦>  <职业>  <歌手>
<周杰伦>  <出生日期>  <1979-01-18>
<周杰伦>  <配偶>  <昆凌>
```

**常见写法**：

-   XML 格式
    
-   Turtle 格式（更简洁）
    
-   JSON-LD（嵌入网页，搜索引擎能抓取）
    

* * *

## 2️⃣ OWL（网络本体语言）

Web Ontology Language

**作用**：在 RDF 基础上，定义**概念之间的关系规则 /即概念之间的关系和推理规则**

* * *

## 两者关系

```
RDF = 建数据库（存事实）
OWL = 写数据库的"表结构+规则"（定义概念怎么关联）
     ↓
   合起来 = 本体（Ontology）= 某个领域的"知识字典"
```

* * *

## 实际应用场景

| 场景 | 怎么用 |
| --- | --- |
| 谷歌知识面板 | 搜索"周杰伦"，右侧显示生日、专辑、相关人物 → 靠网页里的 JSON-LD/RDF 标记 |
| 维基数据（Wikidata） | 全球结构化知识库，QID 标识每个实体，RDF 存储关系 |
| 学术文献检索 | 用 SPARQL 查询"找所有与阿尔茨海默病相关的基因" |

**联盟链 = 精英俱乐部式的共享账本**——比公链高效、比私链开放，但需要成员间协调，适合有信任基础的多方协作场景（如金融、供应链）。

## 回到原文：联盟链"成为节点"的具体含义

> **原文**：需要邀请或申请，得到其中一家公司认可

这意味着：

| 步骤 | 说明 |
| --- | --- |
| 1. 申请 | 你的机构（如供应商）向联盟中的银行提交加入请求 |
| 2. 审批 | 现有成员投票或协商，决定是否允许你加入 |
| 3. 配置 | 给你分配一台服务器，安装联盟链软件，接入网络 |
| 4. 赋权 | 决定你是决策者（可改规则）还是观察者（只能查看） |
