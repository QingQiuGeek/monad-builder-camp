### 网上的证书、证明，怎么让人相信是真的？

现实生活中，您如果要证明"这份学历是真的""这份合同没有被人偷偷改过"，通常要靠**公证处**盖章、或者**官方机构**发证书。可是公证处、机构本身也是"中心化"的——万一它的档案室着火了，或者内部人员被收买改了记录呢？

# 🌰

**生活比喻：一个"永远不会被烧掉、也没人能偷偷改"的公证处**

**EAS（Ethereum Attestation Service，以太坊证明服务）**，就像一个开在区块链上的"公证处"。任何机构、任何人，都可以在上面给某件事情"盖章证明"——比如"某某同学完成了某期学习计划的全部打卡"。

这份"盖章证明"（专业叫**证明 / attestation**）一旦提交上链，就会永远留在区块链账本上：**公开可查、任何人都能验证、没有人能偷偷篡改或撤销**（还记得第 1 课讲过的"改不了、赖不掉"吗？EAS 就是把这个特性用来做"证书"和"记录"）。

## Web3 团队三大角色：Research / Ops / Dev 分别解决什么问题

三个角色对应的是同一个产品从"想清楚"到"做出来"到"有人用"的完整链路：

**Research 负责判断"做什么、为什么做"** → **Dev 负责"把它造出来"** → **Ops 负责"让人知道、让人用、留下来"**

* * *

## Research：解决"这个方向值不值得做"的问题

**核心问题**：在动手投入资源之前，判断一个赛道、一个产品方向、一条技术路线是否真的成立。

**具体做的事**：

-   赛道研究：这个方向的市场现状、竞品、真实需求是什么（比如咱们分析过的Monad高频交互场景，判断"为什么这个场景需要频繁交互""如果链慢/贵会遇到什么问题"）
    
-   技术判断：某项技术（比如EVM兼容、BFT共识）解决的到底是什么问题，值不值得基于它构建
    
-   产品可行性分析：判断一个方向"是否真的需要链上记录，而不是普通数据库就够了"——这正是我们之前反复练习的判断框架
    

**对应到我已经学过的例子**：

-   判断"AI agent支付要不要用区块链"，本质就是一次Research工作——要去查数据（交易量、增速、玩家阵容），而不是凭直觉下结论
    
-   FluxA选择在Base而不是随便一条链上做USDC结算，背后也是一次"手续费/延迟对比"的Research判断
    

**Research 解决的根本问题**：**避免团队在错误的方向上浪费Dev和Ops的资源**——如果方向本身不成立，代码写得再好、社区运营得再热闹也没用。

* * *

## Dev（Tech）：解决"这个东西能不能被造出来、造得对不对"的问题

**核心问题**：把Research判断出的方向，变成一个真实可用、安全可靠的产品。

**具体做的事**：

-   写合约、写前端、搭基础设施（对应你亲手做过的：AI辅助生成NFTBadge合约、Remix编译部署、调用read/write function）
    
-   做安全审查：权限设计对不对、有没有重入攻击风险、命名是否清晰易懂（对应你做过的3个安全检查点）
    
-   保证系统能承受真实的使用压力（对应Monad这类链本身在解决的"高并发下还能不能正常工作"问题）
    

**Dev 解决的根本问题**：**把一个想法，变成一段代码可以证明"这件事真的能运行、而且值得信任"**——这也是为什么你之前学到"Solidity合约部署后几乎没法改"这件事特别重要,Dev工作的严谨程度直接决定产品能不能被信任。

* * *

## Ops：解决"做出来的东西，怎么被人知道、被人用、能留住人"的问题

**核心问题**：技术再好，没人知道、没人用，等于没做。Ops负责把"产品"和"真实的人"连接起来。

**具体做的事**：

-   社区运营：让LXDAO这类组织能持续吸引贡献者、维持活跃度
    
-   活动/黑客松策划：像Monad基金会办黑客松吸引开发者一样，用活动快速积累使用者和口碑
    
-   内容/Meme/传播：把技术特性翻译成普通人能理解、愿意转发的叙事（比如"手续费只要几厘钱"这种硬指标，Ops要想办法变成人人愿意分享的故事）
    
-   用户反馈收集：把真实用户遇到的问题，反馈回Dev和Research，形成闭环
    

**Ops 解决的根本问题**：**弥合"技术上能做到"和"有人真的在用"之间的鸿沟**——你之前学的DevRel，其实就是Ops的一个具体分支：DevRel专门服务开发者群体（写文档、办活动、答疑），是"面向技术受众的Ops"。

* * *

## 三者如何互相依赖，不能单独存在

| 如果缺少... | 会发生什么 |
| --- | --- |
| 缺 Research | Dev可能造出一个没人需要的东西，Ops拼命推广也没用 |
| 缺 Dev | Research判断再准，方向再好，也只是纸上谈兵，做不出真实可用的产品 |
| 缺 Ops | 产品做得再扎实，没人知道，也没人用得起来，团队的努力"锁在抽屉里" |

**一个完整的项目周期，实际上是这三者不断循环**：Research判断方向 → Dev把它做出来 → Ops让人用起来、收集反馈 → 反馈回到Research，判断要不要调整方向 → 循环继续。

## Monad的快链方案

### （1）Monad——"跑得特别快的以太坊表亲"，它到底是怎么做到"快"的？

**以太坊的老办法：**不管有多少笔交易同时涌进来，都得排成一条队，一笔一笔按顺序处理——像超市只开一个收银台，后面排的人再多，也只能一个一个来。

**Monad 的新办法：同时打开很多个"收银台"（并行处理很多笔交易）。只要两笔交易没有互相冲突（比如没有同时修改同一个人的余额），就能真的同时办完；万一恰好"撞车"了，才把冲突的那一两笔单独挑出来，重新排队处理一次。这套思路的专业名字叫"乐观并行执行"。**

"拜占庭容错（BFT）"，就是指一套共识算法/规则，能保证：即使网络里有一部分验证者是坏的、或者出故障了，剩下诚实的大多数,依然能够对账本达成一致的结论。

记住结论：**Monad 靠"很多收银台同时开" + "流水线式确认"，做到了又快、又便宜。**

### （2）六大类天生适合高频链的场景 ★

-   **高频交易 / 状态更新场景：**比如链上订单簿、实时报价撮合——价格和挂单每秒都在变，慢一秒就可能吃亏。
    
-   **Consumer Crypto（面向大众消费者的加密应用）：**比如稳定币支付、日常小额转账，用户体验必须像支付宝一样"秒到账"才有人愿意用。
    
-   **游戏 / 小游戏：**实时对战、道具交互，卡顿一秒游戏就没法玩了。
    
-   **Onchain Social（链上社交）：**点赞、打赏、抢红包这类"随手一点"的高频小动作，需要即时反馈。
    
-   **AI Agent 交互：**就是前面学的——Agent 经常需要连续、高频地调用、支付、结算，慢链会拖垮整个自动化流程。
    
-   **排行榜 / 任务系统：**分数、进度需要频繁更新，大家都盯着实时排名，延迟大就没意思了

### （4）**X402 协议**

**背景：**早在互联网设计之初（HTTP 协议规范里），就预留了一个叫 `402 Payment Required` 的状态码——意思是"这个内容需要付费才能看"。但过去几十年，几乎没人真正启用它，因为一直缺一套"方便的自动扣款方式"，这道门就一直被闲置着。

**X402 协议做的事：X402 是 Coinbase 去年提出的协议。把这道"门"真正装上，而且是专门给 Agent 用的——Agent 访问一个需要付费的服务时，服务器返回"402，请付款"，Agent 的钱包自动**完成支付，服务器验证到账后放行，全程不需要人再点一次"确认支付"。

### （5）批量结算协议

Agent 经常需要做很多"小额、高频"的支付（比如每调用一次 API 就要付几分钱）。如果每一笔都单独在链上确认，速度和手续费都扛不住。

**解决办法：先预授权一个额度，Agent 在额度内先"记账"，很多笔小额交易攒到一起，最后统一结算一次——这就是批量结算协议**。

> **Layer1（L1）**就是以太坊、Monad 这些"主链"本身——账本最终、最权威的那一份。\*\*Layer2（L2）\*\*是搭建在主链上面的"快速通道"：先在快车道上把很多笔交易攒起来处理，最后打包一次性结算回主链，这样既省时间、也省手续费。批量结算协议用的正是这套"先攒后结"的思路。

## 1.EPF

**EPF：以太坊协议研究员计划**（Ethereum Protocol Fellowship）。是以太坊官方（以太坊基金会）办的一个\*\*"核心开发者训练营 + 师徒制"\*\*，专门培养那些想去建设、改进"以太坊这台大机器本身"的人。

以太坊分为 执行层和共识层

-   **执行层（Execution Layer，简称 EL）＝ 账房先生 🧮**它负责"**算账和记状态**"：执行您的每一笔转账、跑智能合约、记住"谁有多少钱、每个合约里存了啥"。教材里一句话总结得好——**执行层管"算账+状态"**，相当于这台"世界计算机"的 CPU 加硬盘。
    
-   **共识层（Consensus Layer，简称 CL）＝ 裁判投票团 ⚖️**它负责"**选块和投票盖章**"：一大群"裁判"（验证者）按规矩投票，决定"这一页账到底算不算数、由谁来写下一页"。**共识层管"选块+投票"**。
    

**PR（Pull Request，修改请求**）＝ 你把代码改好后，提交给项目维护者"请审核、同意就采纳我的修改"。像把改好的一页维修手册交给管理员过目。新人建议从"改动小、范围小"的 PR 做起，更容易通过。

**EIP** 全名是 **以太坊改进提案**（Ethereum Improvement Proposal）。谁要是想改进以太坊（比如"我建议把某个规则改一改，能更省钱/更安全"），就写一份正式文档，**编上一个号**（比如第 7688 号就叫 EIP-7688），然后放到 GitHub 上，大家**公开讨论、辩论、投票**；通过了，就会在以太坊下一次\*\*"升级"\*\*时正式落地。

### （1）Research（约300字）

排行榜和任务系统本质上是"频繁写入 + 频繁读取"的场景。一局游戏结束就要更新分数，一个任务完成就要记录进度，用户会期待这些反馈几乎是实时的——如果延迟感明显，游戏的爽感会立刻被打断。

如果把这类状态放在一条慢链上，会遇到两个具体问题：一是每次更新分数或任务进度都要等区块确认，可能几秒到几十秒不等，玩家会觉得"点了没反应"；二是手续费如果偏高，玩家为了刷一个小任务、争一个排行榜位置去多次交易，成本很快就不划算了，尤其是这类应用本来用户单次价值就很低，高 gas 会直接把这类玩法判死刑。

Monad 的高性能和低延迟正好对上这个痛点：出块快、确认快，意味着"操作—上链—看到结果"这个循环可以做得接近实时，游戏体验不会被链拖累。同时它保持 EVM 兼容，意味着现有的 Solidity 合约、钱包连接方式、开发工具链都能直接搬过来，不需要重新学一套语言或基础设施，这对独立开发者或小团队尤其友好。

排行榜和任务进度是否真的需要上链？如果只是给单个游戏内部记分，普通数据库完全够用，甚至更便宜更快。真正需要链上记录的，是当排行榜结果、任务成就本身要成为**可验证、可跨应用使用的资产**时——比如任务完成后铸造一个可以被其他游戏识别的徽章（跟我之前做的 NFT Badge 类似），或者排名前列能解锁链上可交易的奖励。这时候"上链"才不是噱头，而是让成就变成一种可携带、可组合的资产。

## 4.Go语言

Go（也叫 Golang）是 Google 在 2009 年发布的一种编程语言，主要设计目标是"简单、高效、适合大规模系统开发"。

### 核心特点

-   **编译型语言**：代码编译成机器码直接运行，速度接近 C/C++，比 Python 这类解释型语言快很多
    
-   **语法简洁**：关键字很少（只有 25 个左右），去掉了很多其他语言里"复杂但用得少"的特性（比如没有传统意义上的类继承、没有泛型的历史包袱，虽然后来加了泛型支持）
    
-   **原生并发支持**：这是 Go 最大的亮点。用 `goroutine`（协程）可以极其轻松地写并发程序，一个 `go func(){...}()` 就能起一个轻量级线程，配合 `channel` 做协程间通信，比传统多线程编程简单太多
    
-   **自带垃圾回收**：不用像 C/C++ 一样手动管理内存，但性能损耗比 Java 这类语言小
    
-   **编译速度快**：大型项目编译也很快，这也是 Google 内部选它的原因之一（当时 Google 内部 C++ 项目编译动辄几十分钟）
    

### 常见应用场景

-   **后端/云基础设施**：Docker、Kubernetes、Prometheus 这些云原生领域的明星项目都是 Go 写的
    
-   **高并发服务**：比如 API 服务器、微服务，因为并发模型简单又高效
    
-   **区块链/Web3**：不少区块链底层客户端用 Go 写，比如以太坊的 go-ethereum（Geth）就是最主流的以太坊客户端之一
    

### Solidity和Go的区别

**Solidity —— 应用层（智能合约）**

-   写在链上运行的合约逻辑，比如你之前在 Remix 部署的 NFT Badge 合约
    
-   运行环境是 EVM（以太坊虚拟机），代码上链后由全网节点执行
    
-   你写的是"规则"：谁能铸造、怎么转账、余额怎么算
    

**Go —— 基础设施层（区块链客户端/节点)**

-   写的是**运行区块链本身**的程序，也就是"节点软件"
    
-   最典型的例子是 **go-ethereum（Geth)**，它是以太坊最主流的客户端，负责：打包交易、维护 P2P 网络、验证区块、真正去**执行**EVM 里的 Solidity 字节码
    
-   换句话说，Geth（Go 写的）里包含了一个 EVM 实现，专门用来"读懂并运行"Solidity 编译出来的字节码

# **Web3 学习笔记 Day 4**

## 1.理解合约代码

### （1）整体结构

```solidity
contractNFTBadge{... 这里面所有代码都属于这一台"自动售货机"}
```

`contract NFTBadge { }` 就是这台机器的外壳——里面包住的所有变量和函数，都是"NFTBadge"这一个合约的组成部分,别的合约访问不到里面的东西,除非通过公开的函数

### （2）状态变量

```solidity
mapping(uint256=>address)public badgeOwner;
```

`mapping` 你可以直接理解成**一张查询表/字典**——左边输入一个东西，右边查出对应的另一个东西。

`mapping(uint256 => address)` 的意思是：**"给我一个徽章编号（uint256），我告诉你这个编号属于哪个地址（address）"**。

比如：

```
badgeOwner[0] = 0xAAA... (0号徽章属于这个地址)
badgeOwner[1] = 0xBBB... (1号徽章属于这个地址)
```

这就是咱们之前讲NFT时说的"账本上记的是这一个独一无二的编号目前归谁"——这行代码就是那句话的具体实现。

### （3）权限锁（谁能操作这台机器）

```solidity
modifieronlyOwner(){require(msg.sender== owner,"Not authorized: only owner can mint");_;}
```

`modifier` 是Solidity里一种"可复用的门禁检查"**，你可以理解成一道装在某些函数前面的**安检门\*\*。

-   `msg.sender`：Solidity里的一个内置变量，**永远代表"当前正在调用这个函数的人是谁"**——也就是发起这笔交易的钱包地址
    
-   `require(条件, "报错信息")`：**如果条件不满足，整笔交易立刻失败，并显示报错信息**（还记得咱们讲过的"失败交易也要消耗gas"吗？就是`require`不通过导致的典型场景）
    
-   `_;`：这个下划线是个占位符，意思是"安检通过后，才继续执行被保护的那个函数剩下的代码"

# **Web3 学习笔记 Day 3**

## 1.哈希Hash

**哈希函数**——不管你塞进去的东西多大多小，吐出来的"指纹"永远是固定长度的一串字符。这个过程叫"哈希"，吐出来的结果就叫"哈希值“。

哪怕只改动输入里的一个标点，输出的哈希值会**天翻地覆地变化**，完全看不出规律。这个特性极其重要——它意味着**只要账本上任何一个字被偷偷改动，对应的哈希值立刻会变，篡改行为无所遁形**。

给你一个哈希值，你**没办法反推出原始输入是什么**

### 回到区块链本身：为什么"链"叫链

还记得咱们最开始说的"公共账本，谁也改不了"吗？**哈希正是实现"改不了"这个承诺的技术手段**。

区块链的"块"，是一批批打包好的交易记录。关键设计是：**每一个新区块，都把"上一个区块的哈希值"记录在自己内部**。

你之前贴的转账交易记录里，那个交易本身其实也有一个专属的"**交易哈希（Transaction Hash / TxHash）**"——就是那串很长的字符串，本质上就是"这笔交易的全部内容"经过哈希运算后得到的唯一指纹。你在Monadscan或者MonadVision里粘贴这个哈希去搜索，就能精确定位到这一笔、且只有这一笔交易——因为哪怕交易内容差一个字节，这个哈希都会完全不同。

## 2.NFT代币

NFT = **Non-Fungible Token（非同质化代币）**

区块链是一本"所有人都有一份、谁也改不了"的公共账本。

-   **普通加密货币（比如MON、比特币）**：账本上记的是"数量"——就像账本上写着"A地址有10个单位"，这10个单位彼此完全等价，可以随意拆分、合并
    
-   **NFT**：账本上记的是"这一个独一无二的编号，目前归谁所有"——比如"编号#4728这个东西，现在属于A地址"。这个编号**独一无二，不能拆成一半，也不能跟另一个编号互换成"等价的东西"**
    

## 3.EVM 是什么

**EVM = Ethereum Virtual Machine（以太坊虚拟机）**

还记得咱们讲Solidity合约时的比喻吗——"自动售货机"。EVM就是**运行所有这些自动售货机代码的那台"标准发动机"**。

你写的Solidity代码，不是直接被公共账本读懂的，而是先编译成一种EVM能理解的底层指令（叫bytecode），然后**EVM负责实际执行**这些指令、算出结果、更新账本。

### 什么是"EVM兼容"

**以太坊自己有一台EVM，但这台"发动机"的设计图纸是公开的**。于是很多新链（包括Monad）就直接**照着这套图纸，自己造了一台功能一模一样的发动机**，装在自己的公共账本系统里。

这就是"EVM兼容"的意思：**这条链虽然不是以太坊本身，但它的"发动机"跟以太坊用的是同一套标准**。

## 5.Solidity合约是什么

它是专门为以太坊（以及所有EVM兼容链，比如Monad）设计的编程语言，写出来的代码经过编译后，会被部署到公共账本上，变成一个能被任何人调用的"合约"。

**Solidity合约**：一旦部署到公共账本上，**代码基本没法修改**——因为账本的核心特性就是"不可篡改"，你的合约代码也是账本的一部分。这意味着：**上线前必须反复审计**，一旦有漏洞被黑客发现，钱包里的资产可能被直接转走，没有"下线修复"这个选项。这也是为什么之前提到的"代码审计"在Web3圈子里是个专门的高薪职业。

**智能合约，就是放到区块链上的那一小段程序。**它的特点是：

-   **自动执行：写好规则放上去，只要有人满足条件去触发它，它就自动照章办事，不需要任何人在旁边盯着**。
    
-   **改不了、关不掉：**一旦放上链（部署），它的规则就固定了，谁也没法偷偷改它、也没法把它关停（还是第 3 章那个"不可篡改")。

```solidity
请帮我写一个最小化的 Solidity 智能合约，用于 Web3 训练营的学习练习。

需求：
- 功能是"NFT Badge"——一种不可转让的徽章，用来记录某个地址完成了某项任务/活动
- 只有合约部署者（owner）能够铸造徽章给别人，普通用户不能自己铸造

- 不需要实现完整的 ERC721 标准（不需要 transfer、approve 等功能），因为这是不可转让的荣誉徽章
- 使用 Solidity ^0.8.20
- 代码尽量精简，变量和函数命名清晰易懂，适合初学者阅读
- 生成代码后，请用简单语言解释一下这份合约的结构：有哪些状态变量、哪些函数、权限控制是怎么实现的
```

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract NFTBadge {
    string public name = "Web3 Summer Camp Badge";
    string public symbol = "CAMP";

    address public owner;
    uint256 public nextTokenId;

    mapping(uint256 => address) public badgeOwner;
    mapping(uint256 => string) public tokenURI;
    mapping(address => uint256) public balanceOf;

    event BadgeMinted(address indexed to, uint256 indexed tokenId, string uri);

    modifier onlyOwner() {
        require(msg.sender == owner, "Not authorized: only owner can mint");
        _;
    }

    constructor() {
        owner = msg.sender;
    }

    function mintBadge(address to, string memory uri) public onlyOwner {
        require(to != address(0), "Cannot mint to zero address");

        uint256 tokenId = nextTokenId;
        badgeOwner[tokenId] = to;
        tokenURI[tokenId] = uri;
        balanceOf[to] += 1;

        nextTokenId += 1;

        emit BadgeMinted(to, tokenId, uri);
    }
}
```

### AI的解释

> 这个合约有三组状态变量：`badgeOwner`（记录每个tokenId归谁）、`tokenURI`（记录每个徽章指向的元数据链接）、`balanceOf`（记录每个地址一共拥有几个徽章）。核心函数只有一个——`mintBadge`，被 `onlyOwner` 这个修饰器保护，意味着只有部署合约的那个地址能调用它。每次铸造成功后会触发 `BadgeMinted` 事件，方便外部程序（比如前端界面）监听到"谁在什么时候获得了徽章"。这个合约刻意没有实现转让功能，因为设计目标是"记录荣誉"，不是"可交易资产"。

## 2.概念理解——钱包到底是什么

**MetaMask钱包里并没有存着你的"钱"**。

区块链是一个"所有人都有一份、谁也改不了"的公共账本，你的余额、资产，其实是**记在这本公共账本上的一行数据**，账本本身存在全世界成千上万台电脑里，不存在你的MetaMask里。

那MetaMask里到底装的是什么？——**一把钥匙**。

具体说，是一对"钥匙"：

-   **私钥（Private Key）**：只有你自己知道的一串密码，谁拿到这串密码，谁就能操作账本上属于这个地址的资产。**这个丢了/泄露了，没有客服能帮你找回**（还记得咱们讲过的"去中心化没有客服"吗）

所以MetaMask更准确的定位是：**一把能打开公共账本上"你那一页"的钥匙串**，它本身不存钱。

为什么要选网络

Monad 的技术方案基本是针对这个瓶颈逐条打补丁：

-   **Optimistic Parallel Execution（乐观并行执行）**：交易并行处理，事后解决冲突，而不是像以太坊那样默认串行
    
-   **MonadBFT**：基于 HotStuff 优化的共识算法，专注缩短出块与最终确认时间
    
-   **MonadDB**：为区块链工作负载定制的存储层，让普通硬件也能跑节点，兼顾性能与去中心化
    

宣传的目标是 全兼容以太坊虚拟机（EVM），开发者可以直接部署现有的以太坊智能合约，无需重写代码或学习新的执行环境，同时把吞吐量和延迟做到远超以太坊现状。

Monad 在这套逻辑里扮演什么角色：它不是在发明新概念，而是在解决"**公共账本记得太慢**"这个问题。以太坊这个账本很安全、很公开，但记账速度慢（每秒能处理的交易少），一旦很多人同时来记账，就得排队、加钱插队（也就是 Gas 费飙升）。Monad 想做的，就是让这个公共账本**记得更快、成本更低**，同时保持"公开透明、没有老板"这两个核心特性不变。
