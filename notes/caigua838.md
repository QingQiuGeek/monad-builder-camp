**🔍深读 Moss 仓库:从"这是什么"到"这怎么管人"**

上周读过 README,今天翻整个仓库(目录、issues、标签、贡献流程)。**两次收获完全不同**:

**最大发现:标签体系是一套贡献者路由系统。**

四个维度:难度(starter/intermediate/advanced)、类别(adapter/dex/lending...)、状态(needs-design = 被核心决策卡住了,先别写)、**以及谁能做**。

最后一个维度有两个我第一次见的标签:

| 标签 | 官方描述 |
| --- | --- |
| ready-for-agent | "Fully specified and ready for an AFK agent" |
| maintainer-only | "Reserved for maintainers. Do not start work unless invited." |

`ready-for-agent` 的意思是:**这个任务已经描述到 AI 能无人值守独立完成。** 维护者在分派工作时,已经把"人做"和"AI 做"分开标注了。这不是"我用 AI 写代码",是**把 AI 协作做成了工程管理制度**。

而 `maintainer-only` 补上了我上一次探索漏掉的一半:

> **一个好的贡献漏斗,既要有"从这里进"的指引,也要有"这里别碰"的告示。** 否则新人会把力气花在注定不会被合并的地方,而这种挫败比找不到入口更劝退。

**这是运营思维,不是工程思维。** 开源项目的贡献者增长,和产品的用户增长,是同一个漏斗问题。

**Day 15 | Monad Builder Camp 残酷共学**

今天听区块链钱包产品设计。这场的价值在于:它把我这两周踩的坑,变成了产品设计的取舍框架。

**🔐 热钱包 vs 冷钱包:一张取舍表**

|   | 热钱包 | 冷钱包 |
| --- | --- | --- |
| 联网 | 全程联网 | 全程离线不触网 |
| 设计优先级 | 便捷 > 安全 | 安全 > 便捷 |
| 场景 | 日常高频、体验链上项目 | 大额资金长期存储 |
| 典型特性 | 支持批量自动签名 | 禁止批量自动签名 |
| 代表 | MetaMask、Rabby | 各类硬件钱包 |

"批量自动签名"**:热钱包为了便捷支持它,冷钱包为了安全**禁止\*\*它。这正好接上我上周学的合约安全——批量授权+自动签名,就是钓鱼攻击最爱的组合。**便捷和安全在这里是真的零和,不是口号。**

而我自己用的正是MetaMask(热钱包),课程钱包里只有测试币——**这个选择本身就对了:高频试错用热钱包,里面不放真钱。** 我当时是被camp要求这么做的,今天才知道这是行业规范。

**🚨 一句重复了无数次、但今天有了产品视角的话:**

**"助记词唯一备份丢失后无任何恢复可能,市面宣称可协助恢复助记词的服务均为骗局。"**

这句我这半个月听了至少五遍。但今天不一样——今天是从**产品设计**角度讲的:非托管钱包(OneKey)**无权限**协助恢复,因为它**始终不持有用户私钥**。

这不是"钱包做得不够好",这是**去中心化的定义本身**:你拥有绝对主权,代价是绝对责任。托管钱包(币安)能帮你找回,因为账本记在平台;非托管钱包找不回,因为账本只认私钥。**主权和便利,又一次零和。**

**💡 今天的产品洞察**

整场讲的其实是**同一个权衡的不同切面:去中心化程度 ↔ 用户体验**。

-   热 vs 冷 = 便捷 vs 安全
    
-   托管 vs 非托管 = 可找回 vs 主权
    
-   自动签名 vs 手动确认 = 顺滑 vs 可控
    

而 **AA钱包**(我第一周学的账户抽象)之所以是行业方向,就是想在这条零和线上找一个更好的点:既保留主权,又让普通人不必背负"丢助记词=倾家荡产"。

这跟我一直在说的那句是同一件事:**Web3产品设计的核心矛盾,是"去中心化"和"好用"的对冲。谁能在不牺牲主权的前提下把体验做到Web2级别,谁拿下下一亿用户。**

**Day 12 | Monad Builder Camp 残酷共学**

今天干了两件事:读 Moss README,把 Space 的内容流程设计完。

**📖 Moss:**

**第一次**:我先根据分享会笔记推了一版理解框架,读完 README 发现三处全错——

| 我以为 | 实际 |
| 四步是”发现→加载→模拟→执行” | discover → load → action → simulate,模拟是最后一步 |
| Agent 调用钱包执行交易 | Moss 永不签名、永不发送 |
| 跑在测试网 | Monad 主网(chain ID 143) |

**第二次**:读完之后我总结了一句”Moss 是建立在 Monad 上一个检测并模拟交易的 agent,让用户提前看到交易是否安全”——三处又错了:

-   **Moss 不是 Agent,是 Agent 用的工具**(MCP server)。Agent 是拿工具的人,Moss 是工具箱,它不会自己动。
    
-   **它不只是模拟器,主要在造交易**。四步里前三步都是”造”,最后一步才是”验”。如果只做模拟,它暴露一个 simulate 就够了,但它暴露了四个——**它真正的价值是那句”Agent 不该手搓 calldata”**。模拟是它的良心,不是主业。
    
-   **它从不说”安全”**。它给的是证据,不是结论:“执行后你会少1个MON、多2.3个USDC、**外加授权了某地址无限额度**”——最后那条该不该慌,你自己判断。README 自己反复设限:**“模拟是一道安全网,不是保证”**。
    

修正后的一句话:**Moss 是跑在 Monad 上的工具框架,让 AI Agent 不用手搓代码就能跟链上协议打交道;它负责把交易造出来、在真实链状态上演一遍、把”实际会发生什么”摊给你看——但它永不签名,判断和确认永远是人的事。**

**🔍 验证了昨天的假设:simulate 是规范,不是技术强制**

昨天我提了个假设:Moss 的”强制模拟”可能只是写给 Agent 看的规范,而不是代码层的强制。今天在 README 里找到三处证据:

1.  **文档列表那一行**:「Agent 安全规则 | **强制模拟与意图对齐规则**」——它是一份**写给 Agent 的规则文档**。
    
2.  **“MCP server 只暴露 discover、load、action 和 simulate”** —— 四个独立工具,Agent 想按什么顺序调就按什么顺序调。
    
3.  **最硬的证据在 library 示例里**:
    

const simulation = await simulator.simulate(result);

if (simulation.halted || simulation.results.some(i => i.warnings.length)) {

  throw new Error("simulation failed; do not sign");

}

**这个 throw 是调用方自己写的。**action 返回结果后,你完全可以跳过 simulate 直接签——库不会拦你。

**结论:规范上要求,不是技术强制。而且这是设计,不是疏漏。** 因为 Moss 选择了”永不签名”,它就**没有能力**强制你模拟——像一个不管收银台的导购,没法强制你结账前看价签。它能做的是让不安全的路径**可见**,而不是**不可能**。

**这周我第四次撞到同一句话:签名 ≠ 意图。**

-   tx.origin 漏洞:管理员签了名 ≠ 管理员想干这事
    
-   提示词注入:Agent 收到指令 ≠ 用户下的命令
    
-   未检查 call 返回值:调用了 ≠ 成功了
    
-   Moss:交易构造好了 ≠ 用户同意了
    

**每次的错都一样:把”发生了”当成”授权了”。** 而 Moss 的回答最诚实:**我管不了最后按下确认键的那一下,所以我把对账材料给足你。**

**🎤 Space 内容流程设计**

《不会写代码的人,能在一个晚上部署智能合约吗?

**1️⃣ 互动环节从”现场发交易”改成”现场查我的合约”。** 零基础的人在 Space 里装钱包+配网络+领水龙头+转账,20分钟做不完,现场会死一片。而”打开浏览器搜一个地址”是30秒、人人能做——**而且建立的体感更狠:你没连钱包、没登录、没经我允许,就看到了我做过的一切。** **现场负责”信”,课后负责”做”。**

**2️⃣ 全场只保证一个 aha。** 就是那个反直觉的问题:“我转了0.001,gas花了0.0022,手续费是本金两倍——为什么?”(因为gas按**计算量**收,不按金额。转1块和转一万,活儿一样多。)一说就懂、一辈子记得。这是跟 Speedrun Ethereum 学的”一次一个顿悟”。

**3️⃣ CTA 设了三层**,这是全案最值钱的部分:

| 层级 | 动作 | 门槛 |
| 主 | 发第一笔交易,评论区报 hash | 中 |
| 备选 | 卡住了?把报错截图发评论区 | 低 |

**为什么**:如果只奖励”发出hash”的人,做不到的人会静默流失、并再次确认”我果然不行”。**一个说”允许失败”却只奖励成功的活动,是自相矛盾的。** 我说了卡住是常态,我的 CTA 就必须接住卡住的人。

今天两次打脸都不是AI骗我,是我拿二手材料(会议纪要、自己的总结)推了个听起来通顺的框架,然后没核对就信了。**通顺 ≠ 正确。**

**⚠️ 四个踩出来的知识点**

**1️⃣ 差点部署到假链上。** Remix默认Environment是"Remix VM",给你100个假ETH,部署了也只存在浏览器内存里,刷新就没。识别信号:余额是整数、账户不是自己的、Environment写着VM。真上链要选 Browser Extension 连钱包。

**2️⃣ 自己写的require拦住了自己。** 重复点checkIn,Remix直接弹窗:`execution reverted: "Already checked in today"`。这是我写在代码里那道防线在说话。顺带发现:gas预估会在花钱前预演一遍交易,所以注定失败的交易大多根本发不出去——"失败也扣gas"通常发生在预检通过、上链时条件才变的场景。

**3️⃣ 透明 ≠ 可读。** Remix日志显示Sourcify验证成功,但MonadVision用自己的验证服务,合约页面依然显示 "Contract not verified",只能看到一坨十六进制字节码。链上数据永远透明,但要人类读得懂,依赖ABI和源码验证这层翻译。不同浏览器认不同的验证服务,这是我今天最意外的发现。

**4️⃣ tx hash 不是 block hash。** 写README时差点填错。Remix终端日志里那个 `hash:` 是**区块**hash,交易hash要去区块浏览器的 Creation Txn 字段拿。教训:**本地工具的日志和链上记录冲突时,信链上。链是唯一真相源。**

**🤖 关于用AI写代码**

最大的收获不是"我会写合约了"(我不会),是**我知道该在哪里不信AI**:

-   **业务边界**:它默认按UTC日切,没提这对东八区用户意味着早8点才算新一天
    
-   **权限范围**:必须人工列出所有能改状态的函数,逐个确认谁能调、影响谁
    
-   **安全假设**:"低风险"这个结论本身要人走完检查才能下,不能默认
    

AI能生成技术上正确但业务上未必合适的隐含决策。这三处必须人来定。

**💡 嘉宾分享:AI Agent 安全风险**

提示词注入是Agent最现实的攻击面,而且零技术门槛:把恶意指令藏进Agent会读到的内容里(网页、GitHub仓库、文档),Agent就可能当成用户命令执行。真实案例包括GitHub植入指令诱导安装恶意软件、Grok账号被诱导转移资产。
