# **AI Web3 最小原型开发学习笔记**

## 项目概述

做了一个「AI 辅助的链上交易意图网关」，用户输入自然语言（如"0.1 ETH换成USDC"），系统自动解析意图、查询报价、生成模拟交易回执。整个过程不上链、不连钱包，是一个验证"AI+Web3"可行性的最小原型。

## 技术栈

React + Vite（前端框架）

ethers.js（生成交易哈希）

CoinGecko API（获取ETH实时价格）

硬编码解析（关键词 + 正则表达式）

## 开发步骤

第一步：创建项目

mkdir ai-web3-demo

cd ai-web3-demo

npm create vite@latest . -- --template react

npm install

npm install ethers

npm run dev

第二步：编写前端代码

创建 src/App.jsx，包含：

输入框 + 按钮

意图解析逻辑（硬编码版）

价格查询（CoinGecko API）

模拟交易（生成假哈希）

结果展示区

第三步：运行测试

npm run dev

浏览器访问 [http://localhost:5173](http://localhost:5173)

测试输入：

"0.1 ETH换成USDC"

"0.05 ETH换USDC"

"100 USDC换ETH"

## 遇到的问题及解决办法

### 问题1：权限报错 EPERM

报错内容：

Error: EPERM, Permission denied: \\\\?\\C:\\Users\\29699\\.brainmemory

原因：.brainmemory 文件夹被系统保护，普通终端没有删除权限。

解决办法：以管理员身份运行终端（右键点击cmd → 以管理员身份运行），然后重新创建项目和安装依赖。这个方法从根本上解决了权限问题，以后不会再出现同类报错。

### 问题2：ethers 找不到

报错内容：

Failed to resolve import "ethers" from "src/App.jsx". Does the file exist?

原因：进入项目文件夹后忘记安装 ethers 依赖包，或者安装命令没有执行成功。

解决办法：先确认当前目录是项目根目录 ai-web3-demo，然后执行安装命令：

npm install ethers

安装成功后再启动项目：

npm run dev

### 问题3：找不到 dev 命令

报错内容：

npm error Missing script: "dev"

原因：终端所在目录不对，没有进入项目文件夹，而是在用户根目录 C:\\Users\\29699> 下执行的命令。

解决办法：先进入项目文件夹：

cd ai-web3-demo

然后再执行启动命令：

npm run dev

一定要确认终端路径显示为 C:\\Users\\29699\\ai-web3-demo> 才算正确。

### 问题4：浏览器跳转到 YOYO 文件管理助手

问题现象：打开浏览器输入 [http://localhost:5173，结果跳转到了](http://localhost:5173，结果跳转到了) YOYO 文件管理助手界面，看不到我们的项目页面。

原因：YOYO 是电脑上安装的一个文件管理软件，可能是后台程序常驻运行，或者占用了相关端口，导致浏览器被劫持。

解决办法：

方法一：按 Ctrl + Shift + Esc 打开任务管理器，在进程列表中找到 YOYO 相关程序，右键选择"结束任务"。

方法二：换个端口启动项目：

npm run dev -- --port 5174

然后在浏览器地址栏输入 [http://localhost:5174](http://localhost:5174) 访问。

这个问题的根源在于电脑上安装了第三方软件，跟我们的项目代码没有关系。

AI在科研中的应用

一、用AlphaGo类比提出的核心问题

用AlphaGo打败职业围棋手的例子来引出一个更本质的问题：研究者会被AI取代吗？还是说，研究者可以把AI当成增幅器，变成更强的研究者？

这里的关键不是AI能不能做，而是研究者自身是否能做到两点：第一，有足够的学科理解与判断力；第二，把AI变成自己的助手，而不是把思考外包给AI。

二、案例：把资料输入AI，快速生成深度研究分析

分享提到一个很具体的例子：有科学家把与T-cell sampling相关的资料输入AI，大约100分钟，AI就生成了一份很有深度的实验分析报告。

这份报告不只是摘要或翻译，而是覆盖了研究中常见的关键模块，例如：

第一，实验数据层面的整理与分析，把数据与信息结构化。

第二，背后的逻辑与解释路径，说明为什么能得到这些结论。

第三，机制模型的建立，梳理可能的机制如何串联起来。

第四，实验限制或研究限制，指出哪些地方站不稳、证据不足。

第五，新的因果假设，提出可能的因果链条与需要验证的点。

第六，下一步研究规划，说明接下来应该做哪些实验或验证。

这段想表达的是：AI不只是写得快，而是已经能在一定程度上模拟研究分析流程，帮助研究者缩短从材料堆到研究计划的路径。

三、AI已经在部分重大科学难题上显示出解题能力

分享中提到：AI在蛋白质折叠等几十年级别的难题上，已经取得非常高的准确率，并且在相当大范围内推动了问题的解决。

这用来强调一点：AI的角色不只是文书工具，它在某些领域确实开始具备强分析与强建模能力。

四、建议的方法：先建立自己的知识框架，再用AI放大能力

分享给出的使用姿势更偏向研究方法论，而不是多用AI这么简单。

要先深度学习自己的领域：先有自己的理论、概念框架、方法论地图，先知道什么是关键变量、什么是证据、什么叫因果。

再让AI做放大器：帮你快速梳理材料，帮你生成结构化分析，帮你提出可能的机制、假设和实验路径。

一个很重要的要求是：让AI标注原始资料来源，包括引用、出处、依据来自哪里。这样才能追溯和核验，避免看起来很对但其实是编的情况。

五、警示：不要被AI牵着走，输出必须能理解、能复核

分享强调：使用AI时要避免一种常见陷阱，就是AI说什么就信什么。

更稳的方式是：你先有框架与问题清单，再去问AI，对AI的回答要能解释为什么，而不是只看结果像不像真的。

也就是说，你必须理解输出背后的逻辑，否则AI的内容只是文本，不是知识。

六、一句话结论

AI时代，研究者的分水岭不是会不会用AI，而是能不能在自己已有的知识框架上，把AI当成分析与研究设计的加速器，同时保持对证据、来源、逻辑链条的可解释与可验证。

AI×Web3 学习笔记总结

本次学习主要围绕 AI 辅助开发方法、Web3 技术体系以及链上产品推广实践展开，对 AI 与 Web3 的结合应用有了更系统的认识。

一、AI 辅助开发方法

本次分享重点介绍了 TDD（测试驱动开发）和 SDD（规范驱动开发）。

TDD 的核心流程是：先编写测试，明确预期结果，再完成最小功能实现，最后进行优化重构。通过提前定义验收标准，可以帮助 AI 快速发现问题并进行纠错，避免修改代码时影响原有功能。

SDD 更强调开发前的需求规划，包括明确目标、制定规范、设计方案、拆分任务和记录过程。通过规范化流程，可以让 AI 更准确理解需求，提高复杂项目开发效率。

在 AI 时代，开发者需要提升四项核心能力：

1\. 清晰描述需求；

2\. 拆解复杂任务；

3\. 验证 AI 输出结果；

4\. 设计整体架构。

AI 可以提高开发效率，但无法替代开发者的思考和判断能力。

二、Web3 技术体系学习

分享从以太坊出发介绍了区块链基础架构。区块链可以理解为一个异步、强一致性、可编程的数字状态系统，通过多个节点共识维护链上数据。

以太坊主要分为：

\* P2P 网络层；

\* 共识层；

\* 执行层；

\* 接口层。

普通开发者通常通过 RPC 接口调用区块链能力，不需要直接参与底层网络开发。

三、Web3 五大开发方向

1\. DApp 开发

主要面向用户交互，通过 React、Vue 等前端框架结合钱包和智能合约实现链上应用，是新手较适合的学习方向。

2\. 智能合约开发

主要使用 Solidity 等语言，实现链上的业务规则和资产逻辑，需要理解 EVM 运行机制。

3\. 钱包开发

核心涉及密钥管理、交易签名和账户安全，需要重点关注钓鱼攻击、重放攻击等安全问题。

4\. 节点开发

属于区块链底层方向，需要掌握 Go、Rust、P2P 网络和共识机制。

5\. 安全审计

通过代码分析发现智能合约漏洞，保障项目安全。

对于初学者来说，可以优先学习生态成熟、资料丰富的以太坊，再根据实际需求选择其他公链。

四、链上产品推广实践

课程通过角色扮演活动，让学员从不同用户角度理解链上投票产品的价值。

不同用户群体需要采用不同推广方式：

\* 面向普通用户，强调链上规则公开透明、结果可验证；

\* 面向学生，突出实践价值和职业发展机会；

\* 面向媒体，突出数据公开和可信验证。

同时，课程设置了 Meme 创作活动，鼓励围绕 Monad 和生态项目进行内容传播，提高 Web3 产品推广能力。

五、学习收获

通过本次学习，可以了解到：

\* AI 时代开发者不仅需要会使用工具，更需要具备需求分析、任务拆解和架构设计能力；

\* Web3 不只是金融应用，而是一套利用区块链建立信任和协作的新型技术体系；

\* 实践是理解 Web3 最有效的方式，通过开发 Demo 和参与产品推广，可以更深入理解行业逻辑。

后续学习中，需要持续提升 AI 协作能力，并结合实际项目探索 Web3 应用落地。

# 智能合约源码

本次实验使用 Solidity 编写一个简单计数器合约。

合约功能：

-   保存一段文本信息；
    
-   查询当前信息；
    
-   修改信息；
    
-   记录计数次数；
    
-   查询当前计数。
    

源码如下：

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract MyFirstContract {

    string public message;

    uint256 public counter;


    constructor() {
        message = "Hello Monad";
        counter = 0;
    }


    // Read Function
    function getMessage()
        public
        view
        returns(string memory)
    {
        return message;
    }


    // Write Function
    function updateMessage(
        string memory newMessage
    )
        public
    {
        message = newMessage;
    }


    // Write Function
    function increment()
        public
    {
        counter++;
    }


    // Read Function
    function getCounter()
        public
        view
        returns(uint256)
    {
        return counter;
    }

}
```

### 1\. AI Agent 金融安全探索

学员围绕 AI Agent 自动支付场景展开分享，指出当前主要存在四类安全风险：

-   **意图偏离风险**：AI 对用户需求理解错误导致错误操作；

1.  有限授权；
    
2.  人工监督；
    
3.  关键任务确认；
    
4.  交易全过程可审计；
    
5.  权限隔离。
    

未来 AI Agent 金融应用需要结合区块链链上透明验证能力、钱包权限管理及风控机制，实现安全可控的自动化支付。

AI Agent安全

六大风险：

过度授权：Agent拿高权限后删库，某SaaS平台9秒删光生产数据

提示词注入：恶意指令伪装成摩斯密码/网页代码，诱Agent转币

恶意插件：Clawhub Top100近20%高风险，窃取私钥和API密钥

系统漏洞：OpenCloud网关默认无密码，公网可直接接管

幻觉：让查资料却刷屏发无效字符

供应链攻击：恶意依赖包，未锁版本即中招

防护建议：

最小权限原则，敏感操作设人工二次确认

插件走官方渠道，依赖包锁版本

服务设高复杂度密码

Agent Guard能力：

运行时实时拦截高危操作（删库、窃取数据等）

安装前自动扫描插件后门

全维度安全体检，输出风险报告

全球威胁情报库自动更新

观点

多Agent投票互审思路有效但易失效（模型会遗忘/幻觉）

AI是概率模型，安全问题根治需从底层训练逻辑入手
