## 一、Web3 运营核心定位与目标

1.  **运营本质**
    
    运营相当于生态里的粘合剂，对内协调技术人员、讲师、团队成员；对外对接项目方、开发者 Builder、合作伙伴，兼顾 B 端合作洽谈和 C 端社区维护。
    
    运营一切工作的核心目标就是**增长**，持续完成拉新、提升活跃度、长期留存用户，推动公链或者项目生态持续壮大。
    
    以 Monad 实习计划举例：Co-Learning 学习营、线上分享会、积分排行、黑客松、各类线上活动、社群日常互动、推文宣传，全部都属于运营工作。如果运营缺位，活动参与人数不足，会直接影响公链曝光、实习计划招募、后续黑客松参赛规模。
    
2.  **两类不同运营赛道区分**
    

-   公链 / 基础设施运营（本次分享重点）：主要面向 Builder 开发者。可以把公链类比高速公路，Builder 就是在公路旁搭建商店、设施的建设者；普通链上交易者相当于路上行驶的车辆。公链前期处于基建阶段，优先吸引开发者落地应用，用户交易手续费是生态成熟之后才会产生的收益。
    
-   交易所运营：面向普通散户交易者，重点做用户注册、入金、交易活跃度相关运营。
    

行业现状：Web3 需求量最大两类岗位就是开发者和运营。大部分初创团队运营都是一人多岗，岗位界限不会划分得非常细致。

## 二、一场 Web3 活动（黑客松 / Builder 实习计划）完整 0~1 运营流程

1.  前期 BD 与方案筹备：对接项目方需求，撰写 Proposal 方案；确定目标人群，据此设计活动形式、课程难度、邀约嘉宾、洽谈多方合作赛道；
    
2.  市场宣发 + 活动方案设计：梳理所有公私域流量渠道（LXDAO 推特、TG、微信群等），规划宣传节奏；
    
3.  Builder 招募：多渠道持续触达潜在参与者；
    
4.  社区运营：维护社群，收集参与者疑问，同步各类通知；
    
5.  活动报名、项目征集：搭建报名表单或者活动官网，引导大家报名、自由组队；
    
6.  项目提交、组织评审、同步反馈；
    
7.  活动收尾交付：向合作项目方提交复盘报告，提供核心数据：招募 Builder 数量、组建团队数量、黑客松项目提交数量、生态新增建设者等内容。
    

## 三、活动执行阶段具体工作内容

### 活动前期落地

新建 TG / 微信社群、制作报名表单；安排线上会议、对接评委与嘉宾；持续收集社群问题，同步技术方与项目方，优化参与体验；整理 Github 资源、Demo 相关素材。

### 活动进行中

只发一次通知效果很差，必须多渠道、多种形式反复触达用户：Workshop、AMA、社群提醒、推文轮番宣传，持续提醒大家组队、提交项目。

完整 Builder 成长路径：了解活动 → 报名进群 → 参与各类线上活动 → 动手开发项目 → 提交作品 → 获取奖励与反馈 → 沉淀进入公链生态 → 长期参与开发、实习、大使计划。

### 活动结束后（容易被忽视）

Demo Day 不代表活动结束：跟进优秀 Builder 和优质项目；充当资源桥梁，对接实习、项目孵化、加速器机会；标记高潜力贡献者长期维护；录制会议视频，剪辑后发布到 B 站、Youtube 做二次传播。

## 五、AI 工具在运营工作里的应用

AI 可以全程辅助运营，非常适合没有技术背景的从业者提升效率：

适用场景：调研新项目背景、撰写各类文案（社群通知、中英文推文、公众号长文、主持稿）、撰写 Proposal、活动策划方案、制作宣传图片、整理会议纪要、活动复盘文档。

工具推荐：TG 可接入 AI Agent（对接 Notion / 飞书）、GPT、Grok（推特内置）、Claude。可以多个大模型交叉对比信息，降低错误概率。

重要提醒：AI 只能负责收集信息、产出初稿，整体策略、思路、最终审核必须由人完成。

## 六、数据统计、渠道转化实操

1.  中小型 Web3 活动不需要重型数据分析平台，轻量化操作即可：定期手动统计社群人数、社媒粉丝、推文曝光量、后台报名数据，使用表格整理存档。
    
2.  渠道来源简易统计方案：在报名表单新增「渠道来源」选项，让用户自主勾选（LXDAO 社群、Twitter 等），粗略统计各个渠道引流效果。
    
3.  如果需要统计多名运营各自引流的转化效果：可以分配独立推广链接、渠道标记区分来源；完整商业化链路追踪工具成本较高，大部分初创项目不会采购。
    

运营重点数据指标（可放进复盘、求职作品集）：活动报名人数、线上会议到场人数、黑客松项目提交数量、新增 Builder 数量、社媒曝光量、成功对接实习与孵化项目数量。

## 现场问答重点

Q：公链运营只吸引 Builder，为什么不去面向普通终端用户？

A：Monad 属于底层公链，现阶段处于基建周期，首要目标吸引开发者搭建各类应用；普通交易者属于生态成熟之后的增量用户。交易所业务才是以散户交易者作为核心用户。

Q：零基础如何找到 Web3 运营相关工作？

A：行业运营岗位需求充足，优先在实习计划中积累实战经验，搭建 Twitter 个人作品集，保持主动；报名黑客松运营组获取完整项目履历。

Q：多名运营推广，如何追踪每个人引流用户后续的转化数据？

A：商业化完整链路追踪系统成本较高，普通团队简易方案是使用专属推广链接、表单渠道选项做区分统计。

猫咪AI智能履约项目：传统AI局限性 + Web3链上核心优势总结

## 一、为什么普通传统AI做不了这套猫咪智能投喂履约系统？

**很多人会疑问：现在主流AI Agent（智能体）可以自主执行任务、串联流程，是不是就能替代Web3架构？答案是：普通AI Agent可以做流程自动化，但依然做不了本项目需要的「可信、不可篡改、强约束、可追责的商业化履约闭环」**。传统AI/通用AI Agent存在四大致命短板，无法适配宠物智能投喂履约场景：

**1\. 传统AI只有判断能力，没有强制风控与健康约束能力**

普通AI只能听懂猫咪叫声、判断想吃东西，但**无法自主限制猫咪乱吃零食、无法识别饮食健康问题**。

传统AI会无脑顺从猫咪意图：猫咪一直要猫条就一直下单，出现过度投喂、损害宠物健康，无法实现项目中的「防沉迷、饮食红线、主食优先」智能管控。

**2\. 传统AI无法做标准化、可量化的意图确权**

本项目核心是：**动作打分、声音打分、互动打分，分值≥75分才判定真实意图**。

纯AI只能模糊判断，**没有可信、统一、可追溯的打分确权机制**，容易误判、乱下单，无法作为下单履约的有效依据。

**3\. 传统AI无法衔接真实商业履约、无法自动合规下单**

普通AI只能输出结论（猫咪想吃东西），**不能对接订单、不能审批、不能触发发货履约**，没有完整商业闭环。

**4\. AI Agent日志可篡改、无公信力、无法法定追责**

市面上所有中心化AI Agent的决策日志、执行记录都是**服务器本地存储、可编辑、可删除、可篡改**的。即便Agent能记录操作，也只是普通系统日志，不具备公证效力、不可否认性。一旦出现误投喂、乱下单、消费纠纷，没有任何可信凭证追责，无法支撑真实商业履约场景。同时AI Agent受模型幻觉、上下文漂移影响，预设规则极易被动态绕过，约束不刚性。

* * *

## 二、接入Web3链上+预言机+合约的核心独有优势（本项目关键亮点）

**Web3/区块链不是多余噱头，是解决AI不可信、无监管、无闭环的核心方案**，具体优势如下：

**1\. 实现「AI行为可信化、可审计、可溯源」**

所有猫咪意图打分、AI判断记录、互动记录、审批记录全部**上链存证**。

解决传统AI黑盒问题：每一次下单都有据可查、可审计、可追溯，真正实现AI行为透明化。

**2\. 四层合约强校验，杜绝AI误判乱下单**

传统AI单点判断极易出错，本项目通过**链上四层合约逐阶校验**，多重过滤误判、无效意图、恶意重复请求，极大提升下单准确性。

**3\. 本地预检层+链上双层风控，实现宠物健康风控体系**

Web3架构支撑了独有的风控逻辑：

-   零食消费上限管控
    
-   防沉迷投喂机制
    
-   早中晚饮食健康红线（零食超标强制替换为主食）
    

纯AI无法固化这种**可强制执行、不可篡改的健康规则**，只有链上合约可以永久锁定规则、不被AI幻觉篡改。

**4\. 节约成本、高效合规（本地预检+链上最终确认）**

全部弱校验、预判断在本地AI层完成，**只有真正合规、有效的订单才上链**，大幅减少链上手续费，解决全链运行高成本问题。

**5\. 人机协同审批闭环，更安全、更可控**

预言机触发手机弹窗人工审批，AI负责识别打分，链上负责校验确权，用户负责最终审批，形成「AI识别+机器校验+人工确权」的三重安全闭环，是传统AI完全不具备的。

**6\. 真正实现「AI自主决策的商业化履约闭环」**

传统AI只有感知能力，Web3赋予AI：**确权能力、风控能力、合规能力、履约能力、存证能力**。

从「听懂猫咪、自动执行」升级为「**可信、合规、健康、不可篡改、可法定追溯的全自动宠物喂养商业系统**」，补齐了AI Agent最核心的信任与合规短板。

## 三、一句话核心总结

**普通AI/AI Agent只能“自动化执行投喂流程”，解决不了信任、规则篡改、合规追责、刚性风控问题；Web3+AI Agent架构可以“管住AI幻觉、固化不可篡改规则、合规确权、自动履约、全程存证可追溯”，是唯一能落地商业化、合规化、可追责的宠物AI投喂项目的技术方案。**

    1）`addTodo()` 写入函数（write，消耗 gas）
    
    功能：输入文字、点击添加，生成新 Todo，需要 gas
    

2）`toggleTodo()` 写入函数（write，消耗 gas）

功能：切换任务完成状态，已完成 / 未完成切换，需要 gas

3）`getTodo()` 读取函数（read，无消耗）

功能：查询任务内容，所有读取类命名不带 gas

## 三、代码编写规范要点

1.  不要使用复杂库，不让 AI 写`import`导入外部代码
    
2.  Hardhat 是用来编译、部署智能合约的工具，但属于电脑本地开发软件，需要安装 Node.js 本地环境
    
3.  添加完整注释，Remix 可直接编译

-   链上交互：Testnet 测试网完成合约部署与交互
    
-   gas 区分：写链操作消耗 gas，读链操作不消耗 gas
    
-   分享会 Day3・AI Agent 如何拥有支付能力
    
    核心内容：区分 4 组核心技术概念（API / MCP / Agent / Skill）
    
    ## 一、各概念定义
    
    ### 1\. API
    
    底层数据通道，是开发者预先编写完成的原始数据接口；
    
    作用：提供数据库原始数据输出通路，仅负责传输数据，无计费、自动执行逻辑。
    
    ### 2\. MCP
    
    AI 专用通用翻译 / 接入标准；
    
    定位：一套统一封装规范，套在 API 外层，统一不同 API 的交互格式，让任意 AI 智能体自动识别工具。
    
    ### 3\. Agent（智能体）
    
    区别于普通对话聊天机器人；
    
    核心能力：自主思考、任务拆解、自动调用工具、循环执行完整任务，是发起工具调用的主体。
    
    ### 4\. Skill（标准化工具箱）
    
    内部包含三层完整能力：
    
    1.  底层 API 调用路径：自动对接后端数据库接口；
        
    2.  执行步骤逻辑：发送请求 → 接收数据 → 整理结果；
        
    3.  商业计费规则：内置单次调用定价、联动支付系统自动扣费。
        
    
    * * *
    
    ## 二、手绘示意图逻辑解读
    
    数据库 ↔ API ← MCP 封装层 → Skill
    
    1.  数据库存储原始业务数据；
        
    2.  API 是数据库对外的数据出入口；
        
    3.  MCP 作为统一标准包装 API；
        
    4.  Skill 包裹 MCP+API，叠加执行流程、计费规则，直接供 Agent 调用；
        
    5.  数据流链路：Agent 发起请求 → Skill 通过 MCP 标准调用底层 API → 获取数据库数据返回 Agent，同步触发扣费。
        
    
    ## 三、层级关系总结（自上而下）
    
    `Agent（执行主体） → Skill（商用工具包） → MCP（统一通信标准） → API（底层数据接口） → 数据库（原始数据存储）`

## 一、关键名词通俗解读（直播内讲解内容）

4.  **Mini Demo**：最小可运行 Web3 演示项目，包含智能合约 + 前端交互，用来验证技术流程，是学习产出核心成果。
    
5.  **RPC 接口**：区块链节点通信地址，钱包、部署脚本依靠 RPC 和链上数据交互。
    

## 二、基础实操 1：加密钱包（MetaMask 为主）

### 1\. 核心基础概念

1.  钱包不绑定单条区块链，可自由切换**L1 主网、L2 二层网络、测试网**
    
    -   L2：Arbitrum、OP、Base，用来解决公链不可能三角扩容；
        
    -   测试网：需手动填入 RPC 接口、链 ID 自定义添加，仅做开发测试，资产无真实价值。
        
2.  钱包安全核心规则（重中之重）
    
    -   私钥、助记词 = 钱包资产所有权，泄露资产会被全部转走；
        
    -   **开发钱包与资产钱包必须隔离**，分开创建，不混用；
        
    -   MetaMask 每个账户独立私钥，软件登录密码仅解锁本地，不保护链上资产。
        
3.  两类钱包工具对比
    
    -   MetaMask：浏览器插件，使用最广泛，测试网需要手动开启；
        
    -   另一款专用开发钱包：自带独立开发者模式，一键只展示所有测试网，适合练习。
        

### 2\. 测试网前置操作

2.  切换对应测试网络，完成转账交易，熟悉链上基础交互流程。

针对学员提出的关于 Gas 费计算及数据存储的疑问，讲师进行了详细的原理拆解

1.Gas 费计算逻辑

数据量与费用正相关:明确 Gas 费与操作的数据量(如留言字数)成正比，字数越多导致数据量越大，所需支付的费用越高。

哈希值存储优化:提出通过将复杂字符转化为固定长度的哈希值(Hash)进行存储，理论上可以实现固定额度的 Gas 费消耗，但需进一步验证。

2.合约托管与资金安全

资金托管机制:针对“甲方是否需提前打款”的问题，讲师指出资金托管并非必须，但为了安全起见，建议将约定金额锁定在合约中。

强制执行优势:对比线下合同，链上合约一旦满足条件(如乙方完成工作)，资金将自动划转，避免了线下履约中可能出现的“赖账“风险。
