## 一、整体核心方案：AI 识别猫咪诉求并自动下单履约整套流程

整套系统依托**向量数据库、大模型 LM、本地预检层、预言机 + 区块链合约**四层逻辑，通过识别猫咪叫声、动作判断猫咪意图，经多轮交互确认后自动发起宠物用品下单，同时叠加健康管控、消费限额、多层校验机制，形成完整闭环。

## 二、AI 识别猫咪意图完整流程

1.  **数据召回与初筛**
    
    大模型调取向量数据库存储的猫咪声音、行为数据，匹配距离猫咪当下状态最相近的 3 条参考样本，交由小模型初步判别猫咪诉求。
    
2.  **多轮交互确认意图**
    
    -   依托 ID 向量数据库解析猫叫对应的潜在需求，再由大模型 LM 复核意图；
        
    -   若大模型无法判定需求，会调取饲养教程、模拟猫叫与猫咪持续互动，反复试探直至明确猫咪真实想法；
        
    -   可通过激光笔等设备引导猫咪行为，辅助验证需求。
        
3.  **多维打分判定阈值**
    
    从猫咪动作、叫声、人机互动三个维度打分，当意图分值达标（示例阈值 75 分），大模型正式确认猫咪需求；多轮沟通锁定具体商品后持续提升执行分。
    

## 三、本地预检层：风控、健康、成本前置管控

1.  **消费上限管控**
    
    设置猫咪零食下单上限，防止猫咪反复请求猫条等零食造成过量消费。
    
2.  **猫咪健康防沉迷 / 饮食红线机制**
    
    LM 模型内置健康判断逻辑，若猫咪早、中午已食用零食，夜间会强制判定需主食猫粮，限制零食重复下单，规避损害猫咪健康的行为。
    
3.  **节约链上成本**
    
    所有校验先在本地预检层完成，预检通过后再封装数据上传预言机，避免每次校验区块链合约产生手续费。
    

## 四、预言机 + 区块链合约履约闭环

1.  **客户端通知审批**
    
    数据推送至预言机后，用户手机弹窗推送猫咪需求（想吃的食物品类），需用户手动审批。
    
2.  **四层合约校验**
    
    数据进入合约后，需依次完成四轮流程校验，全部通过才可进入下一环节。
    
3.  **线下履约收尾**
    
    校验完成后触发发货，商品配送完成，整套 AI 识别、下单、履约流程闭环。
    

## 五、底层技术架构梳理

向量数据库（存储猫声 / 行为特征）→大小模型意图识别 + 多轮交互试探→本地预检层（限额、健康风控、成本前置）→预言机推送用户审批→区块链四层合约校验→线下发货履约。

1.  数据模型
    
    每个地址维护自己的 Todo 列表，地址相互隔离

-   text：任务文字
    
-   completed：任务状态（已完成 / 未完成）
    
-   createdAt：创建该任务时的链上时间
    

3.  三大函数
    
    1）`addTodo()` 写入函数（write，消耗 gas）
    
    功能：输入文字、点击添加，生成新 Todo，需要 gas
    

2）`toggleTodo()` 写入函数（write，消耗 gas）

功能：切换任务完成状态，已完成 / 未完成切换，需要 gas

3）`getTodo()` 读取函数（read，无消耗）

功能：查询任务内容，所有读取类命名不带 gas

    核心内容：区分 4 组核心技术概念（API / MCP / Agent / Skill）
    
    ## 一、各概念定义
    
    ### 1\. API
    
    底层数据通道，是开发者预先编写完成的原始数据接口；
    
    作用：提供数据库原始数据输出通路，仅负责传输数据，无计费、自动执行逻辑。
    
    ### 2\. MCP
    
    AI 专用通用翻译 / 接入标准；
    
    定位：一套统一封装规范，套在 API 外层，统一不同 API 的交互格式，让任意 AI 智能体自动识别工具。
    
    ### 3\. Agent（智能体）
    
    区别于普通对话聊天机器人；
    
    核心能力：自主思考、任务拆解、自动调用工具、循环执行完整任务，是发起工具调用的主体。
    
    ### 4\. Skill（标准化工具箱）
    
    内部包含三层完整能力：
    
    1.  底层 API 调用路径：自动对接后端数据库接口；
        
    2.  执行步骤逻辑：发送请求 → 接收数据 → 整理结果；
        
    3.  商业计费规则：内置单次调用定价、联动支付系统自动扣费。
        
    
    * * *
    
    ## 二、手绘示意图逻辑解读
    
    数据库 ↔ API ← MCP 封装层 → Skill
    
    1.  数据库存储原始业务数据；
        
    2.  API 是数据库对外的数据出入口；
        
    3.  MCP 作为统一标准包装 API；
        
    4.  Skill 包裹 MCP+API，叠加执行流程、计费规则，直接供 Agent 调用；
        
    5.  数据流链路：Agent 发起请求 → Skill 通过 MCP 标准调用底层 API → 获取数据库数据返回 Agent，同步触发扣费。
        
    
    ## 三、层级关系总结（自上而下）
    
    `Agent（执行主体） → Skill（商用工具包） → MCP（统一通信标准） → API（底层数据接口） → 数据库（原始数据存储）`

## 一、关键名词通俗解读（直播内讲解内容）

1.  **ABI**：智能合约编译生成的 JSON 接口文档，告诉前端如何调用合约函数、传参；没有 ABI 前端无法和合约交互。
    
2.  **.env 环境变量**：开发配置文件，存放 RPC、私钥等敏感信息，Web2/Web3 通用配置方案。
    
3.  **AA 账户（抽象账户）**：以太坊协议层升级方案，新一代智能钱包，支持账号找回、简化用户操作，以太坊 7702 协议配套升级。
    
4.  **Mini Demo**：最小可运行 Web3 演示项目，包含智能合约 + 前端交互，用来验证技术流程，是学习产出核心成果。
    
5.  **RPC 接口**：区块链节点通信地址，钱包、部署脚本依靠 RPC 和链上数据交互。
    

## 二、基础实操 1：加密钱包（MetaMask 为主）

### 1\. 核心基础概念

1.  钱包不绑定单条区块链，可自由切换**L1 主网、L2 二层网络、测试网**
    
    -   L2：Arbitrum、OP、Base，用来解决公链不可能三角扩容；
        
    -   测试网：需手动填入 RPC 接口、链 ID 自定义添加，仅做开发测试，资产无真实价值。
        
2.  钱包安全核心规则（重中之重）
    
    -   私钥、助记词 = 钱包资产所有权，泄露资产会被全部转走；
        
    -   **开发钱包与资产钱包必须隔离**，分开创建，不混用；
        
    -   MetaMask 每个账户独立私钥，软件登录密码仅解锁本地，不保护链上资产。
        
3.  两类钱包工具对比
    
    -   MetaMask：浏览器插件，使用最广泛，测试网需要手动开启；
        
    -   另一款专用开发钱包：自带独立开发者模式，一键只展示所有测试网，适合练习。
        

### 2\. 测试网前置操作

1.  水龙头领取测试代币（测试 Gas 费）；
    
2.  切换对应测试网络，完成转账交易，熟悉链上基础交互流程。
    

## 三、核心实操：AI Coding 开发 Mini Demo（完整全链路）

1.  **Remix IDE（新手入门推荐，图形化操作）**

### 完整 Mini Demo 闭环（NFT 铸造案例）

步骤 1：AI 生成智能合约代码

直接向 AI 下达需求（如 “写一套 NFT 发行智能合约”），AI 自动输出完整 Solidity 代码。

步骤 2：配置环境、部署合约上链

1.  新建项目目录，安装 Hardhat/Foundry 依赖；
    
2.  创建`.env`环境变量文件：存放**测试网 RPC 地址、开发钱包私钥**；
    
    -   安全规范：`.gitignore`屏蔽.env，禁止上传至 GitHub，防止私钥泄露；仅上传示例.env 模板；
        
    -   风险提醒：第三方 AI 中转工具可能读取.env 窃取私钥，仅信任官方 AI 工具。

3.  用户点击铸造 Mint，唤起 MetaMask 签名交易，支付测试网 Gas；
    
4.  交易上链后，即可铸造 NFT，完成从前端→合约→区块链完整交互。

一、智能合约开发与交互演示

讲师通过 Remix IDE 演示了从代码编写到链上交互的全过程，重点讲解了智能合约的核心特性与执行逻辑。

1.合约部署与核心功能实现

合约功能演示:演示了留言板合约的部署，包含留言(Write)、读取(Read)及触发事件(Event)三个核心函数，实现了链上数据的写入与查询。

交互流程展示:展示了钱包连接、合约部署、留言发送及交易哈希查询的完整操作链路，验证了合约在测试网上的执行效果。

数据不可篡改性验证:通过区块链浏览器查询交易记录，确认了留言内容、时间戳及钱包地址均被永久记录在区块中，体现了区块链的不可篡改特性。

2.智能合约的本质与执行机制

自动化执行逻辑:将智能合约类比为自动贩卖机，强调其“代码即法律”的特性，一旦部署即不可删除或修改，满足条件即自动执行。

Gas 费消耗机制:解释了每次链上交互(如留言)均需消耗 Gas 费，这是交易被记录上链的必要成本，且测试网与主网的费用存在显著差异。

针对学员提出的关于 Gas 费计算及数据存储的疑问，讲师进行了详细的原理拆解

1.Gas 费计算逻辑

数据量与费用正相关:明确 Gas 费与操作的数据量(如留言字数)成正比，字数越多导致数据量越大，所需支付的费用越高。

哈希值存储优化:提出通过将复杂字符转化为固定长度的哈希值(Hash)进行存储，理论上可以实现固定额度的 Gas 费消耗，但需进一步验证。

2.合约托管与资金安全

资金托管机制:针对“甲方是否需提前打款”的问题，讲师指出资金托管并非必须，但为了安全起见，建议将约定金额锁定在合约中。

强制执行优势:对比线下合同，链上合约一旦满足条件(如乙方完成工作)，资金将自动划转，避免了线下履约中可能出现的“赖账“风险。

三、自由交流与答疑环节
