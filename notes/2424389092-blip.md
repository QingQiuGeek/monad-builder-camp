一、安装插件

1\. 打开Anki主界面，顶部菜单栏点击 工具 → 插件（快捷键Ctrl+Shift+A）

2\. 窗口左下角点击 获取插件

3\. 在输入框粘贴插件代码：[1436550454](tel:1436550454) → 点击【确定】开始下载安装

4\. 安装完成后彻底关闭Anki软件，重新打开Anki（必须重启插件才会加载生效）

二、简单配置音源（推荐微软语音，免费音质最好）

1\. 重启Anki后，再次打开 工具 → 插件，在列表找到 AwesomeTTS，选中点击【配置】

2\. 左侧音源选择：Microsoft Edge TTS（无需额外安装，系统自带，发音标准自然）

3\. 语言选择：English (United States) 美式发音，保存设置

三、两种给卡片加读音的方式

方式A：单张卡片手动添加读音（测试用）

1\. 打开你的单词卡片 → 左下角【编辑】

2\. 编辑界面顶部会出现 AwesomeTTS 按钮

3\. 选中你卡片里的【单词】字段，点击AwesomeTTS → Generate Audio

4\. 软件自动生成音频，自动写入卡片音频字段，卡片出现喇叭图标

方式B：整组卡组批量生成读音（你几百条单词表首选）

1\. 顶部【浏览】，打开你的单词卡组，按Ctrl+A全选所有单词卡片

2\. 顶部菜单栏：工具 → AwesomeTTS → Add Audio to Selected Notes

3\. 弹出窗口：

◦ Source Field：选择你的单词字段（读取哪个文字生成读音）

◦ Destination Field：新建/选择一个空白字段用来存放音频

◦ Voice保持美式英文，语速默认1.0即可

4\. 点击【Generate】，等待进度跑完，所有卡片全部自动嵌入真人读音

四、模板显示喇叭播放按钮

1\. 打开卡片 → 编辑 → 卡片模板

2\. 在正面模板加上你的音频字段名

{{单词}}

{{你的音频字段名}}

3\. 保存模板，复习卡片时就会出现喇叭，点击播放读音，全程离线可用

补充优势

1\. 不用折腾Windows系统语音包、不用写复杂TTS代码

2\. 音频文件储存在Anki媒体库，手机同步Anki也能离线播放声音

3\. 支持单词/例句分开配音，自由切换英美发音

常见小问题

• 安装后没看到按钮：一定关闭Anki全部窗口重新打开

• 批量生成卡住：一次不要一次性选中上千张，分批次处理即可

4.  本次分享核心：不教伪造、包装成 Web3 资深从业者，而是教新人把自身已有经历，用 Web3 行业看得懂的语言完成价值转化

# 01 第一部分・简历的本质

简历核心作用：证明「为什么是你」 保研、考研复试、Web3 求职底层逻辑完全一致，和导师筛选学生、团队筛选实习生评判标准相通，简历本质是一次**价值翻译**，由三层核心要素构成：

1.  真实经历：你过去真实完成的所有事情，是企业判断你的基础素材
    
2.  可迁移能力：经历沉淀下来、能复用在 Web3 工作里的硬软技能
    
3.  成长潜力：自主学习、持续为团队创造价值的可能性

# 02 第二部分・求职常见误区

误区认知：没有 Web3 专属经历 = 没有求职优势

1.  **Web3 行业公认「标配经历」（老玩家素材）**
    

-   参加区块链 Hackathon 黑客松  
    
-   编写智能合约  
    
-   担任 DAO 社区贡献者  
    
-   拥有完整链上交互记录  
    

2.  **零基础新人自有素材（无 Web3 标签）**
    

-   专业课程论文  
    
-   传统行业实习经历  
    
-   学生会 / 社团活动  
    
-   考研、保研复试相关项目经历  
    

3.  核心纠正观点 Web3 行业极度看重 **Proof of Work（贡献证明）** 该标准不考核你入行 Web3 的时间长短，核心验证你是否具备持续产出价值、落地做事的能力。

# 03 第三部分・经历翻译器：Web2 经历转化为 Web3 语言

核心逻辑：不纠结经历是否带 Web3 标签，重点证明你具备快速学习、落地创造 Web3 业务价值的通用能力，四类素材对应转化能力：

### 1\. 课程论文 → Research（行业调研研究能力）

对应可迁移技能：海量资料检索、信息整合梳理、行业问题拆解、独立输出分析结论

### 2\. 学生组织 / 社团经历 → Community Operation（社区运营能力）

对应可迁移技能：线上线下活动策划落地、社群日常维护、内容选题产出、多人团队协调统筹

### 3\. 传统行业实习 → 行业深度理解能力

对应可迁移技能：行业信息搜集、市场数据分析、行业逻辑拆解、专业报告输出

# 04 翻译示例・场景一：金融课程论文

### 普通劣质写法

缺陷

2.  无法体现数据分析、逻辑推导等核心能力

# 06 经历量化工具：STAR 法则，让经历可衡量

STAR 完整框架，完整展现工作价值，避免空泛描述

1.  S（Situation 背景）：所处场景、项目基础条件，例：参与大型线上活动整体筹备
    
2.  T（Task 任务）：个人专属负责板块，例：全权负责社群运营板块
    
3.  A（Action 行动）：落地执行的具体动作，例：独立设计宣传内容、日常社群维护、用户互动策划
    
4.  R（Result 结果）：可量化数据成果，例：新增 XX 名参与者、社群活跃度提升 XX%、内容曝光提升 XX  
    Pasted image 20260721134018.png
    

### 对比效果

改写前（无效）：参与活动组织 → 信息单薄，无法体现个人价值 改写后（有效）：完整覆盖背景 + 任务 + 行动 + 量化结果，清晰展示个人能力与产出

# 07 第四部分・新人简历三大核心原则

1.  扬长避短 不要和资深 Web3 从业者比拼代码、链上项目技术经历；无技术背景则重点突出调研、内容创作、行业分析、社群运营、沟通协作等软性通用能力，打造差异化竞争优势。
    
2.  拒绝编造、硬凑虚假 Web3 经历 过往经历必须真实；空泛的「热爱区块链、了解 Web3」毫无说服力，不如落地产出：持续阅读行业研报、自主输出多篇行业分析文章，真实产出才是有效的 Proof of Work。
    
3.  统一使用 STAR 法则撰写所有经历 每一段经历都需要清晰说明：你的工作职责、落地执行方式、最终量化成果，杜绝笼统、无细节的概括性语句。

# 09 结尾总结核心观点

① 求职不要先纠结「我缺少 Web3 相关经历」，转换思维先梳理「我已经拥有哪些可迁移能力」 黑客松、链上项目、DAO 贡献属于加分项，并非新人入行的唯一硬性门槛；  
② 你的调研能力、内容表达、团队组织、自主学习能力，同样是进入 Web3 行业的核心基石； ③ 持续落地、积累属于自己的 Proof of Work，是零基础新人切入 Web3 赛道的最优第一步。

## 一、GitHub 基础认知

1.  **GitHub 是什么** GitHub 是一个程序员的代码托管平台，你可以把它理解成程序员的「网盘 + 朋友圈」。它支持代码备份、他人下载、多人协作修改，是全球开发者共享和协作的核心平台。
    
2.  **项目页面结构**
    
    1.  顶部：项目名，格式为 `用户名/项目名`。
        
    2.  右上角三个核心按钮：
        
        -   **Watch**：关注项目，有更新会收到通知。  
            
        -   **Fork**：复制项目到自己的账号下，可基于原项目二次开发。  
            
        -   **Star**：点赞 + 收藏，是项目受欢迎程度的核心指标。  
            
    3.  中间标签栏：
        
        -   **Code**：存放所有代码文件。  
            
        -   **Issues**：问题清单，用户反馈 Bug、提需求的地方。  
            
        -   **Pull requests (PR)**：代码修改建议，通过后会合并到主线。  
            
        -   Actions/Projects/Wiki：高级功能，新手可暂时忽略。  
            
    4.  下方：文件树 + `README.md`（项目说明书，至关重要）。

## 二、作为使用者：找项目、下载、运行

### 1\. 找项目的 4 种方法

-   **关键词搜索**：在顶部搜索栏直接搜关键词（如 `AI Chat`），结果质量参差不齐，需自己筛选，Star 数越高越靠谱。  
    
-   **每日热门榜**：进入 GitHub 首页，点击「Explore」-「Trending」，查看全球程序员关注的热门项目，建议每周刷一次。  
    
-   **Topic 页面**：进入「Explore」-「Topics」，按标签（如 Python、Chatbot、LLM）筛选，能看到该领域所有热门项目的清单。  
    
-   **Awesome 系列项目**：搜索 `awesome + 领域名`（如 `awesome Python`），这类项目是社区维护的精选清单，能帮你快速入门一个领域，节省大量时间。  
    

### 2\. 判断项目是否值得用

看 4 个核心信号：

-   **Star 数量**：1000+ 基本靠谱，10000+ 属于顶流项目。  
    
-   **最近 Commit 时间**：超过 1 年没更新的项目要警惕，大概率是「死项目」。  
    
-   **Open Issues 数量**：看问题是否被认真回复，没人维护的项目不要用。  
    
-   **README 是否清晰**：没有说明书、说明模糊的项目，千万不要用。  
    

### 3\. 下载到本地

打开项目页面，点击绿色的「Code」按钮，有 3 种下载方式：

-   **Download ZIP**：直接打包下载，适合一次性查看，拿不到历史记录。  
    
-   **Git Clone**：程序员标准做法，复制项目 URL，在终端输入 `git clone 项目URL`，回车即可下载，能拿到完整历史记录。  
    
-   **SSH 克隆**：更高级的方式，需要先配置 SSH 密钥，新手暂时不用。  
    

### 4\. 运行项目

99% 的项目运行方法都写在 `README.md` 里，找到 `Installation` 或 `Quick Start` 部分，照着命令敲就行。 2026 年新手更高效的方式：直接把项目 URL 丢给 AI（如 Codex、Gemini），让 AI 自动看 README、装依赖、解决报错，比自己啃文档高效 10 倍。

## 三、作为上传者：上传项目、推广、维护

### 1\. 上传前准备

-   注册 GitHub 账号（免费，用邮箱即可）。  
    
-   一个像样的项目至少要有 3 个文件：
    
    -   `README.md`：项目说明书，是项目的门面。  
        
    -   `LICENSE`：开源协议，告诉别人能不能商用、能不能修改，新手选 MIT 协议最宽松。  
        
    -   `.gitignore`：上传规则，指定哪些文件不要传（如密码、密钥、临时缓存）。  
        

### 2\. 上传项目

可以手动上传，也可以直接在 AI 开发工具（如 Codex、Gemini）里，让 AI 自动连接 GitHub 并上传项目。

### 3\. 推广项目，获得 Star

-   **写好 README**：
    
    -   第一句话讲清楚项目是什么、解决什么问题，可放截图或动图，纯文字转化率极低。  
        
    -   写清楚安装和运行步骤，尽量用 5 行代码让用户 2 分钟跑起来。  
        
    -   强烈推荐做中英双语 README，能触达海外用户，海外 Star 含金量更高。  
        
-   **主动传播**：
    
    -   海外渠道：Hacker News、Reddit 对应频道、X（原 Twitter）、Product Hunt。  
        
    -   加入 Awesome 列表：找到对应领域的 Awesome 项目，提交 PR 把自己的项目加进去，是最高效的引流方式。  
        
-   **持续维护**：
    
    -   有 Issues 及时回复，有 PR 及时审核。  
        
    -   每周或每月要有 Commit，保持项目活跃。  
        
    -   活跃的项目会形成正向循环：Star 多 → GitHub 推荐 → 更多人发现 → 更多 Star → 更多人改代码。

## 一、核心项目目录结构解析（通俗白话版）

以本次重点探索的 **moss** 项目为例，拆解每一个文件夹/文件的作用，全程不用懂代码，只看功能定位：

### 1\. 核心文件夹（团队开发核心）

-   **packages 核心代码包文件夹**：项目所有功能的本体，是开发者写的核心功能代码。维护者把功能拆分模块化管理，不同文件夹对应不同能力：
    
    -   @themoss/core：项目底层基础规则、校验逻辑
        
    -   @themoss/simulator：链上交易模拟功能
        
    -   @themoss/erc：代币通用功能模块
        
    -   @themoss/protocol-\*：对接不同区块链协议的专属功能
        
    -   @themoss/mcp-server：对外服务对接模块
        
-   **docs 官方文档文件夹**：对外公开的完整教程、开发指南、规则说明，供用户和新人开发者学习使用
    
-   **examples 示例文件夹**：现成的使用案例，新手可以直接照着复刻，快速上手项目功能
    
-   **.github 项目配置文件夹**：GitHub后台自动运行的配置，用来自动化测试、代码检查、版本发布，普通用户无需关注
    
-   **.changeset 版本记录文件夹**：记录每一次功能更新、bug修复，方便管理项目版本迭代
    
-   **.claude/agents AI辅助开发文件夹**：该项目专属AI开发辅助配置，辅助团队高效写代码、改代码
    

### 2\. 根目录关键文件（项目说明书+规则）

-   [**README.md**](http://README.md) **/** [**README.zh-CN.md**](http://README.zh-CN.md)：项目首页自我介绍（英文+中文），是所有人打开项目第一眼看到的内容，包含项目作用、快速上手命令、支持功能
    
-   **LICENSE**：开源协议，规定所有人可以免费使用、修改、二次开发的规则
    
-   [**CONTRIBUTING.md**](http://CONTRIBUTING.md)：贡献指南，告诉外人如何给项目提代码、修bug、新增功能
    
-   [**SECURITY.md**](http://SECURITY.md)：安全说明，记录项目安全规则、漏洞上报方式
    
-   **CODE\_OF\_**[**CONDUCT.md**](http://CONDUCT.md)：社区行为规范，约束所有参与项目的开发者、讨论者
    
-   **package.json / pnpm 系列文件**：项目依赖配置，记录项目需要的工具、运行环境（新手无需看懂内容）

## 二、GitHub 五大核心模块作用（零基础通俗解读）

所有开源项目通用核心模块，看懂就懂90%开源协作逻辑：

### 1\. README（项目门面）

**定位**：项目极简说明书、入门手册

**核心作用**：不用翻代码，3分钟看懂项目是干嘛的、能解决什么问题、怎么快速使用、支持哪些功能、风险提示

**moss项目README重点**：明确说明项目是「AI代理调用区块链协议的工具」，仅模拟交易、不私自签名转账，标注风险（未审计测试版，禁止用于真实资金），附带一键安装、运行命令

### 2\. Docs（完整官方文档）

**定位**：项目完整版百科全书

**与README区别**：README是精简简介，Docs是全套详细教程、架构设计、规则详解、新手入门、问题排查

**moss Docs包含内容**：快速入门指南、MCP工具规则、协议接入教程、AI安全规则、架构决策记录、专业术语手册

### 3\. Issues（问题反馈区）

**定位**：项目bug反馈、功能提议、问题提问专区

**核心作用**：用户发现bug、想要新功能、使用报错、有疑问，都可以在这里发帖；维护者在这里记录待办任务、分配开发工作

**分类场景**：bug报错、新功能需求、优化建议、技术疑问、开发待办

### 4\. Pull Requests（PR 代码合并请求）

**定位**：代码修改提交、审核、更新专区（开源协作核心）

**核心作用**：任何人修改代码、修复bug、新增功能后，提交PR；项目维护者审核代码，没问题就合并到正式项目，完成版本更新

**简单理解**：相当于「提交修改申请+人工审核+正式更新」的流程，保证项目代码质量

### 5\. Discussions（社区讨论区）

**定位**：自由交流社区

**核心作用**：区别于严肃的Issues（正式问题），这里可以自由讨论技术方案、交流使用心得、提问新手问题、规划项目未来方向，无强制任务属性

## 一、Web3 运营核心定位与目标

1.  **运营本质**
    
    运营相当于生态里的粘合剂，对内协调技术人员、讲师、团队成员；对外对接项目方、开发者 Builder、合作伙伴，兼顾 B 端合作洽谈和 C 端社区维护。
    
    运营一切工作的核心目标就是**增长**，持续完成拉新、提升活跃度、长期留存用户，推动公链或者项目生态持续壮大。
    
    以 Monad 实习计划举例：Co-Learning 学习营、线上分享会、积分排行、黑客松、各类线上活动、社群日常互动、推文宣传，全部都属于运营工作。如果运营缺位，活动参与人数不足，会直接影响公链曝光、实习计划招募、后续黑客松参赛规模。
    
2.  **两类不同运营赛道区分**
    

-   公链 / 基础设施运营（本次分享重点）：主要面向 Builder 开发者。可以把公链类比高速公路，Builder 就是在公路旁搭建商店、设施的建设者；普通链上交易者相当于路上行驶的车辆。公链前期处于基建阶段，优先吸引开发者落地应用，用户交易手续费是生态成熟之后才会产生的收益。
    
-   交易所运营：面向普通散户交易者，重点做用户注册、入金、交易活跃度相关运营。
    

行业现状：Web3 需求量最大两类岗位就是开发者和运营。大部分初创团队运营都是一人多岗，岗位界限不会划分得非常细致。

## 二、一场 Web3 活动（黑客松 / Builder 实习计划）完整 0~1 运营流程

1.  前期 BD 与方案筹备：对接项目方需求，撰写 Proposal 方案；确定目标人群，据此设计活动形式、课程难度、邀约嘉宾、洽谈多方合作赛道；
    
2.  市场宣发 + 活动方案设计：梳理所有公私域流量渠道（LXDAO 推特、TG、微信群等），规划宣传节奏；
    
3.  Builder 招募：多渠道持续触达潜在参与者；
    
4.  社区运营：维护社群，收集参与者疑问，同步各类通知；
    
5.  活动报名、项目征集：搭建报名表单或者活动官网，引导大家报名、自由组队；
    
6.  项目提交、组织评审、同步反馈；
    
7.  活动收尾交付：向合作项目方提交复盘报告，提供核心数据：招募 Builder 数量、组建团队数量、黑客松项目提交数量、生态新增建设者等内容。

猫咪AI智能履约项目：传统AI局限性 + Web3链上核心优势总结

## 一、为什么普通传统AI做不了这套猫咪智能投喂履约系统？

**很多人会疑问：现在主流AI Agent（智能体）可以自主执行任务、串联流程，是不是就能替代Web3架构？答案是：普通AI Agent可以做流程自动化，但依然做不了本项目需要的「可信、不可篡改、强约束、可追责的商业化履约闭环」**。传统AI/通用AI Agent存在四大致命短板，无法适配宠物智能投喂履约场景：

**1\. 传统AI只有判断能力，没有强制风控与健康约束能力**

普通AI只能听懂猫咪叫声、判断想吃东西，但**无法自主限制猫咪乱吃零食、无法识别饮食健康问题**。

传统AI会无脑顺从猫咪意图：猫咪一直要猫条就一直下单，出现过度投喂、损害宠物健康，无法实现项目中的「防沉迷、饮食红线、主食优先」智能管控。

**2\. 传统AI无法做标准化、可量化的意图确权**

本项目核心是：**动作打分、声音打分、互动打分，分值≥75分才判定真实意图**。

纯AI只能模糊判断，**没有可信、统一、可追溯的打分确权机制**，容易误判、乱下单，无法作为下单履约的有效依据。

**3\. 传统AI无法衔接真实商业履约、无法自动合规下单**

普通AI只能输出结论（猫咪想吃东西），**不能对接订单、不能审批、不能触发发货履约**，没有完整商业闭环。

**4\. AI Agent日志可篡改、无公信力、无法法定追责**

市面上所有中心化AI Agent的决策日志、执行记录都是**服务器本地存储、可编辑、可删除、可篡改**的。即便Agent能记录操作，也只是普通系统日志，不具备公证效力、不可否认性。一旦出现误投喂、乱下单、消费纠纷，没有任何可信凭证追责，无法支撑真实商业履约场景。同时AI Agent受模型幻觉、上下文漂移影响，预设规则极易被动态绕过，约束不刚性。

* * *

## 二、接入Web3链上+预言机+合约的核心独有优势（本项目关键亮点）

**Web3/区块链不是多余噱头，是解决AI不可信、无监管、无闭环的核心方案**，具体优势如下：

**1\. 实现「AI行为可信化、可审计、可溯源」**

所有猫咪意图打分、AI判断记录、互动记录、审批记录全部**上链存证**。

解决传统AI黑盒问题：每一次下单都有据可查、可审计、可追溯，真正实现AI行为透明化。

**2\. 四层合约强校验，杜绝AI误判乱下单**

传统AI单点判断极易出错，本项目通过**链上四层合约逐阶校验**，多重过滤误判、无效意图、恶意重复请求，极大提升下单准确性。

**3\. 本地预检层+链上双层风控，实现宠物健康风控体系**

Web3架构支撑了独有的风控逻辑：

-   零食消费上限管控
    
-   防沉迷投喂机制
    
-   早中晚饮食健康红线（零食超标强制替换为主食）
    

纯AI无法固化这种**可强制执行、不可篡改的健康规则**，只有链上合约可以永久锁定规则、不被AI幻觉篡改。

**4\. 节约成本、高效合规（本地预检+链上最终确认）**

全部弱校验、预判断在本地AI层完成，**只有真正合规、有效的订单才上链**，大幅减少链上手续费，解决全链运行高成本问题。

**5\. 人机协同审批闭环，更安全、更可控**

预言机触发手机弹窗人工审批，AI负责识别打分，链上负责校验确权，用户负责最终审批，形成「AI识别+机器校验+人工确权」的三重安全闭环，是传统AI完全不具备的。

**6\. 真正实现「AI自主决策的商业化履约闭环」**

传统AI只有感知能力，Web3赋予AI：**确权能力、风控能力、合规能力、履约能力、存证能力**。

从「听懂猫咪、自动执行」升级为「**可信、合规、健康、不可篡改、可法定追溯的全自动宠物喂养商业系统**」，补齐了AI Agent最核心的信任与合规短板。
