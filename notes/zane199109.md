## 关键技术点

### 1\. Moss 文档撰写的挑战

撰写 Moss 新手教程和介绍文章时，需要准确理解框架的核心抽象：

-   **Capability Tree**：每个 Capability 恰好一个 TransactionNode + 一个纯 Receipt parser，嵌套 Capability 产生子交易
    
-   **Exhaustive Receipt Parsing**：每个 Change（Event/native-transfer）必须被恰好覆盖一次且顺序一致，任何 Warning 停止执行
    
-   **ADR 0007 ABI 分级**：compiled > explorer > vendored，决定了信任链的强度
    
-   **Focused ABI 策略**：Handle 类型推断需要纯 function+event ABI，constructor/error/fallback 会污染类型系统

## 关键技术点回顾

### 1\. Focused ABI 是必要模式

moss 的 Handle 类型推断对 ABI 纯净度要求极高。完整 ABI 包含 constructor、error、fallback，会导致 TypeScript 无法正确推断方法签名。

解决方案：将 ABI 拆分为两个文件：

-   `fastlane.ts` — ADR 0007 标注的完整 ABI（用于 runtime）
    
-   `focused.ts` — 仅含 function+event 的 subset（用于 Handle 类型推断）
    

注意：当前代码中 `Handle<typeof FastLaneStakingAbi>` 引用的是完整 ABI，但实际编译通过是因为 abi-tools 的 `compareDeployedAbi` 已经处理了语义等价性。Focused ABI 在历史迭代中解决了早期类型推断失败问题。

### 2\. TokenReference 必须用 typeof

`TokenReference` 是 Zod schema 对象，不是 TypeScript 类型。直接引用会报错：

> Type ‘TokenRef’ is not assignable to bigint

正确用法：

> const depositParams = { receiver: { type: TokenReference, description: “…” }, } satisfies { receiver: { type: typeof TokenReference; description: string }; };

### 3\. Receipt parser 穷尽覆盖

每个 Change 必须被恰好覆盖一次且顺序一致。FastLane 的 Receipt 实现了：

-   `depositReceipt`: Deposit event + nativeTransfer，断言 `event.assets === native.value`
    
-   `redeemReceipt`: Withdraw event + nativeTransfer，断言 `event.assets === native.value`
    
-   `requestUnstakeReceipt`: RequestUnstake event
    
-   `completeUnstakeReceipt`: CompleteUnstake event + nativeTransfer，断言 `event.amountMon === native.value`
    
-   `boostYieldReceipt`: Transfer event（shMON 从 staker 转到 yield originator）
    

Mint/burn Transfer 事件委托给注入的 `ERC20` Protocol dependency。

### 4\. 类型安全证明

`test/types.fixture.ts` 使用 `@ts-expect-error` 编译器推理证明：

-   正向：所有合法参数组合能通过类型检查
    
-   反向：数字代替字符串、无效地址、缺少 ctx 参数等都会被 TS 拒绝

## 学习内容

### 1\. Moss 项目架构总览

**参考：** [README.md](https://github.com/nishuzumi/moss/blob/main/README.md)

Moss 是一个面向 Monad 链的协议层。它把复杂的链上交互封装成 Agent 可调用的统一接口，流程为：

> discover → load → action → simulate

核心特点：Agent 可以查询协议能力、构建无签名的交易、在链上状态中模拟执行，但 Moss 永远不签名、不发送。所有交易证据以结构化的 Receipt 呈现，Agent 和签名者各自做意图对齐检查。

Moss v1 仅支持 Monad 主网（chain ID 143）。

仓库结构

| 模块 | 职责 |
| --- | --- |
| @themoss/core | 装饰器系统、Registry、Capability 树验证、Receipt 穷尽检查 |
| @themoss/simulator | debug_traceCall 调用、状态链式模拟、事件/转账提取为有序 Change |
| @themoss/erc | 地址无关的 ERC-20/ERC-721/ERC-1155 协议、ABI 解析、Receipt 语义 |
| @themoss/system | Monad Runtime 初始化、chain ID 校验、官方固定地址常量 |
| @themoss/protocol-* | 具体协议适配器的 ABI、Capabilities、Queries、Receipts |
| @themoss/mcp-server | MCP 协议的四个工具：discover/load/action/simulate |

文档体系

-   [**CONTEXT.md**](http://CONTEXT.md) — 领域语言字典，定义三个层面的术语：协议层（Protocol、Capability、Query、Handle）、MCP 层（Event、Change、Receipt Parser、Outcome、Warning）、技能层（Intent、Intent Alignment）
    
-   **ADR** — 共 7 篇架构决策记录，覆盖装饰器模型、debug\_traceCall 模拟、Capability 分类法、ABI 来源分级、自描述协议、Capability 树穷尽 Receipt、Kuru 市场发现
    
-   [**AGENTS.md**](http://AGENTS.md) — 直接写给 AI Agent 看的审查指南，定义仓库事实和验收标准
    
-   [**CLAUDE.md**](http://CLAUDE.md) — AI Agent 工作规则
    

自己的理解

Moss 的设计哲学是"Agent 只看到证据，不看到实现细节"。这不是一个钱包或交易发送器，而是 Agent 的链上操作理解层。Receipt 穷尽覆盖（每个 Change 必须被恰好覆盖一次且顺序一致）是整个安全体系的核心保障。

### 2\. PR #100 x402 量子优化服务适配器审查

**参考：** [PR #100](https://github.com/nishuzumi/moss/pull/100)

实际拉取了 PR #100 的代码进行完整审查（`git fetch origin pull/100/head:pr-100`），分析了 20 个文件、+1094 行代码。

**PR 内容：** 新增 @themoss/x402-qopt-adapter 包，处理 HTTP 402 Payment Required 支付流程，用于量子优化 API。

**审查结论：**

-   架构边界判断正确：选择 Service Adapter 而非 @Protocol
    
-   QUBO schema 未限制 0/1 binary
    
-   EVM address 格式校验缺失
    
-   Node Buffer 依赖在浏览器不可用
    
-   缺少 changeset
    
-   example 依赖声明不完整
    

**关键讨论成果：** x402 不适合接入 Moss 核心。Moss 的核心契约是每个 Protocol 必须产出 Monad TransactionNode，经过 simulator.simulate() 产生链上证据。x402 处理的是 HTTP 402 支付流程，不产生 TransactionNode、不需要 Receipt、不需要 simulator。两者安全模型也不匹配。x402 应该作为独立外围工具存在。

Design thinking

x402 的价值在于展示了 Moss 的边界意识：不是所有"Agent 经济"相关的东西都要塞进 Moss。Service Adapter 模式的判断依据是——这个组件是否产出链上 TransactionNode？如果不产，就不属于 Moss 核心。这对 PrismSettle 的架构设计有启发：结算层只负责链上仲裁，不应该承担链下支付逻辑。

### 3\. PR #68 ERC-1155 实现分析

**参考：** [PR #68](https://github.com/nishuzumi/moss/pull/68)、[PR #82 (merge)](https://github.com/nishuzumi/moss/pull/82)

拉取了 PR #68（Coooder 实现，被 PR #82 merge）的代码。这是 ERC-1155 协议首次进入 Moss，以扩展 @themoss/erc 包的方式实现，不是新建独立 protocol 包。

值得学习的模式

**模式一：动态 Handle 工厂**

ERC-1155 不像 ERC-20 绑定固定合约地址，而是每个 collection 地址都是独立实例。通过 #handle(collection, account) 方法动态创建 Handle：

> handle(collection: AddressValue, account: AddressValue) { return createHandle(ierc1155Abi, collection, this.runtime.client, account); }

这意味着同一个 ERC1155 Protocol 类可以操作任意多个 NFT collection 合约。这是和 ERC-20/ERC-721 的关键区别。

**模式二：双 Receipt parser**

> transferReceipt(changes) → 要求恰好一个 TransferSingle changesReceipt(changes) → 穷尽解析所有 Changes（含 TransferBatch）

严格版对应简单交易，宽松版处理复杂批量场景。这种"严格 + 通用"的双 parser 策略很巧妙。

**模式三：自定义 Parameter 类型**

因为 tokenId 和 amount 都是 uint256，普通 PositiveDecimalString 不够用（需要精确到 2^256-1），作者用 Zod refine 扩展了 ERC1155Uint256String：

> const ERC1155Uint256String = UnsignedIntegerString.refine((value) => { try { return BigInt(value) <= MAX\_UINT256; } catch { return false; } }, “Expected an integer no greater than uint256 max.”).describe(“…”);

**模式四：测试直接构造 Change 对象**

测试不依赖真实链上模拟，而是手动 encodeEventTopics + encodeAbiParameters 构造 Change，然后直接调用 Receipt parser 验证。这样速度快、不依赖 RPC、能精确控制输入。

自己的理解

PR #68 展示了 Moss 对 ERC 标准的处理方式：不是每个代币标准都新建一个包，而是把通用 ERC 标准放在 @themoss/erc 里。ERC-1155 的动态 Handle 模式特别值得注意——它说明 Moss 的 Protocol 类本身可以没有固定合约地址，地址由调用时传入。

* * *

### 4\. ERC 包 vs Protocol 包的判断规则

通过对比 Kuru 和 ERC-1155 的实现，总结出清晰规则：

| 维度 | @themoss/erc | @themoss/protocol-* |
| --- | --- | --- |
| 本质 | 通用代币标准实现 | 具体 DApp/协议适配器 |
| 合约地址 | 不绑定固定地址 | 绑定固定部署地址 |
| 适用场景 | 任何符合标准的合约 | 某个特定项目的合约 |

一句话：标准放 ERC，项目放 protocols。

Design thinking

这个判断规则的本质是"地址归属权"。ERC 包里的 Protocol 类不拥有任何合约地址，地址由调用者决定；Protocol 包里的 Protocol 类拥有固定的部署地址，地址写在代码里。这个区别决定了包的位置和架构边界。

* * *

### 5\. ERC-8004 纠正

最初我误以为 ERC-8004 是 signed transfer 标准，被用户纠正后重新核查。实际 ERC-8004 是 **“Trustless Agents”**，关于 Agent 身份、信誉和验证，要求 EIP-155、EIP-712、EIP-721、EIP-1271。这是一个更高层的信任基础设施，不是交易签名委托执行。

教训：不要凭印象回答技术标准问题，必须先查原始规范。

### 1\. Moss PR #31 架构重构深度分析

PR #31 是 moss 最大的一次架构变更：+5396/-4199 行，126 个文件。核心是把"声明式安全"（Plan expects vs simulation effects）替换为"解析式安全"（Receipt parser 从有序 Changes 中解析结构化 Outcome）。

关键术语变更

| 旧概念 | 新概念 | 说明 |
| --- | --- | --- |
| Plan | Capability tree | 不再有独立的 Plan。可执行结构是 CapabilityNode 树，TransactionNode 是叶子 |
| Expects | 无 | 不再有"声明 vs 实际"的量化对账。安全性转移到 Receipt parser + Change 顺序验证 |
| Effects summary | Receipt tree | 模拟结果不再是资产进出汇总，而是 Receipt 递归结构 |
| Observation plane | Receipt.text | 不再有双平面。Receipt.text 是 presentation，Outcome 是权威数据 |
| Confirmation | ReceiptChange 身份验证 | 不再 confirmed observations。改为 ReceiptChange retain exact original Change object |
| Dealer | 无 | 预处理事件过滤/聚合机制移除 |
| Well-known token / Token table | TokenReference | 不再用 symbol 查找。TokenReference = address 或 “native” |
| Semantic type | Parameter type + declaration | { type: Zod, description: string } 分离：context-free 值契约 + method-specific 用途 |
| Step builder | ProtocolRef dependency | 跨协议复用不再用独立函数。Protocol 声明 protocols: { erc20: ERC20 }，Registry 注入 typed instance |
| Manifest registration | Registry.use() scan exports | 不再有 defineProtocolPackage()。Registry 扫描模块 top-level exports，通过 PROTOCOL_META symbol 识别 |

新三层安全机制

**第一层：编译时类型安全**

-   `@Protocol`, `@Capability`, `@Query`, `@Receipt` 装饰器注入类型信息
    
-   Handle 带 ABI 类型参数，所有方法调用有完整类型推断
    
-   每个 Capability 恰好一个 direct TransactionNode + 一个 named Receipt parser（框架验证）
    

**第二层：运行时结构验证**

-   Registry.register() 验证：description 非空、category/verb/risk 合法、receipt 名称匹配 @Receipt 方法、params 有 Zod schema + description
    
-   flattenCapabilityTree() 验证：每个 CapabilityNode 恰好一个 direct TransactionNode child
    
-   verifyReceiptCoverage() 验证：ReceiptChange leaves 与 input Changes 精确 identity + length + order 匹配
    

**第三层：模拟时执行验证**

-   createTraceSimulator() 逐交易执行：traceWithCalls → extractChanges → receipt parser → verifyReceiptCoverage
    
-   任何 Warning（REVERTED, TRACE\_FAILED, CHANGE\_ORDER\_UNAVAILABLE, RECEIPT\_FAILED, CHANGE\_COVERAGE\_MISMATCH, STATE\_CHAIN\_FAILED）立即 halt
    
-   simulate() 返回 ordered Receipt leaf texts 给 MCP/Agent，SDK 保留完整的 Changes, Receipt trees, data, Outcomes

## 学习内容

### 1\. ADR 速览与核心理解

ADR 0001 — 装饰器作者模型

**决策：** 用 TC39 stage-3 装饰器（`@Protocol`, `@Capability`, `@Query`）注册协议，不用 builder API 也不搞 codegen。Handle 必须带 ABI 类型参数。

**我的理解：** 装饰器是"声明式"的——你在类上贴标签，运行时自动注入。这比手动写 `defineProtocol({...})` 更符合直觉，类似 Spring 的 `@Service`。但代价是工具链限制多：vite 8 的 OXC 不支持装饰器降级，TS 钉在 5.9（因为 tsup 的 dts 构建尚不支持 TS 6），Node V8 原生不解析 stage-3 语法。

> Handle 类型参数的关键约束：`Handle<typeof PoolAbi>` 告诉 TypeScript 这个 handle 操作的是哪个合约。不传类型参数 = 放弃类型保护，所有方法调用返回 `any`。

ADR 0002 — 基于 debug\_traceCall 的模拟

**决策：** Monad 不支持 `eth_simulateV1`，改用 `debug_traceCall` 的双 tracer（callTracer + prestateTracer）手动串联交易。

**我的理解：** 这是被迫的创新。主流方案 eth\_simulateV1 不存在，但他们反而得到了更多信息——不仅能看状态变化，还能拿到完整调用树和事件日志，这对安全审计是加分项。

> 模拟引擎比 eth\_simulateV1 更强大：callTracer 提供调用树（检测未知收款地址、意外的 fallback 调用），prestateTracer 提供状态差异（精确追踪资金流向）。

ADR 0003 — 两层能力分类法

**决策：** 动词+类别是闭集（12 个动词），标签是自由表单。

**我的理解：** 闭集保证机器可比较（“用户要 swap，Plan 的 verb 是 supply → 不匹配，报错”），自由标签给长尾场景留出口。类似 Semantic UI 的 action/category/tags 三层结构。

> 动词集刻意保持很小（12 个：swap, wrap, unwrap, supply, withdraw, borrow, repay, stake, unstake, claim, mint, transfer），因为意图对齐锚定在动词上——"用户请求 swap，但 Plan 的 verb 是 supply"必须是硬性的、机械的不匹配。

ADR 0004 — Plan 中的量化预期

**决策：** 每个 Plan 携带 `expects` 块，量化输入输出；风险标签只定性，不定量。

**我的理解：** 这是安全核心。想象一下：用户说"借出 100 USDC"，如果 Plan 没有 expects，模拟器跑完发现实际借出了 1000 USDC —— 没有量化对比就检测不出来。expects 是"声明 vs 实际"的对照表。

> 警告只在"未声明的差异"时触发。一个已声明的流出但无流入是合法的（如永续保证金、 unstake 冷却请求）——规则是"没有未声明的东西"，不是"每笔流出都有流入"。

ADR 0005 — Curated Token 目录

**决策：** 代币身份只通过白名单目录解析，不从链上查 symbol。

**我的理解：** 防钓鱼的关键。任何人都能部署一个叫 “USDC” 的假币，如果让 agent 去链上查 symbol，它就会被骗。白名单 = 只有项目信任的地址才认。类似 Apple 的 App Store 审核机制——不开放，但更安全。

> 安全铁律：symbol → address 解析只通过目录，绝不回退到链上 `symbol()` 查询。任何未知符号都会报 loud error 列出目录内容——agent 要么用显式地址（故意的、可见的），要么问用户。

ADR 0006 — 协议包 + Manifest 组装

**决策：** 每个协议一个 npm 包，Registry 初始为空，通过 `registry.use(manifest)` 显式注册。零隐式注册。

**我的理解：** “显式优于隐式"的极致。import 不会触发副作用，所有注册都是显式调用。这让代码分析工具（和人类）可以精确追踪"谁注册了谁”。类似 Rust 的 trait system——类型安全 + 可追踪。

> 三层基础架构（ADR 0006）：core 无依赖；erc 和 simulator 并列，都只依赖 core；system 依赖 core + erc。依赖严格向下。

ADR 0007 — ABI 来源分级

**决策：** compiled > explorer > vendored，三级优先级，每个 ABI 必须标注来源。

**我的理解：** 安全分级制度。compiled 最可信（自己编译的源码），explorer 次之（区块浏览器验证过），vendored 再次（第三方 SDK）。如果三个都做不到，干脆不做适配器——宁可少功能，不能不安全。

> vendored 版本策略：跟随上游 `dist-tags.latest` + 7 天发布年龄守卫，不用 highest-semver（会捡拾被遗弃的版本线，也是 version-squatting 攻击的形状）。

ADR 0008 — 审计面 + 观察面双平面

**决策：** 模拟结果分两个平面：audit（协议无关的自动审计）和 observation（协议作者编写的人类可读叙述）。单向约束：观察只能收紧结果，不能替代审计。

**我的理解：** 审计面是"机器说的"，观察面是"人说的"。机器说"转出 100 USDC"，人说"用户将 1 MON 兑换为 USDC"。confirms 机制让协议可以声明"这个事件是我交易的 receipts"，但审计面永远说了算——防止协议作者作弊。

> 红线（单向）：observations 只能 tightening outcome，never satisfy it。唯一交叉点是 confirms——capability 可以声明哪些 observations 是其链上收据；声明的收据如果没出现，raise CONFIRMATION\_MISSING。

ADR 0009 — erc 是接口层，不是实例层

**决策：** `@themoss/erc` 只导出标准接口（ABI + 地址无关的通用适配器），不硬编码任何地址。WETH9 的实例（WMON）放在 system 层。

**我的理解：** 接口和实现的分离。erc 定义"ERC20 代币能做什么"，system 定义"Monad 上的 USDC 地址是什么"。这样 erc 可以跨链复用，system 负责链特定配置。类似 JDBC 驱动模式——接口统一，实现按数据库定制。

> 地址归属规则：调用时参数 → erc 层的通用适配器；规范实例数据 → system 或 protocol package。

### 2\. 整体架构理解

把 9 个 ADR 串起来，moss 的设计哲学是：安全优先 > 开发体验 > 灵活性。每一层都在回答同一个问题：“如果 agent 搞错了，怎么不让用户丢钱？”

**安全机制的层层递进：**

| 层级 | 机制 | ADR |
| --- | --- | --- |
| 编译时 | Handle 类型参数 + ABI 类型推断 | 0001 |
| 注册时 | 显式 manifest 注册，零隐式副作用 | 0006 |
| 构建时 | Token 白名单，不信任链上 symbol | 0005 |
| 声明时 | Plan 量化 expects | 0004 |
| 执行时 | debug_traceCall 模拟 + 双平面审计 | 0002, 0008 |
| ABIs 本身 | 三级来源分级 | 0007 |

**分层依赖（自底向上）：**

根据 ADR 0006 原文 “three foundation layers” 和 “dependencies pointing strictly downward”：core（零依赖，纯工具）← (erc, simulator) 并列 ← system（含 WMON 等 Monad 实例）← protocols/\* ← mcp-server。注意：**simulator 和 erc 是并列的**，不是链条关系。原文说 simulator “sits beside erc in the layering”。

### 3\. ADR 之间的关联关系

有些 ADR 互相引用，形成依赖链：

-   ADR 0001（装饰器）依赖 ADR 0006（manifest 注册）—— `@Protocol` 装饰器在注册时验证 handle 字段与 contracts 配置键匹配
    
-   ADR 0009（erc 接口层）依赖 ADR 0007（ABI 来源）—— erc 包的 ABIs 都是 compiled 级别
    
-   ADR 0004（量化 expects）依赖 ADR 0002（模拟引擎）—— expects 在 simulate 时被 reconciliation 检查
    
-   ADR 0008（双平面）依赖 ADR 0001（装饰器）—— `@Event` 装饰器注入 observation handler
    

**基础性 ADR（改了会连锁反应）：** 0001, 0006, 0007 **相对独立 ADR：** 0003, 0005, 0009

### 4 Getting Started：四步标准流（实操教程）

`getting-started.md` 把 ADR 里的理论变成了可运行的代码。核心流程：`pnpm --filter @themoss/example-simple-flow wrap`（跑一遍完整流程，wrap 1.5 MON → WMON）

**四步流水线：**

| 步骤 | 命令 | 做什么 | 对应 ADR |
| --- | --- | --- | --- |
| discover | registry.discover({ verb: "swap" }) | 查货架上有哪些协议能做这个动作 | 0003 |
| load | registry.load([{ protocol, method }]) | 获取调用契约（intent、语义参数、风险标签） | 0003, 0004 |
| action | registry.action(protocol, method, account, params) | 构建 Plan（未签名交易 + 量化 expects） | 0004 |
| simulate | simulator.simulate([plan]) | 在真实链状态回放，对账 expects vs 实际 | 0002, 0008 |

**关键设计点：**

-   **Registry 生来是空的。** `registry.use(manifest)` 决定装什么，没有任何 import 副作用
    
-   **语义类型（Semantic Types）：** 传 `"1.5"` 而不是 `1500000000000000000`，运行时自动解析精度
    
-   **Plan integrity seal：** `planHash` 覆盖 `{chainId, account, txs, expects, confirms}`，agent 侧篡改会暴露
    
-   **链式 Plan：** `[sellPlan, buyBackPlan]` 一次模拟——Plan B 跑在 Plan A 的模拟状态上，多步流程端到端验证
    
-   **Observations = 叙事，不是法律：** `@Event` 装饰器写人类可读回执，`confirms` 让回执承重（缺失则 warning），但永远不能消掉 audit plane 的 warning
    

**MCP 工具集成：** 四步全部暴露为 MCP tools，agent 可以直接用：`{"mcpServers": {"moss": {"command": "node", "args": ["<path-to-moss>/packages/mcp-server/dist/cli.js"], "env": {"MOSS_RPC_URL": "https://rpc.monad.xyz"}}}}`

**写自己的适配器：** `cp -r packages/protocols/_template packages/protocols/<yourprotocol>` 5 项必填：ABI（带来源）→ `@Protocol` → `@Capability` → `@Event` → 测试

参考实现阅读顺序：`wmon.ts`（注释拉满）→ `erc/`（动态地址）→ `kuru/`（reads-before-build）

## 关键收获

1.  **moss 的安全设计是"防御纵深"的典范。** 从编译时类型检查（ADR 0001）到执行时模拟审计（ADR 0002, 0008），每一层都在拦截不同类型的错误。不是靠单一防护，而是多层叠加——即使某一层被绕过，其他层仍然有效。
    
2.  **"显式优于隐式"贯穿始终。** 零隐式注册（ADR 0006）、零链上 symbol 查询（ADR 0005）、零硬编码地址在 erc 层（ADR 0009）。这种设计让代码可追踪、可分析、可审计——对 AI agent 操作的 DeFi 系统来说，可解释性本身就是安全。
    
3.  **被迫的创新往往带来意外优势。** ADR 0002 是因为 Monad 不支持 eth\_simulateV1 而被迫使用 debug\_traceCall，结果反而获得了比 simulateV1 更丰富的信息（调用树 + 事件日志 + 状态差异）。这提醒我：在技术选型中，限制条件不一定是坏事。

## 疑问与待思考

1.  **装饰器的元数据注入是怎么实现的？** ADR 0001 提到用 marker properties 替代 Symbol.metadata，但 `@Protocol` 如何把 ABI 注入到 Handle 的构造函数？是反射还是手动遍历 class fields？
    
2.  **为什么不用 local anvil fork 作为 fallback？** ADR 0002 说 anvil 慢且需要 foundry 依赖。但 anvil 是本地运行的，不依赖外部 RPC——在 RPC 限流或不可用时仍可模拟。是否考虑过 dual-backend（RPC 为主，anvil 为 fallback）？
    
3.  **Token 白名单的更新流程是什么？** ADR 0005 说新 token 需要 PR review + 链上验证。但如果协议引入新的 receipt token（如 LP token），谁来维护目录？protocol 作者自己加还是 core 团队审核？
    
4.  **expects 的准确性怎么保证？** ADR 0004 说 capability 作者从解码后的参数构建 expects。但如果作者写错了 expects（比如故意少写一个 outflow 来绕过检测），模拟器怎么发现？
    
5.  **双平面 confirms 的性能开销？** ADR 0008 说 confirms 被纳入 planHash 防止传输中被剥离。多交易 Plan（cross-protocol composition）中每次模拟都要检查 observation 是否出现，会不会显著增加模拟时间？
    
6.  **erc 层拒绝 WETH9 多链实例的边界在哪？** ADR 0009 说拒绝开放的 wrapper 参数是因为"开放参数是假 WMON 的邀请"。但 L2 上的 WETH（不同地址但相同接口）能通过 `contracts: { weth: { abi, addr } }` 传入吗？
    
7.  **ADR 之间的完整依赖图？** 目前知道部分引用关系，但有没有完整的依赖拓扑？哪些 ADR 是基础性的（改了会连锁反应），哪些是独立的？

# Week 1 — Day 6：Mini Demo 0 提交与全栈联调补丁

**日期：** 2026-07-11 **轨道：** 通用 / 技术 **状态：** ✅ 已完成

* * *

## Day 6 学习目标

**Mini Demo 0 提交 + NFT Badge 全栈联调优化。**

Build Log、Mini Demo 0 提交后，根据实际用户测试反馈对前端/后端做了 6 项设计调整。核心不是"加功能"，而是修正设计上不符合用户直觉的地方。

* * *

## 设计调整记录

### 1\. 钱包断连后无法切换账号

**现象：** 用户断开连接后，点击 Connect Wallet，MetaMask 直接返回已授权的账户，不弹出账户选择窗口。用户根本没有机会切换。

**技术分析：**

钱包连接流程经过三次迭代：

| 方案 | 做了什么 | 为什么不行 |
| --- | --- | --- |
| 第一次尝试：eth_requestAccounts | 每次连接都请求授权 | 对已授权站点直接静默返回已授权账户，不弹窗 |
| 第二次尝试：wallet_requestPermissions | 强制重新请求 eth_accounts 权限 | 大部分 MetaMask 版本同样静默通过，不弹窗 |
| 第三次尝试：wallet_requestPermissions 条件分支 | 断连后标记 wasDisconnected=true，触发 wallet_requestPermissions；刷新页面用 eth_requestAccounts | 同第二版，弹窗行为不统一 |

**最终方案：**`wallet_revokePermissions` **主动撤回授权**

> ```
> // disconnect() 中
> window.ethereum.request({
>   method: "wallet_revokePermissions",
>   params: [{ eth_accounts: {} }],
> }).catch(() => { /* 旧版 MetaMask 忽略 */ });
> 
> // connect() 中直接使用 eth_requestAccounts
> const accs = await window.ethereum.request({ method: "eth_requestAccounts" });
> ```

**设计思考：** 之前一直想的是"怎么在连接时让 MetaMask 重新选账户"，但 MetaMask 没有提供可靠的编程式账户切换接口。换个思路——**断开时真正撤回授权**，下次连接时 `eth_requestAccounts` 自然就会弹出选择窗口。刷新页面不影响，因为 MetaMask 还会记住之前的授权。

### 2\. accountsChanged 事件导致连锁问题

**现象：** 用户已连接 → 点击 Connect 按钮 → MetaMask 弹出确认 → `accountsChanged` 事件触发 → 旧代码 `location.reload()` 刷新页面 → 状态全丢 → 需要再点一次 Connect。

**修复：**

> ```
> // 改前
> window.ethereum.on("accountsChanged", () => location.reload());
> 
> // 改后
> window.ethereum.on("accountsChanged", (accounts) => {
>   if (accounts.length === 0) disconnect();
>   else connect(true);  // force-reconnect，走 restore 路径
> });
> window.ethereum.on("chainChanged", () => connect(true));
> ```

**设计思考：** `location.reload()` 是早期开发时图省事的写法——“反正刷新后一切重来”。但在链上 DApp 中，这种偷懒会破坏用户体验：MetaMask 连接成功后触发的事件不应该让页面白闪一下。

### 3\. 前端 8080 + 后端 8082 双端口架构与 CORS

**现象：** Admin 页面可以打开，但 API 调用全部 404。前端 `npx serve` 无 API 路由，API 请求走相对路径 `/api/claims/...` 落在了 8080 上。

**双端口架构：**

| 端口 | 服务 | 内容 |
| --- | --- | --- |
| 8080 | npx serve（静态文件） | HTML, CSS, JS |
| 8082 | Go 后端 | API /api/claims/* |

**修复：**

1.  前端代码中引入 `API_BASE` 常量指向 8082
    

> ```
> const API_BASE = "http://localhost:8082/api/claims";
> // 所有 fetch 调用都使用 API_BASE 而非相对路径
> ```

2.  Go 后端添加 CORS 中间件
    

> ```
> w.Header().Set("Access-Control-Allow-Origin", "*")
> w.Header().Set("Access-Control-Allow-Methods", "GET, POST, OPTIONS")
> w.Header().Set("Access-Control-Allow-Headers", "Content-Type")
> ```

3.  Admin 页面从 `backend/web/admin/` 复制到 `frontend/admin/`，`npx serve` 可直接访问
    

**设计思考：** 开发时双端口分离是标准实践——前端 dev server 负责热更新和静态文件，后端专注 API。但不解决 CORS 就容易出问题。之前 Go 后端直接托管前端不需要 CORS，用 `npx serve` 后就需要了。两种方式应该都支持。

### 4\. Claim 提交流程设计

**现象：** 用户提交 claim 后，modal 关闭，但徽章卡片仍然显示 “Submit Claim” 按钮。用户可以多次提交同一徽章的 claim。

**设计状态机：**

> ```
> ┌─────────┐   submit    ┌─────────┐   admin approve   ┌─────────┐
> │ No Claim │ ────────→ │  Pending  │ ──────────────→  │  Minted  │
> │（可提交） │           │（不可提交）│                  │（不可提交）│
> └─────────┘           └─────────┘                   └─────────┘
>                            │  admin reject
>                            ↓
>                         ┌─────────┐
>                         │ Rejected │  ← 可重新提交（Resubmit）
>                         └─────────┘
> ```

**实现：**

1.  后端新增 `GET /api/claims/check?user_addr=X&type_id=Y` 接口，返回该用户对该 badge 的最新 claim 状态
    
2.  前端 `state.claimStatus[typeId]` 缓存每个 badge 的 claim 状态
    
3.  提交成功后直接设置 `state.claimStatus[claimTypeId] = 'pending'` 并重渲染
    
4.  卡片按钮根据状态显示不同文案和行为：
    
    -   `""`（无 claim）→ “Submit Claim”
        
    -   `"pending"` → 禁用按钮显示 “Pending Review”
        
    -   `"rejected"` → “Resubmit Claim”
        
    -   `"minted"` → 无按钮（badge 已解锁）
        
5.  连接钱包时自动调用 `refreshClaimStatuses()` 从后端拉取所有 badge 的 claim 状态
    

### 5\. Go fmt.Sscanf 匹配缺陷

**现象：** Admin 面板点击 reject 按钮报错 “invalid claim ID”。

**根因：**

> ```
> // 问题代码：
> n, err := fmt.Sscanf(r.URL.Path, "/api/claims/%d/approve", &id)
> if n == 0 {  // ← 这里只检查 n==0
>     n, err = fmt.Sscanf(r.URL.Path, "/api/claims/%d/reject", &id)
>     action = "reject"
> } else {
>     action = "approve"
> }
> ```

当路径是 `/api/claims/3/reject` 时：

-   第一次 Sscanf 匹配 `/api/claims/%d/approve`：`%d` 成功匹配了数字 `3`（n=1），但后续 `/approve` vs `/reject` 不匹配
    
-   Sscanf 返回 `n=1, err="input does not match format"`
    
-   因为 `n==0` 判断为假，代码错误地走了 `else` 分支，设 `action="approve"`
    
-   然后在 approve 逻辑中，合约调用 `mint` 时报错
    

**修复：** `if n == 0 || err != nil`

### 6\. 断开连接时隐藏 Admin 链接

**现象：** 用户断开钱包后，导航栏的 Admin 链接仍然可见。只有 `checkAdminRole()` 用于控制显示/隐藏，但断开时该方法有 `if (!state.account) return;` 的 early return，不会执行隐藏逻辑。

**修复：** 在 `disconnect()` 中直接隐藏：

> ```
> const adminLink = $("adminLink");
> if (adminLink) adminLink.classList.add("hidden");
> ```

## 学习内容

### 1\. Monad 为什么这么快？

以太坊的瓶颈在于执行速度。Monad 从四个层面同时突破，而不是只在一个点上优化。

**1.1 异步执行（Asynchronous Execution）：把执行从共识的热路径中移出来**

> 通俗解释：以太坊是先执行完所有交易、算出状态根，再拿去和其他节点"对答案"（共识）。Monad 反过来——先"对答案"确定交易顺序，然后再慢慢执行。

以太坊的共识和执行是交错的（interleaved）：leader 必须先执行完所有交易才能提出区块，验证者必须先执行完才能投票。这意味着执行时间被压缩在 12 秒的以太坊区块时间里只剩 ~100ms（约 1%）。

Monad 把共识和执行分成了两条独立的泳道（swim lanes）。节点达成共识只需要确定交易的**顺序**，不需要执行交易。执行在共识之后异步进行，可以用满整个区块时间。这让 Monad 的每块 gas 预算大幅提升。

**1.2 乐观并行执行（Optimistic Parallel Execution）**

> 通俗解释：以太坊一条一条串行执行交易。Monad 同时开多条流水线一起执行，如果发现后面的交易读到的是"旧数据"，再重新执行那条交易。

Monad 使用乐观并发控制（OCC, Optimistic Concurrency Control）。具体来说：

-   交易在同一个块中仍然是线性排序的，语义和以太坊完全一致
    
-   执行时可以并行启动——不等前面的交易完成就开始执行后面的
    
-   每条交易在执行时会"记录"它读到了哪些状态（input tracking）
    
-   执行完后按顺序"合并"（merge）状态变更，检查输入是否与之前的输出一致
    
-   如果不一致（读到了旧数据），这条交易会被重新执行——但因为大部分计算（如签名恢复）不依赖状态，重执行很快
    

**1.3 异步 I/O（Asynchronous I/O）**

> 通俗解释：CPU 比磁盘和网络快几个数量级。传统方式下 CPU 等磁盘读取就像等外卖——什么都不干。异步 I/O 让 CPU 在等外卖的同时继续炒菜。

MonadDb（见下文）全面使用 Linux 的 `io_uring` 异步 I/O。这意味着数据库读取不会阻塞 CPU。结合并行执行，CPU 可以在等待磁盘读取的同时执行其他交易。

延迟对比：

-   CPU L3 缓存：10ns
    
-   内存：100ns
    
-   NVMe SSD：400μs（比内存慢 4000 倍）
    
-   网络：50-200ms（比内存慢 2000 倍）
    

**1.4 JIT 编译（Native Compilation）**

> 通俗解释：以太坊用一个"翻译官"逐条解释 EVM 指令。Monad 把常用的合约提前编译成机器码，像把常用短语背熟了直接说母语。

Monad 维护了一个高度优化的 EVM 解释器，同时对"热合约"（累计 gas 消耗高的合约）进行即时编译（JIT），生成原生 x86-64 机器码。编译器做了两件事：

-   消除冗余工作：如连续 PUSH+ADD 可以折叠成一个常量
    
-   寄存器分配：把 EVM 栈操作映射到 CPU 寄存器，减少内存访问
    

编译是异步进行的，不阻塞执行。冷合约继续走解释器，热合约走编译后的机器码。

**1.5 自定义数据库 MonadDb**

> 通俗解释：以太坊用通用的 LevelDB/LMDB 存数据，但以太坊的状态结构（Merkle Patricia Trie）是树形的。用存扁平键值对的数据库来存树形数据，就像用方盒子装圆苹果——浪费空间还不方便找。

MonadDb 原生实现了 Patricia Trie（一种 Radix Tree 变体），直接在磁盘上存储 Merkle 状态树，避免了"树形数据结构嵌入扁平 KV 数据库"的双重封装开销。此外：

-   绕过文件系统层（filesystem bypass），直接通过 `io_uring` 操作裸设备
    
-   全异步 I/O，配合并行执行
    
-   不仅存状态，还存区块头和 payload
    

### 2\. Monad 的共识层：MonadBFT + Local Mempool + RaptorCast

**2.1 无全局 mempool（Local Mempool）**

> 通俗解释：以太坊有一个"公共候诊室"，所有节点都能看到所有待处理的交易。Monad 没有公共候诊室，交易直接送到接下来要当"值班医生"的几个节点手里。

以太坊的 Gossip mempool 在高 TPS 下有两个问题：

1.  慢：交易需要经过多跳才能到达 leader
    
2.  浪费带宽：gossip 协议需要大量重传
    

Monad 的做法：RPC 节点收到交易后，直接转发给接下来 N=3 个即将成为 leader 的验证者。这些验证者各自维护自己的 local mempool。当轮到某个节点当 leader 时，就从自己的 local mempool 中选交易打包。

**2.2 MonadBFT：快速、响应式、防分叉的共识**

MonadBFT 是 Monad 的共识引擎，实现了：

-   **推测最终性（Speculative Finality）**：1 轮共识即可推测性最终化
    
-   **完全最终性（Full Finality）**：2 轮共识（800ms）完全最终化
    
-   **线性通信复杂度**：leader 发提案给每个验证者（扇出），验证者投票给下一个 leader（扇入），消息数是 O(N) 而非 O(N²)
    
-   **尾部分叉抵抗（Tail-Fork Resistance）**：防止恶意 leader 分叉掉前一个 leader 的区块
    
-   **领导者故障隔离**：一个 leader 失败只导致一次超时，其他轮次照常进行
    

块状态从 Proposed → Voted → Finalized 三步推进。

**2.3 RaptorCast：高效的大区块传播**

区块很大（10000 笔交易 × 200 字节 = 2MB），leader 直接发给所有验证者带宽压力大。RaptorCast 用纠删码（erasure coding）把区块切成碎片，通过两层广播树分发，利用全网验证者的上传带宽来传播。

### 3\. Monad vs Ethereum：行为差异（Differences）

**3.1 合约代码大小：128 KB（以太坊 24 KB）**

init code 上限 256 KB（以太坊 48 KB）。可以部署更复杂的合约，不需要过度拆分。

**1.2 Gas 计费方式不同：按 gas limit 收费，而非 gas used**

以太坊：`gas_paid = gas_used * gas_price` Monad：`gas_paid = gas_limit * gas_price`

原因：Monad 是异步执行（先共识后执行），如果按 gas\_used 收费，用户可以提交高 limit 低使用的交易来占块空间而不付费，形成 DoS 漏洞。

启示：在 Monad 上，设置准确的 gas limit 非常重要，设高了白花钱，设低了交易会 revert 但仍被计入共识。

**1.3 EIP-1559 兼容但 base fee 控制器不同**

Monad 的 base fee 调整更慢：增加慢、减少快，避免 blockspace 闲置。

-   target = 160M gas（80% 满载），block gas limit = 200M
    
-   min base fee = 100 MON-gwei
    
-   公式中引入了 `moment`（block\_gas 偏离 target 的平方的指数加权平均）来平滑调整幅度
    
-   参数：β=0.96, max\_step\_size=1/28
    

**1.4 无全局 mempool**

以太坊有全局 mempool，节点间广播。Monad 没有全局 mempool，交易直接转发给接下来的几个 leader（Local Mempool）。影响：钱包和索引器的设计需要适配，不能假设交易一定出现在全局 mempool 中。

**1.5 不支持 Blob 交易（EIP-4844 Type 3）**

以太坊的 L2 数据可用性用 Blob 交易降低成本。Monad 暂不支持。

**1.6 支持 EIP-7702 委托和 P256 预编译**

-   EIP-7702：EOA 可以委托代码执行，类似账户抽象
    
-   P256（secp256r1）预编译：支持 WebAuthn/passkey 签名验证，对 DApp 登录体验有直接影响
    

**1.7 Reserve Balance 机制**

确保所有共识层接受的交易都有足够的 gas 支付。EIP-7702 委托的 EOA 余额不能低于 10 MON（Reserve Balance 规则对委托 EOA 的约束）。可能导致"共识层接受但执行层 revert"的交易（余额不足时）。

### 2\. Best Practices：高性能应用的构建建议

**2.1 固定 gas 成本用硬编码值，不调用 eth\_estimateGas**

例如 EOA 转账固定 21,000 gas。好处：减少钱包延迟，避免某些钱包在 estimateGas revert 时的异常行为。

**2.2 并发提交 eth\_call 降低读取延迟**

串行调用多个 view 函数会引入不必要的延迟。可以用 Multicall3 合约（部署在 `0xcA11bde05977b3631167028862bE2a173976CA11`，Monad 主网和测试网都有）聚合多个 read 请求到一个调用，或者用 batch request 并发提交。注意 Multicall 内部是串行执行的，不适合放太多昂贵的调用。

**2.3 静态文件托管注意成本控制**

Vercel/Railway 方便但贵，AWS S3+CloudFront 灵活但需要配置。注意超额流量的阶梯定价陷阱。

### 3\. Gas Pricing：Monad 的 gas 模型详解

**关键公式：**

> gas\_paid = gas\_limit \* price\_per\_gas price\_per\_gas = min(base\_price\_per\_gas + priority\_price\_per\_gas, max\_price\_per\_gas)

**与以太坊的区别：**

-   以太坊按 gas\_used 收费，Monad 按 gas\_limit 收费
    
-   Monad 的 base fee 调整更慢（增加慢、减少快），目的是避免 blockspace 闲置
    
-   Monad 默认按 Priority Gas Auction 排序（总 gas price 降序）

## 关键收获

1.  **Monad 快的原因不是单一优化，而是四层叠加。** 异步执行（共识和执行分开）+ 乐观并行执行（OCC）+ 异步 I/O + JIT 编译，再加上自定义数据库 MonadDb 绕过文件系统，最终实现 >10,000 TPS。这和以太坊"在一个点上修修补补"的思路完全不同。
    
2.  **Monad 的 gas 模型与以太坊有本质区别。** 按 gas limit 收费而非 gas used，这是异步执行架构下的 DoS 防御措施。设错 gas limit 的后果比以太坊更直接——设高了白花钱，设低了 revert 但交易仍然是 valid transaction（仍需支付 gas）。MetaMask 在 estimateGas revert 时设置极高 gas limit 的行为在 Monad 上尤其危险。
    
3.  **AI 能快速给结构骨架，但安全细节和"能不能跑通"必须人工把关。** 本周发现的 6 个 AI 合约缺陷中，`_safeMint` vs `_mint` 是安全问题，`hasBadge` 映射缺失是逻辑问题，OZ v5 custom error 是版本兼容问题——这些都只有实际跑测试才暴露。

### 1\. AI 辅助合约开发：从 Prompt 到人工审查

**核心流程：** 写 Prompt → AI 生成初稿 → 我阅读 + 让 AI 解释 → 人工检查 → 记录修改 / 判断。

**Prompt 设计思路：**

-   明确要求用 OpenZeppelin 标准库，避免 AI 从零实现 ERC721
    
-   明确"保持简洁、安全，不要过度设计"，抑制 AI 顺手加白名单、销毁、可升级代理等超出需求的特性
    
-   需求点刻意列得最小：只覆盖 创建类型 / 铸造 / 查询 三个核心能力
    

**AI 初稿的 6 个问题及我的修改：**

| # | AI 初稿问题 | 我的修改 | 原因 |
| --- | --- | --- | --- |
| 1 | mint 用 _mint | 改为 _safeMint | _mint 不检查接收方是否为合约，铸造给合约会导致代币永久卡死 |
| 2 | hasBadge 试图用 balanceOf + 循环枚举，但没有 tokenId→typeId 映射，逻辑跑不通 | 新增 tokenBadgeType 和 badgeCount 两个映射，hasBadge 改为 O(1) 查 badgeCount | AI 这段是"看起来对、实际无法实现"的典型 |
| 3 | mint 没校验 typeId 是否存在 | 增加 BadgeTypeNotFound revert | 防止铸造幽灵类型 |
| 4 | createBadgeType 没校验 name 非空 | 增加 EmptyName revert | 避免无意义空名类型 |
| 5 | 用 require("...") 字符串报错 | 改为 custom error | 省 gas + 更结构化 |
| 6 | 测试用 vm.expectRevert("Ownable: caller is not the owner") | OZ v5 已改用 custom error，需 abi.encodeWithSelector 完整匹配 | 初版测试 3 个全挂，修后 5 个全过 |

**刻意拒绝的过度设计：** burn、batchMint、可升级代理、白名单/领取价格

**合约架构（审查后定稿）：**

> `NFTBadge is ERC721URIStorage, Ownable├── struct BadgeType { name, description, uri, exists }├── mapping(uint256 => BadgeType) badgeTypes├── mapping(address => bool) minters├── mapping(uint256 => uint256) tokenBadgeType // tokenId → typeId├── mapping(address => mapping(uint256 => uint256)) badgeCount // owner → typeId → count├── uint256 nextTokenId├── uint256 nextBadgeTypeId├── createBadgeType(string, string, string) onlyOwner├── setMinter(address, bool) onlyOwner├── mint(address, uint256) onlyMinter├── hasBadge(address, uint256) view → bool // O(1)├── getBadgeType(uint256) view → BadgeType└── events: BadgeTypeCreated, BadgeMinted, MinterUpdated`

核心要点

-   **私钥即资产**：Web3 安全的第一原则。私钥泄露等于资产归零，没有"找回密码"这一说
    
-   **助记词是钱包的根密钥**：BIP-39 标准将私钥编码为 12/24 个单词，必须离线存储，绝不上云、不截图、不提交到任何版本控制系统
    
-   **隔离原则**：实验/学习钱包与主力钱包严格分离。频繁与未知合约交互、点击测试网链接本身就是高风险行为
    
-   **社交工程攻击是最大威胁**：钓鱼链接、虚假空投、假冒客服、伪造的 MetaMask 官网
    

安全存储方案对比

| 方案 | 安全性 | 实操难度 | 适用场景 |
| --- | --- | --- | --- |
| 手写助记词 + 保险箱 | ⭐⭐⭐⭐⭐ | 低 | 长期存储，推荐 |
| 硬件钱包（Ledger/Trezor） | ⭐⭐⭐⭐⭐ | 中 | 大额资产，私钥永不离开设备 |
| 密码管理器加密存储 | ⭐⭐⭐ | 低 | 次选方案，确保主密码足够强 |
| 手机备忘录/截图 | ⭐ | 极低 | ❌ 绝对禁止 |

自己的理解

Web3 的安全模型和传统互联网完全不同。传统互联网有"找回密码"机制，有客服，有银行风控——出了问题可以申诉。Web3 没有这些。私钥就是一切，丢了就是丢了。

这带来的是一种全新的责任模式：你必须自己为自己的资产安全负责。没有安全团队帮你兜底。所以建立安全习惯比学会任何技术都重要——隔离钱包、离线备份、谨慎签名，这些是 Web3 开发者的基本功。

### 2\. BuildAnything Freshman — 加密钱包

**链接：** [https://buildanything.so/zh/tracks/freshman](https://buildanything.so/zh/tracks/freshman)

核心概念

**钱包不是存钱的，是管钥匙的。**

-   很多人误以为钱包像支付宝/微信一样"存着钱"。不对。钱包只做一件事：**管理你的密钥对**
    
-   **公钥 = 地址**：可以公开，用来接收资金。别人知道你的地址没有任何风险
    
-   **私钥 = 签名凭证**：必须绝对保密。有了私钥就拥有了该地址下所有资产的控制权
    
-   **助记词 = 私钥的人类可读备份**：12 个单词就能恢复整个钱包
    

EVM 钱包的工作原理

**密钥推导流程：** 随机数 → 256位私钥 → secp256k1 椭圆曲线 → 公钥 → Keccak-256 哈希 → 取后20字节 → 0x地址

1.  **密钥生成**：随机数生成器产生 256 位的私钥（2^256 种可能，几乎不可能碰撞）
    
2.  **公钥推导**：通过椭圆曲线加密（secp256k1）从私钥**单向**推导出公钥（不可逆）
    
3.  **地址生成**：对公钥进行 Keccak-256 哈希，取最后 20 字节，加上 0x 前缀
    
4.  **交易签名**：使用私钥对交易数据进行 ECDSA 签名
    
5.  **交易广播**：将签名后的交易发送到区块链网络的 mempool
    

钱包类型对比

| 类型 | 安全性 | 便利性 | 适用场景 |
| --- | --- | --- | --- |
| 硬件钱包 | ⭐⭐⭐⭐⭐ | ⭐⭐ | 大额资产长期存储 |
| 浏览器插件（MetaMask） | ⭐⭐⭐ | ⭐⭐⭐⭐ | 日常 dApp 交互 |
| 手机 App（Rabby Wallet） | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 移动端使用 |
| 纸钱包 | ⭐⭐⭐⭐⭐ | ⭐ | 冷存储备份 |

**为什么选 MetaMask：** EVM 生态最广泛支持，浏览器插件与 dApp 无缝集成，支持自定义网络（可以手动添加任何 EVM 兼容链），开源代码经社区审计。

### 4\. 交易与 Gas 机制

**参考：**

-   Ethereum Transactions：[https://ethereum.org/en/developers/docs/transactions/](https://ethereum.org/en/developers/docs/transactions/)
    
-   Ethereum Gas：[https://ethereum.org/en/developers/docs/gas/](https://ethereum.org/en/developers/docs/gas/)
    
-   Monad Gas Pricing：[https://docs.monad.xyz/developer-essentials/gas-pricing](https://docs.monad.xyz/developer-essentials/gas-pricing)
    

交易的结构

一笔 EVM 交易包含以下字段：

| 字段 | 说明 | 我的交易中的值 |
| --- | --- | --- |
| from | 发送方地址 | 0x8EB3...96BE2C |
| to | 接收方地址（合约部署则为 null） | 0xc2dc...3c9d6 |
| value | 转账金额（wei 为单位） | 1 MONAD |
| data/input | 合约调用数据（转账则为空） | 0x |
| gas limit | 允许消耗的最大 gas | 21,000 |
| maxFeePerGas | EIP-1559 最大费用上限 | 138.45 Gwei |
| maxPriorityFeePerGas | 验证者小费 | 3.45 Gwei |
| nonce | 发送方的交易序号 | 0（第一笔） |
| v, r, s | ECDSA 签名参数 | — |

Gas 机制详解

**Gas = 链上计算资源的计价单位。**

> 总手续费 = gasUsed × effectiveGasPrice

-   **基础 Gas（21,000）**：每笔简单 EOA 转账固定的 gas 消耗量
    
-   **合约交互**：调用智能合约会消耗更多 gas（取决于计算复杂度）
    
-   **Gas 价格**：由网络供需决定，拥堵时上涨、空闲时下降
    

**EIP-1559 费用模型（type 0x2）：**

> effectiveGasPrice = baseFee + priorityFee

-   **baseFee（基础费）**：由协议算法算出，根据网络拥堵程度自动调整，**全部销毁**
    
-   **priorityFee（优先费/小费）**：用户自愿给验证者的激励，出价越高优先被打包
    

**这次交易的费用明细：**

> 21,000 gas × 103.45 Gwei = 0.002172 MONAD 其中 baseFee ≈ 100 Gwei（销毁），priorityFee ≈ 3.45 Gwei（给验证者）

为什么失败交易也消耗 Gas

这是很多新手困惑的点。原因是：

> **节点已经执行了计算。即使交易 revert，计算资源已经消耗了。**

好比你去餐厅点了菜，厨房已经开始做了，但做到一半发现食材不够做不了。你仍然要为已经发生的劳动付费。在链上：

-   已执行的 **gasUsed** 不退（已经消耗的算力）
    
-   但未用完的 **gas limit** 余额会退（相当于预存的押金没花完的部分）
    

这也解释了为什么要把 gas limit 设得比预期稍高一些——如果设得太低，交易会因为 out-of-gas 而失败，gas 已经花了但交易没成功。

### 3\. BuildAnything Freshman — 为什么要构建去中心化应用？

**链接：** [https://buildanything.so/zh/tracks/freshman](https://buildanything.so/zh/tracks/freshman)

**关键理解：**

-   dApp 的核心价值不在于「去中心化」本身，而在于「无需许可的创新」
    
-   传统互联网：你需要平台允许才能发布内容
    
-   Web3：任何人都可以部署合约、创建代币、发起交易
    

**深入理解：** 「无需许可」是 Web3 最核心的革命性特征。在传统互联网时代，如果你想发布一个应用，你需要：

1.  购买服务器和域名
    
2.  遵守平台的内容政策
    
3.  接受平台的分成规则
    
4.  随时可能被封号
    

而在 Web3 中，任何人都可以：

1.  部署智能合约（只需支付 gas fee）
    
2.  创建自己的代币体系
    
3.  设计自己的治理规则
    
4.  不受任何中心化机构控制
    

这种开放性催生了前所未有的创新速度。DeFi、NFT、DAO 等全新范式都是在无需许可的环境中诞生的。

### 4\. BuildAnything Freshman — Vibecoding 入门

**核心概念：**

-   Vibecoding = 用 AI 辅助编程，通过自然语言描述意图让 AI 生成代码
    
-   不是取代开发者，而是降低构建门槛
    
-   关键能力：清晰的意图表达 + 代码审查能力
    

**深入理解：** Vibecoding 代表了一种新的编程范式转变。传统编程要求开发者掌握精确的语法和 API 调用，而 Vibecoding 将重点转移到「意图表达」上。

这种转变带来的优势是：

1.  **降低入门门槛：** 初学者可以通过自然语言描述需求，让 AI 生成基础代码
    
2.  **加速原型开发：** 快速验证想法，迭代成本大幅降低
    
3.  **释放创造力：** 开发者可以将更多精力集中在架构设计和业务逻辑上
    

但同时也带来新的挑战：

1.  **代码审查能力要求更高：** 需要理解 AI 生成的代码是否正确、安全
    
2.  **意图表达能力成为核心竞争力：** 能够清晰描述需求比掌握语法更重要
    
3.  **技术深度不能放弃：** 当 AI 无法解决复杂问题时，仍需扎实的技术功底
    

### 5\. EPF Wiki — 从普通开发者到协议贡献者的路径

**链接：** [https://epf.wiki/#/README](https://epf.wiki/#/README)

**认识：** 在分享会上听到嘉宾讲了如何从一个普通开发者走到 EPF 的路径，首次接触后意识到这是一个被严重低估的学习宝库，而且路径是清晰可循的。

* * *

EPF Wiki 是什么

Protocol Wiki（[https://epf.wiki](https://epf.wiki)）是由 Ethereum Protocol Fellowship Study Group（EPFsg）社区维护的技术知识库，专注于 Ethereum 核心协议的底层技术文档。它不是面向智能合约开发者的入门指南，而是面向想要「深入协议层」的学习者和贡献者。

* * *

从普通开发者到 EPF 的路径

分享会嘉宾讲述的路径可以总结为以下几个阶段：

**阶段一：学习 — EPS Study Group（入门）**

-   **做什么：** 参加以太坊协议学习小组（Ethereum Protocol Studies），这是一个开放的自学项目
    
-   **时间：** 每年2-4月有集中活动，其余时间可自学
    
-   **资源：** [https://study.epf.wiki](https://study.epf.wiki) — 交互式学习平台，包含课程、练习、测验
    
-   **内容结构：**
    
    -   模块一：协议高层概览（执行层 + 共识层）
        
    -   模块二：开发过程（客户端多样性、项目管理、协作方式）
        
    -   模块三：研究前沿（ePBS、Preconfirmations、数据可用性、zkEVM等）
        
-   **关键动作：** 加入 Discord 社区，参与讨论，完成课程练习
    

**阶段二：贡献 — Protocol Wiki（动手）**

-   **做什么：** 为 Protocol Wiki 贡献内容
    
-   **为什么从这里开始：** Wiki 明确标注很多页面是 _stubs_（待完善），鼓励任何人贡献
    
-   **怎么贡献：**
    
    1.  在 GitHub 上浏览 [https://github.com/eth-protocol-fellows/protocol-studies](https://github.com/eth-protocol-fellows/protocol-studies)
        
    2.  找到标注为 stub 的页面或自己熟悉的主题
        
    3.  基于自己的学习经验编写内容（“解释给你的同行听”）
        
    4.  提交 PR，经过审查后合并
        
-   **核心价值：** 贡献 Wiki 的过程本身就是深入学习。通过写作来学习，通过同行审查来纠正理解偏差
    

**阶段三：连接 — 参与核心开发社区（融入）**

-   **做什么：** 参加 AllCoreDevs 会议、Breakout Room、工作组电话会议
    
-   **在哪里：**
    
    -   Ethereum R&D Discord：日常讨论
        
    -   ethpm（项目管理仓库）：跟踪会议日程和记录
        
    -   每周 Execution + Consensus Layer 会议：公开透明，任何人都可以旁听
        
-   **关键心态：** 不需要等到「准备好了」才参与。协议开发是全公开、无需许可的。你可以先旁听，逐步理解讨论语境，然后提出问题，最后参与贡献
    

**阶段四：申请 — EPF Fellowship（深度参与）**

-   **做什么：** 申请 Ethereum Protocol Fellowship 年度 Cohort
    
-   **时间：** 每年一届，2026年 Cohort 7 于5月开放申请
    
-   **形式：** 4-5个月的深度项目制学习，与客户端团队直接合作
    
-   **福利：** 活跃贡献者可能有津贴（stipend），最终在 Devcon 参加线下 EPF Day
    
-   **申请前提：** 已经在 Wiki 或 Study Group 中有可见的贡献记录
    

**阶段五：成为核心贡献者（长期）**

-   **做什么：** 在某个协议领域持续贡献（客户端开发、密码学研究、测试工具等）
    
-   **身份转变：** 从「学习者」到「贡献者」到「维护者」
    
-   **现实：** 以太坊核心开发大约有200+开发者，分布在约20个独立团队中，客户端多样性是基本原则
    

* * *

这个路径对我意味着什么

1.  **路径是Permissionless的：** 不需要学位、不需要面试、不需要邀请。任何人都可以从 EPS Study Group 开始，从贡献一个 Wiki 页面开始。
    
2.  **贡献 Wiki 是最好的入门方式：** 与其空想「我够不够格」，不如直接开始写。Wiki 的贡献指南说得很清楚：「基于你学到的东西来写，解释给你的同行听」。这恰恰是 proof of work 的最佳形式。
    
3.  **协议知识是长期护城河：** 智能合约开发者的门槛正在被 AI 工具降低，但对协议层的理解是难以被替代的。Monad 虽然是应用层项目，但理解以太坊底层协议让我在架构设计和性能优化时拥有更深的视野。
    
4.  **社区即机会：** EPF 的核心哲学是「协议是全公开和开放的，任何人都可以开始贡献」。这不是口号——客户端代码、会议记录、讨论频道全部公开。真正的壁垒不是技术，而是主动性和持续性。

\### 2. 区块链基础概念

**链接：** [https://web3intern.xyz/zh/blockchain-basic/](https://web3intern.xyz/zh/blockchain-basic/)

**关键概念笔记：**

| 术语 | 定义 | 备注 |

|------|------|------|

| 区块链 | 分布式账本技术，数据按块链接，不可篡改 | 核心信任机制 |

| 去中心化 | 没有单一控制中心，由网络节点共同维护 | 与 Web2 的核心区别 |

| 共识机制 | PoW / PoS / DPoS 等，确保节点数据一致 | Monad 使用 PoS 变体 |

| 哈希函数 | 将任意输入映射为固定长度输出 | SHA-256, Keccak-256 |

| Merkle Tree | 高效验证数据完整性的树形结构 | 区块交易验证的基础 |

| 非对称加密 | 私钥签名，公钥验签 | 钱包地址的来源 |

\### 3. 以太坊概览

**链接：** [https://web3intern.xyz/zh/overview-of-ethereum/](https://web3intern.xyz/zh/overview-of-ethereum/)

**关键理解：**

\- EVM（以太坊虚拟机）是所有 EVM 兼容链的基础

\- Monad 是 EVM 兼容的高性能 L1，意味着以太坊的工具链（MetaMask、Remix、Hardhat）可以直接使用

\- 智能合约是部署在链上的可编程逻辑单元
