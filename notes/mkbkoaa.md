**Space / 小型活动策划 Brief**

**活动名称**

**Reverse AMA｜如果由社区来面试项目方，你会问什么？**

**一、活动目标（Why）**

**为什么要做这场 Space？**

传统 AMA 往往由项目方介绍产品、路线图和生态规划，容易形成单向输出，听众参与度有限，讨论也容易停留在宣传层面。

本次活动采用 Reverse AMA（反向 AMA）形式，将提问权交给社区，由听众主导讨论，项目方负责回答。

活动希望解决三个问题：

**学习问题**

-   帮助 Web3 新人学会如何判断一个项目，而不是被动接受信息。
    
-   通过真实提问了解项目运营、产品和社区建设，而不仅关注市场热点。
    

**社区问题**

-   提升社区成员参与感，鼓励用户从”听众”转变为”贡献者”。
    
-   建立开放、透明的交流氛围，增强社区信任。
    

**传播问题**

-   通过高质量提问形成可传播内容，提高 Space 的讨论价值和社区影响力。
    

**二、目标听众（Who）**

本次活动主要面向：

**Web3 新人与社区活跃成员**

包括：

-   想了解 Web3 项目的新人
    
-   Bootcamp 学员
    
-   社区 Contributor
    
-   对项目运营感兴趣的 Web2 从业者
    

次要受众：

-   项目运营
    
-   Builder
    
-   社区 Manager
    

用户画像：

已经接触过 Web3，希望了解一个项目真正值得关注哪些方面，并学习如何与项目方有效交流。

**三、核心主题（One Sentence）**

**“如果今天由社区来面试项目方，什么问题最值得被回答？”**

所有讨论围绕这一主题展开，不延伸至币价预测、投资建议或市场行情。

**四、核心问题（Discussion）**

主持人提前整理社区最关注的五类问题，保证讨论聚焦。

**Question 1：项目为什么值得做？**

-   项目想解决什么问题？
    
-   为什么选择这个赛道？
    
-   与同类项目相比最大的差异是什么？
    

**Question 2：社区真正需要什么样的贡献者？**

-   社区如何定义高质量贡献？
    
-   新人有哪些低门槛参与方式？
    
-   项目更看重活跃度还是专业能力？
    

**Question 3：项目如何看待长期增长？**

-   社区增长最重要的指标是什么？
    
-   如何保持社区活跃而不是短期热度？
    
-   是否有长期 Contributor 激励机制？
    

**Question 4：项目踩过哪些坑？**

-   社区运营失误

**Monad 与以太坊显著差异及高性能开发最佳实践总结**（智能合约开发者视角）

**1\. 虚拟机（EVM）差异**

-   **合约代码大小限制大幅放宽**：
    
    -   合约代码最大 128 KB（以太坊为 24 KB）
        
    -   初始化代码最大 256 KB（以太坊为 48 KB）
        
-   **Gas 定价调整**：因 Monad 执行优化，部分操作码和预编译程序的 Gas 价格已重新调整（详见操作码定价文档）。
    
-   **新增预编译**：支持符合 EIP-7951 的 **secp256r1 (P256)** 验证预编译（地址 0x0100），便于链上验证 WebAuthn / Passkey 签名。
    

**2\. 交易（Transaction）机制差异**

-   **Gas 费用计算方式不同**：
    
    -   按 **Gas Limit** 而非实际 Gas 使用量扣费。
        
    -   公式：value + gas\_bid \* gas\_limit（异步执行的 DoS 防护措施）。
        
-   **储备余额（Reserve Balance）机制**：
    
    -   确保纳入共识的交易都能支付费用。
        
    -   可能出现**最终执行回滚但仍收取 Gas** 的交易（例如余额不足），这类交易仍是有效的。
        
-   **其他**：
    
    -   **不支持 EIP-4844 Blob 交易**（Type 3）。
        
    -   **无全局内存池**：交易通过本地内存池转发给后续领导者，提高效率。
        

**3\. EIP-7702 授权相关特殊规则**

-   已委托（Delegated）的 EOA 账户，余额**不能低于 10 MON**；取消委托后可低于 10 MON。
    
-   被委托的 EOA 在作为智能合约调用时，**禁用 CREATE 和 CREATE2 操作码**。
    

**4\. 历史数据（History）访问**

-   由于高吞吐量，完整节点**无法支持任意历史状态查询**（存储成本过高）。
    
-   需参考“历史数据”部分了解具体限制，并据此设计应用。
    

**5\. 高性能应用构建最佳实践**

**网站托管与成本控制**

-   **Vercel / Railway** 等无服务器平台使用方便，但流量大时费用较高。
    
-   **推荐生产环境使用 AWS** 等云服务商实现精细成本控制：
    
    -   **静态文件 + CDN**：Amazon S3 + CloudFront
        
    -   **无服务器函数**：AWS Lambda
        
    -   **容器化应用**：Amazon ECS 或 EKS
        
    -   **数据库**：Amazon RDS
        
-   **定价注意事项**：仔细查看阶梯价格，避免被低价促销套餐误导（超出限额后费用会急剧上升）。
    

**RPC 调用优化**

-   **eth\_estimateGas**：
    
    -   对于 Gas 消耗固定的操作（例如原生代币转账固定 21,000 Gas），**直接硬编码**，不要每次调用 eth\_estimateGas。
        
    -   可显著提升用户操作速度，并避免部分钱包在回滚时的异常行为。
        
-   **eth\_call 批量优化**：
    
    -   避免串行多次请求，推荐**并发或批量提交**以降低延迟。
        
    -   **推荐方案**：
        
        1.  使用 **Multicall3** 合约（已部署在 Monad 主网和测试网，地址：0xcA11bde05977b3631167028862bE2a173976CA11），聚合多个读取操作。
            
        2.  复杂场景可部署**自定义聚合合约**，通过单个 eth\_call 返回所有数据。
            
    -   必要时考虑切换到专业索引器进一步提升性能。
        

Monad 在执行效率和 Gas 模型上与以太坊有明显差异，开发者需重点适配**更大合约体积**、**储备余额规则**、**历史状态限制**和**批量 RPC 调用**。合理利用硬编码 Gas 和 Multicall 可显著提升应用用户体验和性能。建议结合官方文档中的 Gas 定价和历史数据部分进行具体实现。

# 2026-07-10
<!-- DAILY_CHECKIN_2026-07-10_START -->



**Monad vs 以太坊：开发者必知的核心差异**

**1\. 虚拟机（VM）**

-   **合约代码上限**：128 KB（以太坊仅 24 KB），初始化代码上限 256 KB（以太坊 48 KB）。能写更大、更复杂的合约。
    
-   部分操作码和预编译 gas 价格调整（冷访问更贵，部分加密运算更贵）。
    
-   新增 **P256 签名验证**（地址 0x0100），支持 WebAuthn / Passkey 签名验证。
    

**2\. 交易规则**

-   **Gas 费用按 gas limit 收取**（不是实际用量）。扣款 = value + gas\_price × gas\_limit。
    
-   引入 **Reserve Balance（储备余额）机制**（约 10 MON）：
    
    -   防止 DoS 攻击。
        
    -   可能看到“余额不足”而失败的交易，但仍扣 gas（正常回滚）。
        
-   不支持 **Blob 交易**（EIP-4844）。
    
-   无全局内存池，交易转发给未来几个出块者。
    

**3\. EIP-7702 委托**

-   委托后的 EOA 余额**不能低于 10 MON**（取消委托后可以）。
    
-   委托状态下，合约代码**禁止使用 CREATE / CREATE2**。
    

**4\. 历史数据**

-   因高吞吐量，完整节点**不保存任意历史状态**（节省存储）。
    

**一句话总结**：Monad 大部分和以太坊一样，但为了极致性能，在 Gas 计费、合约大小、历史数据访问上做了调整。开发时建议用 Monad 专属 Foundry 测试。

# 2026-07-09
<!-- DAILY_CHECKIN_2026-07-09_START -->




今天的内容感觉比较偏技术，对于非计算机专业的学习者来看，其实有一定难度。

**DApp 架构和开发流程简要总结**：去中心化 + 多技术栈协作（合约 + 前端 + Indexer），重点解决链上查询低效和安全问题。

**核心架构（区别于传统 Web 应用）**

-   **前端**：React/Vue 等构建 UI，通过**钱包**（MetaMask）和 **RPC 节点** 与区块链交互（只读调用 + 交易签名）。
    
-   **智能合约**：核心业务逻辑，用 **Solidity** 编写，部署在区块链（EVM）上，自动执行、不可篡改。
    
-   **Indexer（检索器）**：监听合约 Events，将链上数据同步到传统数据库，供前端高效查询。
    
-   **区块链 + 去中心化存储**（IPFS/Arweave）：存储状态/交易和大文件。
    

**开发流程**

1.  **需求分析**：定功能、选链（以太坊等）、设计 UX。
    
2.  **智能合约**：编写、测试、安全审计。
    
3.  **Indexer**：用 Ponder/Subgraph 等捕获事件、入库。
    
4.  **前端**：集成钱包、交互逻辑（推荐 Viem/Wagmi）。
    
5.  **交互与部署**：Hardhat/Foundry 部署合约，前端上 IPFS/Vercel，上线维护。
