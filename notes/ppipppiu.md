## 一、学习目标

本节主要学习如何使用 AI 辅助生成 Solidity 智能合约，并通过 Remix 完成基础的合约编译、部署和交互。

重点不是直接相信 AI 生成的代码，而是要理解代码结构、检查合约逻辑，并知道部署后各个概念之间的关系。

需要掌握：

1.  如何用 AI 生成一个最小 Solidity 合约。
    
2.  如何在 Remix 中编译、检查、部署和测试合约。
    
3.  理解合约源码、ABI、合约地址、读函数、写函数和交易哈希之间的关系。
    
4.  理解一次合约交互背后发生了什么。

3.  AI 可以帮助生成代码、解释代码、检查错误，但不能完全替代人工判断。

## 三、核心概念整理

### 1\. 智能合约源码

智能合约源码就是用 Solidity 写出来的代码。

它定义了：

1.  合约能存储什么数据。
    
2.  用户可以调用哪些函数。
    
3.  哪些操作会修改链上状态。
    
4.  哪些操作只是读取链上数据。
    
5.  用户之间的权限边界如何区分。
    

* * *

### 2\. ABI

ABI 可以理解为“前端和合约沟通的接口说明书”。

它告诉外部程序：

1.  这个合约有哪些函数。
    
2.  每个函数需要传入什么参数。
    
3.  每个函数会返回什么结果。
    
4.  哪些函数可以读取，哪些函数会发起交易。
    

前端页面或钱包想调用合约，一般需要合约地址和 ABI。

* * *

### 3\. 合约地址

合约部署成功后，区块链会给这个合约生成一个唯一地址。

合约地址的作用：

1.  用来定位链上的某个合约。
    
2.  用户或前端可以通过地址找到并调用合约。
    
3.  可以在区块链浏览器中查看合约的部署和交互记录。
    

* * *

### 4\. Read Function

Read function 是只读取链上数据的函数。

特点：

1.  不修改链上状态。
    
2.  通常不需要支付 Gas。
    
3.  调用后可以立即看到返回结果。
    
4.  常见关键字包括 `view` 和 `pure`。
    

例如：查询自己的 Todo 列表。

* * *

### 5\. Write Function

Write function 是会修改链上状态的函数。

特点：

1.  会改变合约中的数据。
    
2.  需要发起交易。
    
3.  通常需要支付 Gas。
    
4.  交易需要被链确认后才算执行完成。
    

例如：创建 Todo、完成 Todo。

* * *

### 6\. Transaction Hash

Transaction Hash 是一笔交易的唯一编号。

作用：

1.  用来查询交易是否成功。
    
2.  用来查看交易发起人、目标合约、Gas 消耗等信息。
    
3.  可以在区块链浏览器中追踪交易记录。
    
4.  每次 write function 调用都会产生一个 transaction hash。
    

* * *

## 四、Remix 实践流程

### 第一步：创建合约文件

在 Remix 中创建新的 `.sol` 文件，例如：

`OnchainTodo.sol`

然后把 Solidity 合约代码粘贴进去。

* * *

### 第二步：编译合约

进入 Solidity Compiler 面板，选择和代码匹配的 Solidity 版本。

如果编译成功，通常会看到：

1.  左侧出现绿色对勾。
    
2.  合约名称可以在编译结果中被识别。
    
3.  没有红色报错信息。
    

* * *

### 第三步：部署合约

进入 Deploy & Run Transactions 面板。

初学阶段可以选择：

`Remix VM`

这样是在浏览器模拟环境中部署，不需要真实钱包，也不会消耗真实资产。

点击 Deploy 后，如果部署成功，会在 Deployed Contracts 区域看到合约实例。

* * *

### 第四步：交互测试合约

部署后可以测试合约函数。

常见操作：

1.  调用 `createTodo` 创建任务。
    
2.  调用 `getMyTodos` 查询任务列表。
    
3.  调用 `completeTodo` 标记任务完成。
    
4.  再次调用 `getMyTodos` 检查状态是否变化。
    

* * *

## 五、最小合约实践理解

这次实践的核心是完成一个最小可运行的 Onchain Todo 合约。

合约功能包括：

1.  每个用户通过 `msg.sender` 拥有自己的 Todo 列表。
    
2.  每条 Todo 包含任务内容、完成状态和创建时间。
    
3.  用户可以创建自己的 Todo。
    
4.  用户只能完成自己列表中的任务。
    
5.  用户可以查询自己的全部 Todo。
    

* * *

## 六、需要重点理解的 Solidity 语法

### 1\. SPDX License Identifier

用于声明代码许可证。

常见写法：

`// SPDX-License-Identifier: MIT`

作用是说明这份代码使用什么开源协议。

* * *

### 2\. pragma solidity

用于指定 Solidity 编译器版本。

例如：

`pragma solidity ^0.8.0;`

意思是使用 Solidity 0.8.x 版本进行编译。

* * *

### 3\. struct

`struct` 用来定义一条 Todo 的数据结构。

例如一条 Todo 可以包含：

1.  `text`：任务内容。
    
2.  `isCompleted`：是否完成。
    
3.  `createdAt`：创建时间。
    

* * *

### 4\. mapping

`mapping(address => Todo[])` 表示：

每一个用户地址，对应一个 Todo 数组。

这样不同钱包地址之间的数据就是分开的。

* * *

### 5\. msg.sender

`msg.sender` 表示当前调用合约的人，也就是当前交易或调用的发起地址。

在 Todo 合约中，`msg.sender` 用来区分不同用户。

例如：

1.  A 地址创建的 Todo 存在 A 的列表里。
    
2.  B 地址创建的 Todo 存在 B 的列表里。
    
3.  A 不能直接修改 B 的 Todo 列表。
    

* * *

### 6\. require

`require` 用来做条件检查。

如果条件不满足，交易会失败并回滚。

例如：

1.  检查任务编号是否存在。
    
2.  防止用户操作不存在的 Todo。
    
3.  防止错误数据写入链上。
    

* * *

### 7\. event

`event` 用来记录链上事件日志。

作用：

1.  方便前端监听合约变化。
    
2.  方便区块链浏览器展示操作记录。
    
3.  方便后续做链上数据分析。
    

* * *

## 七、链上状态修改与读取区别

### 会修改链上状态的操作

这些操作属于 write function，需要发起交易：

1.  创建 Todo。
    
2.  标记 Todo 为完成。
    
3.  修改合约中存储的数据。
    

* * *

### 只读取链上数据的操作

这些操作属于 read function，不会修改链上状态：

1.  查询自己的 Todo 列表。
    
2.  查看某个变量的值。
    
3.  调用 `view` 类型函数。
    

* * *

## 一、学习目标

本节主要学习 Web3 钱包、测试网、交易和区块浏览器的基础使用流程，并完成一次测试网交互。

需要掌握的内容：

1.  创建课程专用钱包，不使用主力钱包做课程实验。
    
2.  添加 Monad Testnet。
    
3.  领取测试币。
    
4.  完成一次测试网转账或应用交互。
    
5.  使用 block explorer 查看并解释一笔交易记录。
    

## 二、钱包基础

1.  什么是钱包？
    

Web3 钱包不是单纯“存钱的地方”，而是用来管理链上账户和发起签名操作的工具。

钱包的主要作用包括：

查看账户地址和余额；

发起转账；

连接 DApp；

对交易或消息进行签名；

管理多个链上账户。

2.  钱包、账户、地址、私钥的关系
    

钱包 Wallet：管理链上账户的工具，例如 MetaMask。

账户 Account：链上的身份。

地址 Address：可以公开分享的收款地址。

私钥 Private Key：控制账户资产的核心凭证，不能泄露。

助记词 Seed Phrase：恢复钱包的凭证，不能截图、上传或提交。

3.  为什么要创建课程专用钱包？
    

因为测试网实验可能会连接不熟悉的网站或 DApp。为了降低风险，应该单独创建一个实验钱包。

基本原则：

不使用主力钱包；

不导入真实资产账户；

不在 GitHub、截图或作业中提交助记词；

不随便授权不熟悉的网站。

## 三、Monad Testnet 基础

1.  什么是测试网？
    

测试网是用于开发、测试和学习的区块链网络。测试网资产没有真实金融价值，适合用来练习转账、部署合约和测试 DApp。

Monad 的 testnet 是主要测试网络环境，适合用于完成课程中的钱包交互、测试币领取和交易记录查询。

2.  添加 Monad Testnet 的目的
    

添加 Monad Testnet 后，钱包才能连接 Monad 测试网络，并在该网络上完成测试交易。

添加网络时通常需要关注：

Network Name：网络名称；

Chain ID：链 ID，用来区分不同区块链网络；

RPC URL：钱包连接区块链节点的入口；

Currency Symbol：原生代币符号；

Block Explorer：用于查询交易、地址和区块信息。

3.  什么是测试币？
    

测试币是测试网上使用的代币，主要用于支付测试网交易中的 gas fee。

测试币没有真实价值，主要用于：

发起测试转账；

调用测试网 DApp；

部署或测试智能合约；

生成可以在 block explorer 中查询的交易记录。

## 四、交易基础

1.  什么是链上交易？
    

链上交易是由账户发起并签名的操作，会被记录到区块链上。

常见交易包括：

从一个地址转账到另一个地址；

调用智能合约；

与 DApp 交互；

部署智能合约。

2.  一笔交易通常包含什么信息？
    

from：发送方地址。

to：接收方地址或合约地址。

value：转账金额。

gas fee：交易手续费。

nonce：账户发出的第几笔交易。

signature：钱包私钥签名生成的授权证明。

transaction hash：交易哈希，用来唯一定位一笔交易。

3.  Transaction Hash 是什么？
    

Transaction Hash 可以理解为一笔交易的唯一编号。

完成转账或 DApp 交互后，可以复制 transaction hash 到 block explorer 中查询交易详情。

## 五、Gas 基础

1.  什么是 Gas？
    

Gas 是衡量链上操作所需计算资源的单位。

只要交易需要被区块链执行，通常就需要支付 gas fee。

2.  为什么需要 Gas？
    

Gas 的作用包括：

支付网络计算资源；

防止垃圾交易攻击；

防止恶意或无限循环的合约消耗网络资源；

激励验证者处理交易。

3.  Gas Fee 如何理解？
    

Gas Fee 可以简单理解为：

Gas Fee = 使用的 Gas 数量 × 每单位 Gas 的价格

在实际使用钱包时，钱包一般会自动估算推荐 gas fee。初学阶段通常不需要手动修改。

## 六、Block Explorer 基础

1.  什么是 Block Explorer？
    

Block Explorer 是区块链浏览器，用来查看链上公开数据。

可以查询的内容包括：

地址余额；

地址历史交易；

某一笔交易详情；

区块信息；

合约地址；

Gas 使用情况；

交易是否成功。

2.  查看一笔交易时重点看什么？
    

Transaction Hash：交易唯一编号。

Status：判断交易是否成功。

From：发送方地址。

To：接收方地址或合约地址。

Value：转账金额。

Gas Used：实际消耗的 gas。

Timestamp：交易发生时间。

Block：交易被打包进哪个区块。

3.  如何解释一笔交易？
    

可以按照以下顺序解释：

第一，这笔交易由哪个地址发起；

第二，发送到了哪个地址或合约；

第三，交易类型是普通转账还是合约交互；

第四，交易是否成功；

第五，消耗了多少 gas；

第六，交易被记录在哪个区块中；

第七，交易哈希是什么。

## 七、实操流程记录

本次实操可以按照以下流程完成：

1.  安装 MetaMask 或其他 EVM 钱包。
    
2.  创建课程专用钱包。
    
3.  备份助记词，但不要上传、截图或提交。
    
4.  添加 Monad Testnet。
    
5.  通过 faucet 领取测试币。
    
6.  完成一次测试网转账或 DApp 交互。
    
7.  在 block explorer 中查询 transaction hash。
    
8.  记录交易状态、发送地址、接收地址、gas 使用情况等信息。
    

## 八、本节小结

通过本节学习，我理解了 Web3 钱包、测试网、交易、Gas 和 Block Explorer 的基本关系。

钱包用于管理账户和发起签名；测试网用于无真实资产风险的实验；测试币用于支付测试网 Gas；交易哈希可以在区块浏览器中查询交易详情。

后续进行智能合约部署或 DApp 交互时，也会继续用到类似的钱包连接、签名确认、支付 Gas 和查询交易记录的流程。

## 1\. Web3 的工作方式

### 1.1 核心特点：远程协作 + 异步沟通

Web3 团队通常分布在不同国家和时区，因此远程办公和异步协作很常见。工作重点不是“在线多久”，而是是否按时完成明确的交付物。

远程协作中最重要的是：

-   **信息留痕**：关键结论、决策和任务不要只留在聊天记录中，要同步到 Notion、文档、GitHub Issue 等地方。
    
-   **明确时区**：沟通截止时间要标明时区，例如 `UTC+8 18:00 前提交`。
    
-   **明确责任人**：每个任务需要写清楚负责人、交付物和截止时间。
    
-   **会议后整理纪要**：记录结论、行动项、负责人和 Deadline。
    

### 1.2 常见工具

### 1.3 远程工作沟通原则

接到任务时，需要确认三个内容：

```text

2. 什么时候完成：截止时间是什么？
3. 按什么标准完成：参考什么格式、目标或优先级？
```

可以使用下面的表达方式：

```text
我理解本次需要在周三前提交方案初稿，
优先按照 X 框架完成。
如果方向需要调整，希望可以在周四上午前反馈。
```

向上汇报可以使用：

```text
进展 + 卡点 + 建议 + 需要的支持
```

例如：

```text
目前已完成数据整理和初步分析，整体进度约 70%。
当前卡点是缺少部分字段说明。
建议先按照现有字段完成基础版本，
需要确认后续是否会补充用户标签数据。
```

## 3\. Web3 安全重点

### 3.1 最重要的原则

```text
助记词和私钥绝不告诉任何人。
```

任何要求你输入助记词、私钥、钱包密码，或要求连接钱包并签名的陌生网页，都需要高度警惕。

尤其是以下情况：

```text
“领取空投”
“领取奖学金”
“验证身份”
“钱包需要升级”
“官方客服协助处理问题”
“连接钱包完成任务”
```

这些常常是钓鱼骗局。

### 3.2 常见骗局

1\. 假招聘 / 假面试软件

骗子以 Web3 公司、实习机会或内推为名，要求下载“内部面试软件”“会议软件”或“资料包”。

风险：

```text
- 木马窃取浏览器 Cookie
- 窃取邮箱、Telegram、Discord 账号
- 窃取钱包文件和密码
- 录屏、键盘记录、远程控制电脑
```

原则：

```text
正式面试优先使用 Zoom、Google Meet、Teams 等公开工具。
不要随意下载陌生人发来的安装包。
```

2\. 假官方账号 / 假客服

骗子会在 Telegram、Discord、X 等平台冒充项目方、管理员、HR 或客服。

常见话术：

```text
账号异常，需要验证
获得空投资格
需要连接钱包
需要发送验证码
需要转账验证
```

原则：

```text
不要相信主动私聊你的“官方人员”。
遇到问题时，自己从官网或官方频道进入。
```

3\. 钓鱼网站

钓鱼网站的域名可能与官网非常像，例如只改一个字母、后缀或使用相似字符。

原则：

```text
不要通过私聊链接进入官网。
优先手动输入官网地址。
仔细检查域名是否正确。
```

4\. 恶意插件和木马

恶意浏览器插件可能会盗取钱包信息、替换转账地址或读取浏览器数据。

原则：

```text
- 软件和插件只从官网或官方应用商店下载
- 不安装来源不明的浏览器插件
- 不使用破解软件或未知工具
- 转账前核对完整地址
```

5\. 地址替换 / 剪贴板劫持

木马会监控剪贴板，当你复制钱包地址时，自动替换成攻击者地址。

原则：

```text
不要只核对地址前后几位。
转账前尽量核对完整地址。
优先使用地址簿、白名单或 ENS。
```

6\. 账号安全问题

邮箱、手机号和密码管理不当，可能导致多个账号同时被盗。

建议：

```text
- 每个平台使用不同的强密码
- 使用密码管理器
- 重要账号开启 2FA
- 优先使用验证器或硬件密钥，不依赖短信验证码
- 不要在浏览器中保存助记词
```

安全防护清单强调：陌生链接不点、陌生软件不装、助记词和私钥离线保存、转账前核对完整地址，并定期检查和撤销不必要的钱包授权。

## 4\. Web3 求职与项目选择风险

Web3 项目可能采用远程协作、灵活用工、人民币加 Token、全 USDT 等不同模式。

入职前需要确认：

```text
1. 合同主体是谁？
2. 是否有明确的工作内容和薪资结构？
3. 薪资中法币和 Token 的比例是多少？
4. Token 是否有锁定期、归属期或价格波动风险？
5. 项目是否存在拉人头、多级返利、高收益承诺等问题？
6. 项目是否有白皮书、官网、公开团队信息和真实产品？
```

需要警惕以下项目特征：

```text
- 高收益承诺
- 拉人头返利
- 多级分佣
- 充值后抽奖、提现的模式
- 只强调 Token 和收益，不强调产品和真实业务
- 要求员工参与发币、拉新、私下收款或资金流转
```

即使只是技术、运营或推广人员，如果明知项目涉嫌非法集资、传销、赌博或洗钱，仍可能存在法律风险。

## 1\. 区块链与 Web3 基础

### 1.1 区块链是什么

区块链可以理解为一种由多个节点共同维护的分布式账本。

-   **区块（Block）**：存储一批交易记录和相关信息。
    
-   **链（Chain）**：每个区块会保存前一个区块的哈希摘要，因此按时间顺序连接起来。
    
-   **节点（Node）**：参与保存账本、验证交易和维护网络的设备或参与者。
    
-   **分布式账本**：多个节点保存相同版本的数据，不依赖单一服务器。
    

区块链的核心逻辑可以概括为：  
  
用户发起交易  
→ 交易广播到网络  
→ 节点验证交易是否合法  
→ 验证后的交易被打包进区块  
→ 新区块加入链上  
→ 全网账本同步更新

### 1.2 区块链的主要特点

-   **不可篡改**：修改历史区块会影响后续区块的哈希，因此需要同时修改大量记录，成本极高。
    
-   **公开透明**：链上交易记录通常可被查询。
    
-   **伪匿名**：地址本身不直接对应真实身份，但地址与身份关联后，交易记录仍可被追踪。
    
-   **去中心化**：账本由多个节点共同维护，不依赖单一平台或机构。
    
-   **可追溯**：链上交易通常可以沿着区块顺序追踪历史记录。
    

需要注意：区块链并不是“绝对匿名”，更准确地说是“地址匿名、交易公开”。

### 1.3 公链、联盟链和私链

| 类型 | 节点加入方式 | 数据可见性 | 管理方式 | 常见场景 |
| --- | --- | --- | --- | --- |
| 公链 | 任何人可参与 | 通常公开 | 去中心化共识 | 加密货币、DeFi、NFT |
| 联盟链 | 需邀请或审批 | 联盟成员可见 | 多机构共同治理 | 供应链、金融协作 |
| 私链 | 由组织控制 | 内部可见 | 中心化管理 | 企业内部系统、审计 |

可以简单理解为：

```text
公链：开放互联网
联盟链：多个机构共享的系统
私链：企业内部数据库
```

### 1.4 Web2、Web3 与 Web 3.0

| 维度 | Web2 | Web3 | Web 3.0 |
| --- | --- | --- | --- |
| 核心逻辑 | 平台集中管理 | 用户拥有资产和身份 | 数据语义化、便于机器理解 |
| 数据控制权 | 平台 | 用户/钱包 | 依赖标准化数据结构 |
| 典型技术 | 云服务、数据库、平台账号 | 区块链、钱包、智能合约 | RDF、OWL、知识图谱 |
| 代表应用 | 微信、抖音、淘宝 | Uniswap、ENS、MetaMask | 知识图谱、语义搜索 |

重点区分：

-   **Web3 不等于 Web 3.0。**
    
-   Web3 更关注去中心化、数字资产所有权和无需信任的协作。
    
-   Web 3.0 更关注让机器理解数据之间的关系。
    

### 1.5 去中心化的价值与挑战

主要价值

-   降低对中心化平台的依赖。
    
-   用户可以通过私钥直接控制资产和账户。
    
-   网络不容易因单点故障而整体失效。
    
-   智能合约可以让规则自动执行。
    
-   开发者可以在开放协议基础上继续组合和创新。
    

主要挑战

-   **扩展性问题**：节点多、共识复杂，交易吞吐量和确认速度可能受限。
    
-   **安全问题**：区块链不可篡改，但智能合约漏洞仍可能造成损失。
    
-   **用户体验问题**：私钥管理、Gas 费、钱包操作对新手不友好。
    
-   **监管问题**：匿名性、跨境交易和资产属性会带来合规风险。

## 2\. 以太坊基础

### 2.1 以太坊是什么

以太坊不仅是一条区块链，也可以理解为一个支持智能合约的全球共享计算平台。

比特币更偏向于“去中心化货币”，而以太坊进一步支持开发者部署程序和应用：

```text
比特币：主要解决点对点价值转移

以太坊：价值转移 + 智能合约 + 去中心化应用
```

以太坊的核心价值在于：

-   支持智能合约自动执行规则。
    
-   支持构建 DApp。
    
-   支持 DeFi、NFT、DAO 等 Web3 应用。
    
-   提供开放、可组合的基础设施。
    

### 2.2 ETH、Gas 与 Gas Fee

-   **ETH**：以太坊网络的原生代币。
    
-   **Gas**：衡量一笔操作需要消耗多少计算资源的单位。
    
-   **Gas Fee**：用户执行交易或调用智能合约时支付的手续费。
    

可以理解为：

```text
ETH：支付网络使用费的资产
Gas：操作所需计算资源的计量单位
Gas Fee：实际支付给网络的手续费
```

例如：

-   转账 ETH 需要 Gas Fee。
    
-   铸造 NFT 需要 Gas Fee。
    
-   与 DeFi 合约交互需要 Gas Fee。
    
-   部署智能合约也需要 Gas Fee。
    

Gas Fee 的作用不只是收费，也可以防止恶意用户无限发送计算任务占用网络资源。

### 2.3 智能合约

智能合约是部署在区块链上的程序。

它会按照预设规则自动运行，不需要银行、平台或人工中介来执行。

例如，一个简单的交易规则可以是：

```text
当用户支付指定金额
→ 合约自动确认付款
→ 合约自动将 NFT 转给用户
→ 所有记录写入链上
```

智能合约的优点：

-   自动执行。
    
-   规则公开。
    
-   可验证。
    
-   减少中介依赖。
    

但也需要注意：

-   合约代码可能有漏洞。
    
-   合约一旦部署，修改成本较高。
    
-   错误逻辑可能造成真实资产损失。
    

### 2.4 以太坊账户

以太坊中主要有两类账户：

| 账户类型 | 英文 | 控制方式 | 用途 |
| --- | --- | --- | --- |
| 外部账户 | EOA | 私钥控制 | 用户钱包、发起交易 |
| 合约账户 | CA | 智能合约代码控制 | 部署和运行 DApp 逻辑 |

简单理解：

```text
EOA = 用户钱包账户
CA = 智能合约账户
```

用户通常通过钱包控制 EOA，再由 EOA 发起交易与合约账户交互。

私钥是控制 EOA 的唯一凭证，因此不能泄露、截图、发送给他人或存储在不安全的位置。

### 2.5 PoW 与 PoS

以太坊早期使用 PoW（Proof of Work，工作量证明），后来在 The Merge 后转向 PoS（Proof of Stake，权益证明）。

| 机制 | PoW | PoS |
| --- | --- | --- |
| 参与者 | 矿工 | 验证者 |
| 主要资源 | 算力、电力 | 质押资产 |
| 出块方式 | 计算竞争 | 随机选择验证者 |
| 能耗 | 较高 | 明显更低 |
| 风险约束 | 硬件与电力成本 | 质押资产可能被罚没 |

PoS 中，验证者需要质押 ETH 参与网络验证；作恶或严重失职可能触发 Slashing，即部分质押资产被罚没。

### 2.6 Layer 1 与 Layer 2

以太坊主网可以理解为 Layer 1。

Layer 2 是建立在主网之上的扩容方案，目标是降低手续费、提高交易速度，同时继承主网的安全性。

基本逻辑：

```text
大量交易先在 Layer 2 处理
→ 将结果或证明批量提交到以太坊主网
→ 主网负责最终验证与安全保障
```

常见 Layer 2 / Rollup 项目包括：

-   Arbitrum
    
-   Optimism
    
-   zkSync Era
    
-   Scroll
    

以太坊主网更像“结算层和安全层”，Layer 2 更像“高频交易处理层”。

## 3\. Vibecoding 入门

### 3.1 什么是 Vibecoding

Vibecoding 可以理解为：通过自然语言描述需求，让 AI 智能体帮助完成代码编写、页面搭建、功能实现和问题修复。

重点不是完全不懂技术，而是能够：

-   明确自己要做什么。
    
-   清楚描述需求。
    
-   审视 AI 输出。
    
-   持续反馈和迭代。
    

### 3.2 Vibecoding 的基本循环

```text
描述需求
→ AI 构建代码和文件
→ 查看运行结果
→ 反馈需要修改的地方
→ 重复迭代
```

第一版通常不会完全符合预期，因此迭代是正常过程，而不是失败。

### 3.3 Prompt 的重要性

Prompt 是给 AI 智能体的指令。

模糊 Prompt：

```text
帮我做一个网站。
```

更有效的 Prompt：

```text
帮我搭建一个个人作品集网站。

要求：
- 深色背景，白色文字；
- 包含页眉、个人简介、项目展示卡片和页脚；
- 页面需要适配手机端；
- 项目卡片包含标题、描述和跳转链接；
- 使用简洁的无衬线字体。
```

一个高质量 Prompt 通常包含：

```text
目标：要做什么
用户：给谁使用
页面：包含哪些模块
功能：用户可以做什么
样式：颜色、布局、视觉风格
限制：技术要求、数据要求、响应式要求
```

核心原则：

> 不需要一开始就会写代码，但需要尽量清楚地表达你想构建什么。

### 3.4 处理错误的方式

当 AI 生成的项目出现问题时：

1.  不要只说“报错了”。
    
2.  复制完整错误信息。
    
3.  描述页面上实际看到的现象。
    
4.  告诉 AI 预期结果是什么。
    
5.  让 AI 基于完整上下文排查问题。
    

示例：

```text
当前问题：
点击“提交”后页面空白。

浏览器控制台完整报错如下：
[粘贴完整报错]

预期行为：
提交成功后应该显示“报名成功”提示，并跳转到结果页面。

请先定位问题原因，再修改代码。
```

完整错误信息通常比自己概括错误更有价值。

### 3.5 使用 AI 智能体的安全原则

-   优先在沙箱或云端开发环境中运行智能体。
    
-   不要提供真实私钥、密码、API Key 或带资金的钱包信息。
    
-   将 AI 生成的代码视为需要审查的外部代码。
    
-   涉及资金、用户数据、钱包交互时必须额外检查。
    
-   使用开发测试账号，而不是自己的主钱包。
    

可以使用 Replit、GitHub Codespaces 等隔离环境进行练习。

## 4\. 今日重点概念速记

| 概念 | 简要理解 |
| --- | --- |
| 区块链 | 多节点共同维护的分布式账本 |
| 区块 | 存放交易记录的数据单元 |
| 哈希 | 用于校验数据是否被修改的摘要 |
| 节点 | 保存账本、验证交易、维护网络的参与者 |
| 钱包 | 管理地址和私钥的工具 |
| 地址 | 链上的公开账户标识，可用于收款和查询 |
| 私钥 | 控制钱包资产的唯一凭证，绝不能泄露 |
| 交易 | 用户在链上发起的操作，如转账或调用合约 |
| ETH | 以太坊网络的原生代币 |
| Gas | 链上操作消耗的计算资源单位 |
| Gas Fee | 执行交易或调用合约需要支付的手续费 |
| 智能合约 | 部署在链上、按规则自动执行的程序 |
| EOA | 由私钥控制的用户账户 |
| CA | 由智能合约代码控制的账户 |
| PoS | 通过质押资产参与网络验证的共识机制 |
| Layer 2 | 在主网之外处理大量交易、降低成本的扩容方案 |
| Vibecoding | 用自然语言驱动 AI 完成开发和迭代的方式 |

## 5\. 今日个人理解

Web3 的核心不只是“加密货币”，而是尝试通过区块链、钱包和智能合约，把数字资产所有权、交易规则和协作关系从中心化平台部分转移给用户和公开协议。

以太坊的重要性在于，它让区块链从“记录转账”扩展到“可以运行程序和应用”。

对于开发者或 Web3 从业者来说，除了理解链、钱包、Gas 和智能合约等基础概念，也需要具备持续构建和验证的能力。Vibecoding 可以降低原型开发门槛，但不意味着可以忽略安全、审查和需求表达能力。
