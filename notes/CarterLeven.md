# Web3 技术分享：如何从 0 到 1 使用 AI 开发

## Map 1: AI 时代软件工程的开发闭环

### 两个生命周期

-   **0 到 1：产出证明核心价值的 Demo**
    
    -   **核心三问**：必须明确**什么场景**、**解决了什么痛点**、**核心交互流程是怎样的**。
        
    -   **AI 优势**：利用大模型的高发散性，快速堆砌出第一版可行性验证代码。
        
-   **1 到 99：把 Demo 演进为可不断使用产品**
    
    -   从“跑得通”转变为“高可用、高并发、可持续维护”的线上系统。
        
    -   **核心重构**：有了 AI 的加持，**重构的成本被降低了 80%**。
        
    -   **为什么要不断重构？** 因为 AI 给出单片代码的速度极快，但缺乏全局架构观，生成精度低且极易产生“方向漂移（Context Drift）”。必须通过不断重构，纠正 AI 带来的代码屎山，锁死系统边界。
        

### 2\. 从 AI 到 Agent 的能力跃迁

-   **AI (LLM)**：主要停留在“一问一答”的文本生成和信息检索阶段。
    
-   **Agent (智能体)**：具备**核心意图拆解、工具调用、自动化测试与闭环执行任务**的能力。它不仅能给你写出代码，还能自主在本地运行指令、捕获报错并完成自我纠错。
    

### **3\. 三位一体的 AI 自动化工程化落地流派**

\[SDD: 规范驱动\] ➔ 先对齐意图与任务拆分 ↓ \[TDD: 测试驱动\] ➔ 用自动化断言逼 AI 填空与纠错 ↓ \[[AGENTS.md](http://AGENTS.md)\] ➔ 将成功经验持久化为长期记忆

-   **📋 SDD（规范驱动开发：Specification-Driven Development）**
    
    -   **核心思想**：先对齐意图，再生成代码。在 AI 动手前，先死锁输入、输出和边界条件。
        
    -   **黄金流程**：需求（为什么做） $\\rightarrow$ 规范（做成什么） $\\rightarrow$ 方案（如何实现） $\\rightarrow$ 任务（如何拆分） $\\rightarrow$ 实现（代码测试） $\\rightarrow$ 归档（保留结果）。
        
    -   **工具选择**：新项目可参考 `Spec Kit`，已有项目增量变更可用 `OpenSpec`。
        
-   **🧪 TDD（测试驱动开发：Test-Driven Development）**
    
    -   **核心思想**：给 AI 打造一个快速纠错回路。用数学和逻辑语言框死正确性，杜绝肉眼 debug 的低效。
        
    -   **实施 5 步循环**：写测试（定义期望） $\\rightarrow$ 测试失败（红灯证明问题） $\\rightarrow$ 最小实现（让 AI 填空） $\\rightarrow$ 测试通过（绿灯获得证据） $\\rightarrow$ 重构（保持行为并优化性能）。
        
-   **💾 AGENTS.md（持久化长期指令约定）**
    
    -   **核心思想**：一次性 Prompt 负责当前任务；持久化指令负责**反复生效**的工程约定，拒绝给 AI 当无记性的复读机。
        
    -   **落地配置**：在项目根目录建立 `AGENTS.md`（或全局配置 `~/.claude/CLAUDE.md`），写入该项目专属的构建/测试命令、编码风格与架构禁止事项。
        
    -   **安全红线**：**绝对不要在里面写入钱包私钥或临时需求**。
        

## Map 2: Web3 全景技术栈与职业大分流

> **入局铁律**：五个方向共享区块链、密码学、网络、测试和安全基础。 **战略建议**：作为开发者入门，优先选择 **ETH（以太坊生态）** 去学习开发，因为其 EVM 标准和基础设施在全网最通用、最具普适价值与商业沉淀。若已有明确的高频金融思路，再横向迁移至其他高性能兼容链（如 Monad）。

### 💻 1. DApp 开发（前端与链交互）

-   **核心内容**：负责用户交互界面（React / Next.js）、钱包唤起连接（Wagmi / RainbowKit） 以及链上数据的抓取与展示。
    
-   **极客演进**：放弃臃肿的 `ethers.js`，全面拥抱轻量极速的 `viem`，利用 `Promise.all` 机制在前端本地实现 RPC 批量请求（Multicall），压榨每一毫秒的网络响应延迟。
    
-   **数据分流**：节点只负责最新的高频 Read/Write，复杂的历史账本与衍生K线数据，全面交由 `Envio` 或 `Goldsky` 这类高吞吐量的低延迟索引器（Indexer）进行异步解析。
    

### ⚙️ 2. 智能合约（链上核心逻辑）

-   **核心内容**：使用 **Solidity** 语言将资产、权限和协议规则硬编码在区块链虚拟机（EVM）中。
    
-   **核心开发框架**：全面淘汰基于 JS 且运行缓慢的 Hardhat 框架，统一采用由 Rust 编写的 `Foundry` 全家桶，实现“纯 Solidity 编写测试”的高效自动化 TDD 流程。
    
-   **Gas 优化与防爆**：引入卷到极致的底层库 `Solady`，或在核心撮合逻辑中适度使用 `Yul` **汇编操作码**。针对 Monad 独特的“按 Gas Limit 扣费”的防 DoS 机制，必须在合约端死守绝对精简的空间开销。
    

### 🔑 3. 钱包开发（工具与安全底座）

-   **核心内容**：掌管私钥的绝对安全、账户管理、签名以及交易体验。
    
-   **技术演进**：从传统的 `BIP-39/32/44` 助记词分层确定性钱包，向 `ERC-4337` 账户抽象（AA）以及 `WebAuthn / Passkey` 消费级体验演进。
    
-   **链端配合**：利用链端对 `secp256r1` (P256) 曲线的原生预编译加速，打通用户直接调用手机 FaceID 刷脸实现毫秒级链上安全签名的无摩擦通道。
    

### 🏎️ 4. 节点开发（底层共识与分布式系统）

-   **核心内容**：使用 Go / Rust 语言改写区块链底层的执行客户端与共识客户端。
    
-   **极客前沿**：研究用 Rust 重写的超高性能客户端 `Reth`、节点间打手势通信的网络库 `libp2p`、以及彻底打破传统单线程 I/O 阻塞的异步并发存储架构（如 MonadDB）。

## 为什么 Monad 改变了高频交互的竞技场？

以往我们谈论高性能公链，往往只关注单体 TPS 的数字增加。但 Monad 的真正价值在于，它为了跑通“异步执行”而被迫对以太坊的底层规则进行了大刀阔斧的重构。正是这些硬性的技术改动，彻底打破了传统 Web3 应用“操作必弹窗、等待必转圈、高频必瘫痪”的魔咒，让去中心化产品第一次具备了比肩中心化互联网体验的可能。

## 一、 异步执行引出的金融防线：从根本上稳固高频交互环境

在以太坊的逻辑里，网络是先执行完交易再收取 Gas 费的，这就像在餐馆吃完再结账。但 Monad 采用了颠覆性的异步执行架构，在共识节点对交易进行排序和打包时，这些交易尚未被实际执行。如果沿用旧制，黑客就可以伪造大量表面上设置了巨额 Gas 限制、但实际一运行就报错退出的垃圾交易，用极低的成本瘫痪整个网络。为了封死这种 DoS 攻击通道，Monad 做出了一项极具魄力的规定：**无论你最终实际消耗了多少 Gas，网络一律直接扣除你设定的 Gas Limit 全额费用。**

这一看似冷酷的“霸王条款”，配合其特有的“基础费上涨慢、下跌快”的全新 Gas 控制器，反而为链上高频交互构筑了一条坚不可摧的金融防线。由于恶意攻击的成本变得高不可攀，网络即便在极端高频的吞吐状态下，也能维持极度便宜且稳定的 Gas 环境。这直接让去中心化限价订单薄（Orderbook DEX）和全链游戏（Full on-chain games）成为了现实，做市商和玩家可以疯狂挂单、撤单、交互，而不用担心 Gas 像以前一样瞬间飙升或网络崩溃。

## 二、 密码学与接口的极致压榨：将高频体验推向“无感”

高频产品体验的另一个核心痛点在于“等待摩擦”。以太坊的前端应用在发送每笔交易前，钱包都必须向节点发起一次网络请求来估算油费。这种串行的等待不仅增加了网络往返的延迟，还常常因为节点报错让用户卡死在加载界面。Monad 的技术栈打破了这一繁琐流程，它极力推行在固定成本的操作中直接由前端“硬编码”写死 Gas 限制，并要求开发者使用高并发的 RPC 批处理和本地 Nonce 追踪技术。当这些隐藏在底层的网络往返时间被压缩到极致，钱包的弹窗和响应速度可以平白快上几十毫秒，实现真正的点击即响应。

更让人兴奋的是，Monad 在虚拟机层面原生支持了全新的密码学验证曲线（EIP-7951）。这意味着未来的高频应用不需要再借助复杂的合约转换，就能直接在链上高效验证手机硬件自带的生物识别签名。用户不再需要面对繁琐的私钥或频繁跳出的钱包确认，只需刷一下 FaceID 或者按一下指纹，就能在毫秒间完成一次安全的链上高频交互，彻底抹平了 Web2 与 Web3 之间的用户感知鸿沟。

## 三、 合约体积的 5 倍暴涨：释放复杂智能引擎的终极算力

在以太坊上写过复杂逻辑的开发者，都曾深切体会过被 24 KB 合约大小限制支配的痛苦。为了塞下更复杂的业务，开发者不得不把完整的逻辑拆散到多个相互调用的子合约中，这不仅让架构变得支离破碎，更因为跨合约调用的嵌套极大地拖慢了执行效率。Monad 直接解开了这一枷锁，将最大合约体积极限暴力提升了 5 倍，拓展到了 128 KB。

这多出来的空间，绝不仅仅是多写几行代码那么简单。它允许开发者将完整的全链游戏世界观、多步复杂的动态资产套利策略、甚至轻量级的链上 AI 决策与模型验证逻辑，毫无保留地打包进同一个单体合约中高效运转。当庞大、复杂的“链上 AI 代理”得以摆脱空间束缚、在免受跨合约延迟折磨的单体合约中肆意倾泻算力时，再配合 Monad 强悍的高并发状态读取生态，整个链上生态的自动化、高频化以及智能化协作，才真正迎来了质的飞跃。

## 自主学习

### 1\. 合约源码 (Source Code)

-   **本质**：用 Solidity 或 Vyper 等语言编写的人类可读的代码。
    
-   **大白话**：产品的“设计图纸”。
    

### 2\. ABI (Application Binary Interface)

-   **本质**：一个 JSON 格式的列表，描述了合约有哪些函数、需要什么输入、返回什么输出。
    
-   **大白话**：给机器看的“产品使用说明书”或“接口 API”。没有它，外部程序就不知道怎么跟那一堆看不懂的十六进制机器码（Bytecode）交互。
    

### 3\. 合约地址 (Contract Address)

-   **本质**：合约在区块链上的唯一身份证（如 `0x1f9840...`）。
    
-   **大白话**：智能合约的“收货地址”或“官方网址”。
    

### 4\. Read / Write Function (读/写函数)

-   **Read (读)**：
    
    -   **特点**：纯查询（如查余额 `balanceOf`），不消耗 Gas。
        
    -   **大白话**：只读不写，相当于刷微博看帖。
        
-   **Write (写)**：
    
    -   **特点**：修改状态（如转账 `transfer`），必须由钱包签名，消耗 Gas。
        
    -   **大白话**：改变数据，相当于发微博、改密码。
        

### 5\. Transaction Hash (交易哈希 / Tx Hash)

-   **本质**：区块链上任何一个“写操作”成功或失败后，系统生成的唯一 64 位十六进制哈希值。
    
-   **大白话**：银行转账成功的“电子回单号”。你可以用它在区块链浏览器（如 Etherscan）上查询这笔操作是成功了、失败了，还是还在排队。
