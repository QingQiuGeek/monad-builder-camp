# Monad Streak Pet Demo 设计学习笔记

## 1\. Demo 的核心想法

这次 Demo 的名字是 **Monad Streak Pet**，中文可以理解为“Monad 链上打卡宠物”。

最开始我只是想做一个普通的链上打卡合约，但后来发现单纯的 `checkIn()` 对新人来说不够有吸引力。所以我把“每日链上交互”包装成了一个更容易理解的动作：**喂养宠物**。

这个 Demo 的核心逻辑是：

每日链上交互 = 喂养宠物 = 宠物成长

用户每天在 Monad Testnet 上完成一次操作，宠物就会获得成长值、累计喂养次数和等级提升。

* * *

## 2\. 为什么选择宠物养成

宠物养成比普通打卡更适合社区传播。

普通用户可能不理解什么是智能合约、write function、gas、transaction hash，但很容易理解“每天喂一次宠物”。这样可以降低 Web3 新人的心理门槛。

同时，宠物养成天然适合做连续打卡、等级成长、状态变化、分享截图和排行榜。它不只是一个技术 Demo，也可以扩展成一个社区活动或新人 onboarding 工具。

* * *

## 3\. 产品设定

Demo 名称：

Monad Streak Pet

宠物名称：

Momo

宠物设定：

Momo 是一只紫色链上小怪兽，身体里有一个代表 Monad 能量的六边形核心。它不吃普通食物，只吃用户每天在 Monad Testnet 上完成的链上交互。

核心口号：

Feed onchain. Grow with every streak.

中文理解：

每天一次链上喂养，养大你的 Monad 宠物。

* * *

## 4\. 合约功能设计

这个 Demo 的合约叫：

MonadStreakPet

主要功能包括：

createPet(string name)

用户创建自己的链上宠物。

```
feed()
```

用户每天喂养一次宠物。这个函数会修改链上状态，所以是 write function，需要钱包确认，也会产生 transaction hash。

```
getPetCore(address user)
```

查询宠物的基础信息，比如名字、创建时间、累计喂养次数、上次喂养日期、当前 streak 和最好 streak。

```
getPetStatus(address user)
```

查询宠物状态，比如等级、今天是否已经喂养、当前成长阶段。

```
hasFedToday(address user)
```

查询某个地址今天是否已经喂养宠物。

* * *

## 5\. 宠物成长规则

宠物根据累计喂养次数升级：

| 等级 | 条件 | 形态 |
| --- | --- | --- |
| Level 0 | 0 次喂养 | Egg |
| Level 1 | 1-2 次喂养 | Hatchling |
| Level 2 | 3-6 次喂养 | Baby Pet |
| Level 3 | 7-13 次喂养 | Speed Pup |
| Level 4 | 14-29 次喂养 | Monad Beast |
| Level 5 | 30+ 次喂养 | Monad Guardian |

这个规则比较简单，适合 Week 1 Demo。后续可以继续增加更多成长机制，比如特殊形态、连续打卡奖励、Badge 或排行榜。

* * *

## 6\. 哪些数据上链，哪些不上链

这次 Demo 让我学到，不是所有内容都需要放到链上。

适合上链的数据：

```
宠物名字
是否创建宠物
累计喂养次数
上一次喂养日期
连续喂养天数
最好连续记录
宠物等级
```

这些数据代表用户真实参与记录，可以通过区块浏览器公开验证。

不一定需要上链的数据：

```
宠物图片
动画效果
分享卡片
Meme 文案
前端 UI
活动介绍
```

这些内容更适合放在前端、GitHub、assets 文件夹或普通数据库里。合约负责记录关键状态，前端负责展示体验。

* * *

## 7\. 遇到的问题和修改

在开发过程中，我遇到了一个 Remix 编译错误：

```
Stack too deep
```

原因是 AI 最开始生成的 `getPet()` 函数一次性返回太多字段，导致 Solidity 编译器处理不过来。

我后来把一个大的查询函数拆成两个小函数：

```
getPetCore()
getPetStatus()
```

这样代码结构更清晰，也成功解决了编译问题。

这个过程让我理解到：AI 可以生成初稿，但代码是否能编译、函数结构是否合理，仍然需要人工判断和修改。

* * *

## 8\. AI 在这个 Demo 中的作用

AI 主要帮助我完成了：

```
产品方向设计
Solidity 合约初稿
合约结构解释
README 文档整理
宠物形象设定
Ops / Meme 传播点设计
GitHub 展示结构设计
```

但 AI 不能替我完成最终判断。比如网络是否选对、合约是否能编译、是否有权限问题、哪些数据应该上链、是否提交了敏感信息，这些都必须由我自己检查。

* * *

## 9\. 这个 Demo 和 Ops 方向的关系

我选择 Week 2 继续走 **Ops** 方向，因为这个 Demo 不只是一个合约练习，也可以变成一个社区增长活动。

它可以扩展成：

```
每日喂养打卡
新人 onboarding 任务
宠物等级排行榜
连续 streak 活动
宠物状态分享卡
Monad Meme 活动
链上参与 Badge
```

相比纯技术项目，我更关注的是如何把链上交互变成普通新人愿意参与、愿意分享、愿意持续完成的活动。

* * *

## 10\. 本次 Demo 最大的学习收获

这次 Demo 让我理解到，一个链上产品不只是“把数据写到区块链上”，更重要的是设计一个让用户愿意交互的理由。

普通的合约函数对新人来说可能很抽象，但如果把它包装成“喂养宠物”“连续成长”“分享状态”，用户会更容易理解，也更愿意参与。

同时，我也理解了链上产品的基本分工：

```
合约：记录关键状态和真实链上行为
前端：展示宠物形象和用户体验
区块浏览器：验证交易和合约状态
AI：辅助生成初稿和整理思路
人工判断：负责安全、逻辑和最终决策
```

这个 Demo 是我从“完成链上操作”走向“设计链上产品”的第一步。

## Monad 合约部署学习笔记

今天学习了如何把一个最小 Solidity 合约部署到 Monad Testnet，并完成基础合约交互。

整个流程可以理解为：

**合约源码 → 编译 → 连接钱包 → 部署到测试网 → 获得合约地址 → 调用合约函数 → 区块浏览器验证**

首先，我在 Remix 中创建了 `CheckIn.sol` 文件，并写入一个简单的打卡合约。这个合约可以记录用户是否打卡，以及累计打卡次数。

然后，我使用 Remix 的 Solidity Compiler 对合约进行编译。编译的作用是把 Solidity 代码转换成区块链可以识别的字节码。如果代码有语法错误，部署前就会报错。

接着，我用课程专用钱包连接 Remix，并确认钱包网络是 Monad Testnet，而不是 Ethereum 主网。这个步骤很重要，因为如果网络选错，可能会使用真实 ETH 支付 gas。

之后，我点击 Deploy，把合约部署到 Monad Testnet。部署成功后，链上会生成一个合约地址。合约地址就像这个程序在区块链上的“门牌号”，之后所有交互都要通过这个地址完成。

部署合约本身也是一笔链上交易，所以会生成 deployment transaction hash。这个 hash 可以在区块浏览器中查询，用来验证合约是否部署成功。

合约部署完成后，我测试了 read function 和 write function。read function 只是读取链上数据，不会修改状态，所以一般不需要 gas，也不会生成交易 hash。write function 会修改链上状态，例如调用 `checkIn()` 完成打卡，所以需要钱包确认，并会产生新的 transaction hash。

最后，我在区块浏览器中查看了合约地址、部署交易和交互交易，理解了链上操作是公开可查的。

这次学习让我明白，智能合约部署不是简单地“上传代码”，而是把代码编译后写入区块链。部署成功后，合约会拥有独立地址，用户可以通过 read / write function 与它交互。整个过程需要特别注意钱包网络、gas 费用、交易 hash 和区块浏览器验证。

今天完成了 Monad Week 1 的基础学习和打卡任务。开始创建了一个专门用于测试网的钱包。之后，学习了如何添加 Monad Testnet，领取测试网资产 MON，并完成了一次测试网转账或链上交互。通过这一步，我第一次理解了链上交易的基本流程：钱包发起交易、网络处理交易、交易被打包上链，最后生成 transaction hash。

然后使用区块浏览器查看交易详情，学习了 `from`、`to`、`value`、`gas`、`transaction fee` 和 `status` 的含义。`from` 是发起交易的钱包地址，`to` 是接收方地址或合约地址，`value` 表示转账金额，`gas` 和手续费代表链上执行交易所消耗的成本。我也理解了为什么失败交易仍然可能消耗 gas，因为节点已经对交易进行了处理和计算。

最后开始学习 AI 辅助开发，用 AI 生成了一个最小 Solidity 打卡合约，并在 Remix 中完成编译和测试。这个合约可以记录用户是否打卡、累计打卡次数，并限制同一个地址每天只能打卡一次。通过人工检查，我学习了 `mapping`、`event`、`function`、`msg.sender`、`require` 和 `block.timestamp` 等基础概念。

今天最大的收获是：链上产品和普通互联网产品不同，用户的操作、资产和交易记录都会直接写入区块链，公开可查且不可随意撤回。AI 可以帮助我快速生成代码和解释逻辑，但最终仍然需要我自己检查合约是否能编译、函数是否符合预期、是否存在权限问题和潜在安全风险。
