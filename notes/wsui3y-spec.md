# 2026-07-15

今天的学习重点，是从“知道一个岗位大概做什么”，进一步深入到“这个岗位实际是怎么开展工作的、最终需要产出什么”。课程分别从 Research、Ops 和 Dev 三个方向拆解了真实工作流程，也让我对不同方向所需要的能力有了更具体的认识。

### 一、Research：从阅读概念转向拆解机制

Research 方向的重点不是简单摘抄资料或总结名词，而是要学会阅读 EIP、MIP、ERC 和治理提案，理解一个协议或产品为什么会出现、想解决什么问题，以及它的方案会对哪些用户和生态参与者产生影响。

阅读提案时，可以按照几个部分进行拆解：首先了解提案提出的背景和当前存在的问题；其次分析它给出的解决方案和底层机制；然后判断会影响哪些角色，例如用户、开发者、节点、协议方或治理参与者；最后再分析提案可能带来的收益、风险、争议和长期影响。这样才能从“看懂文字”进一步走向“理解设计逻辑”。

除了阅读提案，还需要选择一个真实的 Web3 产品或协议进行分析，例如 Uniswap、Aave、Lido、Pendle、Morpho 等。分析时不能只介绍它是什么，而要回答几个核心问题：它解决了什么问题、目标用户是谁、用户为什么愿意使用、协议如何产生收益、收益和激励从哪里来、主要风险是什么，以及它与同类产品相比有什么优势。

我理解到，研究类工作的核心并不是资料搜集得多，而是能否从大量信息中提炼出关键机制，并用新人也能听懂的语言进行解释。一个好的 Research Builder 不仅需要理解协议，还需要具备结构化表达、风险判断和独立思考能力。

### 二、Ops：把一个主题组织成完整活动

Ops 方向今天主要学习了 Space 活动的策划流程。一次完整的 Space 并不是临时邀请嘉宾聊天，而是需要围绕一个清晰主题设计活动标题、简介、议程、主持串词、核心问题、互动环节和收尾行动。

活动开始前，需要先明确目标：这场 Space 是为了科普、项目宣传、社区增长，还是帮助用户了解某个赛道。目标不同，邀请的嘉宾、讨论的问题和宣传内容也会不同。

在议程设计上，可以先用简短的开场介绍活动背景和嘉宾，再围绕三到五个核心问题逐步展开。问题之间需要有逻辑，例如先谈行业背景，再谈具体案例，最后讨论未来趋势或给新人的建议。主持人除了提问，也需要控制时间、承接嘉宾回答，并在内容偏离主题时及时拉回。

互动环节也很重要，可以通过聊天区提问、观众连麦、投票或案例讨论增加参与感。结束时还需要设计明确的 CTA，例如关注项目账号、加入社群、填写反馈表或参与后续活动。宣传文案也不能只写时间和链接，而要让用户快速知道“这场活动和我有什么关系、我能获得什么”。

通过这部分学习，我意识到 Web3 运营并不只是发推文和维护社群，而是需要把内容、用户、嘉宾、项目和传播渠道连接起来。好的运营既要有执行力，也要有内容判断、节奏设计和沟通能力。

### 三、Dev：学习 AI Coding 的协作流程

Dev 方向主要学习如何借助 AI 阅读 Web3、AI、钱包、合约或 API 文档，并逐步生成代码框架和技术方案。AI Coding 并不是把需求丢给 AI 后直接复制代码，而是一个需要不断确认、测试和修改的协作过程。

比较完整的流程是：先明确自己要解决的问题和最终功能，再阅读相关文档，整理关键接口、参数和限制；之后让 AI 根据文档生成基础代码框架，再由人工检查代码结构、安全性和业务逻辑；最后通过实际运行、测试和报错信息不断修改。

其中，“AI Collaboration Log”也很重要，需要记录向 AI 提出的需求、AI 生成的内容、出现的问题，以及自己做了哪些人工修改。这样不仅能够证明自己的参与过程，也可以帮助后续复盘哪些提示词有效、哪些错误容易重复出现。

我也进一步理解到，AI 可以降低开发和阅读文档的门槛，但不能替代开发者的判断。尤其是钱包、智能合约和资产相关功能，如果不了解代码背后的权限、调用方式和风险，只靠复制 AI 代码，很容易出现安全问题。因此 Dev 方向不仅需要会使用 AI，还需要具备基本的代码阅读、调试和验证能力。

### 四、三个方向的区别与联系

通过今天的学习，我发现 Research、Ops 和 Dev 虽然产出不同，但本质上都需要完成“理解问题—拆解信息—形成方案—输出结果”的过程。

Research 更重视机制分析、资料判断和结构化表达；Ops 更重视活动策划、内容传播、社区沟通和执行；Dev 更重视技术文档、代码实现、测试和协作流程。三者并不是完全割裂的，一个项目通常需要研究人员理解市场和协议，需要开发者完成产品，也需要运营把产品介绍给真实用户。

对我来说，Research 方向目前比较吸引我的地方，是它能够帮助我真正理解一个协议为什么成立，而不是只停留在概念层面。同时，设计和运营能力也可以成为辅助优势，例如把复杂的研究内容整理成更清晰的图表、页面或内容，让新人更容易理解。今天的任务让我对不同方向的真实工作内容和产出形式有了更明确的认识，也为后续选择和深入学习提供了参考。

Day 2 学习笔记｜把 Research 方向变成可执行 Scope

今天的学习重点，是把“我想做研究”进一步变成一个具体、可执行的研究任务，而不是只停留在兴趣层面。

我目前选择的是 Research 研究向。Research Builder 的核心，不只是收集资料和总结内容，而是要先提出一个清晰的 Research Question，再明确研究对象、资料来源、研究边界和最终产出。

在选择研究对象时，可以从 DeFiLlama Top TVL 协议，或 Uniswap、Morpho、Pendle 等代表性产品中挑选一个具体项目。研究时需要先说明：我到底想研究什么问题，这个问题为什么值得研究，以及我要从哪些角度展开。

今天我也进一步理解了，一份完整的研究不能把事实、数据、假设、推断和个人判断混在一起。事实和数据需要有来源；假设需要说明依据；推断要展示思考过程；个人判断则要明确表达这是自己的观点，而不是把它写成确定结论。

因此，Research Question 不能太宽泛。比如“研究某个 DeFi 协议”就过于笼统，更合适的方式是围绕一个具体问题展开，例如：

·某协议的核心机制如何吸引用户和资金？它相比同类产品有哪些优势、限制和潜在风险？

资料来源也不能只依赖单一文章或 AI 总结，而应该尽量结合多类信息，包括项目官方文档、DefiLlama 数据、链上数据、研究报告、社区讨论和同类产品资料，并对不同来源进行交叉验证。

我认为研究任务最重要的是控制边界。短时间内不可能把一个大型协议的所有内容都研究完整，所以需要先选定一个具体对象和一个核心问题，再围绕产品定位、核心机制、用户需求、数据表现和竞争关系进行分析，避免研究范围不断扩大。

Week1 一周学习总结

DAY 3｜钱包、安全与第一笔测试网交易学习笔记

今天的学习主要围绕“如何安全地使用钱包，并在测试网上完成一笔链上记录”展开。因为之前已经接触过钱包、助记词、Gas、交易哈希、区块浏览器，也在 Remix + MetaMask 里做过简单合约部署，所以今天更多是在把这些零散概念串成一条完整的操作流程。

一、钱包与安全意识

今天重新梳理了加密钱包的基础逻辑。钱包本身并不是“存币的软件”，更准确地说，它是用来管理地址、私钥和链上交互权限的工具。真正重要的是私钥和助记词，它们代表了资产控制权。

这部分我最大的收获是：课程实验一定要使用单独的钱包，不要直接使用主力钱包。因为学习过程中会频繁连接网站、添加测试网、领取测试币、发起交易或授权，如果直接用主力钱包会有不必要的风险。

另外也再次明确了几个安全原则：

1\. 助记词和私钥不能截图、不能上传、不能发给任何人。

2\. 不随便连接陌生网站，不随便签名。

3\. 课程钱包和主力钱包要分开使用。

4\. 测试币没有真实价值，但操作流程和真实链上交互很接近，所以仍然要认真对待。

二、Monad Testnet 与测试币

今天也学习了 Monad Testnet 的使用流程。测试网的作用是让开发者和学习者在不消耗真实资产的情况下，体验链上交易、合约交互和应用测试。

我按照流程添加了 Monad Testnet，并领取了测试币。通过这个过程，我更清楚地理解了“测试币”的作用：它不是用来投资或交易的，而是用来支付测试网上的 Gas，帮助我们模拟真实链上的操作。

三、第一笔测试网交易 / 应用交互

今天的实操重点是完成一次测试网交易或应用交互。相比只看概念，真正点一次确认、等待交易上链、再去 explorer 里查交易记录，会更直观地理解链上操作的过程。

一笔交易大概包含这些信息：

1.From：发起交易的钱包地址

[2.To](http://2.To)：接收地址或合约地址

3.Value：转账金额，或交互中涉及的数值

4.Gas Fee：这笔交易消耗的手续费

5.Transaction Hash：这笔交易的唯一编号

6.Status：交易是否成功

其中我觉得 transaction hash 很重要。它有点像链上世界里的“操作凭证”，只要有 hash，就可以在区块浏览器里查到这笔交易的状态和细节。

四、区块浏览器的理解

今天也学习了如何使用 block explorer 查看交易。以前我对 explorer 的理解比较模糊，只觉得它像一个查询工具。今天操作之后发现，它其实是理解链上透明性的关键入口。

在区块浏览器里，可以看到交易是否成功、交易双方地址、区块高度、Gas 消耗、时间等信息。也就是说，链上操作不是只停留在钱包界面里，而是会被记录到网络上，并且可以公开查询。

五、今日总结

整个流程重新走了一遍：

1.使用 Remix 编写并编译 Solidity 合约

2.使用 Browser Extension 连接 MetaMask，并切换到 Monad Testnet

3.部署合约到 Monad Testnet

4.调用 write function（leaveMessage） 完成留言，再调用 read function（getLastMessage、getLastSender） 验证数据是否成功写入和读取

5.获取 Contract Address、Deployment Transaction Hash、Interaction Transaction Hash

过程中也顺便弄清楚了几个之前一直比较模糊的概念：

1.Contract Address：合约部署到链上后生成的唯一地址，用于后续与合约交互。

2.Deployment Transaction Hash：部署合约对应的链上交易记录，可以用于查询部署是否成功。

3.Interaction Transaction Hash：每次调用 write function 都会生成一笔新的交易，也会对应一个唯一的 Transaction Hash。  
[4.read](http://4.read) function 不会产生链上交易，只负责读取数据；write function 会修改链上状态，因此需要钱包签名并支付 Gas（测试网使用测试币）。

今天最大的收获还是第一次完整走通了编写合约 → 编译 → 部署 → 调用合约 → GitHub 整理项目这一整套流程。中间遇到了不少问题，比如 Remix 和 MetaMask 的连接、README 的编写、GitHub 上传等，但都通过查资料和不断尝试解决了。相比单纯看教程，自己完整实践一遍之后，对智能合约开发流程有了更清晰的理解，也没那么陌生了
