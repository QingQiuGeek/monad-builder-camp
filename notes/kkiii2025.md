## 非标教育：AI 可能改变的不是课程内容，而是学习路径

传统教育具有很强的标准化特征：

-   统一课程；
    
-   统一进度；
    
-   统一题目；
    
-   统一答案；
    
-   统一评价方式。
    

但真实学习过程是非标准的。不同人的知识基础、兴趣、理解速度、表达方式和目标都不同。但非标教育最大的难题不是内容生成，而是：如何证明学习真的发生了？因此，AI 教育产品不能只评价“用户是否完成任务”，还要判断：

-   用户是否理解为什么这样做；
    
-   能否迁移到新问题；
    
-   能否发现 AI 的错误；
    
-   在没有 AI 时是否仍具备基本判断力；
    
-   是否形成了自己的知识结构。
    

## AI-native 孩子的思维方式：从“知识安全感”转向“执行安全感”

过去的学习者很容易形成一种“知识安全感”：只有当我能背出一个概念、公式或原理时，我才认为自己掌握了它。当无法完整复述知识时，人会产生愧疚感，觉得自己“不够努力”或者“没有真正学会”。

AI-native 的学习者可能更倾向于：我不一定能完整背出概念，但我知道如何借助 AI、工具和资料，把事情跑通。

而是知识的作用发生了变化。

-   **Web3 运营核心落地方法**
    
    -   **运营本质认知**
        
        -   核心角色定位：Web3 运营是产品运营官、社区粘合剂、增长操盘手三者的复合角色，绝非简单的社媒内容发布与活动执行。
            
        -   内容提效方法：使用 AI 先产出 10 个内容创作维度，选择冲突感最强的角度扩写后人工微调，可快速完成社媒内容产出。
            
    -   **分岗位实操标准**
        
        -   社区运营：核心是制造用户高光时刻，提前通过悬念物料、用户投票、老带新活动积累期待感，避免机器人刷屏的无效操作。
            
        -   活动运营：采用阶梯式递进任务设计，通过多步权益反馈筛选真实用户，沉淀愿意主动传播的核心铁粉，避免纯转发抽奖引来羊毛党。
            
        -   开发者运营：优先响应 GitHub 上开发者的高频诉求，通过定向激励、针对性黑客松奖项搭建生态，将核心反馈用户转化为生态布道者。
            
    -   **内容创作优化建议**：制作宣传素材时，使用参考图作为生成依据比纯文字描述的效果更稳定，可避免生成内容过于抽象的问题。
        

-   **AI 辅助开发（Vibe Coding）成长路径**
    
    -   **六层能力分级框架**
        
        -   L1 静态页面层：对应个人主页、落地页等静态内容，基础代码可完全由 AI 生成，核心能力要求为审美把控，可拆分素材多工具整合。
            
        -   L2 可交互前端层：对应动态交互页面，需掌握 React、状态管理、API 调用基础逻辑，代码可由 AI 生成，仅需自行完成 Vercel 基础配置。
            
        -   L3 全栈应用层：对应前后端加数据库的完整应用，优先推荐使用 Supabase 作为数据库，仅需具备基础 SQL 代码审阅能力即可落地。
            
        -   L4 身份支付层：对应登录体系、支付接入等涉及资金安全的模块，AI 可辅助生成配置，但必须自行完成安全校验，避免资金损失。
            
        -   L5 工程化系统层：对应 Docker、云服务器等工程化内容，仅需具备代码审阅能力，无需自主编写，可根据需求逐步接触学习。
            
        -   L6 分布式系统层：对应大型企业级微服务架构，可先行了解逻辑，核心需保留代码审阅能力，应对运行过程中的异常问题。
            
    -   **核心落地原则**
        
        -   能力成长规律：随着层级提升，AI 的辅助占比逐步降低，对人的原创能力、安全把控能力要求逐步提升，需优先掌握核心校验能力。
            
        -   工具操作方法：优先掌握 GitHub 基础操作，通过 Vercel 可一键完成 GitHub 项目的公网部署，内容更新后可自动同步，无需复杂配置。
            
    -   **跨 Agent 持久化记忆工具分享**
        
        -   **工具核心定位**：是由用户自主维护、可被兼容 Agent 读取的跨 Agent 统一上下文层，所有内容的写入与更新均需经过用户确认。
            
        -   **核心价值**：解决不同 AI 工具、不同对话场景下的上下文割裂问题，避免用户反复手动复制粘贴背景信息，保障 AI 协作的连续性。
            
        -   **落地逻辑**：通过索引匹配当前场景的相关上下文，优先读取摘要内容，信息不足时再读取全文，所有内容的存取权限完全由用户掌控。
            
        -   **后续计划**：当前工具处于早期迭代阶段，开放社群成员试用邀请，欢迎反馈优化建议，同时可对接相关方向的组队开发需求。

钱包 vs 交易所：

钱包上的资产控制权在个人，而在交易所上注册的是一个平台账户，资产实际由交易所统一保管，你看到的是账户里的余额记录（中心化）。相当于把钱放在自家保险箱与银行的区别。

助记词 vs 私钥：

助记词能生成、恢复整个钱包，包括钱包里的多个账户和私钥。私钥主要用来证明某个操作是本人同意的、是你在链上行使控制权的根本凭证、证明一项操作确实获得了账户主人的授权。开发者特别关心怎么安全保存它，因为一旦被复制，系统可能分不清真正的主人是谁。

私钥是其中一间房的钥匙，助记词是可以重新配出整栋房子所有钥匙的总图纸。

ERC-20 vs ERC-721：

ERC-20 通常用来制作代币，可以类比为人民币，你的100元何我的100元可以互换，该标准规定了余额查询、转账和授权别人使用代币等基本功能。

ERC-721 通常用来制作 NFT，是“每个都有独立编号的物品”，不能直接互换。

### AI Agent 能力飞跃的 4 大底层原因

1.  **大模型本体变强**：基础推理、逻辑理解能力大幅提升，复杂问题处理更成熟
    
2.  **超长上下文窗口**：可读取海量代码、项目文档、历史交互记录，推理更精准
    
3.  **工具生态完善**：支持 CLI 命令行、代码执行、Bash、外部 API 调用，工具箱丰富
    
4.  **完整记忆闭环**：持久化存储交互记忆，自动总结用户习惯形成专属 Skill，实现个性化迭代
    

### 两大适配 AI 的开发范式

（1）TDD 测试驱动开发（提升代码健壮性首选）

流程：**先写测试用例定义预期结果 → 测试失败 → 最小代码实现 → 测试通过 → 按需重构** 优势：给 AI 搭建自动纠错回路，防止改代码破坏原有逻辑，AI 可自动执行`cargo test`等自检。

（2）SDD 规范驱动开发（大型项目标准化）

流程：明确需求目的 → 制定开发规范 → 设计技术方案 → 拆解细分任务 → 落地实现 → 文档留痕溯源 配套工具：

### 开发者和 AI 协作必备 4 项能力

1.  **会描述**：写高质量 Prompt，清晰说明用户、场景、约束、验收标准
    
2.  **会拆解**：拆分复杂任务为可执行、可验证小步骤，理清依赖与风险
    
3.  **会验证**：自动化测试、手动校验、日志埋点，人工把控最终交付质量
    
4.  **会设计**：自主掌握架构边界、数据流、长期质量体系，避免完全依赖 AI
    

### Web3开发赛道

├─ 强耦合组合：DApp ↔ 智能合约（项目标配） ├─ 独立赛道1：钱包开发（项目极少自研） ├─ 独立赛道2：底层节点开发（公链团队专属） └─ 独立赛道3：安全审计（第三方上线必备） 通用底层基础：密码学 + 区块链底层原理

理解 Monad ，首先要明白 Monad 的高频交互场景并非针对所有需要高频交互的产品。

  
适合 Monad 的产品通常需要满足：**高频交互 × 资产状态变化 × 需要开放可组合/可信结算**。  
不适合的场景则是：**高频but低价值、无资产、无跨平台组合需求、只需要平台自己记账**。例如普通社交 feed、浏览记录、经验值、非交易型游戏行为，大多数应该放数据库，链上只记录最终结算或资产变更。

如果只是点赞、评论、普通积分、内部排行榜，普通数据库更好；如果每次交互都会改变可转让资产或公共状态，Monad 的价值才明显 —— Monad 的机会不是把数据库替换掉，而是让过去因为 Ethereum 太慢太贵而无法上链的应用变得可用，尤其是预测市场、onchain CLOB/perps、agent 微支付、高组合性的 DeFi 消费应用、可验证资金池游戏。

## 7.09 学习笔记

Replit 网站无需梯子。

**Monad 的由来：**

2009 年，Bitcoin 证明“价值可以在没有中间人的情况下转移”，但不能在它上面构建完整应用。

2015 年，Ethereum 证明“去中心化计算确实可行”，它的目标不是让去中心化应用用起来像普通软件一样快。

2025年，Monad 继承了 Ethereum 建立起来的一切，然后继续问下一个问题：**如果去中心化应用快到像普通软件一样，会发生什么？**

**Monad 设计目标**：10,000 TPS、400ms 出块、800ms 确定性最终性（deterministic finality）

**主网和测试网：**Mainnet 使用具有真实价值的代币；testnet 上的代币没有价值，是用来安全实验的。

**Chain ID** 是一个用来标识某条区块链的唯一编号。

**RPC（Remote Procedure Call，远程过程调用）**是钱包或应用用来与区块链对话的一个 URL，相当于“给区块链电话”。RPC 也分 Public RPC 和 Paid RPC，面向真实用户上线时使用Paid RPC，一般测试就用Public RPC。

**Faucet（水龙头）**是一个免费发放 testnet 代币的网站。官方 Monad faucet： [faucet.monad.xyz](http://faucet.monad.xyz)

**Block explorer** 用于查询链上的任何东西：交易、钱包、合约，以及它们的历史。

大多数钱包也支持通过 [chainlist.org](http://chainlist.org) 一键添加。

**Supabase：**一项免费服务，开箱即用地提供**数据库**和**文件存储**。

**数据库**存放结构化数据（文本、数字、日期、各种关系），**文件存储**存放复杂的文件（路径 URL 放在数据库，文件字节放Sotrage）。

**给应用加上 Storage：**

在侧栏的 Storage **创建一个 Bucket**，暂时保持 Public bucket 关闭

**设置一条 Storage 策略**，New policy → 选 template → 选 Allow access to JPG images in a public folder to anonymous users → Allowed operation模块中勾选 SELECT（读取 / 下载）和 INSERT（上传。（后面在 authentication 那节课再把它收紧）

**需要在表里有一个地方存放每个上传文件的路径**。告诉智能体我要加一列来引用文件，它会给我用于输入 SQL Editor 中去运行的内容

接下来只需告诉智能体：**在应用里上传文件、显示这些文件、限制用户可以上传什么**相应的需求
