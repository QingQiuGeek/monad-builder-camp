# 2026-07-09
<!-- DAILY_CHECKIN_2026-07-09_START -->


随着 AI Agent 的发展，它已经不再只是一个负责回答问题的聊天工具，而是开始具备“替用户行动”的能力。例如，它可以调用钱包、执行链上交易、管理资金、访问 API，甚至连接不同的工具和服务。这种能力让 Agent 变成了高效率的数字助手，但同时也带来了新的安全风险。

在 AI Agent 时代，最大的风险之一是 Prompt Injection。攻击者可以在网页、文件或外部内容中隐藏恶意指令，诱导 Agent 忽略原本规则，执行转账、泄露信息或调用危险工具。其次是权限滥用问题，如果 Agent 拥有钱包、API Key 或系统权限，一旦被恶意指令控制，损失会比普通聊天错误严重得多。私钥泄露、恶意 Tool Call、MCP 插件安全，也都会成为开发者必须重视的问题。

因此，AI Agent 的安全核心不是“完全信任 AI”，而是建立边界。开发者应该采用最小权限原则，只给 Agent 必要权限；对转账、删除、授权等高风险操作加入人工确认；避免让 Agent 接触明文私钥；同时对 Tool Call 和 MCP 服务进行审计、监控和限制。总结来说，AI Agent 越像一个真实员工，就越需要像管理真实员工一样设置权限、流程和安全检查。

# 2026-07-07

完整链路：

Prompt → 合约初稿 → 人工检查 → Remix 编译 → 部署 → read / write 测试 → Explorer 验证

* * *

一、使用 AI 生成合约

1.  编写 Prompt
    

我正在学习 Solidity，请帮我生成一个最小可运行的 Onchain Todo 合约。

要求如下：

Solidity 版本为 ^0.8.20  
每个地址维护自己的 Todo 列表  
Todo 需要包含 text、completed、createdAt 三个字段  
提供 addTodo()、toggleTodo()、getTodos() 三个函数  
不要使用复杂库  
添加完整注释  
保证 Remix 可以直接编译

在写 Prompt 时，我刻意限制了三个核心条件：

第一，锁定 Solidity 版本为 ^0.8.20，避免 AI 输出不兼容版本的代码。

第二，明确数据模型：每个地址维护自己的 Todo 列表，并且每条 Todo 只包含 text、completed、createdAt 三个字段。

第三，限定函数清单，只保留 addTodo()、toggleTodo()、getTodos() 三个入口，避免合约变得过于复杂。

最后一句“保证 Remix 可以直接编译”，是为了防止 AI 输出 Hardhat 配置、外部依赖库或带 import 的复杂写法，从而保证代码可以直接在 Remix 中运行。

* * *

2.  AI 输出的合约代码
    

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

/**
 * @title OnchainTodo
 * @author Web3-Todo
 * @notice 链上待办事项合约 —— 每个钱包地址拥有独立的 Todo 列表
 * @dev 设计目标：最小可运行、无外部依赖、可直接在 Remix 中编译部署
 *
 * 核心功能：
 * - addTodo()    添加一条待办
 * - toggleTodo() 切换某条待办的完成状态
 * - getTodos()   查询当前调用者的全部待办
 */
contract OnchainTodo {
    struct Todo {
        string text;
        bool completed;
        uint256 createdAt;
    }

    mapping(address => Todo[]) private todosByOwner;

    event TodoAdded(
        address indexed owner,
        uint256 indexed index,
        string text,
        uint256 createdAt
    );

    event TodoToggled(
        address indexed owner,
        uint256 indexed index,
        bool completed
    );

    function addTodo(string calldata text) external {
        require(bytes(text).length > 0, "Todo text cannot be empty");

        Todo[] storage list = todosByOwner[msg.sender];
        uint256 index = list.length;

        list.push(
            Todo({
                text: text,
                completed: false,
                createdAt: block.timestamp
            })
        );

        emit TodoAdded(msg.sender, index, text, block.timestamp);
    }

    function toggleTodo(uint256 index) external {
        Todo[] storage list = todosByOwner[msg.sender];
        require(index < list.length, "Todo index out of bounds");

        list[index].completed = !list[index].completed;

        emit TodoToggled(msg.sender, index, list[index].completed);
    }

    function getTodos() external view returns (Todo[] memory) {
        return todosByOwner[msg.sender];
    }
}
```

* * *

3.  让 AI 解释代码，并进行人工核对
    

拿到代码后，我没有直接部署，而是继续让 AI 解释代码逻辑：

请不要修改代码，而是逐行解释：

contract 是什么  
struct 为什么这样设计  
mapping 的作用  
addTodo 做了什么  
toggleTodo 做了什么  
getTodos 为什么是 view  
storage 和 memory 有什么区别

AI 解释完成后，我按照 Prompt 的要求对代码进行核对：

`mapping(address => Todo[])` 实现了“每个地址维护自己的 Todo 列表”。这里不是通过 `onlyOwner` 控制权限，而是使用 `msg.sender` 作为 key，让每个钱包地址只能访问和修改自己的列表。

`addTodo()` 和 `toggleTodo()` 都属于 write 操作，会修改链上的 storage 数据，因此调用时需要钱包签名，并且需要消耗 Gas。

`getTodos()` 是 view 函数，只负责读取数据，不修改链上状态。在 Remix 中调用时属于静态调用，不需要真正发送交易，也不会消耗 Gas。

合约中也设置了边界检查：当 text 为空时，`addTodo()` 会 revert；当 index 超出数组范围时，`toggleTodo()` 会 revert；如果当前地址没有添加过 Todo，`getTodos()` 会返回空数组 `[]`。

AI 额外添加了 `TodoAdded` 和 `TodoToggled` 两个 Event。虽然 Prompt 中没有强制要求，但我最终保留了它们，因为 Event 可以让后续在区块浏览器 Explorer 中查看 Logs，更方便验证交易执行结果。

* * *

4.  部署前人工检查
    

在部署之前，我对合约做了以下检查：

Remix 可以正常编译通过。

三个函数与 Prompt 要求一致，分别是 `addTodo()`、`toggleTodo()`、`getTodos()`。

`getTodos()` 返回的是当前地址的完整 Todo 列表，而不是按单个 id 查询。

权限模型符合预期：每个地址只能修改自己的 Todo 列表。

合约没有引入不必要的库、继承或复杂逻辑。

变量命名和注释清晰，方便阅读和理解。

最终我没有修改 AI 生成的代码，部署版本与 AI 初稿保持一致。

* * *

二、Remix 编译

打开 Remix IDE 后，我新建了一个 Web3-Todo 工作区，并在 `contracts/OnchainTodo.sol` 文件中粘贴完整合约代码。

然后点击左侧 Solidity Compiler，也就是 S 图标，选择 0.8.20 或以上版本的编译器，点击 `Compile OnchainTodo.sol`。

编译完成后，Solidity Compiler 图标出现绿色对号，说明合约编译成功，没有语法错误，也没有版本兼容问题。

* * *

三、部署到 Monad Testnet

1.  连接钱包并准备部署
    

部署前提与 Day 1 相同：Rabby 钱包已经添加 Monad Testnet，Chain ID 为 10143，并且账户中有测试 MON 作为 Gas 费。

编译通过后，我点击 Remix 左侧的 `Deploy & Run Transactions` 面板，在 Environment 中选择 `Browser Extension`。

随后 Remix 会连接浏览器钱包扩展。我这里使用的是 Rabby Wallet，如果使用 MetaMask 等其他钱包插件，操作逻辑也是一样的。

连接成功后，我确认 Remix 面板顶部显示的网络为 `Monad Testnet (10143)`，Contract 选择已经编译完成的 `OnchainTodo`，并确认合约旁边带有 Compiled 绿标。

由于这个合约没有 constructor 参数，所以不需要额外输入部署参数，直接点击 Deploy 即可。

* * *

2.  Rabby 钱包确认部署
    

点击 Deploy 后，Rabby 钱包弹出合约部署确认窗口。

我在确认窗口中主要检查了两点：

第一，当前网络是否为 Monad Testnet。

第二，本次操作是否为部署合约交易。

确认无误后，点击签名并提交。

提交完成后，回到 Remix 底部终端，看到绿色对号，说明合约部署成功。

部署成功后，我记录了两个关键信息：

Transaction Hash：用于查询本次部署交易。  
Contract Address：合约部署成功后生成的链上地址，也是之后调用合约和在 Explorer 中验证合约的核心地址。

* * *

四、read / write 测试

合约部署完成后，Remix 的 Deployed Contracts 区域会出现已经部署好的 `OnchainTodo` 合约实例。

我先测试 write 函数 `addTodo()`。

在 `addTodo` 输入框中填入一条待办事项，例如：

`学习 Solidity Todo 合约`

然后点击 `transact`。由于 `addTodo()` 会修改链上数据，所以 Rabby 会再次弹出签名确认窗口。确认网络和交易内容无误后，我点击签名并提交。

交易成功后，Remix 终端出现绿色对号，说明 Todo 已经成功写入链上 storage。

接着，我测试 read 函数 `getTodos()`。

点击 `getTodos()` 后，Remix 返回当前钱包地址下的 Todo 列表。返回结果中可以看到 text、completed 和 createdAt 三个字段：

text 表示待办事项内容。  
completed 初始值为 false，表示还未完成。  
createdAt 是创建时间戳，来自 `block.timestamp`。

随后，我继续测试 `toggleTodo()`。

由于刚刚添加的是当前地址的第一条 Todo，它在数组中的 index 为 0。因此我在 `toggleTodo()` 中输入 `0`，点击 transact，并在钱包中确认签名。

交易成功后，再次调用 `getTodos()`，可以看到这条 Todo 的 completed 状态已经从 false 变成 true，说明 `toggleTodo()` 成功修改了链上状态。

* * *

五、Explorer 验证

为了确认交易确实发生在 Monad Testnet 上，我使用交易哈希和合约地址在 Monad Explorer 中进行验证。

首先，我用部署时生成的 Transaction Hash 查询部署交易。

在 Explorer 中可以看到：

交易状态为成功。  
网络为 Monad Testnet。  
交易类型是合约部署。  
From 地址是我的钱包地址。  
To 显示为合约创建。  
页面中可以看到部署后生成的 Contract Address。

然后，我再使用 Contract Address 查询合约地址，可以看到该合约已经真实存在于 Monad Testnet 上。

接着，我查看后续调用 `addTodo()` 和 `toggleTodo()` 产生的交易记录。

`addTodo()` 对应的是一次写入链上 Todo 数据的交易。  
`toggleTodo()` 对应的是一次修改 Todo 完成状态的交易。  
这两次操作都需要钱包签名，并且都会产生交易哈希。

同时，因为合约中定义了 `TodoAdded` 和 `TodoToggled` 两个 Event，所以在 Explorer 的 Logs 区域可以看到对应事件记录。这样可以进一步验证合约不仅部署成功，而且函数调用也正常执行。

* * *

六、最终总结

本次作业完成了一个最小可运行的 Onchain Todo 合约，从 Prompt 编写、AI 生成合约、人工检查、Remix 编译、部署到 Monad Testnet，再到 read / write 测试和 Explorer 验证，形成了一条完整的链上开发流程。

这次练习让我理解了几个关键点：

第一，Prompt 写得越明确，AI 输出的合约越接近可部署版本。

第二，AI 可以帮助生成初稿，但部署前必须人工检查合约逻辑，尤其是权限、数据结构、函数行为和边界条件。

第三，`addTodo()` 和 `toggleTodo()` 是 write 操作，会修改链上状态，需要签名并消耗 Gas。

第四，`getTodos()` 是 read 操作，只读取数据，不修改链上状态。

第五，Explorer 验证是链上开发中非常重要的一步，因为它可以证明交易和合约确实真实存在于区块链上。

最终结果：合约编译成功、部署成功、Todo 添加成功、Todo 状态切换成功，并且可以在 Explorer 中查询到对应交易和事件记录。
